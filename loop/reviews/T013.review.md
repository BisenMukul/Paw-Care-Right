## Security findings (adversarial)

1. **email_verified linking check (account-takeover vector) — SAFE.** `social-auth.service.ts:74` gates the entire link/provision branch on `if (identity.email && identity.emailVerified)`. An email that is present but **unverified** (or `email_verified:false`) falls through to `throw new UnauthorizedException()` (line 95). It can never `update` an existing user's sub column. Unit `no verified email and no sub match throws UnauthorizedException` (spec:144) and e2e `new sub with no verified email → 401` (e2e:251) both prove fail-closed. Verifier maps `email_verified` from signed claims only (`apple-token-verifier.ts:40`), and claims are trustworthy because the whole token is RS256-signature-verified against the JWKS.

2. **Uniform 401 / no reason leakage — SAFE.** Every verification failure in `AppleTokenVerifier.verify` collapses in a bare `catch { throw new UnauthorizedException(); }` (apple-token-verifier.ts:52–56); empty/non-string `sub` → same (42–44). `SocialAuthService` throws bare `UnauthorizedException` for unresolved provider (line 35) and no-linkable-identity (line 95). `AllExceptionsFilter` maps 401 → `UNAUTHORIZED` (all-exceptions.filter.ts:16). tampered/wrong-aud/wrong-iss/expired e2e cases all assert 401 — indistinguishable.

3. **Identity token never logged — SAFE.** The verifier passes no token to any logger; it throws a token-free exception. `AllExceptionsFilter` logs only `code: message` + stack (filter:38–41) — the token is not in the exception. Unit `never logs the identity token, even on failure` (apple-token-verifier.spec.ts:146) spies console log/warn/error and asserts no call contains the token. No `console.*` in production source.

4. **JWKS seam cannot bypass prod verification — SAFE.** Production always binds `APPLE_JWKS_RESOLVER` → `createRemoteJWKSet(new URL(APPLE_JWKS_URL))` (auth.module.ts:42) — this is the required cached Apple JWKS. The only override is via `Test.createTestingModule(...).overrideProvider(APPLE_JWKS_RESOLVER).useValue(createLocalJWKSet(...))` in the e2e (auth-social.e2e-spec.ts:43–46). No env flag or code path swaps the resolver at runtime; verification strictness (`issuer: APPLE_ISSUER`, `audience: config.appleClientId`, `algorithms: ["RS256"]`, exp enforced by jose) is identical in prod and test (apple-token-verifier.ts:33–37).

5. **Same session + same provisioning path — SAFE.** `login()` returns `authService.issueSession(provisioned)` (social-auth.service.ts:41) — the same method `verifyOtp` calls (auth.service.ts:77); response shape identical to `otp/verify`. New accounts go through `authService.provisionNewAccount` (social-auth.service.ts:86) — the same method `provisionOrGetUser` uses (auth.service.ts:141), not a parallel copy. Refresh token from the same `RefreshTokenService.issue` family machinery.

6. **P2002 race — SAFE.** Transaction wrapped; on `P2002` (social-auth.service.ts:147–149) it re-queries by the provider's sub column and returns that session if found, else rethrows (48–61). Unit `re-fetches by sub on a P2002 race` (spec:155) proves it. e2e `same token twice returns the same user (idempotent by sub)` (e2e:234) confirms dedupe-by-sub.

7. **Provider abstraction (T014-ready) — SAFE.** `SocialTokenVerifier` interface + `SOCIAL_TOKEN_VERIFIERS` registry (social-verifier.ts) built into a `Map` in the constructor (social-auth.service.ts:28); Google is addable by registering a second verifier + widening the DTO array — no reshaping. Unit-tested (`unresolved provider throws UnauthorizedException without verifying`, spec:188). The google branches in the sub-column switches are deliberate provider-agnostic shaping (R6), **not** a Google verifier — no Google JWKS/claims/DTO value exists.

8. **DTO allowlist — SAFE.** `@IsIn(["apple"])` (social-auth.dto.ts:14) + global `whitelist + forbidNonWhitelisted` pipe → non-apple provider is a 400 `VALIDATION_FAILED` before any verifier/DB work. e2e `unknown provider → 400` (e2e:259) asserts 400 + `VALIDATION_FAILED` (not 401).
