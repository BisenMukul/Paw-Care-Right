## Acceptance criteria — evidence

### AC1 — Join-replaces transaction (the crux) — PASS
`households.service.ts:80-146`. Invariant comment quoted at lines 35-48: *"every user has EXACTLY ONE household `Membership` at all times … `acceptInvite` therefore implements JOIN-REPLACES: within the SAME transaction that creates the joiner's new MEMBER membership …, their current membership is removed first."* Step order inside `$transaction` (103-145):
1. Atomic claim (107-110) → 2. **pets guard `petCount > 0` → `ConflictException` (118-123) BEFORE any delete** → 3. leave: OWNER→`household.delete` (125-127), MEMBER→`membership.delete` (128-129) → 4. `membership.create MEMBER` (134-136) → 5. return target `{id,name}`.
- Pet-data safety: the 409 throw at 121-123 precedes both delete branches, and rolls back the claim. **Pets are never deleted on join.** e2e `pets-present blocks join-replaces` (`households.e2e-spec.ts:197-217`) proves 409 + B still owns "B's Cat" afterward.
- Non-owner joiner drops only the membership row (129); someone else's household untouched.
- Delete-before-create in one txn ⇒ membership count never observably exceeds 1. e2e happy path (161-194) proves B's old household row is `null` (179-180), B's membership in A is `MEMBER` (182-185), invite `usedAt`/`usedById` set (187-189), and **B's `GET /v1/pets` resolves to A's pets** (191-193) — the from-membership guard invariant holds with zero 404.

### AC2 — Single-use + expiry + anti-probing — PASS
- **No TOCTOU:** the authoritative single-use claim is `tx.householdInvite.updateMany({ where: { id, usedAt: null }, data: { usedAt, usedById } })` INSIDE the txn (107-110); `claimed.count !== 1 → NotFoundException` (112-114). The pre-check read at 81-88 is advisory only; two racing callers both passing it still resolve to exactly one winner (count 1) and one loser (count 0 → 404). Reuse e2e (139-159): first accept 200, second 404 NOT_FOUND.
- **Uniform 404** for not-found / expired / used: single `throw new NotFoundException()` with no message (83-88) — identical wire response. e2es prove indistinguishability: expired→404 (121-137), used→404 (139-159), unknown-well-formed→404 (264-273). An attacker cannot distinguish "invalid" from "live-but-used/expired".
- **7-day expiry at creation:** `INVITE_TTL_MS = 7*24*60*60*1000` (service:8), `expiresAt = now + TTL` (54); e2e asserts ~now+7d (103-118).
