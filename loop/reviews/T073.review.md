## Adversarial probe 1 — out-of-order truth table (built from `rc-webhook.state.ts`)
Gate at `state.ts:68`: `lastEventAt !== null && eventTsMs < lastEventAt.getTime()` → `skip:"stale"`, evaluated BEFORE the type switch.

| Sequence (applied, then arriving) | Arriving ts vs stored | Result | Final state | Safe? |
|---|---|---|---|---|
| EXPIRATION(t2) → RENEWAL(t1<t2) | strictly older | skip:stale | stays FREE/expired | yes (tested state+e2e) |
| RENEWAL(t2) → CANCELLATION(t1<t2) | strictly older | skip:stale | stays PREMIUM/active | yes (tested state) |
| RENEWAL(t1) → RENEWAL(t2>t1) | newer | upsert | PREMIUM/active | yes (tested) |
| RENEWAL(ts) → CANCELLATION(ts equal) | not strictly older | upsert applied | PREMIUM/cancelled | yes — entitlement stays PREMIUM |
| unknown TRANSFER(older) after RENEWAL | strictly older | skip:stale (gate beats type) | unchanged | yes (tested :168) |
| PRODUCT_CHANGE(newer) family→individual | newer | upsert, plan=product_id, householdId re-stamped | PREMIUM/active, new plan | yes — entitlement flip delegated to T072 clock/plan |

- **Equal-timestamp tie (R2):** applied in arrival order. Deterministic given arrival order; dedupe covers true replays. One theoretical unsafe ordering exists — EXPIRATION then RENEWAL at *identical* ms would resurrect PREMIUM — but distinct RC events sharing an exact `event_timestamp_ms` is astronomically rare and RC does not emit it. Documented R2. **Non-blocking, carry-forward.**
- **Missing-timestamp event:** `resolveEventTimestampMs` falls back to `receivedNow` (state.ts:22), i.e. "now" = newest, so a timestampless event is never gated and always applies. Direction is normally safe (a null-ts EXPIRATION drops premium); the only adverse case is a hypothetical timestampless RENEWAL arriving after a real EXPIRATION, which RC does not produce (RENEWAL always carries `purchased_at_ms`/`event_timestamp_ms`). **Non-blocking, noted.**

## Adversarial probe 2 — transaction / concurrent-delivery race
- Dedupe is enforced by the `ProcessedWebhookEvent` PK inside the same `$transaction` as the upsert (`service.ts:35-79`). Two *parallel* deliveries of the same eventId serialize on the unique index: the loser blocks until the winner commits, then receives P2002 → caught → duplicate no-op → its tx rolls back → its upsert never runs. Double-apply is prevented under parallelism, not just sequential replay.
- **P2002 scoping is correct.** `isUniqueConstraintViolation` (service.ts:124) checks only `code === "P2002"` with no `target`, but it is invoked *exclusively* inside `recordEventOrDetectDuplicate` whose try-block contains a single statement — `tx.processedWebhookEvent.create` (service.ts:90-100). The `subscription.upsert` is OUTSIDE that try, so a Subscription-level P2002 (e.g. two events for the same `rcAppUserId` racing the create path) propagates → rollback → 500 → RC retries → update path self-heals. No misclassification today. **Non-blocking robustness note:** if future edits add statements to that try-block, a foreign unique violation could be swallowed as a dedupe hit — a `target`/`meta` assertion on `eventId` would harden it. Recommend for a future pass, not required now.

## Adversarial probe 4 — unknown-user semantics
- `$RCAnonymousID:…` and unknown UUIDs: `resolveHouseholdId` (service.ts:109-122) finds no membership and no existing row → returns null → upsert skipped, but the `ProcessedWebhookEvent` dedupe row IS written and 200 acked (service.ts:58-63). e2e `:203-214` confirms no `Subscription` row is created. FK `Subscription.rcAppUserId → User` violation is thereby avoided.
- **Carry-forward (T074/T075):** because the dedupe row persists for a skipped unknown-user event, a purchase made *before* the user account is linked is effectively LOST until RC sends the next event for that user — the original event will never reprocess after signup. Matches plan decision 5 / risk R7. Acceptable for T073 (documented), but T074 paywall / T075 quota should not assume the very first purchase event always lands a `Subscription` row.
