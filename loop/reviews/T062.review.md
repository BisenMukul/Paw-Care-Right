## Duty 6 — deviation #1 mathematical evaluation (SOUND — a strength)
Claim: reusing a byte-identical startAt with DAILY/WEEKLY absent DST is a vacuous tz-change test, so MONTHLY;BYMONTHDAY=15 is required.
Verified: for DAILY absent DST, occurrence n = `Date.UTC(anchorDay+n, h,m,s) - constOffset = startAt + n·24h`, independent of the constant offset ⇒ reusing the same startAt across two no-DST zones yields IDENTICAL instants (vacuous). WEEKLY likewise (7·24h multiples). MONTHLY;BYMONTHDAY anchors to a calendar day (the 15th) at startAt's wall-clock-of-day, which DIFFERS between zones (same UTC startAt reads as 09:00 IST vs 23:30 EDT), so the instants genuinely diverge (Jun 15 vs Jun 16 03:30Z — a full day apart). The service-level tz-sweep tests reuse the SAME startAt with only tz changed and demonstrate real divergence — non-vacuous. Argument is mathematically sound.
Minor (non-blocking): the `timezone-matrix.spec.ts` §TZ-CHANGE cell itself uses DAILY with two DIFFERENT startAts + `after=startAt`, so that particular cell is only an identity round-trip + "distinct startAts differ" — the weaker form. It is honestly labeled, and the genuine reuse-same-startAt divergence is proven in `reminders-service.tz-sweep.spec.ts` and `scheduler-tz-sweep.spec.ts`. Coverage as a whole is non-vacuous.
