### Non-blocking (carry-forward to T070 sweep / M6 gate)
1. **Path-traversal hardening in the shared namespace guard.** Both `viewUrls` and the pre-existing `confirmUpload` gate only on `key.startsWith("pets/<petId>/original/")`. A crafted key such as `pets/<petId>/original/../../<otherPetId>/original/x.jpg` passes `startsWith`, and `deriveThumbKey` (first-`/original/`ŌåÆ`/thumb/`) yields `pets/<petId>/thumb/../../<otherPetId>/original/x.jpg`. On real S3 keys are literal (`..` not resolved ŌåÆ harmless 404), but a MinIO/gateway that normalizes `..` could resolve to another pet's object behind a presigned GET. Exploitation also requires guessing UUIDs. This is a *pre-existing pattern faithfully mirrored by T069*, not a regression ŌĆö but `viewUrls` newly makes it a **read** primitive, so recommend the M6 security sweep add a `..`/normalization rejection (or regex-validate the key shape) to the shared guard. Note: also add a matching `confirmUpload` case.
2. **No test covers a photo-bearing row surviving a page append without re-render.** The T067 render-count proof uses `photoKeys:[]` fixtures, so it's vacuous for rows that mount the strip. The architecture (separate memoized strip + stable props) closes the hole structurally and the new integration test mounts a real strip, but T070's factory/coverage pass should add a photo-bearing-row append assertion to lock it.
3. **Per-row presign fan-out.** Each visible photo-bearing row fires its own `usePhotoViewUrls` POST (one batch of Ōēż6 keys per row, not per photo). Bounded by SectionList virtualization and cheap (presign is local compute, no S3 I/O), so acceptable for S ŌĆö but worth a batched-endpoint note if timeline photo density grows.
4. **404ing rendition renders expo-image's own empty state, not the `bg-brand-100` View placeholder** (the View placeholder only shows when the key is absent from the response, which never happens since the endpoint always signs). Cosmetic, matches the plan's documented risk. No crash.
