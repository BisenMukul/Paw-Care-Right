# T058 — Notification preferences + quiet hours — CHECKER review (re-review after fix)

## Original verdict: FAIL (recorded for history)
The quiet-hours deferred job carried only `{ reminderEventId }`, and `PushProcessor` called
`sendForEvent(id)` which loops the ENTIRE household. In a multi-member household where one
member set quiet hours and another did not, the non-quiet co-member was pushed at reminder
time and pushed AGAIN when the quiet member's deferred job re-ran at window-end (event still
`status:"SENT"`; resend happened whether the collapse claim had expired — `setNx` wins — or was
still alive — `owner === event.id` "re-enter" branch). This reliably double-notified co-members,
violating the plan Decision-2 guarantee "no path double-pushes a user" and the T057 per-user
collapse invariant, and was uncovered because every defer test used a single-member household.

## Fix delta reviewed (4 files)
1. `push.contract.ts` — `PushJobData.userId?: string` added, documented as set ONLY on a
   quiet-hours deferred re-enqueue; scheduler (T056) jobs still omit it. Correct.
2. `push.processor.ts` — passes it through: `sendForEvent(job.data.reminderEventId, job.data.userId)`. Correct.
3. `push-sender.service.ts` — `sendForEvent(reminderEventId, targetUserId?)`. The recipient
   list is narrowed BEFORE the per-user loop:
   `recipientUserIds = targetUserId !== undefined ? allRecipientUserIds.filter(id => id === targetUserId) : allRecipientUserIds`
   (lines 123-128). Because the filter gates the loop itself, EVERY per-user side effect is
   short-circuited for non-target members — the collapse `setNx` claim, the quiet-hours
   re-defer enqueue, the group query, the device fetch, the send, and (downstream, via an empty
   `outgoing`) the receipts enqueue. Not merely the final push call. The deferred enqueue now
   includes `userId` in the payload (`{ reminderEventId: event.id, userId }`, line 154) and the
   suffixed jobId `${event.id}:defer:${userId}:${windowEndMinuteEpoch}` (line 156) is intact per
   the plan. The targeted user still runs its own type-off / quiet-window re-check and its own
   claim unchanged.
4. `push-sender.service.spec.ts` — new "[checker regression]" test (line 555).

## Verification
- **All side effects gated, not just the push:** confirmed by code structure — the filter
  produces `recipientUserIds` that the entire `for` loop iterates, so a non-target member is
  never entered (no `setNx`, no re-defer `pushQueue.add`, no `device.findMany`, no receipts).
- **Deferred enqueue carries `userId` + intact jobId:** confirmed (lines 152-162); the original
  scheduler job path (no `userId`) still fans out to every recipient exactly once.
- **Regression test is non-vacuous (code-read mutation):** the test drives the original run
  (user-A defers, user-B sent once — asserts deferred payload `{ reminderEventId, userId: "user-A" }`
  and first-run recipients `["token-b"]`), then the window-end re-run `sendForEvent(EVENT_ID, "user-A")`.
  It asserts the second run's recipients are exactly `["token-a"]` (604), user-B is never
  re-fetched (608, 611), user-B's total sends across both runs === 1 and user-A === 1 (615-617),
  and exactly two distinct collapse claims total (620-622). Removing the filter makes
  `sendForEvent(EVENT_ID, "user-A")` reprocess user-B: line 604 would see `["token-a","token-b"]`,
  line 616 would count token-b twice, line 608 device-fetch delta would be +2 not +1, and line
  611 would find a user-B fetch — four independent assertions fail. Non-vacuous.
- **Edge — deferred `userId` no longer a household member:** `allRecipientUserIds.filter(id => id === targetUserId)`
  yields `[]`, the loop body never runs, `outgoing` stays empty, and `sendForEvent` returns
  cleanly at the `outgoing.length === 0` guard. No throw. Handled sanely.
- **T057 + T058 assertions intact:** all pre-existing tests present and unweakened — collapse
  3→1 count, losing-event-sends-nothing, winner-retry-re-enter, per-user distinct claims,
  collapse-key minute epoch, DeviceNotRegistered ticket/receipt prune, chunk isolation/throw,
  plus the T058 type-off suppression (no send/prune/defer), mixed-group count=1, defer-before-claim
  (`setNx` not called, `pushAdd` once), and outside-window normal send. Only additive change is
  the new regression test.
- **Gates:** coordinator independently verified typecheck green and api 56 suites / 581 tests
  (the +1 over the prior 580 is exactly this added regression test). Consistent; not re-run in
  full here per instruction (single-file api jest runs hang on the BullMQ open-handle leak).

The blocking double-push path is closed at its root, the fix has zero collateral side effects
for co-members, the edge case exits cleanly, and the regression is genuinely enforced. No new
scope, no forbidden patterns, no §7 safety impact (care-reminder scheduling only).

VERDICT: PASS
