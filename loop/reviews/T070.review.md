## Adversarial probe 1 — `isKeyInPetNamespace` attack keys (photos.constants.ts:26-34)
Read char-by-char. prefix = `pets/<petId>/original/`. Logic: startsWith(prefix) AND remainder non-empty AND no `/` AND no `..`. Attack results (prefix here = `pets/pet-1/original/`):
- `pets/pet-1/original/../../other-pet/original/x.jpg` → remainder has `/` → **REJECTED**.
- `pets/pet-1/original/..` → remainder `..` → **REJECTED**.
- `pets/pet-1/original/a/../b.jpg` → has `/` → **REJECTED**.
- bare prefix (remainder empty) / key === prefix → **REJECTED**.
- foreign prefix `pets/other-pet/original/abc.jpg` → startsWith false → **REJECTED**. (Prefix-confusion e.g. `pets/pet-10/...` also fails startsWith because prefix ends in `/original/`.)
- backslash `pets/pet-1/original/a\b.jpg` → no `/`, no `..` → **PASSES**. Not exploitable: `\` is a literal S3/MinIO key char, not a separator; `deriveThumbKey`/`deriveMainKey` only string-replace `/original/`→`/thumb|main/` and the trailing ext, so the derived key stays `pets/pet-1/thumb/a\b.jpg` — inside pet-1's namespace. No cross-tenant reach.
- URL-encoded `pets/pet-1/original/%2e%2ex.jpg` → no literal `/` or `..` → **PASSES**. Not exploitable: verified no decode occurs between DTO and presign — `viewUrls` passes the key literally into `deriveThumbKey`/`deriveMainKey` (pure string ops) → `storage.getPresignedGetUrl({key})`; the AWS SDK URL-encodes the literal key when signing, so `%2e%2e` becomes `%252e%252e` on the wire and the object key is the literal `%2e%2ex.jpg` — never resolves to `..`. Storage treats keys literally end-to-end, satisfying the plan's precondition. (Any `%2e%2e/…` variant with a slash is rejected outright.)
- Verdict on the helper: closes the traversal READ primitive; the two literal-char keys that pass are provably non-exploitable given literal-key storage. The "no `/`, no `..`" remainder rule (vs strict uuid regex) correctly keeps the legit `never-uploaded.jpg` → 404 path (constants.spec regression test present).
