## Truncation algebra (the sharp edge) — verified from code, not just tests
`buildVetSummary` (build-vet-summary.ts:188-210): `parts=[header, weight]`; `budget = 2500 - footer.length - SECTION_SEPARATOR.length(2)`; final = `parts.join("\n\n") + "\n\n" + footer`.
- Every list-section push goes through `fits(parts, candidate, budget)` = `[...parts, candidate].join("\n\n").length <= budget`. After a push, `parts` includes the candidate, so the invariant `parts.join("\n\n").length <= budget` holds cumulatively (each push re-checks against the current, growing `parts`). Final length ≤ `budget + 2 + footer.length = 2500`. **Exact-fit boundary** uses `<=`, so a section filling remaining budget exactly is allowed and still yields ≤2500. ✓
- **Marker overflow**: marker block pushed only if it fits; else if `included>0` push the included-only block; else push nothing and return `true` (no lower section). Footer still appended. ✓
- **Footer never dropped/truncated**: appended unconditionally as the literal tail `${SEP}${footer}`, its length reserved in `budget` up front, never a truncation candidate. Terminates **every** path including EMPTY (proven structurally, corroborated by the `it.each` footer-final test over GOLDEN/OVERFLOW/EMPTY). ✓
- **Empty sections** consume separator budget only via `fits`-guarded pushes; tiny, all present in EMPTY. ✓
- **Length metric**: `String.length` (UTF-16 code units) throughout — consistent with the AC/enforcement metric. Note excerpt `slice(0,120)` can split a surrogate pair (cosmetic mojibake, never a length or safety breach). Non-blocking.

Non-blocking theoretical note (carry-forward): header + weight blocks are pushed WITHOUT a budget check (plan decision 2 — "bounded, tiny"). A pathologically long `pet.name` (thousands of chars) could in principle push header+weight+footer over 2500. In practice pet name is length-bounded by the pets DTO upstream, so unreachable; acknowledged design assumption, not a defect for this task.
