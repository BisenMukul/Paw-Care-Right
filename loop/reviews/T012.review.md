## Security review — 8 points (code evidence)

**1. Rotation/theft defense (A1 fix) — VERIFIED CORRECT.** `refresh-token.service.ts#rotate`:
- (a) Reuse family-revocation runs **outside** the transaction: `updateMany({where:{familyId,revokedAt:null}})` at `:47-50` then `throw` at `:51` — both precede the `$transaction` opened at `:59`. The revocation is auto-committed and persists before the 401 (fails toward revocation). ✓
- (b) Happy-path rotation is a conditional write inside the tx: `tx.refreshToken.updateMany({where:{id:row.id, rotatedAt:null, revokedAt:null}, data:{rotatedAt}})` `:62-65`, `if (rotated.count===0) throw UnauthorizedException` `:67-69` — race guard, stronger than a bare `update`. Under Postgres READ COMMITTED the losing concurrent rotate re-evaluates the WHERE after the winner commits → count 0 → 401. ✓
- (c) New token issued in the **same** familyId: `this.issueWith(tx, row.userId, row.familyId)` `:71`. ✓
- Ordering of checks: unknown→401 (`:38-40`), reuse→family-revoke+401 (`:42-52`), natural expiry→401 **without** family-revoke (`:54-57`). Matches the deliberate theft-vs-expiry model. Unit spec `refresh-token.service.spec.ts` covers valid-rotate, race-loss (count 0), already-rotated, already-revoked, expired-no-revoke, unknown, revokeFamily, and revokeFamily-unknown-no-op.

**2. Hashed at rest — VERIFIED.** Refresh tokens: persisted only as `tokenHash: this.hash(token)` where `hash=createHash("sha256")…digest("hex")` (`refresh-token.service.ts:105,113-115`); raw 256-bit `randomBytes(32).base64url` (`:97`) is returned once, never stored/logged. OTP codes: persisted only as `createHmac("sha256", config.otpHmacSecret)` digest (`otp.service.ts:79-81,28`), TTL 600s (`:30`), single-use `del` on match (`:46`), attempt-capped (`:52`). Grep for logger/plaintext persistence of `code`/`token` found **only** the dev-transport line (point 5). ✓

**3. Anti-enumeration — VERIFIED.** `requestOtp` unconditionally returns `{ok:true}` — no user-exists branch (`auth.service.ts:52-56`). `verifyOtp` throws a single uniform `UnauthorizedException()` for wrong/expired/capped/unknown (`:58-65`, comment `:62-63`). `logout`→`revokeFamily` is a no-op on unknown token (`refresh-token.service.ts:81-84`) and controller always 200 (`auth.controller.ts:45-50`). Verify failure surfaces as bare 401 `UNAUTHORIZED` with no reason (filter maps `HttpStatus` → code generically). ✓

**4. Rate limit — VERIFIED, correctly scoped.** `rate-limit.guard.ts`: `INCR` (`:29`), `EXPIRE 60` only on first hit `count===1` (`:31-33`), `count>5` → `setHeader("Retry-After","60")` + throw 429 (`:35-38`). Applied **only** to `otp/request` (`auth.controller.ts:22`). No global throttler — `app.module.ts` diff adds only `AuthModule`; no `ThrottlerModule`/helmet/CORS (correctly deferred to T017). `Retry-After` survives the filter (`all-exceptions.filter.ts` uses `response.status().json()`, which does not strip prior headers) — proven by the AC5 header assertion. ✓

**5. Dev transport — VERIFIED.** `otp-transport.ts`: `sendOtp` logs `OTP for ${email}: ${code}` via Nest `Logger` **only** when `config.nodeEnv !== "production"` (`:23-26`); prod path logs a generic warn, never email+code (`:31`). Bound behind the `OTP_TRANSPORT` Symbol DI token (`auth.module.ts:32`); interface-only, no nodemailer/SMTP dependency added. e2e overrides with `CapturingOtpTransport` (`auth.e2e-spec.ts:49-52`). ✓

**6. JWT — VERIFIED.** Access token 15m (`ACCESS_TOKEN_TTL="15m"`, signed at `auth.service.ts:111-113`). Minimal claims — only `{ sub: userId }` (+ iat/exp); no email/household in the JWT. Secret from validated env via `JwtModule.registerAsync` → `config.jwtSecret` (`auth.module.ts:21-24`). `env.schema.ts` adds `JWT_SECRET`/`OTP_HMAC_SECRET` as `z.string().min(1).default(<clearly dev-insecure>)`. Only obviously-dev placeholder secrets committed (`.env.example`); no real `.env` touched. ✓

**7. Auto-provision — VERIFIED.** `provisionOrGetUser` creates `user` → `household(ownerId)` → `membership(role:"OWNER")` in one `prisma.$transaction` on first verify (`auth.service.ts:118-133`); known-email path reuses the earliest owned household, throws `InternalServerErrorException` on the (T012-impossible) no-household invariant break (`:135-144`) rather than silently creating. e2e asserts the User/Household/OWNER-Membership rows (`auth.e2e-spec.ts:124-139`); unit spec covers create-path, reuse-path, and invariant-break. ✓

**8. Attempt cap — VERIFIED (explicit test present).** `otp.service.ts:50-55`: on mismatch `attempts+1`; when `attempts >= OTP_MAX_ATTEMPTS(5)` the key is `del`'d (invalidated), else re-stored. Trace: 5 wrong guesses (0→5) then the correct code finds no record → false → 401. Covered by e2e `:168` and unit `otp.service.spec.ts:71-89`. ✓

## Non-blocking observations (reported, not FAIL)
- **OTP TTL reset on each wrong attempt:** `verifyCode` re-`set`s the record with a fresh `OTP_TTL_SECONDS` on a non-capping mismatch (`otp.service.ts:58`), resetting the 10-min clock per guess. Immaterial to brute-force resistance — the 5-attempt single-use cap bounds an attacker to ≤5 guesses over a 1e6 space regardless of TTL. Matches plan risk #9.
- **OTP consumed before provisioning:** `verifyCode` deletes the code on match before `provisionOrGetUser` runs; a provisioning failure consumes the code (no replay). Safe direction.
- **New dependency justification:** `@nestjs/jwt@^11` (official Nest JWT tooling for the mandated NestJS stack) is the sole new dep; `pnpm-lock.yaml` diff adds only `@nestjs/jwt@11.0.2`. CLAUDE §2.7 requires the one-line justification in `loop/journal.md` at commit/DoD time (not part of this working-tree diff) — flagged for the committer, not a code defect.
