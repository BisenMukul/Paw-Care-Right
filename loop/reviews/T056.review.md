## The #1 hunt — infinite-loop / off-by-one on advance: SAFE

## Non-blocking (T057/T062 carry-forwards)
1. The 24h backfill window `[now-24h, now]` is inclusive on both ends, so a daily reminder aligned exactly on both boundaries would enumerate 2 `MISSED` rows in a nominal-24h window. This is why the 50h fixture (not 48h) is used — sound reasoning, since 48h would land occurrences exactly on `from` and `now` and yield count 2, breaking the "exactly once" assertion. 2. Decision-3 semantics: if the best-effort `pushQueue.add` fails, the event stays `SENT` with no delivery while `nextFireAt` still advances (at-most-once push). Accepted per plan (non-emergency reminders; T062 agenda-vs-fired reconciliation). Preserves the never-double-push guarantee.
3. Backfill advance is unguarded (no `nextFireAt:dueAt` predicate) — safe only because boot is single-threaded; documented as such. If backfill were ever invoked concurrently with a tick this would race. No action needed for T056.
