  `metadataAccessor.isProcessor(instance.constructor || metatype)` (bull.explorer.js:42-64). The
  `overrideProvider(CheckRunnerProcessor).useValue({ process: async () => undefined })` yields an
  instance whose `constructor` is `Object` (no `@Processor` reflect-metadata) → filtered out → no

  instantiated (override replaces it). Empirically confirmed: 2 full runs clean, no race.
- **Justified.** The file's own prior docstring anticipated T043 consuming the queue; wiring the
  processor into WorkersModule/AppModule made a live worker boot inside this GET-read-path suite
  that manually stages statuses — a real §-neutral test-infra race, not a product bug. Fixing it
  in the same commit is correct.
- **No assertion semantics changed.** Diff is exactly the header comment rewrite + two
  `overrideProvider(...).useValue(...)` insertions (beforeAll + premium bootstrap). Grep for
  added expect/status/get/post lines returned nothing. Verified.

## Deviation #2 — residual cross-file worker race (evaluated for severity)
The ~10 other e2e suites boot `AppModule` WITHOUT the override, so a live `CheckRunnerProcessor`
Worker attaches to the shared `pawcareright-checks` queue. Traced behavior: no `AI_TEXT_PROVIDER=
fake` is set anywhere in the api jest env (jest.setup.ts is only `reflect-metadata`; no env in
package.json jest block / global-setup), so `getTextProvider()` → `OllamaTextProvider` at
`https://ollama.com` with placeholder key `example-ollama-cloud-api-key`. If such a worker
dequeues a `pawcareright-checks` job (only ever enqueued by checks.e2e's POST), it loads the
shared-Postgres row, transitions it, calls the provider → network failure with the placeholder
key → `runTriage` absorbs → **SAFE_FALLBACK → FALLBACK write** (itself §5-safe). The only harm is
a foreign worker flipping a check row's status that checks.e2e asserts on — the observed
one-time RUNNING/DONE flake.
- **Severity: LOW, CI-flake-only, non-safety.** Observed once, not reproduced in 6 executor runs
  + my 2 = 8 clean runs. The primary suite is now deterministic. **Non-blocking.**
- **Recommendation nuance for M4:** the executor's suggested `AI_TEXT_PROVIDER=fake` in
  global-setup would only remove the wasteful network call — a fake-provider worker would STILL
  dequeue and process the job and flip row status (arguably *faster*, so the race is not
  eliminated). The robust fix is to prevent foreign workers from processing checks.e2e's jobs:
  apply the same `CheckRunnerProcessor` no-op override across the AppModule-booting e2e files
  (ideally via a shared test-module helper), or gate the processor's registration behind a test
  env flag. Recommended as a small follow-up hygiene task, not a T043 blocker.
