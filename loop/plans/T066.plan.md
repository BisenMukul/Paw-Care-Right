# Plan — T066: Quick-log actions (weight / note+photo / vet visit)

## Objective (from card)
Wire the pet-home quick actions so a user can log a **note+photo** and a **vet visit**
in 2 taps (the weight action already landed in T065). Each action must create the
correct HealthLog `kind` (integration-tested at the client boundary), and both forms
must validate via the shared `packages/types` Zod schemas (`noteValueSchema`,
`vetVisitValueSchema`) before posting.

## Safety statement (CLAUDE.md §7 / PRODUCT_SPEC §5)
These are **plain record-keeping** surfaces, not AI-result surfaces. They produce no AI
output and touch no disclaimer / emergency / triage / safe-fallback code
(`<VetDisclaimer/>`, the Emergency interstitial, `src/checks/*` are all unaffected).
Note and vet-visit entries are **as-entered** free text only. The VET_VISIT schema
carries **NO** medication/dose field and this task adds none (mirrors
`Reminder.medNameAsEntered` posture, CLAUDE §7 rule 2); no field, string, or placeholder
in this task suggests, requests, or computes a drug name, dose, or any medical advice.
No "diagnosis"/"possible causes"/interpretive copy anywhere. This task does not weaken
any Safety Policy surface. **No SAFETY-ESCALATION applies.**

## Key decisions (planner-made — checker, scrutinize these)
1. **NO api / types / data / prisma change — this is a mobile-only task (the main scope
   call).** The photo upload endpoint discovered in the repo is **pet-scoped, not
   check-scoped**: `POST /v1/pets/:petId/photo-upload-url` + `/v1/pets/:petId/photo-upload-confirm`
   (`apps/api/src/photos/photos.controller.ts`), producing keys under
   `pets/<petId>/original/<uuid>.<ext>`. The T064 `CreateLogDto` already accepts
   `photoKeys?: string[]` (`@ArrayMaxSize(HEALTH_LOG_PHOTO_KEYS_MAX=6)`) and the service
   already stores them. Therefore health-log photos **reuse the existing pet-scoped
   upload machinery verbatim** via `uploadIntakePhoto(petId, uri, onProgress)`
   (`apps/mobile/src/api/intake-photos-api.ts`). This is option (b) from the card:
   reuse the generic pet-scoped media endpoint — **no new upload-URL endpoint is added.**
   (`uploadIntakePhoto` is misleadingly named but is a generic pet-scoped presign→PUT→confirm
   helper; we reuse it directly rather than rename it, since renaming would touch
   out-of-scope intake code.)
2. **Entry shape mirrors T065's weight route: dedicated form routes, not pet-home modals.**
   New flat routes `app/note/[petId].tsx` and `app/vet-visit/[petId].tsx` (mirror
   `app/weight/[petId].tsx`). The 2-tap path is: **tap 1** = pet-home quick action →
   navigates straight to the form route (the form is the screen body, immediately
   visible — no intermediate "add" button, unlike the chart-first weight screen);
   **tap 2** = Save. Each screen gates on `usePet` for the mandated loading/error/
   empty/offline states (§6), then `router.back()` on successful save.
3. **Forms validate through the shared Zod schemas directly** (AC "validate via shared
   schemas"). A pure mobile module `src/health-logs/health-log-forms.ts` wraps
   `noteValueSchema` / `vetVisitValueSchema` (`safeParse`), trims input, drops empty
   optional strings to `undefined`, and maps failures to string error keys. Forms call
   these pure validators before ever calling the mutation — mirrors how
   `add-weight-form.tsx` gates on `parseDisplayToGrams`.
4. **`onSuccess` invalidates a new `healthTimelineKeys.pet(petId)` namespace** (forward
   contract for T067's timeline list). No query consumes it yet, so invalidation is a
   harmless no-op today and the honest pattern once T067's `useHealthTimeline` lands
   (same posture as `useAddWeight`→`weightSeriesKeys`). Flagged for the checker as the
   one cross-task contract introduced here.
5. **Scope trims kept minimal for the S size, each an optional/deferrable field:**
   - **Photos attach to the NOTE action only** (the card says "note+photo, vet visit").
     The vet-visit form is text-only. `photoKeys` is a top-level create field so this is
     purely a UX choice, not a schema limit.
   - **`vetVisitValueSchema.costMicroUsd` is OMITTED from the v1 UI.** It is optional in
     the schema; money entry (locale/decimal → integer micro-USD) is an error-prone
     rabbit hole disproportionate to an S task. The form ships `reason` (required),
     `clinicName` (optional), `notes` (optional). A vet visit with those fields absent
     validates fine.
   - **`occurredAt` = now** for both actions (no date picker), mirroring weight.
6. **`photoKeys` is included in the POST body only when non-empty** (mirrors weight
   omitting `photoKeys` entirely); an empty attach set posts a bare NOTE.

## Files to create/modify (exhaustive — executor may touch NOTHING else)

### Create — pure logic
- `apps/mobile/src/health-logs/health-log-forms.ts` — pure, React-free validators over
  the shared schemas. Imports `noteValueSchema`, `vetVisitValueSchema`, `NoteValue`,
  `VetVisitValue` from `@pawcareright/types`. Exports:
  - `type NoteFormError = "empty" | "tooLong";`
  - `validateNoteForm(text: string): { ok: true; value: NoteValue } | { ok: false; error: NoteFormError }`
    — trims; empty → `{ ok:false, error:"empty" }`; else `noteValueSchema.safeParse({ text })`;
    on failure (max 2000) → `"tooLong"`; success → `{ ok:true, value }`.
  - `type VetVisitFieldError = "empty" | "tooLong";`
  - `type VetVisitFormErrors = { reason?: VetVisitFieldError; clinicName?: VetVisitFieldError; notes?: VetVisitFieldError };`
  - `validateVetVisitForm(raw: { reason: string; clinicName: string; notes: string }): { ok: true; value: VetVisitValue } | { ok: false; errors: VetVisitFormErrors }`
    — trims all; builds a candidate that **omits** `clinicName`/`notes` keys when their
    trimmed value is empty (so optional-absent is valid); `reason` empty → `errors.reason="empty"`
    (short-circuit before parse so the "required" message is precise); else
    `vetVisitValueSchema.safeParse(candidate)`, mapping each failing `issue.path[0]`
    (`reason`/`clinicName`/`notes`) to `"tooLong"`; success → `{ ok:true, value }`.
    Never returns `costMicroUsd` in the candidate (decision 5).

## Risks & edge cases (checker: scrutinize)
- **Upload fails mid-form:** the picker keeps the slot in `failed` with a retry; a failed
  photo contributes no key, so Save still posts the note with only the successfully
  uploaded keys — the note is never blocked by a photo failure. Tested.
- **Offline:** each screen renders its offline banner (via `useIsOffline`) and gates like
  the weight screen; a Save attempt while offline surfaces the mutation error path (no crash).
  §6 offline state covered by the gate.
- **`photoKeys` > 6:** the picker disables adding at `HEALTH_LOG_PHOTO_KEYS_MAX`; the api's
  `@ArrayMaxSize(6)` is the server backstop. Tested at the limit.
- **Empty note / whitespace-only reason:** rejected client-side by the shared-schema
  validators before any POST; inline error shown, mutation not called. Tested.
- **`strictObject` schemas:** sending an empty-string optional (`clinicName:""`) would fail
  `min(1)`; the validator omits empty optionals to `undefined` so they are dropped from the
  payload — verified in `health-log-forms.test.ts`.
- **Effect-loop in the photo picker:** the `onKeysChange` effect is keyed on the joined-keys
  string (not array identity), replicating the guard proven in `PhotoPromptQuestion`.
- **QuickActions signature change** ripples into the pet-home snapshot; snapshot is
  refreshed and the two nav assertions added — `onPress` handlers are not serialized so the
  existing weight/reminders assertions are unaffected.
- **Decisions 1 (mobile-only reuse of the pet-scoped upload endpoint), 4 (new timeline key
  namespace), and 5 (photo-on-note-only + cost omitted)** are the only non-forced choices —
  flagged here for review.
