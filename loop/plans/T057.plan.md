# Plan — T057: Expo push sender

## Objective (from card)
Consume `pawcareright-push` `{ reminderEventId }` jobs (produced by T056) and actually deliver the push: batch-send in chunks of 100, poll delivery receipts later, prune `Device` rows that come back `DeviceNotRegistered` (in tickets **and** receipts), and collapse simultaneous reminders for the **same user in the same minute** into a single push that shows a count. All logic unit-tested against a mocked Expo client.

## Safety statement
Pure notification-delivery infrastructure. It renders **no** AI output and touches **none** of the `PRODUCT_SPEC §5` safety surfaces — no disclaimers, no emergency escalation, no medication dosing — so §5 is not weakened and this task is safe to plan.
- **Push copy is pinned and generic** (see "Send pipeline spec"). Exactly two body forms: single → `"Care reminder: {reminder.title}"`; collapsed (count ≥ 2) → `"{count} care reminders due"`. `title` = the shared `APP_DISPLAY_NAME` constant only. The body **never** adds medical advice, dosing, drug names, the vet-confirm note, or the word "diagnosis"; it echoes only the reminder title the user themselves created, to that user's own devices. Med-type reminders are shown by their stored title only — the sender never derives or suggests a dose (CLAUDE §7 rules 1–2).
- **Tokens are credentials-adjacent: never logged.** Every log line is id/count-keyed (`reminderEventId`, `deviceId`, `userId`, chunk index, counts) — never `expoPushToken`, never `reminder.title`/`type`/`medNameAsEntered` (CLAUDE §6, mirrors the check-runner/scheduler "ids only" rule).

## Key decisions (numbered; restated under "Risks & decisions")
1. **Transport = the `expo-server-sdk` npm package, wrapped behind an injectable port `ExpoPushClient`.** New dependency (justification line required in `loop/journal.md`): it is the official, actively-maintained Expo server SDK; it owns token-format validation (`Expo.isExpoPushToken`), gzip/APNs/FCM routing, and the typed ticket/receipt error shapes, which are the fiddly, stable-but-easy-to-get-wrong parts. The **port normalizes** SDK tickets/receipts into tiny local shapes (`{ status, id?, errorCode? }`), so the sender never imports SDK types and unit tests mock a 3-method interface ("mocked Expo SDK"). Chunking-at-100, collapse, pruning and receipt orchestration stay in **our** service (so they are directly testable), not delegated to the SDK.
2. **Collapse = per-user, deduplicated by an atomic Redis claim per `(userId, dueMinute)`.** The push queue delivers one job per event (T056, `jobId = eventId`), so collapse must happen at send time across jobs. Each recipient user's `(userId, minuteEpoch)` is claimed via `SET key ownerEventId NX EX`; the event that wins the claim sends ONE push whose count = the number of still-eligible (`status = SENT`) events due that minute across that user's households; events that lose the claim skip that user (they are already counted by the winner). The claim stores the **winner event id** so the winner's own BullMQ retries re-enter (get == self ⇒ proceed) while other events are excluded — retry-safe. Chosen over "leader = min(eventId)" (which races to 0-or-2 pushes) and over a per-`(user,event)` DB claim table (needs a migration).
3. **Event status: read-only within the existing enum — no migration.** `ReminderEventStatus` has no `DELIVERED`/`FAILED`; T056 already creates events `SENT` (= "handed to the push pipeline") with `sentAt`. The sender treats `status === "SENT"` as "eligible to notify" and **writes nothing** to `ReminderEvent` (so a reminder the user completed/snoozed in the gap — status moved to `DONE`/`SNOOZED` — is naturally excluded). Delivery truth (tickets/receipts) is observable via logs + the device-pruning side effect; it is deliberately **not** persisted on the event (no column exists, none is added).
4. **Receipts polled on a separate delayed queue.** Ticket ids are collected at send time and enqueued as a single delayed job on a new `pawcareright-push-receipts` queue (`+15 min`); its worker calls `getReceipts`, prunes any device whose receipt is `DeviceNotRegistered`, logs other receipt errors. Keeps the send path fast and the receipt path independently retryable.
5. **Retry/backoff is set here (it was T056's explicit hand-off).** T056's plan states retry/backoff for the push job is T057's concern; so this task adds `attempts`+`backoff` to the existing `pushQueue.add(...)` opts in `reminder-scheduler.service.ts` (the only change to that file). The receipt job sets its own `attempts`+`backoff`. On a transient Expo/network error the sender **throws after finishing the other chunks** (per-chunk isolation), letting BullMQ retry the whole job.

## Send pipeline spec (`sendForEvent(reminderEventId)`)
1. Load the event with its reminder → pet → household → memberships:
   `prisma.reminderEvent.findUnique({ where: { id }, include: { reminder: { include: { pet: { include: { household: { include: { memberships: { select: { userId: true } } } } } } } } } })`.
   - If `null` (redelivered job for a deleted row) → log `push_event_missing`, return.
   - If `event.status !== "SENT"` (already `DONE`/`SNOOZED`/`MISSED`) → log `push_event_not_eligible`, return.
2. `minuteEpoch = Math.floor(event.dueAt.getTime() / 60_000)`; `minuteStart = new Date(minuteEpoch * 60_000)`, `minuteEnd = new Date((minuteEpoch + 1) * 60_000)`.
3. `recipientUserIds = event.reminder.pet.household.memberships.map(m => m.userId)` (dedup).
4. Build the outgoing message list (array of `{ message: PushMessage; deviceId: string }`) by iterating recipients:
   For each `userId`:
   a. **Claim:** `won = await redis.setNx(collapseKey(userId, minuteEpoch), event.id, COLLAPSE_CLAIM_TTL_SECONDS)`. If `!won`, `owner = await redis.get(collapseKey(...))`; if `owner !== event.id` → **skip this user** (another event owns the collapse). If `owner === event.id` (this is a retry of the winner) → proceed.
   b. **Collapse group + count:** `group = prisma.reminderEvent.findMany({ where: { status: "SENT", dueAt: { gte: minuteStart, lt: minuteEnd }, reminder: { pet: { household: { memberships: { some: { userId } } } } } }, include: { reminder: { select: { title: true } } } })`. `count = group.length` (≥ 1). `singleTitle = group[0].reminder.title` (used only when `count === 1`).
   c. **Devices:** `devices = prisma.device.findMany({ where: { userId }, select: { id: true, expoPushToken: true } })`. Filter to `expoClient.isValidToken(token)` (log+skip invalids by count, no prune). If none → continue.
   d. For each valid device push `{ message: { to: token, title: APP_DISPLAY_NAME, body: buildReminderPushBody({ count, singleTitle }), data: { reminderEventId: event.id, count, kind: "reminder" }, sound: "default" }, deviceId }`.
5. If the message list is empty → return (nothing to send).
6. **Chunk + send (batch 100):** slice the list into chunks of `EXPO_PUSH_CHUNK_SIZE` (100), preserving order. For each chunk (sequential), `tickets = await expoClient.sendChunk(chunk.map(x => x.message))` inside try/catch:
   - **catch (transient/5xx/network):** log `push_chunk_failed` (chunk index only), set `hadTransientError = true`, **continue to the next chunk** (per-chunk isolation).
   - **success:** zip `tickets[i]` to `chunk[i].deviceId` (Expo returns tickets in input order):
     - `status === "error"` && `errorCode === "DeviceNotRegistered"` → add `deviceId` to `prune` set.
     - `status === "error"` (other codes) → log `push_ticket_error` (`deviceId`, `errorCode`).
     - `status === "ok"` && `id` → push `{ ticketId: id, deviceId }` to `receiptTargets`.
7. **Prune** (immediate ticket-level): if `prune` non-empty → `prisma.device.deleteMany({ where: { id: { in: [...prune] } } })`; log `push_device_pruned` with count + ids.
8. **Enqueue receipt check:** if `receiptTargets` non-empty → `receiptsQueue.add(PUSH_RECEIPT_JOB_NAME, { tickets: receiptTargets }, { jobId: \`receipt:${event.id}\`, delay: RECEIPT_CHECK_DELAY_MS, attempts: 3, backoff: { type: "exponential", delay: 60_000 }, removeOnComplete: true, removeOnFail: false })` (best-effort try/catch; a queue failure is logged, not thrown).
9. Log `push_sent` (`reminderEventId`, recipient count, message count, chunk count, pruned count, receipt count) — ids/counts only.
10. If `hadTransientError` → `throw new Error("expo_send_transient_failure")` so BullMQ retries the job (decision 5). (Note: retry re-sends already-delivered chunks — at-least-once, documented Risk 4.)

## Collapse spec
- **Key:** `collapseKey(userId, minuteEpoch)` → `pawcareright:push:collapse:${userId}:${minuteEpoch}` (CLAUDE §1a Redis prefix). TTL `COLLAPSE_CLAIM_TTL_SECONDS = 3600`.
- **Semantics:** exactly one push per `(user, minute)`. Winner = the event that first `SETNX`-claims the key; it sends one message per device with `count` = number of `SENT` events due that minute across that user's households. Non-winner events skip that user. The stored owner id lets the winner's retries re-enter without a second distinct push.
- **Body:** `buildReminderPushBody({ count, singleTitle })` → `count >= 2 ? \`${count} care reminders due\` : \`Care reminder: ${singleTitle}\``.
- **Cross-household:** a user in multiple households gets a single collapsed push covering all their due reminders that minute (query is by membership, not household). "Different users" → different keys → separate pushes. "Different minutes" → different `minuteEpoch` → separate keys → separate pushes.

## Receipt & pruning spec (`checkReceipts(tickets)`)
1. `receipts = await expoClient.getReceipts(tickets.map(t => t.ticketId))` (adapter chunks internally).
2. Build a `ticketId → deviceId` map from the payload. For each ticket id present in `receipts`:
   - `status === "error"` && `errorCode === "DeviceNotRegistered"` → collect `deviceId` for pruning.
   - `status === "error"` (other codes, e.g. `MessageRateExceeded`) → log `push_receipt_error` (`deviceId`, `errorCode`), do **not** prune.
   - `status === "ok"` → no-op.
   - ticket id absent from `receipts` (not ready yet) → log at debug, no-op (a later attempt/job may re-poll; not persisted — accepted, Risk 5).
3. Prune: `prisma.device.deleteMany({ where: { id: { in: [...deviceIds] } } })` if non-empty; log count.
4. Pruning = **delete** the `Device` row (matches `DevicesService`, which deletes rows; the unique `expoPushToken` is freed so a future re-register succeeds). `deleteMany` is idempotent (a row already gone ⇒ count 0, no throw).

## Risks & decisions (numbered — for the checker to scrutinize)
1. **New dependency `expo-server-sdk` (decision 1).** Justification (journal): official, actively-maintained Expo server SDK; provides token validation + typed ticket/receipt error shapes + transport (gzip/APNs/FCM). Alternative hand-rolled HTTPS to `exp.host` rejected (more bespoke error/receipt code, less clearly "mocked Expo SDK"). Risk: an unmaintained/oversized dep — checker verifies the release recency and that it is imported in exactly one file (`expo-push.client.ts`).
2. **Per-user collapse via Redis `SETNX` claim (decision 2).** Race: a sibling event materialized by T056 *after* the winner already queried its group is not counted and — because the key is claimed — is not separately pushed (**under-count / rare silent drop of that one reminder's push**). Safe direction: reminders are non-emergency, the event still exists in-app, and no double-push ever occurs per user. Mitigations noted but deferred: T056 materializes all due events in one synchronous `tick()` before the push worker typically drains; a small push-enqueue delay could further settle the group. Checker: confirm no path double-pushes a single user in a minute and that a losing/duplicate event never throws.
3. **Event is read-only; delivery state not persisted (decision 3).** Within the existing enum there is no `DELIVERED`/`FAILED`; adding one is a migration this task avoids. Consequence: delivery success/failure is only observable via logs + device pruning, not queryable on `ReminderEvent`. Judged acceptable for T057; a future reconciliation task can add it if needed. Checker: confirm the sender performs no `reminderEvent.update`/`create`.
4. **At-least-once delivery (decision 5).** A transient failure on a later chunk throws to retry the whole job, re-sending earlier already-delivered chunks ⇒ possible duplicate notifications. Accepted (Expo is at-least-once anyway; reminders are low-stakes). Checker: confirm per-chunk try/catch isolates chunks and the throw happens only after all chunks are attempted.
5. **Modifying T056's `pushQueue.add` opts (decision 5).** Adds `attempts`/`backoff` to `reminder-scheduler.service.ts`. This is the hand-off T056's own plan (§Push queue contract) assigned to T057. Risk: touching another task's file — mitigated because the existing spec uses `expect.objectContaining`, so added opts don't break it (spec untouched). Checker: verify the T056 spec still passes and only the opts object changed.
6. **No Expo access token wired.** `new Expo()` runs unauthenticated (subject to Expo's default rate limits); adding `EXPO_ACCESS_TOKEN` would require an env-schema change in `packages/config` (out of scope). Accepted for now; a future task can inject it. Checker: confirm no direct `process.env` read (CLAUDE §6 "config via validated env").
