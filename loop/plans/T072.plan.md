# Plan — T072: Subscription mirror model + entitlement service

## Objective (from card)
Add the `Subscription` mirror model (ARCHITECTURE §3) + a household-aware entitlement
service that resolves whether the caller is premium (own sub OR a household member's
active **family** sub), exposed at `GET /billing/entitlement`. Model + read-side only —
T073 writes rows via the RC webhook. No mobile changes, no gating of any feature.

## Key decisions (checker: scrutinize)
1. **Cardinality — CONFLICT WITH ARCHITECTURE, follow the card.** ARCHITECTURE §3's ERD
   draws `Household ||--o| Subscription` (≤1 sub per household). The card's acceptance
   paths ("*non-family sub of another member*", "*multiple rows pick the strongest*")
   require **per-purchaser** rows, i.e. `Household ||--o{ Subscription` and
   `User ||--o| Subscription`. Per ARCHITECTURE line 3 ("where a task card and this doc
   disagree, journal it and follow the task card") we model **one row per RevenueCat app
   user**. The row's **field list stays exactly ARCHITECTURE §3**:
   `(householdId, rcAppUserId, entitlement FREE|PREMIUM, plan?, status, expiresAt, rawEventJson)`
   — no `store`/`userId`/`isFamily`/`eventId` columns added (avoid speculative scope).
   Executor MUST note this cardinality conflict in `loop/journal.md`.
2. **`rcAppUserId` is the primary key AND the FK to `User.id`, and the T073 upsert key.**
   T071 set purchaser identity = backend `userId` (`Purchases.logIn(user.id)`), so
   RevenueCat's `app_user_id` **equals** the backend `userId`. Therefore `rcAppUserId`
   both identifies the purchaser (FK → `User.id`, `onDelete: Cascade`) and is the unique
   idempotent upsert key T073 will `upsert({ where: { rcAppUserId } })` on. No separate
   `userId` column (it would duplicate `rcAppUserId`). RC **event-id** idempotency is
   T073's concern — no column here.
3. **Family detection via a shared constant, not a stored column.** A sub is "family" iff
   `plan === FAMILY_PLAN_PRODUCT_ID` (`"pawcareright_family_annual"`, new constant in
   `packages/types`). Keeps the model faithful to ARCHITECTURE (no extra `isFamily`
   column) and gives one server-side source of truth for the family product id.
4. **Resolution = two bounded queries (no N+1), then an in-memory strongest-pick.**
   Given caller `userId` + caller **current** `householdId`:
   - **own** = `subscription.findUnique({ where: { rcAppUserId: userId } })` (PK lookup, 0/1).
   - **family** = `subscription.findMany({ where: { householdId } })` (indexed; household
     sub-count is tiny), then in-memory keep `plan === FAMILY_PLAN_PRODUCT_ID`.
   Own sub is matched by **identity** (survives household moves); family grant is scoped to
   the caller's **current** household (leaving the household drops the family benefit).
5. **Active/expiry predicate (the grace decision).** A row entitles iff
   `entitlement === "PREMIUM" && (expiresAt === null || expiresAt.getTime() + GRACE_MS > now)`.
   The **clock is authoritative for expiry**: a PREMIUM row whose `expiresAt` is past is
   NOT entitled even if a renewal webhook is late (conservative — losing premium is
   fail-safe, never a safety issue). `GRACE_MS = SUBSCRIPTION_GRACE_MS = 0` (documented
   single knob): RevenueCat already folds billing-retry grace into the `expiresAt` it
   sends, so we add none. The raw `status` string is stored for traceability but is **not**
   the resolution authority (avoids parsing RC's many status values); `entitlement` +
   clock decide. `entitlement === "PREMIUM"` with `expiresAt` future but `status` odd →
   entitled; `entitlement === "FREE"` (expired row) → not entitled regardless of clock.
6. **Strongest-pick + response shape.** Precedence **own > family > none**. If own is
   active → `source: "own"`. Else among active family rows pick the one with the latest
   `expiresAt` (`null` = farthest future) → `source: "family"`. Else `source: "none"`.
   Response (shared Zod `billingEntitlementSchema` in `packages/types`):
   `{ entitled: boolean, source: "own"|"family"|"none", plan: string|null, expiresAt: string|null }`
   (`expiresAt` = ISO string of the entitling row, or `null`). Service returns this exact
   shape (Date → `toISOString()`), so the return type === the schema type.

## Risks & edge cases (checker: scrutinize)
- **R1 — ARCHITECTURE cardinality conflict (decision 1):** ERD says ≤1 sub/household; the
  card's ACs need per-user rows. We follow the card and journal it. Field list stays exactly
  §3. If the checker reads ARCHITECTURE literally this will look like a deviation — it is the
  documented, card-authorized one.
- **R2 — user leaves household mid-sub:** own sub is matched by `rcAppUserId` (identity), so
  the purchaser keeps their own premium after moving. A **family** sub's grant follows its
  stored `householdId`: after the purchaser moves, members remaining in the old household
  keep the family benefit AND the purchaser keeps it via the own-path — so a single family
  sub can transiently benefit two households. This over-grants (never a safety issue) and
  self-corrects on the next T073 webhook; flagged for T073.
- **R3 — clock skew vs late RC events (decision 5):** the clock is authoritative for expiry,
  so a delayed renewal webhook briefly shows the user as expired (fail-safe: lose premium,
  never a §5 surface). `GRACE_MS = 0` because RC already bakes billing-retry grace into
  `expiresAt`; the single constant is the tuning point if that assumption changes.
- **R4 — status vs expiresAt disagreement:** resolution ignores the raw `status` string and
  uses `entitlement === "PREMIUM"` + clock. Contradictory rows (FREE+future, PREMIUM+past)
  resolve conservatively to not-entitled / entitled-until-expiry respectively.
- **R5 — two family subs in one household:** `pickEntitlement` picks the active family row
  with the latest `expiresAt` (`null` = farthest future); covered by a named test.
- **R6 — household deletion cascades the sub:** `Subscription.householdId` FK is
  `onDelete: Cascade`; JOIN-REPLACES deletes an empty owner household, which would drop that
  purchaser's mirror row. Row self-heals on the next T073 webhook (upsert by `rcAppUserId`).
  Non-safety billing gap; flagged for T073. Kept Cascade to honor the non-null `householdId`.
- **R7 — `FAMILY_PLAN_PRODUCT_ID` duplicates the mobile `PRODUCT_IDS.family` string.** The
  `packages/types` constant is the server source of truth; a `entitlement.spec.ts` note pins
  the exact value. Mobile is not edited this task (T071-owned); drift is a checker/journal note.
