# Plan — T014: Google Sign-In

## Objective (from card)
Extend `POST /v1/auth/social` to verify Google `id_token`s by adding a second `SocialTokenVerifier`
(`GoogleTokenVerifier`) behind the existing provider-agnostic abstraction (T013), widening the DTO
allowlist and the `SOCIAL_TOKEN_VERIFIERS` registry to `[apple, google]`. Reuse the *entire* existing
link-by-sub / link-by-verified-email / provision / P2002-race path unchanged. Same acceptance matrix as
T013, applied to Google; the provider abstraction is unit-tested. No new provisioning logic, no schema
change, no new dependency.

## Tests → acceptance-criteria map

### AC "Same test matrix as T013 for Google" → `apps/api/test/auth-social.e2e-spec.ts` (Google cases; `GOOGLE_JWKS_RESOLVER` overridden with a local keyset; offline/deterministic)
- **valid → session** → `valid Google token for a new user provisions account + returns a session`: POST a
  Google token (correct iss/aud/sub, `email_verified: true`) → **200**; non-empty `accessToken`/`refreshToken`,
  `user.email` === token email, string `householdId`; DB `User.googleSub` === sub, owned `Household`, `OWNER`
  `Membership`; decoded access-token `sub` === new user id.
- **tampered → 401** → token signed by a key absent from the Google JWKS → **401**, `error.code === "UNAUTHORIZED"`,
  no `accessToken`/`refreshToken`, NOT 500.
- **wrong audience → 401**, **wrong issuer → 401** (iss not in `GOOGLE_ISSUERS`), **expired → 401** — each signed
  by the valid Google key with one bad claim → all uniform **401**.
- **existing-email links, not duplicates** → `Google sign-in with an email matching an existing user links, does
  not duplicate`: seed user E (no googleSub) + household + membership via Prisma; POST a Google token (email E
  verified, fresh sub G) → **200**, `user.id` === existing id; `user.findMany({email:E})` length **1**, that
  user's `googleSub` === G, decoded sub === existing id.
- **idempotent by sub** → `same Google token twice returns the same user`: two POSTs, same token → both 200, same
  `user.id`, exactly one `User` row for that googleSub.
- **no verified email → 401** → `new Google sub with no verified email → 401`: fresh sub, `email_verified:false`
  (or absent), no existing googleSub → **401** (fail-closed; cannot provision, cannot link).
- **CROSS-provider link (the natural "existing-email links not duplicates" for Google)** →
  `existing Apple-linked user signing in with Google links to the SAME user`: seed a user with `appleSub` set +
  email E (verified-equivalent) + household + membership via Prisma; POST a Google token (email E verified, fresh
  sub G) → **200**, `user.id` === existing id; users-with-E count **1**; that user's `googleSub` === G AND
  `appleSub` unchanged; decoded sub === existing id. Proves cross-provider linking, no duplicate account.
- **(forced edit) unknown provider → 400** → the existing test, with provider value changed `"google"` →
  `"facebook"`: still **400**, `error.code === "VALIDATION_FAILED"` (allowlist rejects non-`[apple,google]`).

## Interfaces/contracts (executor must match)

### Google verifier — `google-token-verifier.ts`
```
export const GOOGLE_ISSUERS = ["https://accounts.google.com", "accounts.google.com"];
export const GOOGLE_JWKS_URL = "https://www.googleapis.com/oauth2/v3/certs";
export const GOOGLE_JWKS_RESOLVER = Symbol("GOOGLE_JWKS_RESOLVER"); // -> jose key-resolver fn (JWTVerifyGetKey)
```
- Constructor: `@Inject(GOOGLE_JWKS_RESOLVER) private readonly keyResolver: JWTVerifyGetKey`,
  `private readonly config: AppConfigService`. `readonly provider = "google" as const`.
- `verify(token)`:
  `await jwtVerify(token, this.keyResolver, { issuer: [...GOOGLE_ISSUERS], audience: this.config.googleClientId, algorithms: ["RS256"] })`
  then: `email = typeof payload.email === "string" ? payload.email : null`;
  `emailVerified = payload.email_verified === true` (**strict boolean**, no string coercion — R4);
  guard `if (typeof payload.sub !== "string" || payload.sub.length === 0) throw new UnauthorizedException()`.
  Catch ALL errors → `throw new UnauthorizedException()`. Never log `token`.
  Note: spread `[...GOOGLE_ISSUERS]` so the value is `string[]` (jose `issuer` accepts `string | string[]`;
  a `readonly` array from `as const` would not be assignable — declare `GOOGLE_ISSUERS` as a plain `string[]`
  to keep it simple, or spread at the call site).

### DTO widening — `social-auth.dto.ts`
```
const SUPPORTED_PROVIDERS: SocialProvider[] = ["apple", "google"];
```
(`@IsIn(SUPPORTED_PROVIDERS)` + `@ApiProperty({ enum: SUPPORTED_PROVIDERS })` follow automatically; the global

## Risks & the design decisions the planner made (scrutinize)
- **R1 — DTO widening forces a one-line edit to the T013 e2e (decision, unavoidable):** the existing
  `unknown provider → 400` test posts `{ provider: "google", identityToken: "x" }`. Once `"google"` is a valid
  provider, that body passes validation, reaches the Google verifier, and `jwtVerify("x", …)` fails → **401**,
  not 400 — the test WOULD regress no matter where Google tests live. Minimal fix: change the provider value to
  `"facebook"` (still non-allowlisted → still 400 `VALIDATION_FAILED`), preserving the test's intent. This is the
  ONLY behavioral edit to a T013 spec; all Apple assertions are untouched. ("specs unedited" is honored for the
  T012 specs and the T013 *unit* specs, which stay byte-identical.)
- **R2 — e2e layout = extend the existing file, not a new one (decision):** since R1 already forces an edit to
  `auth-social.e2e-spec.ts`, a separate Google file would still leave that edit AND duplicate the whole harness
  (keypair, JWKS, prisma cleanup, app setup). Extending shares one app (both resolvers overridden), one
  `createdUserIds`/`afterEach`, and one prisma client — "cleanup shared", same endpoint, cohesive. Must add the
  `GOOGLE_JWKS_RESOLVER` override so no real network fetch fires.
- **R3 — accept BOTH Google issuers (decision):** `issuer: [...GOOGLE_ISSUERS]` accepts a token whose `iss` is
  either `https://accounts.google.com` or `accounts.google.com`, because Google legitimately emits both forms;
  rejecting one would 401 valid users. Every other issuer still → 401. Unit-tested for both accepted forms and a
  rejected third. Spread to `string[]` to satisfy jose's `issuer: string | string[]` typing.
- **R4 — strict boolean `email_verified` (decision, differs from Apple):** Google's OIDC id_token emits
  `email_verified` as a real JSON boolean, so `emailVerified = payload.email_verified === true` — a string
  `"true"` (or any non-boolean) maps to `false`. Unit-locked. This is deliberately stricter than Apple's
  `=== true || === "true"`. It fails UPWARD: a mis-typed/absent verification claim yields 401 (no sign-in),
  never a silent link to a victim's account — consistent with the fail-closed link gate in
  `social-auth.service.ts` (`if (identity.email && identity.emailVerified)`).
- **R5 — zero service-layer change is correct (decision, verified by reading):** `social-auth.service.ts` and
  `auth.service.ts` already switch/handle `google → googleSub` in every relevant method
  (`buildSubWhere`/`buildSubData`/`buildNewAccountParams`, `NewAccountParams.googleSub`, `provisionNewAccount`).
  Adding the Google verifier + widening the DTO/registry is the whole task. If the executor finds a service-layer
  change necessary, that signals a misread — STOP and flag.
- **R6 — no migration / no new dep:** `User.googleSub String? @unique` exists (T011); `jose` is already a
  dependency (used by the Apple verifier). Neither `schema.prisma`/migrations nor `package.json`/lockfile changes.
- **R7 — uniform 401 / never log the token:** the Google verifier mirrors Apple's bare
  `catch { throw new UnauthorizedException(); }` and empty-`sub` guard — every failure mode (bad signature,
  iss, aud, exp, alg, malformed) collapses to an indistinguishable 401 rendered by `AllExceptionsFilter` as
  `{ error: { code: "UNAUTHORIZED", … } }`; the identity token is never passed to a logger.
- **R8 — file count (~8, within the ~8-10 hint):** 2 create (verifier + its unit spec) + 6 modify (DTO, module,
  env schema, config getter, `.env.example`, the shared e2e). Each maps to an AC or a gate; nothing speculative.
- **Safety:** T014 touches no Safety-Policy surface (PRODUCT_SPEC §5) — no disclaimers, emergency escalation, or
  dosing copy. NOT a SAFETY-ESCALATION.
