# Plan — T032: Intake schema & question flows

## Objective (from card)
Add the symptom-intake data model to `packages/types`: ~12 symptom categories, each with dynamic follow-up question definitions typed as a discriminated union (`single` / `multi` / `scale` / `duration` / `photoPrompt`), exported as DATA (not just types) so downstream renderers are schema-driven; plus a fail-closed `parseIntake` that validates a completed intake (mirroring T030's `parseTriage`). Plan only — no code.

## Safety statement (§5 surface — planned, NOT blocked)
T032 does not weaken any Safety Policy surface. The intake question copy is user-facing prose (CLAUDE §7), so every `prompt` / `label` / `helpText` / scale label / option label MUST be §7-clean: no "diagnosis/diagnose", no medication names/dosing, no treatment instructions — questions only *collect symptoms*, they never advise. A mechanical scan test enforces this (see Tests). The intake feeds the deterministic red-flag layer (T031) and the AI triage layer (T033) which own the fail-upward behavior; T032 only structures the input. NOT a SAFETY-ESCALATION.

## Placement & dependency-direction decision (flagged R1)
Everything ships in **`packages/types`** (the card says so). It imports `Species`/`Sex` from `./pet` and, if used, `petIdSchema` from `./branded-ids` — no new deps, zod only. It CANNOT import `packages/ai` (dependency direction is `ai → types`), so the intake→`RedFlagIntake`/`RedFlagSign` **mapper does NOT ship in T032** — see R2 and the mapping table. T032 only *designs option values so the mapper is trivial later.*

## The RedFlagSign mapper does NOT ship here (flagged R2)
`packages/ai/src/rules/types.ts` owns `RED_FLAG_SIGNS` (snake_case vocabulary) and `RedFlagIntake`. A mapper from a completed intake to `RedFlagIntake.signs` would have to import that vocabulary from `packages/ai`, which `packages/types` may not do. Re-declaring the vocabulary in `types` would be a forbidden duplicate interface (CLAUDE §6 "never hand-write a duplicate interface"). Therefore the mapper belongs to the **api layer (T042)**, which already depends on both `types` and `ai`. T032's obligation is only to shape the red-flag-adjacent categories (`urinary`, `breathing`, `injury`) with **stable option `value` strings that map 1:1** to `RedFlagSign`. The intended mapping is documented below as a contract for T042 (data, not code).

## Conventions to follow (ratified in T030/T031)
- `z.strictObject` at EVERY object level (extra keys rejected — matches "unknown … rejected" AC spirit).
- `z.discriminatedUnion("type", […])` for both question defs and answers.
- `parseIntake` mirrors `parseTriage` exactly: accepts `unknown`; if `string`, `JSON.parse` in try/catch → `INVALID_JSON: <msg>`; else `safeParse`; returns a discriminated `{ ok:true; value } | { ok:false; reason; issues? }`; **NEVER throws, NEVER mutates**.
- `noUncheckedIndexedAccess`: record/array lookups return `T | undefined` — handle defensively in the refine and in `getCategoryDef`.
- String data ids are **kebab-case** (category ids, question ids, option values); this is a *different namespace* from the snake_case `RedFlagSign` vocabulary (bridged by the T042 mapper).
- Question copy is externalized DATA here (like the breeds dataset); i18n is T110. Note this in a header comment so the checker does not flag the §6 "no hardcoded user-facing strings in components" rule — this is a data package, not a component.

## Interfaces/contracts (executor must match)
```ts
import { z } from "zod";
import type { } from "./pet"; // Species/Sex not needed in the schema itself — see note below

// ---- enums ----
export const SYMPTOM_CATEGORIES = [
  "vomiting","diarrhea","not-eating","limping","skin-itch","eyes",
  "ears","urinary","breathing","behavior","injury","other",
] as const;                                  // exactly 12 (card: "~12")
export const symptomCategorySchema = z.enum(SYMPTOM_CATEGORIES);
export type SymptomCategory = z.infer<typeof symptomCategorySchema>;

export const DURATION_UNITS = ["minutes","hours","days","weeks"] as const;
export const durationUnitSchema = z.enum(DURATION_UNITS);
export type DurationUnit = z.infer<typeof durationUnitSchema>;

export const QUESTION_TYPES = ["single","multi","scale","duration","photoPrompt"] as const;

// ---- question-def discriminated union (the schema DATA renderers consume) ----
export const optionSchema = z.strictObject({ value: z.string().min(1), label: z.string().min(1) });
// base fields present on every question: id, prompt, required, optional helpText
// single:      + options: optionSchema[] (min 2)
// multi:       + options: optionSchema[] (min 2) + maxSelections?: int>0
// scale:       + min:int, max:int, minLabel:string, maxLabel:string   (refine: max > min)
// duration:    + units: durationUnitSchema[] (min 1)
// photoPrompt: + maxPhotos: int 1..3
export const questionDefSchema = z.discriminatedUnion("type", [ /* 5 members */ ]);
export type QuestionDef = z.infer<typeof questionDefSchema>;

export const categoryDefSchema = z.strictObject({
  id: symptomCategorySchema,
  label: z.string().min(1),
  questions: z.array(questionDefSchema).min(1),
});
export type CategoryDef = z.infer<typeof categoryDefSchema>;

export const INTAKE_CATEGORIES: readonly CategoryDef[]; // the 12-entry dataset below
export function getCategoryDef(id: string): CategoryDef | undefined;

// ---- completed intake + answers ----
// single:      { questionId, type:"single", value:string }
// multi:       { questionId, type:"multi", values:string[] (min 1) }
// scale:       { questionId, type:"scale", value:int }
// duration:    { questionId, type:"duration", value:number>0, unit:DurationUnit }
// photoPrompt: { questionId, type:"photoPrompt", photoKeys:string[] (max 3) }
export const answerSchema = z.discriminatedUnion("type", [ /* 5 members */ ]);
export const completedIntakeSchema; // strictObject{ category, answers:answer[], freeText?:string(max 2000) } + superRefine
export type CompletedIntake = z.infer<typeof completedIntakeSchema>;

export type ParseIntakeResult =
  | { ok: true; value: CompletedIntake }
  | { ok: false; reason: string; issues?: z.core.$ZodIssue[] };
export function parseIntake(raw: unknown): ParseIntakeResult; // never throws; JSON.parses strings first
```
Note: the schema itself needs no `Species`/`Sex` (species context is applied by T042). Import `Species`/`Sex` ONLY if a helper references them; otherwise no cross-file import is required beyond zod. Do NOT add `petId` to the completed intake — the pet is a path param at T042 (decision R3).

## Validation semantics (the `.superRefine` on `completedIntakeSchema`)
Structural shape (each answer is a well-formed union member; category is a known enum) is checked by the base schema. The refine adds the cross-field, category-aware checks (it references the static `INTAKE_CATEGORIES` data):
1. Look up `getCategoryDef(data.category)`; if `undefined` (defensive — enum already guards) → issue and stop.
2. Build `questionId → QuestionDef` map for that category.
3. For each `answers[i]`:
   - unknown `questionId` (not in the category) → issue at `["answers", i, "questionId"]`. **(unknown question rejected)**
   - `answer.type !== question.type` → issue at `["answers", i, "type"]`. **(type-mismatch rejected)**
   - value checks by type:
     - `single`: `value` must be one of the question's option values → else issue at `[…, "value"]`.
     - `multi`: every element in the question's option values, elements unique, and `values.length ≤ maxSelections` when set → else issue.
     - `scale`: `question.min ≤ value ≤ question.max` → else issue at `[…, "value"]`. **(scale out-of-range rejected)**
     - `duration`: `unit` must be in the question's `units` list → else issue at `[…, "unit"]`.
     - `photoPrompt`: `photoKeys.length ≤ question.maxPhotos` → else issue.
   - duplicate `questionId` across answers → issue (each question answered at most once).
4. Every `required` question in the category must have a matching answer → else issue at `["answers"]`. **(completed fixture must answer all required questions)**
All issues REJECT — the refine never mutates. `parseIntake` returns them flattened in `reason` (`path.join('.')+': '+message` joined by `; `) plus raw `issues`, exactly like `parseTriage`.

## Tests to write (map to acceptance criteria) — `intake.spec.ts`
Build a `validIntakeFor(categoryId)` factory that, given a `CategoryDef`, emits a schema-valid `CompletedIntake` answering every question correctly (single→first option value, multi→[first option value], scale→min, duration→{value:2, unit:first allowed unit}, photoPrompt→{photoKeys:[]}). Used as the base for both the positive fixtures and the mutate-to-negative tests. Deterministic table loops (no `fast-check` — R5, no new dep).

- **AC "Schema tests per category" → `every category definition is well-formed`** (`it.each(INTAKE_CATEGORIES)`):
  - `categoryDefSchema.safeParse(def).success === true` for each of the 12.
  - question `id`s unique within the category; every option `value` unique within its question; scale `max > min`; photoPrompt `maxPhotos` in 1..3; duration `units` non-empty.
  - Also (module-level): `INTAKE_CATEGORIES.map(c=>c.id)` equals `SYMPTOM_CATEGORIES` as a set (no missing/extra category — supports T044's "no hardcoded list"), length 12.
  - `getCategoryDef` returns the def for each known id and `undefined` for `"nope"`.
- **AC "completed-intake fixture per category validates" → `a completed intake validates for every category`** (`it.each(SYMPTOM_CATEGORIES)`):
  - `parseIntake(validIntakeFor(id)).ok === true`.
  - and `completedIntakeSchema.safeParse(validIntakeFor(id)).success === true`.
  - one variant per category adds a `freeText` string → still ok.
- **AC "unknown category rejected" → `unknown category is rejected`**:
  - `parseIntake({ category:"not-a-category", answers:[] })` → `{ok:false}` (enum), issue path includes `category`.
  - typo/case variants (`"Vomiting"`, `"URINARY"`, `""`) → each `{ok:false}`.
- **unknown question id → `answers referencing an unknown question are rejected`**: take a valid fixture, change one answer's `questionId` to `"ghost"` → `{ok:false}`, path `answers.0.questionId`.
- **type mismatch → `type-mismatched answers are rejected`**: replace a `single` answer with a `scale`-typed answer for the same `questionId` → `{ok:false}`, path `answers.*.type`.
- **scale out of range → `scale answers outside min/max are rejected`**: set a scale answer `value` to `0`, `6`, `1.5` → each `{ok:false}`.
- **single not in options → `single answers must use a defined option`**: set `value:"made-up"` → `{ok:false}`.
- **multi checks → `multi answers respect options and maxSelections`**: value not in options → reject; duplicate values → reject; (if a question sets `maxSelections`, exceeding it → reject — else document N/A).
- **duration unit → `duration answers must use an offered unit`**: set `unit:"weeks"` on a question offering only `["hours","days"]` → reject; non-positive `value` (0/-1) → reject.
- **photo cap → `photoPrompt answers respect maxPhotos`**: `photoKeys` length `maxPhotos+1` → reject.
- **required coverage → `missing a required question is rejected`**: drop one required answer from a valid fixture → `{ok:false}` path `answers`; dropping only an optional question → still ok.
- **strict extra key → `extra keys are rejected`**: extra top-level key on the intake, and extra key inside an answer object → each `{ok:false}` (strictObject).
- **JSON string handling → `parseIntake accepts a JSON string and reports INVALID_JSON`**: serialized valid fixture → ok; truncated `'{"category":'` → `{ok:false, reason startsWith "INVALID_JSON"}`.
- **never throws → `parseIntake never throws on hostile input`**: `null`, `42`, `[]`, BigInt, Symbol-keyed object, 200-deep nested → each wrapped in `expect(...).not.toThrow()` and returns `{ok:false}`.
- **§7 safety copy scan → `no intake copy contains diagnosis or dosing language`**: concatenate every `label`, `prompt`, `helpText`, option `label`, and scale `minLabel`/`maxLabel` across `INTAKE_CATEGORIES`; assert NONE matches `/diagnos/i` and NONE matches the dosing/drug pattern `/(\bmg\b|\bml\b|\bdose|dosage|milligram|per kg|administer|tablet|ibuprofen|paracetamol|acetaminophen|aspirin|benadryl|diphenhydramine|metacam|tramadol)/i`. (The pattern literal lives only in the test.)
- **mapping-contract anchors → `red-flag-adjacent option values are stable`**: assert the specific option values the T042 mapper relies on exist — `urinary/difficulty` has `straining` and `cannot-urinate`; `urinary/blood-in-urine` has `yes`; `breathing/character` has `labored`,`open-mouth-cat`,`gasping`; `breathing/gum-color` has `pale-white`,`blue-purple`; `injury/what` has `hit-by-vehicle`; `injury/bleeding` has `heavy`; `injury/consciousness` has `unresponsive`. ## Risks & design decisions the planner made (for CHECKER scrutiny)
- **R1 — All in `packages/types`, zod-only, no new import beyond `./pet` if needed.** Card mandates it; renderers/api/ai all consume from `types`. The intake schema itself needs neither `Species` nor `Sex` (species context applied at T042), so the executor should import from `./pet` ONLY if a helper actually references those types — otherwise no cross-file import. Flag if the checker expects species inside the intake: it is deliberately excluded (R3).
- **R2 — The RedFlagSign mapper ships in T042 (api), NOT here.** `packages/types` cannot import `packages/ai`'s `RED_FLAG_SIGNS`; duplicating the vocabulary is a forbidden duplicate interface. T032 instead pins stable option `value`s (mapping table above) and a test that guards them. This is the single most consequential boundary decision — scrutinize that the pinned values genuinely cover the red-flag-adjacent signs T031 expects.
- **R3 — Completed intake carries `category` + `answers` + optional `freeText` only; no `petId`, species, or photos-at-root.** The pet is a path param at T042; photos are per-`photoPrompt`-answer `photoKeys`. Keeps the value object self-contained and fixtures UUID-free. Trade-off: T042 must join intake with pet context — expected.
- **R4 — Required-question enforcement is IN the refine.** "A completed-intake fixture … validates" implies a *complete* intake answers its required questions; so the refine rejects a missing required answer. Optional questions (`contents`, most `photo`/`signs` extras) may be omitted. If the checker deems required-enforcement too strict, note it is scoped to `required:true` questions only and every valid fixture answers them.
- **R5 — Table-driven "property" tests via hand-rolled loops, not `fast-check`.** CLAUDE §2 rule 7 (no new dep). The category/answer matrices are bounded and give deterministic fail-closed coverage.
- **R6 — Discriminated-union answers mirror the question-def union on `type`.** This lets Zod validate answer *shape* structurally and lets the refine do a cheap `answer.type === question.type` gate before value checks. Alternative (untyped answers) would push all validation into the refine and lose static types for consumers — rejected.
- **R7 — kebab-case for all data ids (categories, questions, options), distinct from snake_case `RedFlagSign`.** One convention inside T032; the T042 mapper bridges the two namespaces. If the checker prefers snake_case category ids to match `RED_FLAG_SIGNS`, note the namespaces are intentionally separate and bridged by data, so either is safe — kebab chosen to match T031's kebab payload-key convention.
- **R8 — Exactly 12 categories (card says "~12").** A module-level test asserts `INTAKE_CATEGORIES` covers `SYMPTOM_CATEGORIES` exactly, so downstream (T044) can render "no hardcoded list."
