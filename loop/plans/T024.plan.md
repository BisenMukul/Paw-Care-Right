# Plan — T024: Mobile add-pet wizard (L)

## Objective (from card)
Build a multi-step add-pet wizard in `apps/mobile` (species → breed autocomplete → details → photo via `expo-image-picker` with client-side compress → done) whose draft is a **resumable zustand store persisted to MMKV**, that is **completable with only species + name** (every other field/step skippable), performs the create + photo-upload orchestration, and lands on a minimal pet-home screen. Component tests cover per-step validation and the skip path; a `<90s` manual completion check is journaled (deferred to the M1 device milestone).

## Headless-verification reality (read first — mirrors T018/T023)
Headless Linux container: **no device/emulator/Expo Go, no live API**. All tests mock the native modules (`expo-image-picker`, `expo-image-manipulator`, `expo-image`, `react-native-mmkv`, `expo-router`) and the app-level api hook modules / `global.fetch`. Two AC clauses cannot be device-run here and are handled explicitly:
1. **"journal manual check <90s completion"** — CANNOT be timed on-device here. The plan provides (a) a verbatim **manual device script** (below) journaled now and run at **M1**, and (b) the automated stand-in: the `add-pet-done` component test drives the full minimal (species+name) path through the store to the mocked create call and the pet-home redirect, proving the wizard is completable end-to-end.
2. **`expo-image-picker` native picker / OS permission** only work in a dev build. Headless, the module is jest-mocked; tests assert the picker is invoked and its result flows through `compressImage` into the draft.

**RNTL v14 `render` is async — every test must `await render(...)`** (T008 Amendment A1).

## Interfaces/contracts (executor must match)
**Draft + store (`src/pets/add-pet-store.ts`)**
```ts
import type { Pet, Sex, Species } from "@pawcareright/types";
interface AddPetDraft {
  species: Species | null; name: string;
  breedSlug: string | null; breedName: string | null;
  sex: Sex | null; neutered: boolean | null;
  birthDate: string | null;          // "YYYY-MM-DD" (UI); serialized to ISO for the payload
  ageEstimateMonths: number | null; weightGrams: number | null;
  photoUri: string | null;           // compressed local file:// uri
  createdPetId: string | null;       // set once POST /v1/pets succeeds → idempotent resume
}
interface AddPetState {
  draft: AddPetDraft; stepIndex: number;
  setField<K extends keyof AddPetDraft>(key: K, value: AddPetDraft[K]): void;
  setStep(index: number): void;
  setCreatedPetId(id: string): void;
  reset(): void;                     // restores INITIAL_DRAFT + stepIndex 0
}
export function buildCreatePetPayload(draft: AddPetDraft): CreatePetPayload; // omits null/empty; birthDate→ISO; species+name always present
```
- Persist: `persist(creator, { name: "pawcareright.add-pet-draft", storage: createJSONStorage(() => mmkvStorage), partialize: (s) => ({ draft: s.draft, stepIndex: s.stepIndex }) })`.
- MMKV `StateStorage` adapter (mirror `src/api/query.ts` usage — `createMMKV()` → `getString/set/remove`):
```ts
const mmkv = createMMKV();
const mmkvStorage: StateStorage = {
  getItem: (n) => mmkv.getString(n) ?? null,
  setItem: (n, v) => { mmkv.set(n, v); },
  removeItem: (n) => { mmkv.remove(n); },
};
```
- `buildCreatePetPayload` builds `{ species, name, ...optional }` then validates with `createPetSchema.safeParse` (from `@pawcareright/types`); on failure the caller surfaces the details-step error (should not happen if the step validated).

**Pet create payload (matches T021 `CreatePetDto`)**: `{ species: "DOG"|"CAT"; name: string; sex?: "MALE"|"FEMALE"|"UNKNOWN"; neutered?: boolean; breedSlug?: string; birthDate?: string /*ISO*/; ageEstimateMonths?: number; weightGrams?: number }` — `birthDate` XOR `ageEstimateMonths` (server-authoritative; client mirrors).

**Photo upload orchestration (`uploadPetPhoto` in `pets-api.ts`; matches T023)**:
```
const blob = await (await fetch(localUri)).blob();          // file:// → Blob, .size = contentLength
const { uploadUrl, key } = await apiClient.post<{uploadUrl:string;key:string}>(
  `/v1/pets/${petId}/photo-upload-url`, { contentType: "image/jpeg", contentLength: blob.size });
const put = await fetch(uploadUrl, { method:"PUT", body: blob, headers:{ "Content-Type":"image/jpeg" } });
if (!put.ok) throw new Error("upload failed");              // plain fetch — presigned S3 URL, NO auth header
await apiClient.post(`/v1/pets/${petId}/photo-upload-confirm`, { key });
```
(Compression already applied at the photo step → the blob is JPEG ≤1600px, well under the 5MB/type limits.)

**Done-screen orchestration order (idempotent)**:
1. `petId = draft.createdPetId ?? (await createPet.mutateAsync(buildCreatePetPayload(draft))).id`; if just created → `setCreatedPetId(petId)`.
2. if `draft.photoUri` → `try { await uploadPetPhoto(petId, draft.photoUri) } catch { /* non-fatal: pet exists, R3 */ }`.
3. `queryClient.invalidateQueries({ queryKey: ["pets"] })`.
4. capture `localPhoto = draft.photoUri`; `reset()`.
5. `router.replace({ pathname: "/pets/[id]", params: { id: petId, localPhoto: localPhoto ?? "" } })`.
Any throw in step 1 → set error state, do NOT reset; Retry re-invokes (createdPetId still null → single create).

**Breed row**: `{ slug: string; name: string }` (subset of `@pawcareright/data`'s `Breed`; use `Breed[]` as the query result type). Endpoint: `GET /v1/breeds?species=DOG&q=lab` (public).

**`compressImage`**: uses `expo-image-manipulator` to resize longest edge → 1600 (no enlargement), format JPEG, compress `0.7`; returns `{ uri, width, height }`.

## Risks & the design decisions the planner made (checker: scrutinize each)
- **R1 — Native module versions via `expo install` (STOP condition).** `expo-image-picker`/`expo-image-manipulator`/`expo-image` MUST be SDK-57-pinned by `expo install`. If a compatible version can't resolve → BLOCK, do not free-float.
- **R2 — Wizard routing + draft store location.** Chosen: a root-level `app/add-pet/` modal Stack group with one screen per step (species→breed→details→photo→done) + a minimal `app/pets/[id].tsx`, both registered on the existing root `<Stack>` (minimal edit; `(tabs)` untouched, T008/T018 tests stay green). The draft is a **separate** zustand store `src/pets/add-pet-store.ts` (NOT the auth store), persisted to a **second MMKV instance**. `src/api/query.ts`'s "only the root layout imports mmkv" comment is a *web-bundle* isolation note; on mobile a second instance is fine and the store is never imported by web. Alternative (nesting under `(app)/(tabs)`) rejected as churn. Back on step 1 closes the wizard; Back elsewhere = previous step.
- **R3 — Photo upload is non-fatal; create is the atomic success point.** Decision: `POST /v1/pets` success (tracked by persisted `createdPetId`) is what "completes" the wizard; a subsequent photo-upload failure does NOT block landing on pet home (the pet exists; photo can be retried later). This keeps create idempotent under Retry (no duplicate pets). Trade-off: a failed upload silently yields a photo-less pet for now — flagged. The create failure path IS surfaced + retryable (AC).
- **R4 — Pet home is deliberately minimal.** Name + species + breed + local-photo-or-placeholder only; no remote photo (no display-URL endpoint yet), no timeline/care/edit. Full pet profile is a later task. `expo-image` is used for the local preview/avatar (satisfies the §6 image-caching intent) and future remote use.
- **R5 — `birthDate` via plain `YYYY-MM-DD` text, no native date picker.** Avoids a new dependency while still allowing BOTH `birthDate` and `ageEstimateMonths` to be entered so the XOR rule is reachable and testable (AC). UX is minimal by design; a real date picker is a later polish task. Flagged.
- **R6 — `expo-image-manipulator` API-shape localized in `compress-image.ts` (STOP condition).** SDK 57 may expose the new context API (`ImageManipulator.manipulate(...).renderAsync()/saveAsync()`) rather than legacy `manipulateAsync`. The executor uses whatever the pinned version exports, confined to this one file; the unit test mocks the module to that same shape. If neither resize API is available in the pinned version → BLOCK.
- **R7 — `contentLength` via RN `fetch(file://).blob().size`, no `expo-file-system`.** The presign DTO requires an integer `contentLength`; deriving it from the compressed blob avoids a new dep and also gives the PUT body. If the runtime cannot produce a Blob from a `file://` uri → fallback would be `expo-file-system` `getInfoAsync().size` (a new dep) → that is a BLOCK/flag, not a silent add. Headless tests mock `uploadPetPhoto`, so this is a device-time risk only.
- **R8 — `@pawcareright/types` consumed for Pet/Species/Sex + `createPetSchema` (not a duplicate interface).** Unlike T018's auth (no shared schema existed), `packages/types` DOES export `petSchema`/`createPetSchema`/`Species`/`Sex`; importing them satisfies §6 "never hand-write a duplicate interface" and reuses the server XOR rule client-side. Requires building `@pawcareright/types` dist before mobile typecheck/test (Step 2), mirroring the api-client precedent. If jest/Metro can't resolve the workspace ESM package → BLOCK rather than duplicating types.
