## Interfaces/contracts (executor must match these shapes)
Intended `.github/workflows/ci.yml` shape (exact keys; executor fills indentation/quoting to valid YAML):
```yaml
name: CI
on:
  push:
    branches: [main]        # default branch; adjust only if `git symbolic-ref` shows a different default
  pull_request:
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
jobs:
  build:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: pawcareright
          POSTGRES_PASSWORD: pawcareright
          POSTGRES_DB: pawcareright
        ports: ["5432:5432"]
        options: >-
          --health-cmd "pg_isready -U pawcareright -d pawcareright"
          --health-interval 5s --health-timeout 5s --health-retries 10
      redis:
        image: redis:7
        ports: ["6379:6379"]
        options: >-
          --health-cmd "redis-cli ping" --health-interval 5s
          --health-timeout 3s --health-retries 10
    env:
      DATABASE_URL: postgresql://pawcareright:pawcareright@localhost:5432/pawcareright?schema=public
      REDIS_URL: redis://localhost:6379
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with: { version: 10.34.3 }
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: pnpm }
      - run: pnpm i --frozen-lockfile      # triggers api postinstall `prisma generate` (no DB needed to generate)
      - run: pnpm typecheck
      - run: pnpm lint
      - run: pnpm test
      - run: pnpm build
  ai-evals:                                 # STUB — activates at T040 (packages/ai). turbo no-op today (0 tasks → exit 0).
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4

      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: pnpm }

```

## Risks & the design decisions the planner made (checker: scrutinize each)
- **R1 — Workflow file + triggers.** Single new file `.github/workflows/ci.yml`; triggers `push` (default branch, assumed `main` — executor confirms via `git symbolic-ref --short HEAD` / repo default) + `pull_request`. `concurrency` with `cancel-in-progress: true` keyed on `github.ref` to avoid stacked runs. Decision: `main` as the push branch — flag if the repo default differs.
- **R2 — pnpm + Node + cache.** `pnpm/action-setup@v4` pinned to `10.34.3` (matches `packageManager`), `actions/setup-node@v4` with `cache: 'pnpm'` (the card's "pnpm cache") — setup-node's built-in pnpm cache keyed on `pnpm-lock.yaml`; `action-setup` runs BEFORE `setup-node` so the `pnpm` binary exists for the cache step. Actions pinned to `@v4` majors.
- **R3 — ONE `build` job with sequential steps, not parallel jobs.** The card lists an ordered pipeline "typecheck → lint → test → build"; a single job keeps the postgres+redis `services` alive for the `test` step and runs the four as separate `pnpm <task>` invocations in order. Decision: separate sequential steps (each its own turbo run), NOT one combined `turbo run typecheck lint test build` — this matches the M0 gate style and avoids intra-invocation turbo concurrency races (R5). Trade-off: no per-step parallelism, but correctness + services-availability win for a scaffold CI.
- **R4 — Services must match the api test's expected URLs exactly.** The `apps/api` health e2e (`test/app.e2e-spec.ts`) asserts `{ db: "ok", redis: "ok" }`, so it needs live Postgres+Redis at `localhost:5432`/`localhost:6379` with `pawcareright` creds — mirrored from `env.schema.ts` defaults and `docker-compose.yml`. The workflow also sets `DATABASE_URL`/`REDIS_URL` env explicitly (belt-and-suspenders) so it does not silently rely on the Zod defaults. No migrations exist until T011, so the health check runs against an empty schema (it only pings connectivity, does not query app tables) — confirmed by reading the e2e. `prisma generate` runs via the api `postinstall` during `pnpm i` and needs no DB connection.
- **R5 — typecheck→test ordering & `.next/types` race.** Running the four as separate sequential `pnpm <task>` invocations means each is an independent `turbo run`; `pnpm typecheck` already triggers web's `build` via `apps/web/turbo.json` (`typecheck dependsOn ^build, build`), so `.next/types` exists before later steps. Mobile typecheck uses a glob-only include (no generated-file dependency, per T008). Decision: rely on the existing per-app turbo.json ordering + sequential steps rather than forcing an explicit pre-`build`. Flag: if CI `test` ever fails on a missing generated type, the fix is to reorder (`build` before `test`) — but the M0 sequential gate already passes locally in this order, so this is not pre-emptively added (avoid over-engineering, §2).
- **R6 — `ai-evals` is a SEPARATE stub job.** `pnpm test:ai-evals` → `turbo run test:ai-evals`; no workspace defines that script yet (packages/ai doesn't exist), so turbo matches 0 tasks and exits 0. The job is separate (not a step in `build`) so it is independently green/visible and trivially "activated" at T040 by real eval scripts. No services attached. Risk: a turbo version quirk making an empty task set non-zero — mitigated by the explicit local check `pnpm test:ai-evals; echo $?` (must be 0); if a future turbo prints non-zero on empty sets the executor should flag, not paper over with `|| true` (that would hide real T040 failures) — instead BLOCK and note.
- **R7 — `--frozen-lockfile`.** CI installs use `pnpm i --frozen-lockfile` so a drifted lockfile fails the run. The committed `pnpm-lock.yaml` is in sync after T008; this task adds no deps, so the lockfile must NOT change (verified via `git diff --exit-code pnpm-lock.yaml`).
- **R8 — README badge owner/repo.** Prefer the real remote (`git remote get-url origin`); documented fallback `BisenMukul/Paw-Care-Right`. The badge is cosmetic and does not gate CI; a wrong slug only renders a broken image, not a build failure. Decision: derive from remote, fall back to the known slug.
- **R9 — Local validation without `act`.** AC "workflow valid (`act`-style dry OR yaml lint)" is satisfied by PyYAML `safe_load` (+ `actionlint` if present) plus the proof that every `run:` maps to a real, locally-green root script. `act` (Docker-in-Docker) is explicitly NOT required in this headless container. Stated so the checker does not demand a full `act` run.
- **Safety:** T009 is pure CI/build infrastructure + a README badge. It touches NO AI prompt, disclaimer, emergency-escalation, or dosing surface (`PRODUCT_SPEC §5`). No safety policy surface is weakened. No SAFETY-ESCALATION.
