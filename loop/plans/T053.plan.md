# Plan — T053: Reminder models + migration

## Objective (from card)
Add the `Reminder` + `ReminderEvent` Prisma models per ARCHITECTURE §3 (with the mandated `ReminderEvent(dueAt, status)` index), a shared **RRULE-subset validator** so an invalid `rrule` is rejected at the DTO layer, and a pure **`computeNextFireAt`** helper (unit-tested) that maintains `nextFireAt`. Ship the migration + the create DTO + custom `@IsRRule` validator **without any controller/service/module** (CRUD is T055). "Do" only: models, migration, rrule validation, nextFireAt helper. "Accept": migration clean; invalid rrule rejected at DTO layer.

## Safety statement (read first)
- This task adds a **data model + input validation only**. It is NOT a PRODUCT_SPEC §5 medical-content surface: no AI, no triage, no user-facing guidance copy, no emergency/hotline logic, no disclaimers touched. It cannot weaken any §5 safety surface, so it is planned (not blocked).
- Reminders are **care-critical for timing** (a missed/mistimed reminder is a reliability defect, tested end-to-end in T056/T062), but T053 ships no scheduler and no delivery — only the schema + validators the engine will consume.
- `medNameAsEntered` is added as a **nullable column only**. It exists so the T061 medication tracker can record a name/dose *exactly as the user or their vet entered it*. Per CLAUDE §7 / §5, the med tracker **never suggests** drug names or dosages. T053 authors **no** medication copy, no dosing strings, and no default value for this column. Any such content lands (and is red-team/lint-checked) in T054/T061, not here.
- Validator error messages authored here are developer/API-facing (mirrors existing DTOs) and contain **no** medical, dosing, or "diagnos*" language.

## Key decisions (each is a scrutiny point — see Risks)
1. **Zero new dependency.** The RRULE handling is a small **hand-rolled RFC5545 subset** validator/parser in `packages/types` — NOT the `rrule` npm package. Rationale: CLAUDE §2 rule 7 requires an added dep to have a release < 6 months old, which cannot be verified offline for `rrule`; the product needs only simple recurrences (yearly vaccines, monthly/every-N parasite & flea/tick, daily/N-per-day meds — see T054/T060/T061), so a full RFC5545 engine is over-engineering (golden rule 2). The full range-**expansion** engine (T055 agenda, T056 worker) may revisit a library; T053 deliberately keeps its surface to *validation* + *single next occurrence*.
2. **Shared validator in `packages/types`, tz math in `apps/api`.** The pure string validator/parser lives in `packages/types` (mobile T060's schedule builder, the api DTO, and the future engine must all agree on "valid rrule"; the parser uses **no `Intl`**, so it is Hermes-safe for mobile). The `computeNextFireAt` helper (which needs IANA-timezone/DST math via Node's full-ICU `Intl`) lives api-side, mirroring the existing "server logic ≠ shared data shape" split documented in `packages/types/src/check-status.ts`.
3. **`Reminder.type` is a validated `String`; `ReminderEvent.status` is a Prisma enum.** ARCHITECTURE §3 writes `status: PENDING|SENT|DONE|SNOOZED|MISSED` (a closed set → Prisma `enum ReminderEventStatus`, mirroring the `CheckStatus` pattern + the tuple-in-types convention). §3 writes `type` with **no** enumerated set → store as `String` validated by a shared Zod `reminderTypeSchema`, matching the repo precedent of storing evolving vocab as validated `String` (`TriageResult.urgency`, `CheckFollowUp.response`). This lets T054/T061 extend the reminder-type vocabulary **without a DB migration**.
4. **Timezone model = UTC instants + IANA string.** `startAt`, `nextFireAt` (Reminder) and `dueAt` (ReminderEvent) are stored as UTC `DateTime`; `Reminder.timezone` stores the IANA tz id (e.g. `Europe/Paris`). Wall-clock recurrence is anchored in that tz. This is the standard reminders-correct model and what T055/T056/T062 DST tests assume.
5. **No controller/service/module.** T053 ships the `CreateReminderDto` + `@IsRRule` decorator + `computeNextFireAt` as standalone, directly-unit-tested units; the endpoint that wires them is T055. No `reminders.module.ts` is created (would be speculative/unused).

## Data model & migration spec
Add to `schema.prisma` (uuid ids, `TIMESTAMP(3)` timestamps, enum-before-model ordering — matching existing conventions):

```prisma
enum ReminderEventStatus {
  PENDING
  SENT
  DONE
  SNOOZED
  MISSED
}

model Reminder {
  id               String   @id @default(uuid())
  petId            String
  type             String   // validated by reminderTypeSchema at the DTO boundary (String mirrors TriageResult.urgency)
  title            String
  rrule            String   // RFC5545 subset, validated by @IsRRule at the DTO boundary
  timezone         String   // IANA tz id (e.g. "Europe/Paris")
  startAt          DateTime // DTSTART anchor (UTC) — recurrence origin; nextFireAt/agenda computed from this
  nextFireAt       DateTime // UTC; set on create (first occurrence >= startAt), advanced by the T056 worker
  medNameAsEntered String?  // T061 med subtype: name as entered; never suggested
  active           Boolean  @default(true) // T058/T060 pause/disable
  templateKey      String?  // T054/T055/T059 template-pack provenance + instantiation idempotency
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  pet    Pet             @relation(fields: [petId], references: [id], onDelete: Cascade)
  events ReminderEvent[]

  @@index([petId])
}

model ReminderEvent {
  id          String              @id @default(uuid())
  reminderId  String
  dueAt       DateTime            // UTC
  status      ReminderEventStatus @default(PENDING)
  sentAt      DateTime?           // set by T056/T057 on push
  completedAt DateTime?           // set by T055 complete endpoint / T061 MED_GIVEN
  createdAt   DateTime            @default(now())

  reminder Reminder @relation(fields: [reminderId], references: [id], onDelete: Cascade)

  @@unique([reminderId, dueAt])
  @@index([dueAt, status])
  @@index([reminderId])
}
```
And on `model Pet`, add the back-relation field (next to `checks SymptomCheck[]`):
```prisma
  reminders Reminder[]
```
Notes for the executor:
- `onDelete: Cascade` on both FKs mirrors `SymptomCheck.pet` (deleting a pet/reminder removes its reminders/events).
- The `@@index([dueAt, status])` is the ARCHITECTURE §3 mandated composite (the T056 minute-tick scans `ReminderEvent(dueAt<=now, PENDING)`). `@@index([reminderId])` satisfies CLAUDE §6 "every FK indexed". `@@unique([reminderId, dueAt])` gives the T056 event-generation idempotency (no two events for the same reminder+instant; distinct doses have distinct `dueAt`).
- `rrule` is required (non-null) per §3; a one-off reminder is expressed as a `COUNT=1` rule.

## RRULE-subset validation semantics (`packages/types/src/rrule.ts`)
`parseRRule` accepts an RFC5545 `RRULE`-style value (semicolon-separated `KEY=VALUE` parts; a leading `RRULE:` prefix is tolerated and stripped). Supported subset:
- `FREQ` — **required**, one of `DAILY | WEEKLY | MONTHLY | YEARLY` (`RRULE_FREQUENCIES`).
- `INTERVAL` — optional positive integer (default 1) → covers "every-N" (T060).
- `BYDAY` — optional comma list of `MO,TU,WE,TH,FR,SA,SU`.
- `BYMONTHDAY` — optional comma list of integers 1..31.
- `COUNT` — optional positive integer (one-off = `COUNT=1`).
- `UNTIL` — optional RFC5545 UTC timestamp `YYYYMMDDTHHMMSSZ`.
Rejection rules (each returns `{ ok: false, reason }`): empty/whitespace, missing `FREQ`, unknown `FREQ` value, unknown/unsupported key, duplicate key, non-positive/non-integer `INTERVAL`/`COUNT`, `BYDAY` token not in the weekday set, `BYMONTHDAY` outside 1..31 or non-integer, malformed `UNTIL`, both `COUNT` and `UNTIL` present (RFC5545 forbids). `isValidRRule` = `parseRRule(...).ok`. No `Intl`/timezone use here (Hermes-safe).

## computeNextFireAt semantics (`apps/api/src/reminders/next-fire-at.ts`)
`computeNextFireAt(rule: ParsedRRule, startAt: Date, after: Date, timeZone: string): Date | null`
- Returns the earliest occurrence instant `>= after` (UTC `Date`), anchored to the **wall-clock time-of-day of `startAt` interpreted in `timeZone`**, honoring `FREQ`/`INTERVAL`/`BYDAY`/`BYMONTHDAY`. Returns `null` when the series is exhausted (`COUNT`/`UNTIL` passed).
- DST-correct: the same local wall-clock time (e.g. 09:00 `Europe/Paris`) maps to the correct UTC instant on both sides of a DST transition. Implement the tz↔UTC conversion with a small internal `Intl.DateTimeFormat`-based offset helper (full-ICU on Node) — no new dependency.
- Pure/total: never throws; `MONTHLY` on `BYMONTHDAY=31` skips months without a 31st (do NOT roll into the next month). T053 only needs the **single next** occurrence; range-expansion is T055.
- On reminder create (T055) the caller sets `nextFireAt = computeNextFireAt(rule, startAt, startAt, timeZone)`; T056 advances it. T053 ships and unit-tests the helper; no endpoint calls it yet.

- **AC1 "Migration clean"** → `apps/api/test/reminders-schema.e2e-spec.ts` (introspective, mirrors `checks-schema.e2e-spec.ts`; `global-setup.ts` runs `prisma migrate deploy` against a fresh DB — the suite passing proves the migration applied cleanly):

  - `Reminder` has a `petId` FK index; both FKs are `ON DELETE CASCADE` (`information_schema.referential_constraints` / `table_constraints`).
  - PKs on `Reminder.id` and `ReminderEvent.id`.
- **AC2 "invalid rrule rejected at DTO layer"** → `apps/api/src/reminders/dto/create-reminder.dto.spec.ts` (`plainToInstance(CreateReminderDto, payload)` + `validate()`):
  - a payload with a malformed `rrule` (e.g. `"FREQ=NOPE"`) → `validate()` returns an error whose constraints include the `IsRRule` key; a valid payload (e.g. `"FREQ=WEEKLY;BYDAY=MO"`) → no errors.
  - also asserts an invalid `timezone` and an out-of-set `type` are rejected (guards for the whole DTO).
- **rrule matrix** → `packages/types/src/rrule.spec.ts`:
  - valid: `FREQ=DAILY`, `FREQ=WEEKLY;INTERVAL=2;BYDAY=MO,WE`, `FREQ=MONTHLY;BYMONTHDAY=31`, `FREQ=YEARLY`, `FREQ=DAILY;COUNT=1`, `FREQ=DAILY;UNTIL=20261231T090000Z`, `RRULE:FREQ=DAILY` (prefix tolerated).
  - invalid: `""`, `INTERVAL=2` (no FREQ), `FREQ=HOURLY`, `FREQ=DAILY;INTERVAL=0`, `FREQ=DAILY;INTERVAL=-1`, `FREQ=WEEKLY;BYDAY=XX`, `FREQ=MONTHLY;BYMONTHDAY=32`, `FREQ=DAILY;FOO=1`, `FREQ=DAILY;FREQ=WEEKLY` (dup), `FREQ=DAILY;COUNT=1;UNTIL=20261231T090000Z`.
- **reminder enums** → `packages/types/src/reminder.spec.ts`: `reminderEventStatusSchema` accepts each label / rejects `"NOPE"`; `reminderTypeSchema` accepts each `REMINDER_TYPES` value / rejects unknown.
- **nextFireAt** → `apps/api/src/reminders/next-fire-at.spec.ts`:
  - `DAILY` from a start → next day same wall-clock; `WEEKLY;BYDAY=MO`; `MONTHLY;BYMONTHDAY=15`; `YEARLY`; `INTERVAL=2` spacing.
  - `MONTHLY;BYMONTHDAY=31` skips a 31-less month.
  - `COUNT`/`UNTIL` exhaustion → `null`.
  - **DST-adjacent**: e.g. a `DAILY` 09:00 reminder in `America/New_York` across the March spring-forward returns instants whose local wall-clock stays 09:00 (assert the UTC offset shifts by one hour across the boundary). Keeps T055/T056/T062 honest early.

## Risks & decisions the planner made (numbered — for the checker)
1. **No `rrule` dependency; hand-rolled subset.** Chosen because CLAUDE §2 rule 7's "release < 6 months" bar can't be verified offline and the product needs only simple recurrences (over-engineering to pull a full RFC5545 engine). Risk: a hand-rolled parser could miss an edge case a mature lib handles — mitigated by the explicit valid/invalid matrix and by scoping the shared piece to *validation* + *single-next-occurrence* only. T055/T056 may still adopt a library for full range-expansion; that decision is deferred and journaled there, not forced here.
2. **`computeNextFireAt` does IANA/DST math with zero deps via `Intl`.** Node's full-ICU makes this correct; kept api-side so mobile/Hermes never imports it. Risk: DST offset math is the classic reminders trap — mitigated by the mandated DST-adjacent unit test. If the executor finds tz-correct single-occurrence math infeasible without a library, mark blocked + journal (CLAUDE §3) rather than shipping approximate math.
3. **`type` as validated `String`, `status` as Prisma enum.** Follows the §3 notation distinction (`type` open vs `status: A|B` closed) and the repo's `urgency`/`response` String precedent, so T054/T061 can grow the type vocab without a migration. Risk: `type` isn't DB-enforced — mitigated by `@IsIn(REMINDER_TYPES)` + `reminderTypeSchema` at the boundary.
4. **Forward columns (`startAt`, `active`, `templateKey`, `sentAt`, `completedAt`) + `@@unique([reminderId, dueAt])`.** Added ahead of their consumers to avoid a re-migration mid-phase, mirroring the T041 SymptomCheck precedent; each is justified against a named card (T055/T056/T057/T058/T059/T061). `startAt` is not in §3's field list but is math-essential as the DTSTART anchor. Risk: a column proves unused/mis-shaped later — accepted as cheaper than re-migrating; each is nullable/defaulted so it's inert until used.
5. **No controller/service/module shipped.** DTO + validator + helper are tested as standalone units; the AC's "DTO layer" rejection is proven via `class-validator`'s `validate()`, not an HTTP request. Risk: the DTO/validator aren't yet exercised through Nest's `ValidationPipe` — acceptable because the pipe is global (proven in existing e2e suites) and simply runs these same decorators; wiring is T055.
6. **`rrule` required (non-null); one-offs use `COUNT=1`.** Matches §3 (no `?`). Risk: a future genuinely-scheduleless reminder would need a schema change — none is in the P5 backlog, so accepted.
