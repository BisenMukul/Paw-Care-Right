# Plan — T044: Mobile: check entry + category picker (M)

## Objective (from card)
Turn the pet-home "Something wrong?" CTA into the entry point of the symptom-check flow: a screen showing a schema-driven grid of the 12 intake categories (label + icon), plus a recent-checks shortcut section, and the file-based route scaffold for the flow. This task builds ONLY the entry screen + category grid + the documented navigation contract for T045+ — no intake form, no submission, no result/emergency surfaces.

## Safety statement
Entry UI only. This screen renders no AI output, no diagnosis, no dosing, and touches NO Safety-Policy surface (no `<VetDisclaimer/>`, no emergency interstitial, no hotline copy) — those arrive at T048/T049. The category labels shown ("Vomiting", "Diarrhea", …) come from `INTAKE_CATEGORIES` in `packages/types` (already §7-reviewed schema DATA, not component strings). The mobile-added copy (grid title/subtitle, recent-checks empty state, offline banner) is neutral and non-medical: it collects intent and navigates, it never advises. The flow this screen *opens* eventually ends in §5 surfaces, but this task adds none of that medical copy. No Safety Policy surface is weakened (PRODUCT_SPEC §5 / CLAUDE.md §7). No SAFETY-ESCALATION.

## Key decisions
1. **Route:** new `app/check/index.tsx` (the grid). Registered in the root Stack. The intake screen `check/[category]` and any `check/_layout` are T045's to create — this task only documents the navigation contract to it.
2. **Entry point:** the pet-home CTA (`app/pets/[id].tsx`) is re-targeted from `/coming-soon` to `/check`, carrying `petId` (the active pet whose home this is). Minimal diff: one `onPress` target + drop the now-unused `symptom-check` coming-soon param.
3. **Icons:** emoji only — no icon library (no new deps; `@expo/vector-icons` not assumed present). A local `Partial<Record<SymptomCategory,string>>` map with a default fallback so adding a category to `packages/types` needs ZERO mobile change (preserves the T045 mutation-resistance pattern).
4. **Recent-checks shortcut:** honest empty-state placeholder wired for T050. `packages/types` exposes only `CheckStatus` (no check-summary shape) and neither `packages/api-client` nor `apps/mobile/src/api` has any checks hook/endpoint client. Wiring the T042 list endpoint is real work that belongs to T050 (Check history). So T044 ships a structured, testable empty section — not a live list.
5. **Data-driven grid:** a presentational `CategoryGrid` takes a `categories` prop defaulting to `INTAKE_CATEGORIES`, letting a test inject a synthetic schema to prove the grid is data-driven (mutation-resistant), mirroring the T045 AC pattern.

## Risks & the decisions the planner made (for the checker to scrutinize)
- **R1 (recent-checks = placeholder).** No check-summary type (`packages/types` has only `CheckStatus`) and no checks client hook exist; the T042 list endpoint is unwired on the client. Decision: ship a structured empty-state section (`check-recent-empty`) wired for T050, not a live list. Keeps scope tight and honest.
- **R2 (emoji icons + fallback).** No icon dependency is present/allowed, so icons are emoji via a `Partial<Record<...>>` map with a default fallback. This deliberately lets a new category added to `packages/types` render with zero mobile change (T045 mutation-resistance).
- **R3 (nav target not yet built).** The grid pushes to `/check/[category]`, which **T045** creates. Before T045 that route does not exist at runtime; tests mock the router so they pass. This is exactly the "route scaffolding" the card asks for, and T044→T045 are sequential — acceptable, documented.
- **R4 (screen states N/A).** The grid is local schema data (no fetch) → no loading/error/empty states; offline is a non-blocking banner. Stated explicitly so the checker does not flag "missing §6 states".
- **R5 (entry-point diff).** The pet-home CTA is re-pointed to `/check`; the single affected assertion in `pet-home-screen.test.tsx` is updated. Snapshot markup is unchanged (onPress not serialized), so `pet-home-snapshot` stays green.
- **R6 (petId passthrough).** The grid needs `petId` from the CTA. Deep-linking directly to `/check` with no `petId` passes `undefined` downstream — acceptable for this scaffold; the real entry (pet-home CTA) always supplies it. Hardening (guard/redirect) is deferred.
- **R7 (`categories` prop on `CategoryGrid`).** The grid exposes an optional `categories` prop (default `INTAKE_CATEGORIES`) purely to enable the injected-schema mutation-resistance test. Minor, justified API surface; the screen relies on the default.
