## Interfaces/contracts (executor must match)

**`defineEnv` signature (`src/env/index.ts`):**
```ts
export function defineEnv<Schema extends z.ZodType>(
  schema: Schema,
  source?: NodeJS.ProcessEnv,
): z.infer<Schema>;
```
Behavioral contract:
- `source` defaults to `process.env` (so tests pass an explicit object and never mutate real env).
- Use `schema.safeParse(source)`. On success return `result.data` (typed as `z.infer<Schema>`).
- On failure, build a report from **every** issue: one line per issue formatted `  - <path.join(".")>: <message>`, joined by `\n`, under a heading, and `throw new Error(...)`. Exact message shape:
  ```
  Invalid environment variables:
    - GEMINI_API_KEY: Required
    - AI_TEXT_MODEL: Required
  ```
- Access issues via `result.error.issues` (stable in Zod v3 and v4). If the installed Zod major exposes a different issue accessor, adjust ŌĆö but the "list all keys, formatted, not a raw ZodError dump" contract is fixed.

## Risks & the design decisions the planner made (scrutinize these)
- **A ŌĆö Test runner = Jest + ts-jest in CommonJS mode (the ESM-safe mechanic).** CLAUDE.md ┬¦3 locks Jest, so Vitest is out. `packages/config` is `type: module`, which normally forces the painful `NODE_OPTIONS=--experimental-vm-modules` + cross-platform env-var wrapper path. We avoid all of it: ts-jest transpiles the `.ts` source **to CommonJS** using `tsconfig.spec.json` (`module: CommonJS`), and Jest runs in its default CJS runtime. Result: no `--experimental-vm-modules`, no `cross-env` (which would be a stale/unmaintained dep ŌĆö CLAUDE.md ┬¦7 risk), works on Windows + Node 20 unchanged. `type: module` only affects untransformed `.js`; every spec/source file here goes through ts-jest, so it is inert.
- **B ŌĆö Jest config lives in `package.json`, not a `jest.config.js/.cjs` file.** A standalone JS config in a `type: module` package would be ESM (or need a `.cjs` extension) and would add a JavaScript file to the `eslint .` surface (risking `no-undef`/`module` lint failures) and force an eslint-ignore edit. Putting the config in `package.json` (valid JSON, ts-jest transform tuple included) keeps the change smaller and leaves `eslint.config.mjs` untouched.
- **C ŌĆö Spec files stay under `src` and are covered by `pnpm typecheck`.** The main `tsconfig.json` includes `["src"]`, so `env.spec.ts` is typechecked too; this needs `@types/jest` (globals) + `@types/node` (`__dirname`, `node:fs`/`node:path`) as devDeps, which the base tsconfig picks up automatically (no `types` restriction on the base). This is intentional ŌĆö the executor's `typecheck` validates the tests, not just the impl.
- **D ŌĆö `defineEnv` is bounded by `z.ZodType`, not `z.ZodObject<ZodRawShape>`.** Using the base `z.ZodType` bound with `z.infer<Schema>` return is stable across Zod v3/v4 and sidesteps v4's tightened `ZodObject` generics. `zod` is pinned `^4` (current stable in 2026). Risk: if the installed major changes the `.issues` accessor or `z.infer` behavior, the executor must reconcile (contract in Interfaces is fixed; only the accessor may move).
- **E ŌĆö Scope kept to helper + `.env.example`; no exported `aiEnvSchema`.** The card mandates only the generic helper and the seeded example file. A ready-made provider schema is deferred to the consumer task (T006/api). Building it now would be speculative (CLAUDE.md ┬¦2 rule 2).
- **F ŌĆö `.env.example` seeds only the six AI keys.** No other env key is used anywhere in the repo yet, so "every key used across the repo" is exactly these six today; later config-adding tasks extend this file (per the card's standing instruction).
- **Safety:** No Safety Policy surface (disclaimers, emergency escalation, dosing) is touched ŌĆö pure env/config plumbing. Not blocked.
