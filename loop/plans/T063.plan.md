# Plan ‚Äî T063: HealthLog model + migration (Phase 6 kickoff)

## Objective (from card)
Add the `HealthLog` table exactly per ARCHITECTURE ¬ß3 ‚Äî `kind` enum `WEIGHT|MEAL|NOTE|VET_VISIT|MED_GIVEN|CHECK_REF`, `valueJson`, `photoKeys[]` (reusing the T023 upload pipeline's keys), `occurredAt` ‚Äî with a clean migration and, in `packages/types`, kind-specific Zod `valueJson` validators (WEIGHT grams int>0, VET_VISIT fields, CHECK_REF backlink shape, etc.). Model + migration + validators + specs ONLY. No endpoints (T064), no UI (T065+), no aggregation (T066/T067).

## Safety statement
This task adds a data-model table + Zod validators. It touches NO active Safety Policy surface (PRODUCT_SPEC ¬ß5 / CLAUDE ¬ß7): no `<VetDisclaimer/>`, no emergency interstitial, no AI-result path, no user-facing prose. Two kinds brush the medication surface and are constrained accordingly: **MED_GIVEN** and **VET_VISIT** `valueJson` carry ONLY descriptive, user-/system-recorded "as entered" fields (mirroring the existing `Reminder.medNameAsEntered`/`medDoseAsEntered` "recorded, never suggested" precedent, CLAUDE ¬ß7 rule 2). No validator emits, requires, suggests, or computes a drug name recommendation, a dose, or an administration instruction; there is no diagnosis language anywhere (the schemas hold no prose). Not a SAFETY-ESCALATION; safe to plan.

## Key decisions (the minimum design choices ‚Äî scrutinize these)
1. **`kind` is a Prisma enum `HealthLogKind`, not a `String`.** ARCHITECTURE ¬ß3 gives an exact CLOSED set (`WEIGHT|MEAL|NOTE|VET_VISIT|MED_GIVEN|CHECK_REF`), the same closed-`|`-set style as `CheckStatus`/`ReminderEventStatus` (which are Prisma enums) ‚Äî unlike `Reminder.type` which ARCHITECTURE left as open vocab (hence `String`). So `HealthLogKind` is a Prisma enum mirrored verbatim, in this exact order, by `HEALTH_LOG_KINDS` in `packages/types` (same mirror contract `CHECK_STATUSES` has with `enum CheckStatus`).
2. **Backlinks (CHECK_REF ‚Üí SymptomCheck, MED_GIVEN ‚Üí ReminderEvent) live INSIDE `valueJson`, NOT as dedicated FK columns.** ARCHITECTURE ¬ß3's authoritative field list for HealthLog is exactly `(id, petId, kind, valueJson, photoKeys[], occurredAt)` ‚Äî it lists NO `checkId`/`reminderEventId` column. ARCHITECTURE wins over the card's shorthand "CHECK_REF backlink". So `checkRefValueSchema` carries `{ checkId }` and `medGivenValueSchema` carries `{ reminderEventId, ... }`. Trade-off (loss of DB-level referential integrity on the backlink) is stated in R1 for the checker.
3. **MED_GIVEN is a read-time PROJECTION (T061 contract ‚Üí T064), not necessarily a persisted row.** The T061 contract is: MED_GIVEN = `ReminderEvent(type=MEDICATION, status=DONE, completedAt)`. T064 will UNION those events into the timeline as MED_GIVEN entries. T063 therefore adds NO `reminderEventId` column and NO write path for MED_GIVEN; it only ships the `MED_GIVEN` enum member (ARCHITECTURE lists it) and its `valueJson` validator so T064's projection is typed/validated by the single source. See R2.
4. **`valueJson` is validated by a pure `parseHealthLogValue(kind, raw)` gate that never throws and fails closed** ‚Äî mirroring `parseIntake`/`parseTriage` (`resultJson` is "validated by Zod at write AND read"). The api will call it on write AND read. Lookup is `HEALTH_LOG_VALUE_SCHEMAS[kind].safeParse(raw)`; an unknown kind or invalid JSON string returns `{ ok:false }`, never a throw.
5. **Every `valueJson` variant is a `z.strictObject`** (rejects unknown keys), exactly like `intake.ts`'s answer/question unions. Forward-compat extra keys are intentionally rejected (safety-conservative). See R7.
6. **Field sets for MEAL/NOTE/VET_VISIT/MED_GIVEN are planner-chosen minimums** ‚Äî ARCHITECTURE spells out only WEIGHT ("weight grams int>0") and says "visit fields, etc." Chosen minimal, safety-clean shapes are enumerated in the file spec below and in R3 for scrutiny. Because `valueJson` is schemaless at the DB layer, any later widening (T065+) is a `packages/types`-only change with NO migration.
7. **One index only: `@@index([petId, occurredAt(sort: Desc)])`** ‚Äî exactly ARCHITECTURE ¬ß3's index line. Its leading `petId` also serves the FK lookup (Postgres leftmost-prefix), so no separate `petId` index is added ‚Äî identical to `SymptomCheck` which has only `@@index([petId, createdAt(sort: Desc)])` for its Pet FK. No speculative `(petId, kind, occurredAt)` composite for T064's possible `?kind=` filter or T066 weight-series ‚Äî the `petId` prefix already bounds the scan to one pet's logs (small). Avoiding a speculative index is the anti-over-engineering choice (CLAUDE ¬ß2). See R5.
8. **HealthLog is a mutable user resource ‚Üí include `createdAt` + `updatedAt`** (matches `Pet`/`SymptomCheck`/`Reminder`, the user-editable models), unlike the immutable system records `TriageResult`/`ReminderEvent` (createdAt only). `occurredAt` (the real-world event time) is distinct from `createdAt` (row insert time). See R4.

## Risks & the decisions the planner made
- **R1 ‚Äî Backlinks in `valueJson`, not FK columns.** Deliberate, because ARCHITECTURE ¬ß3's authoritative field list has no `checkId`/`reminderEventId` column and ARCHITECTURE wins over the card's shorthand. Cost: no DB-level referential integrity or cascade on the backlink, and no cheap reverse "find logs for check X" lookup. If the CHECKER judges a real FK column is required, that is a deviation FROM ARCHITECTURE (journal it, don't silently add). Mitigation: `checkId`/`reminderEventId` are validated (uuid/non-empty) by Zod at write & read.
- **R2 ‚Äî MED_GIVEN persisted vs projected.** Decision 3 keeps it projection-only (no column, no write path), matching the T061 contract. If T064 instead chooses to persist MED_GIVEN rows, the same `medGivenValueSchema` serves unchanged ‚Äî this plan does not block that, it just adds no storage machinery now.
- **R3 ‚Äî Planner-chosen field sets for MEAL/NOTE/VET_VISIT/MED_GIVEN.** ARCHITECTURE details only WEIGHT + "visit fields, etc." The minimal shapes above are the smallest that make each kind meaningful and safety-clean. Because `valueJson` is schemaless in Postgres, widening later is a `packages/types`-only change (no migration) ‚Äî so choosing minimal now is low-regret.
- **R4 ‚Äî `updatedAt` on a possibly-immutable log.** Included to match the user-editable-resource precedent (Pet/SymptomCheck/Reminder). Harmless if logs turn out immutable. If the checker prefers the `TriageResult`/`ReminderEvent` createdAt-only pattern, drop `updatedAt` (single-line change) ‚Äî flagged so it is a conscious call, not an oversight.
- **R5 ‚Äî One index only.** Matches ARCHITECTURE exactly and avoids over-engineering. `petId`-prefix bounds every T064/T066 query to one pet's (small) log set. If real perf data later shows a `kind` filter is hot, add the composite in that task.
- **R6 ‚Äî Migration ordering.** New timestamp MUST be strictly greater than `20260715130000`; `prisma migrate status` in self-verify confirms ordering + no shadow-DB drift. Prefer `migrate dev` generation so the SQL matches Prisma exactly (JSONB/array/default forms).
- **R7 ‚Äî `strictObject` rejects unknown `valueJson` keys.** Safety-conservative (mirrors `intake.ts`). If a future client sends extra keys they are rejected, not silently stored; that is the intended fail-closed posture. Revisit only if a task explicitly needs additive-forward `valueJson`.
- **Safety (re-asserted):** VET_VISIT carries no med/dose field; MED_GIVEN name/dose are optional, record-only "as entered" strings ‚Äî no validator suggests, requires, or computes any drug or dose. Enforced by the dedicated safety spec case.
