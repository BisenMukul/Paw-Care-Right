## Interfaces/contracts (executor must match)

### Verifier abstraction â€” `social-verifier.ts`
```
export type SocialProvider = "apple" | "google";
export interface VerifiedSocialIdentity {
  provider: SocialProvider;
  subject: string;          // the token `sub` claim
  email: string | null;
  emailVerified: boolean;
}
export interface SocialTokenVerifier {
  readonly provider: SocialProvider;
  verify(identityToken: string): Promise<VerifiedSocialIdentity>;
}
export const SOCIAL_TOKEN_VERIFIERS = Symbol("SOCIAL_TOKEN_VERIFIERS"); // -> SocialTokenVerifier[]
```

### Apple verifier â€” `apple-token-verifier.ts`
```
export const APPLE_ISSUER = "https://appleid.apple.com";
export const APPLE_JWKS_URL = "https://appleid.apple.com/auth/keys";
export const APPLE_JWKS_RESOLVER = Symbol("APPLE_JWKS_RESOLVER"); // -> jose key-resolver fn (JWTVerifyGetKey)
```
- Constructor: `@Inject(APPLE_JWKS_RESOLVER) keyResolver`, `AppConfigService config`.
- `verify(token)`: `await jwtVerify(token, this.keyResolver, { issuer: APPLE_ISSUER, audience: this.config.appleClientId, algorithms: ["RS256"] })`; then `subject = payload.sub`, `email = typeof payload.email === "string" ? payload.email : null`, `emailVerified = payload.email_verified === true || payload.email_verified === "true"`. Catch ALL errors â†’ `throw new UnauthorizedException()`. Never log `token`.

### DTO â€” `social-auth.dto.ts`
```

```
(ValidationPipe is `whitelist + forbidNonWhitelisted` â†’ extra fields/unknown provider â†’ 400.)

### Endpoint
`POST /v1/auth/social` â€” public-by-absence (no guard until T015). `@HttpCode(200)`. Body `SocialAuthDto`. Response = `AuthTokens` (identical to `otp/verify`: `{ accessToken, refreshToken, user: { id, email }, householdId }`).

### Linking + provision + race semantics â€” `SocialAuthService`
Providerâ†’sub-column map (switch, no `any`): `apple â†’ appleSub`, `google â†’ googleSub` (google column exists; unreachable in T013 but keeps the method provider-agnostic for T014). Inside one `prisma.$transaction`:
1. find `User` by the provider's sub column === `identity.subject` â†’ if found, resolve its owned household (oldest; missing â†’ `InternalServerErrorException`, mirroring the existing invariant) â†’ `ProvisionedUser`.
2. else if `identity.email && identity.emailVerified`:
   - find `User` by `email` (normalized lower/trim). If found â†’ `update` that user setting the sub column = `identity.subject`; resolve its household â†’ `ProvisionedUser` (**links, no new user**).
   - else â†’ `authService.provisionNewAccount(tx, { email, [subColumn]: subject })` (same account shape as OTP first-verify).
3. else (no verified email, no sub match) â†’ `throw new UnauthorizedException()` (R4).
Race safety: wrap the transaction; on Prisma `P2002` (unique violation on the sub or email) â†’ re-query `User` by the sub column; if found â†’ build `ProvisionedUser` from it; else rethrow. Then `login()` calls `authService.issueSession(provisioned)`.

### AuthService reuse surface
- `export interface ProvisionedUser { userId; email; householdId }`
- `issueSession(provisioned: ProvisionedUser): Promise<AuthTokens>` (public; used by `verifyOtp` + `SocialAuthService`).
- `provisionNewAccount(tx, { email; appleSub?; googleSub? }): Promise<ProvisionedUser>` (public; used by `provisionOrGetUser` + `SocialAuthService`). Writes User(email, sub?, default locale/region) + Household(default name, owner) + Membership(OWNER) â€” identical to today's inline creates.

## Risks & the design decisions the planner made (scrutinize)
- **R1 â€” jose version (decision):** Use **`jose` (^5)**, not ^6. The api's `tsconfig.json` sets `moduleResolution: "Node"` (classic) + `module: "CommonJS"`; jose v5 ships a CJS build reachable via `main`, whereas jose v6 is `exports`-only ESM that classic Node resolution won't resolve and Nest/ts-jest CJS won't `require`. v5 is actively maintained and provides `jwtVerify`, `createRemoteJWKSet`, `createLocalJWKSet`, `SignJWT`, `generateKeyPair`, `exportJWK`. If even v5 fails typecheck/CJS interop, STOP and flag â€” do not switch libraries. Justification line for `loop/journal.md`: "jose ^5 â€” standard JOSE impl; createRemoteJWKSet gives cached Apple JWKS verification; CJS build compatible with api moduleResolution:Node."
- **R2 â€” JWKS seam via DI provider override (decision):** the seam is the injectable `APPLE_JWKS_RESOLVER` (a jose key-resolver fn), overridden in the e2e with a local keyset â€” no local HTTP server, no network, deterministic. Alternative (configurable URL + spun-up server) rejected as heavier/flakier.
- **R3 â€” unknown provider = 400, not 401 (decision):** the DTO `@IsIn(["apple"])` makes any non-apple provider a validation failure (400 `VALIDATION_FAILED`) before any verifier runs. Consistent with the pipe; keeps 401 exclusively for token-verification failures.
- **R4 â€” no-verified-email policy = 401 (decision, fails closed):** if the token has no sub match AND no verified email, we CANNOT provision (`User.email` is required + unique), so we return 401 rather than inventing a placeholder email. Requiring `emailVerified===true` to link/create also prevents linking to a victim's existing account via an unverified/spoofed email claim. Apple sets `email_verified` true for real + relay addresses, so normal first-auth flows are unaffected.
- **R5 â€” AuthService refactor could regress OTP (mitigated):** `issueSession`/`provisionNewAccount` are pure extractions; `verifyOtp`/`provisionOrGetUser` keep the same query sequence and return shape, so `auth.service.spec.ts` (which asserts exact create call-counts + result equality) and `auth.e2e-spec.ts` stay green. The self-verify sequence re-runs both.
- **R6 â€” provider-agnostic linking touches `googleSub` (decision, deliberate shaping for T014):** the sub-column switch maps googleâ†’googleSub even though google is unreachable in T013. This is the minimal shaping the look-ahead requires (T014 adds only the Google verifier + widens the DTO/`SOCIAL_TOKEN_VERIFIERS` arrays); it is NOT speculative Google verification (no Google JWKS/claims/DTO value). `googleSub` already exists in the schema.
- **R7 â€” uniform 401 / no token leakage:** every verification failure path (bad signature, iss, aud, exp, alg, malformed) resolves to a bare `UnauthorizedException` â†’ the existing `AllExceptionsFilter` renders `{ error: { code:"UNAUTHORIZED", ... } }`. The identity token is never logged and failure reasons are never surfaced.
- **R8 â€” file count (~15) slightly exceeds the ~10-14 hint:** the overage is the regenerated `pnpm-lock.yaml` plus two focused unit specs (verifier + linking); each earns its place against an AC or a gate. No feature file is superfluous.
- **Safety:** T013 touches no Safety-Policy surface (no disclaimers, emergency escalation, or dosing copy) â€” not a SAFETY-ESCALATION.
