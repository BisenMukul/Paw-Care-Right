# Plan ‚Äî T075: Free-tier gating (wire quota to real entitlements)

## Objective (from card)
Wire the quota service to real (T072) entitlements so free = 1 lifetime check, 5 food
lookups/day, 1 pet, and chat/sharing are premium-only; every gated surface returns a
consistent 402-class error that drives a mobile upsell sheet. Counters are server-side and
survive reinstall. T042's rules-BEFORE-quota + emergency-bypass ordering MUST NOT regress.

## Safety statement (PRODUCT_SPEC ¬ß5 / CLAUDE.md ¬ß7 ‚Äî LOAD-BEARING, non-negotiable)
- **Quota/entitlement NEVER gates a ¬ß5 safety surface.** On the checks path the deterministic
  red-flag rules run and short-circuit BEFORE any quota consideration (`checks.service.ts`
  step 4‚Üí5 today). This plan changes ONLY *what the quota branch consults* (step 6's
  resolver); it does not move, weaken, or add code on the red-flag branch (step 5), and adds
  no new caller upstream of red-flag evaluation. An explicit e2e re-asserts: **after the free
  check quota is exhausted, a red-flag intake still returns 201 (emergency bypass intact).**
- No emergency interstitial, `<VetDisclaimer/>`, hotline, dosing, or "diagnos*" copy is
  touched anywhere in this task. The mobile change to the check-flow screen is limited to the
  neutral "See plans" button's *navigation target* ‚Äî no safety copy, no emergency code.
- The upsell sheet is always dismissible and is structurally incapable of appearing on the
  emergency path: the checks mutation is tagged `meta.skipUpsell` so the global 402 handler
  never fires for a check, and the red-flag branch returns 201 (never a 402), so it cannot
  trigger an upsell at all.
- No SAFETY-ESCALATION: the card is implementable without weakening any safety surface.

## Key decisions (each is a design choice ‚Äî scrutinize)
1. **One 402 error code for ALL gated surfaces: reuse the existing `PAYMENT_REQUIRED`
   (HTTP 402).** No new enum member. Rationale: the error contract maps by HTTP status
   (`all-exceptions.filter.ts` `STATUS_TO_CODE[402]="PAYMENT_REQUIRED"`), the api-client maps
   402‚Üí`PAYMENT_REQUIRED`, and mobile ALREADY keys off `err.code === "PAYMENT_REQUIRED"`
   (`use-check-submission.ts`). A second code (e.g. `PREMIUM_REQUIRED`) would require a
   custom exception carrying the code plus filter changes, for marginal UX gain. "Consistent
   402-class error" (card) is best served by a single code. Rejected alternative recorded in
   Risks R1.
2. **Extract ONE shared entitlement resolver; do not duplicate billing logic.** Replace the
   v1 `StaticEntitlementResolver` (everyone FREE) with a `BillingEntitlementResolver`
   (`ENTITLEMENT_RESOLVER` token) backed by T072's `BillingService.getEntitlement(userId,
   householdId)`. Mapping: `entitled ? {tier:"PREMIUM"} : {tier:"FREE"}`, always
   `bypassQuota:false`. ALL gated surfaces (checks, pets, invites) consume this ONE token ‚Äî
   giving a single billing‚Üíquota mapping and a single e2e override seam (override
   `ENTITLEMENT_RESOLVER` ‚Üí everything premium at once, mirroring the existing checks e2e).
3. **Resolver interface gains `householdId`.** `resolve(userId, householdId)` ‚Äî entitlement
   is household-scoped (SPEC ¬ß7: "Premium entitlement is household-scoped"). Every call site
   already has both ids. The existing checks e2e/unit mocks are arg-agnostic (`resolve: async
   () => ‚Ä¶`), so they keep passing.
4. **Premium checks keep the fair-use cap.** `bypassQuota:false` for real premium ‚áí the
   `PREMIUM.checks = {month, 30}` limit in `quota.constants.ts` is enforced (SPEC "fair-use
   30/mo"). The pre-existing checks e2e premium mock (`bypassQuota:true`) stays valid as a
   mock; the new matrix asserts premium simply is not blocked at the free limit (2 checks
   succeed, well under 30).
5. **Pets/invites gates are entitlement-tier checks, not Redis counters.** Pet count is
   authoritative from `Pet` rows (already server-side, already reinstall-safe); the free gate
   is "resolve entitlement; if FREE and non-deleted pet count ‚â• `FREE_MAX_PETS` (1) ‚Üí 402".
   Invites is a pure feature-lock: "if FREE ‚Üí 402" (no counter). Only the `checks`/`food`
   metrics use Redis counters.
6. **Food-lookup + chat gating = NOT-APPLICABLE-YET (no endpoint exists).** Grep confirms
   there is no food/toxin HTTP endpoint (T035 shipped only `packages/data` + an in-package
   lookup service; the `THROTTLE_FOOD` config is explicitly "routes don't exist yet") and no
   chat module (Phase 8). The `foodLookups` Redis metric + limits already exist in
   `quota.constants.ts` and are LEFT AS-IS, ready for the future endpoint. Per CLAUDE.md ¬ß2
   (no speculative features / over-engineering is a defect) this task adds NO chat/food
   endpoints and NO speculative gate constants for them; when those endpoints land they call
   the same `ENTITLEMENT_RESOLVER` + (for food) the existing `QuotaService.consume(‚Ä¶,
   "foodLookups", ‚Ä¶)`. This is documented, not built.
7. **Mobile: one central 402 interceptor + one reusable upsell sheet.** A TanStack
   `MutationCache.onError` (wired in mobile's `query.ts`, the single place the app's
   `QueryClient` is built) shows a global `<UpsellSheet/>` whenever a mutation fails with an
   `ApiError` of code `PAYMENT_REQUIRED`, EXCEPT when the mutation sets `meta.skipUpsell`.
   The checks mutation sets `meta.skipUpsell:true` and keeps its bespoke full-screen "quota"
   state (whose dead upgrade button is wired to `/paywall`), avoiding a double UI on the
   ¬ß5-adjacent check flow. `createQueryClient` is extended to accept a `mutationCache`
   passthrough (backward-compatible).

## Risks & the decisions the planner made (for the checker)
- **R1 (decision 1 ‚Äî the primary decision) single vs two error codes.** Chose one
  `PAYMENT_REQUIRED` for both quota-exhausted (checks/pets) and feature-locked (invites).
  Trade-off: the upsell sheet cannot vary copy by cause. Accepted because the card demands a
  "consistent 402-class error", the whole error stack already routes 402‚Üí`PAYMENT_REQUIRED`,
  and mobile already keys on it ‚Äî a second code would touch the filter/contract for little
  gain. If differentiated copy is later needed, add a code then, not speculatively.
- **R2 (decision 5) pets count TOCTOU.** Count-then-create is not atomic: two concurrent
  `POST /pets` on a free household could both observe count 0 and both create (overspend by
  one). Non-safety, rare, worst case one extra pet. Accepted (matches the codebase's
  pragmatic posture); checks use atomic Redis `INCR` and are unaffected.
- **R3 (decision 4) premium expiring mid-session.** Entitlement is resolved per-request via
  `getEntitlement` (fresh DB read + clock), never cached in-session, so a lapse takes effect
  on the next gated request ‚Äî correct fail-toward-free. `expiresAt` grace is T072's
  `entitlement.util` and untouched.
- **R4 family-member counts.** The 1-pet gate counts pets in the *household*, and entitlement
  is household-scoped, so a premium family member lifts the gate for the whole household and a
  free household is capped at one pet regardless of member count ‚Äî matches SPEC ¬ß7 intent.
- **R5 daily-window tz (future food metric).** The `foodLookups` day bucket is UTC
  (`windowBucket`), so "5/day" resets at UTC midnight; noted for when the food endpoint lands.
  Not activated by this task.
- **R6 double-UI on the check flow.** Mitigated by `meta.skipUpsell` on `useCreateCheck`; the
  interceptor test pins that suppression. If a future gated mutation forgets the meta it will
  correctly show the global sheet (safe default).
- **R7 module coupling / cycles.** `QuotaModule` now imports `BillingModule`
  (Billing‚ÜíPrisma only, no cycle); `PetsModule`/`HouseholdsModule` import `QuotaModule` for
  the token. Verified acyclic: nothing Billing/Quota depends on Pets/Households.
