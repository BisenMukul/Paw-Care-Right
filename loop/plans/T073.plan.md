# Plan — T073: RevenueCat webhook handler

## Objective (from card)
Add `POST /billing/rc-webhook`: auth-header verify (constant-time), idempotent
event processing (event-id dedupe), a state machine over
INITIAL_PURCHASE/RENEWAL/CANCELLATION/EXPIRATION/BILLING_ISSUE/PRODUCT_CHANGE that
upserts the T072 `Subscription` mirror row, out-of-order safe, raw event retained,
bad auth → 401.

## Safety statement (PRODUCT_SPEC §5)
This task touches **no** Safety Policy surface: no AI output, no `<VetDisclaimer/>`,
no emergency/red-flag interstitial, no medication dosing/med-name copy, no user-facing
symptom prose. It is server-side billing/entitlement plumbing only. No SAFETY-ESCALATION.
Fail-direction note: entitlement resolution (T072) keys off `entitlement` + `expiresAt` +
the clock (never `status`); this handler only mirrors RC state, so the worst-case bug is a
user briefly losing premium (safe) rather than a safety-content regression.

## Key decisions (each is a design choice — scrutinize)
1. **Auth model = shared secret in the `Authorization` header, NOT HMAC.** RevenueCat
   webhooks send a fixed, operator-configured `Authorization` header value verbatim; there is
   no request-body signature. The guard compares the incoming header against
   `RC_WEBHOOK_AUTH_TOKEN` (validated env) using a **length-safe constant-time** compare:
   `timingSafeEqual(sha256(incoming), sha256(secret))` (SHA-256 both sides so unequal lengths
   don't leak via `timingSafeEqual` throwing). Missing/empty header → 401. No new dependency
   (`node:crypto`). Because auth is header-only, **no raw-body parsing is required** — the JSON
   body parser stays as-is.
2. **Dedupe store = new `ProcessedWebhookEvent` table** (`eventId @id`, `processedAt`), not a
   column on `Subscription` (one subscription receives many events; a set-column can't dedupe).
   Dedupe is enforced by the PK: attempting to insert an already-seen `eventId` throws Prisma
   `P2002` → treated as a replay no-op.
3. **Atomicity for replay-safety = one interactive `prisma.$transaction`** wrapping *both* the
   `ProcessedWebhookEvent` insert *and* the `Subscription` upsert. If the upsert fails, the
   dedupe row rolls back too, so RC's retry reprocesses cleanly. A genuine infra error is left to
   throw → 500 → RC retries (correct). Only *successfully handled* outcomes ack 200.
4. **Ordering discipline = event-timestamp staleness gate.** Each write stamps
   `Subscription.lastEventAt` (new nullable column) from the event's ordering clock
   (`event_timestamp_ms ?? purchased_at_ms ?? received-now`). An incoming event whose ordering
   timestamp is **strictly older** than the stored `lastEventAt` is **stale → skip state change**
   (but is still dedupe-recorded and acked). This makes a late RENEWAL arriving after an
   EXPIRATION unable to resurrect premium, and a stale CANCELLATION unable to override a newer
   RENEWAL. Equal timestamps are applied in arrival order (see Risk R2).
5. **Unknown / anonymous `app_user_id` = resolve-household-or-skip, never 500.** RC may send
   ids that aren't backend users (e.g. `$RCAnonymousID:…`). Household is resolved from the
   purchaser's **current single membership** (v1 = one household per user); if none resolves,
   fall back to an existing row's `householdId`; if neither exists → the id is not a real
   member → **skip the upsert (no `Subscription` row written), log ids-only, still ack 200 +
   dedupe-record.** This also prevents the `Subscription.rcAppUserId → User` FK violation.
6. **`householdId` re-stamped at WRITE time** from the purchaser's current membership on every
   applied event (carry-forward ③), so a household move is reflected; only when no current
   membership resolves do we preserve the existing row's `householdId`.
7. **Sandbox handling = process both environments identically**, retaining `environment` inside
   `rawEventJson` for audit. No env-gating (keeps local/dev/test flows working); documented as a
   risk, not enforced separation.
8. **Inline processing, no BullMQ.** The handler does two DB statements in one transaction — no
   AI/image/push/long-running work — so §6's worker rule does not apply. Idempotency + atomic
   dedupe + always-ack-on-handled give the retry-safety §6 demands of a webhook.
9. **Raw body captured un-stripped.** The controller types `@Body()` as a plain object (no
   class-validator DTO), so the global `whitelist/forbidNonWhitelisted` pipe does not strip
   unknown RC fields; the Zod envelope (`.passthrough()`) validates only the consumed subset, and
   the **entire raw body** is stored in `Subscription.rawEventJson`.

  PREMIUM, status="billing_issue", expiresAt=from expiration_at_ms (grace via T072 clock);
  EXPIRATION → FREE, status="expired", expiresAt=from expiration_at_ms; any other type →
  `skip:"unknown_type"`. Stale gate applied first: strictly-older ts → `skip:"stale"`.

## Tests to write (map to acceptance criteria)
- **AC "table-driven per event type"** → `rc-webhook.state.spec.ts`: one row per type asserting
  `computeSubscriptionUpdate` output — INITIAL_PURCHASE→PREMIUM/active/expiresAt set;
  RENEWAL→PREMIUM/active/new expiresAt; PRODUCT_CHANGE→PREMIUM/active/new plan; CANCELLATION→
  PREMIUM/cancelled/expiresAt kept-from-event; BILLING_ISSUE→PREMIUM/billing_issue;
  EXPIRATION→FREE/expired; unknown type→`skip:"unknown_type"`; every applied case sets
  `lastEventAt` from the event ts.
- **AC "incl. out-of-order delivery"** → `rc-webhook.state.spec.ts`: (a) EXPIRATION applied, then
  RENEWAL with older ts → `skip:"stale"` (stays expired); (b) RENEWAL applied, then CANCELLATION
  with older ts → `skip:"stale"`; plus `billing-webhook.e2e-spec.ts` real round-trip: POST
  RENEWAL then POST older-ts EXPIRATION → final row still PREMIUM; POST EXPIRATION then older-ts
  RENEWAL → final row still FREE/expired.
- **AC "replay-safe"** → `rc-webhook.service.spec.ts`: dedupe-insert throws P2002 → no upsert
  called, returns `{received:true}`. `billing-webhook.e2e-spec.ts`: same `eventId` POSTed twice →
  both 200, exactly one processing (row unchanged on 2nd, single `ProcessedWebhookEvent`).
- **AC "bad signature 401"** → `rc-webhook.guard.spec.ts`: correct token→allow; wrong→401;
  missing header→401; empty→401; different-length→401 (no throw). `billing-webhook.e2e-spec.ts`:
  wrong/missing `Authorization` → 401 with error envelope `code:"UNAUTHORIZED"`.
- **Carry-forward: unknown/anonymous user → 200, no row, logged** →
  `rc-webhook.service.spec.ts`: membership.findMany→[] and no existing row → no upsert, dedupe
  recorded, `{received:true}`. e2e: POST with `app_user_id:"$RCAnonymousID:x"` → 200, no
  `Subscription` row created.
- **Carry-forward: unknown event type → 200-ack + dedupe-recorded + no state change** →
  state.spec (skip) + e2e: POST `type:"TRANSFER"` → 200, dedupe row present, no Subscription write.
- **Carry-forward: householdId re-stamped at write** → `rc-webhook.service.spec.ts`: upsert data
  carries the household resolved from current membership.

## Risks & edge cases (for the checker)
- **R1 (decision 1) HMAC-vs-header:** if RC is actually configured for a signed body in this
  project, header-only auth is wrong. Mitigation: RC's documented webhook auth *is* a static
  Authorization header; HMAC is not offered. Constant-time compare avoids timing oracles.
- **R2 (decision 4) timestamp ties:** two distinct events with identical ordering ms are applied
  in arrival order (equal is not "strictly older"). Astronomically rare; dedupe already covers
  true replays; fail-direction is safe (losing premium early). Documented, not defended further.
- **R3 (decision 5) FK race — user deleted mid-event:** membership resolves, user deleted before
  upsert → FK violation inside the tx → tx rolls back → 500 → RC retries → on retry no membership
  → unknown-user skip. Self-heals; no orphan dedupe row (atomic tx).
- **R4 RC retry storms:** every retry of an already-processed `eventId` is a cheap P2002 no-op
  ack; a 2xx tells RC to stop. Genuine failures stay non-2xx so RC keeps retrying.
- **R5 PRODUCT_CHANGE family↔individual:** handled by writing `plan=product_id` + re-stamping
  `householdId`; T072 resolution then grants/revokes family coverage automatically. No special
  case.
- **R6 (decision 7) sandbox purchases** share the same DB and could grant premium in dev/test;
  accepted for this task (retained in `rawEventJson`); real env separation is future work.
- **R7 dedupe-recording unknown-user/unknown-type events:** we record `ProcessedWebhookEvent`
  even when no Subscription is written, so replays stay cheap and consistent; harmless because the
  table has no FK to `User`.
- **R8 global-pipe stripping (decision 9):** must NOT bind a class-validator DTO to `@Body()`, or
  `forbidNonWhitelisted` will 400 on RC's extra fields and lose the raw event. `@Body()` typed as
  a plain object is deliberate.
