## Interfaces/contracts (executor must match)

### `ApiClientConfig` additions (`client.ts`)
```ts
export interface ApiClientConfig {
  baseUrl: string;
  getAuthToken?: () => string | null | Promise<string | null>;
  fetch?: typeof fetch;
  /** Called on a 401 to obtain a fresh access token; null ⇒ session is dead. Single-flighted by the client. */
  refreshSession?: () => Promise<string | null>;
  /** Called after a refresh yields null/throws, or a retried request is still 401. */
  onSessionExpired?: () => void | Promise<void>;
}
```
Single-flight contract (per client instance): a closure-scoped `let refreshInFlight: Promise<string | null> | null`. First 401 sets it to `refreshSession()` (cleared in `finally`); concurrent 401s `await` the same promise. On resolved non-null token → retry the ORIGINAL request EXACTLY once with `Authorization: Bearer <token>` (override, not via `getAuthToken`). If that retry is 401, or token was null, or `refreshSession` threw → `await onSessionExpired?.()` then throw the normalized 401 `ApiError`. Non-401 responses and network throws are unchanged. The refresh call itself must never be intercepted — on mobile it runs through the bare `authClient` (below), so the interceptor never sees it. Only 401 triggers this; `shouldRetry` still governs query-level retries.

## Risks & the design decisions the planner made (for CHECKER scrutiny)
1. **Single-flight lives in the client (closure), delegating to a store `refreshSession` that is itself idempotently single-flighted.** Both the client's 401 path and the store's `restore()` funnel through one `refreshSession`, so overlapping refreshes cannot occur. Alternative (mutex only in store) rejected: the client must still dedupe concurrent 401s across in-flight requests before calling out.
2. **Retry uses the token RETURNED by `refreshSession` (explicit `Authorization` override), not a re-read via `getAuthToken`.** Decouples the retry from store-update timing and makes the AC header assertion deterministic.
3. **Recursion avoidance = a bare `authClient`.** The refresh/logout endpoints go through a client with no `refreshSession`, so a 401 there cannot re-enter the interceptor. This requires repointing `auth-api.ts` (in scope).
4. **`refreshSession` returns null on failure and does NOT mutate auth state; `sessionExpired` performs the local logout.** Keeps refresh side-effect-free and puts the single logout (clear + signedOut, no network) behind `onSessionExpired` — satisfying "refresh failure logs out" without a network call on a dead session. `restore()` is refactored to keep its existing observable behavior (checker: diff must show T018 store assertions still pass verbatim).
5. **Promote `@tanstack/react-query-persist-client` to a core `dependency`.** It is platform-agnostic JS (no native binary), so putting it in the core chunk does NOT break the web/isolation guarantee — only `react-native-mmkv` (native) must stay subpath/optional-peer. This lets the core export `PersistedApiQueryProvider` typed with `Persister`. Alternative (structural/`unknown` typing to keep it optional) rejected as less type-safe for equal isolation. Justify in journal (CLAUDE §2 rule 7).
6. **`useIsOffline` tested at the store primitive, not via React render.** api-client's jest env is `node` with no RTL; adding RTL would be a new dep. The hook is a thin `useSyncExternalStore` wrapper over `subscribe`/`getIsOfflineSnapshot`, which are unit-tested directly. Risk: the React binding itself is exercised only at mobile integration.
7. **Persister is now unit-tested via `jest.mock("react-native-mmkv")` + a fake mmkv** (revising T010 risk #7, which left it untested). The fake implements the methods the persister actually calls — `getString`, `set`, `remove` (note: the code uses `remove`, not `delete`) — cast to `MMKV`.
8. **New deps:** `expo-network` (mobile, SDK-pinned `~57.0.x`, Expo-sanctioned `addNetworkStateListener`; preferred over netinfo for SDK pinning) and the persist-client promotion (Risk 5). Nothing else. strict TS, no `any`, no `console.*`.
9. **jest.setup native mocks** (`react-native-mmkv`, `expo-network`) are added defensively so future renders of the layout stay headless; existing mocks and the 19 T018 tests remain green.
