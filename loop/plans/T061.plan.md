# Plan — T061: Medication tracker

## Objective (from card)
Add a **MEDICATION reminder subtype** that records a med **name + dose exactly as entered** and a **course length**, generating **per-dose reminder occurrences** (e.g. 2×/day for 10 days = 20 events), surfacing the given dose as a **record** on the agenda, and treating a completed MEDICATION dose as the **MED_GIVEN** timeline source — with **zero suggestion surfaces** (free-text only; no drug/dose autocomplete, presets, defaults, or example values), enforced mechanically by reusing the T038 detector as a lint test. Accept = (AC1) course-generation tests (2×/day × 10 days = 20 materializable occurrences); (AC2) no string this module adds trips the T038 dosage/drug detector.

## Safety statement (CLAUDE §7 rule 2 — THE load-bearing constraint; PRODUCT_SPEC §5)
This task touches the exact surface §7.2 governs: "the med tracker records what a vet prescribed; it never suggests." **This plan is *record, not recommend* on every surface, and adds a mechanical gate that proves it.** No AI is involved; no `<VetDisclaimer/>`, emergency-interstitial, or existing dosing-prohibition copy is changed or weakened. Per surface:
- **Med create form (mobile):** `medNameAsEntered` and `medDoseAsEntered` are **free-text `TextInput`s only**. There is **NO** drug-name autocomplete, **NO** dose preset chips, **NO** default/prefilled dose, and **placeholders contain no dose example** (no numbers, no units like "mg/ml", no drug names). A **static, non-editable disclaimer** ("Enter exactly what your vet prescribed. We never suggest medications or doses.") is always shown. The app never generates, completes, ranks, or defaults a drug name or a dose.
- **Course scheduling:** the app schedules *reminders* only (when to be reminded). `courseLengthDays` and dose *times* are reminder cadence, never dosing advice. The recurrence uses the locked RRULE subset; no dosing math is performed.
- **Agenda / detail:** shows the med name + dose **verbatim as the user entered them** (a record of user input), never app-authored text.
- **Detector gate:** every user-facing string this task adds lives in one SSOT (`MEDICATION_STATIC_COPY`) that is scanned by the actual T038 `scanUnsafe`; the build fails if any string looks like a dose/drug suggestion.

This is **not** a SAFETY-ESCALATION: correctly implemented, the task *strengthens* the §7.2 no-suggestion posture. It is planned. **If the executor cannot keep the detector spec green with neutral copy, it must fix the copy, never weaken/skip the spec.**

## Key decisions (numbered; scrutinize under Risks)

1. **A medication course = N sibling `Reminder`s (one per daily dose time), grouped by a shared `courseId`; each sibling recurs `FREQ=DAILY;COUNT=<courseLengthDays>`.** The locked RRULE grammar (`packages/types/src/rrule.ts`) fires **once per day** at `startAt`'s wall-clock time-of-day and has **no BYHOUR / multiple-times-per-day** support, so intra-day multiplicity is *not* expressible in a single rule. Least-invasive design consistent with the locked grammar: 2×/day → **2** sibling reminders (distinct `startAt` times), each `FREQ=DAILY;COUNT=10` → 10 occurrences each → **20** total. No grammar change.
2. **Course termination is grammar-native via `COUNT`, not a new column.** `parseRRule` already parses `COUNT` and `computeNextFireAt`/`occurrencesBetween` already honor it (`occurrenceIndex > rule.count → null`). `COUNT=<courseLengthDays>` deterministically ends each sibling after exactly `courseLengthDays` occurrences. **No `endsAt` column, no scheduler deactivation added.**
3. **`courseId` groups siblings; NO dedicated index.** Access to a course's siblings is always **pet-scoped**, which the existing `@@index([petId])` already covers — mirroring the `templateKey` precedent (queried via a pet-scoped `where`, no dedicated index). Adding a `courseId` index now would be speculative (CLAUDE §2: over-engineering is a defect). Just the nullable column.
4. **`medDoseAsEntered` is a new nullable column** (`medNameAsEntered` already exists from prior tasks). The card lists name **and** dose as distinct fields; storing dose separately (vs. folding into the name free-text) keeps the record structured and lets the agenda show it as its own line.
5. **MED_GIVEN write path = the existing completion record; NO HealthLog table is created here.** The `HealthLog(kind: … MED_GIVEN …)` model is owned by **T063** (Phase 6) and the Timeline API by **T064**; T061 must not steal that scope. The T060 `completeOccurrence` upsert already durably and **idempotently** records a given dose (`status=DONE, completedAt=now` on `ReminderEvent`, keyed `reminderId_dueAt`). **Contract established here:** a MEDICATION `Reminder`'s `ReminderEvent` with `status=DONE` + `completedAt` set **is** the MED_GIVEN entry; T064 projects these. T061 changes **no** completion behavior and adds **no** timeline table; it adds a test proving the MED_GIVEN record is durable + idempotent, and documents the projection contract.
6. **Course generation lives in a pure module `buildMedicationCourse(...)`** (`apps/api/src/reminders/medication-course.ts`), unit-tested against `occurrencesBetween` (AC1 counts **materializable** occurrences, not arithmetic). The service `createMedicationCourse` maps its specs into a single `$transaction` of `reminder.create`s sharing one `crypto.randomUUID()` `courseId`.
7. **Detector-reuse lint test lives in `packages/ai` and imports the *actual* T038 `scanUnsafe` from `./detector`; the med copy it scans is the SSOT `MEDICATION_STATIC_COPY` defined in `packages/types`.** Rationale: (a) `scanUnsafe` is **not** exported from `@pawcareright/ai`'s public index and should not be (it is red-team internals) — so the test must live *inside* `packages/ai` to import it locally; (b) `packages/ai` already value-imports `@pawcareright/types` (e.g. `SAFE_FALLBACK`, `FOOD_SAFETY_FALLBACK`), so the copy must live where **both** the med UI and the ai spec can import it **without an app→app or ai→app cycle** — i.e. `@pawcareright/types`; (c) there is direct precedent for safety-critical user-facing constants living in `@pawcareright/types` (`SAFE_FALLBACK`, `FOOD_SAFETY_FALLBACK`). Mobile `strings.ts` imports the same constants, so **the tested string is byte-identical to the rendered string** — no drift, which is exactly what a §7.2 mechanical gate needs. `scanUnsafe` takes a `TriageResult`; the spec constructs one (reusing `detector.spec.ts`'s `resultWith` pattern) with every `MEDICATION_STATIC_COPY` string placed in a scanned string[] field, and asserts `[] `. A positive-control case (a known bad string) guards against a vacuous test.
8. **Agenda carries the med record via two optional fields.** `agendaEntrySchema` (T060, `packages/types/src/agenda.ts`) gains optional `medNameAsEntered?`, `medDoseAsEntered?`; the API `agenda()` virtual-seed + event-map and `toAgendaEntry`/`ReminderResponse` propagate them from the reminder row (already loaded). Small, additive, and directly satisfies "agenda rows for med reminders show name + dose as entered."
9. **Client computes per-dose UTC anchors; server stays tz-simple.** The mobile form combines the chosen start date + each dose time in the device tz to an ISO instant (reusing the proven `combineLocalDateTime` pattern from `edit.tsx`) and sends `doseStartAts: string[]` + the device `timezone`. The server does no wall-clock math; each sibling stores that `timezone` so `computeNextFireAt` keeps the dose at the same local wall-clock across DST (module is DST-correct).

## Endpoint & contract specs the executor must match

## Risks & edge cases (numbered — for the checker to scrutinize)
1. **RRULE has no intra-day multiplicity (decision 1).** Verified in `rrule.ts`/`next-fire-at.ts`: one fire/day at `startAt`'s wall-clock. Sibling-per-dose-time is the grammar-compatible model. Checker: confirm no grammar edit and that `buildMedicationCourse` emits `FREQ=DAILY;COUNT=N` per sibling.
2. **Course termination via `COUNT` (decision 2).** `computeNextFireAt` returns `null` after `count` occurrences; `occurrencesBetween` stops. Checker: confirm the AC1 sum is bounded to exactly `times × days` over an *unbounded* window (proves COUNT terminates, not the window).
3. **Occurrence bound.** `courseLengthDays ≤ 365 < MAX_OCCURRENCES (500)` and `doseStartAts ≤ 12`, so no sibling can exhaust the expander's cap. Checker: confirm DTO `@Max(365)`/`@ArrayMaxSize(12)`.
4. **DST within a course.** `computeNextFireAt` is DST-correct and anchors to the tz wall-clock; a dose stays at the same local time across a transition (siblings store the creation `timezone`). Checker: no UTC-fixed offset assumptions.
5. **Timezone change mid-course (traveling user).** Siblings keep their creation tz; the dose fires at the original-tz wall-clock. Accepted — identical to every other reminder's behavior; not in scope to re-home.
6. **Completing after course end.** A `dueAt` beyond `COUNT` is not a genuine occurrence → T060 `assertOccurrence` returns `400`. You cannot complete a non-existent dose. Checker: confirm no bypass.
7. **Double-complete a dose = one MED_GIVEN.** T060 `completeOccurrence` upsert on `reminderId_dueAt` is idempotent. Checker: confirm the idempotency test and that T061 changes **no** completion logic.
8. **MED_GIVEN with no HealthLog table (decision 5).** HealthLog/timeline are T063/T064; T061 must not create them. The DONE MEDICATION event is the durable MED_GIVEN record T064 will project. Checker: confirm no HealthLog model/migration/endpoint is added and that this is documented, not silently dropped.
9. **Detector false-positive on chosen copy (AC2).** The neutral copy must avoid digits+units, drug names (`DRUG_NAME_TOKENS`), admin-frequency phrasing, and `diagnos*`. The spec (with positive control) is the guard; if it fails, fix the **copy**, never the spec. Checker: confirm the spec scans the SSOT via the real `scanUnsafe` and is non-vacuous.
10. **Med copy lives in `packages/types` (decisions 7).** Deviates from "strings in per-app `strings.ts`" but matches the `SAFE_FALLBACK`/`FOOD_SAFETY_FALLBACK` safety-constant precedent and guarantees the tested string == the rendered string (no drift). This is the ONE notable design deviation — flagged intentionally.
11. **No `courseId` index (decision 3).** Access is pet-scoped (existing `@@index([petId])`), mirroring `templateKey`. Adding an unused index would be over-engineering (§2). Checker: confirm no course query pattern goes un-indexed.
12. **Duplicate dose times.** Identical `doseStartAts` are de-duped server-side so the agenda never shows two identical rows; the returned `reminderCount` reflects the de-duped set.
