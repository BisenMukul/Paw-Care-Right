# Plan — T012: Email OTP auth (request/verify) + JWT + refresh rotation

## Objective (from card)
Ship passwordless email-OTP auth on `apps/api`: `POST /auth/otp/request` (rate-limited 5/min/IP, always-200 anti-enumeration, dev transport logs the code behind an interface), `POST /auth/otp/verify` → access JWT (15m) + rotating hashed refresh token (30d), `POST /auth/refresh` (rotation with reuse/theft family-revoke), `POST /auth/logout` (revoke the whole token family). Auto-provision `User` + `Household` + OWNER `Membership` on first successful verify. All under `/v1/auth`, class-validator DTOs, Swagger, Nest Logger, strict TS.

Not a Safety Policy surface (PRODUCT_SPEC §5 = AI guidance/disclaimers/dosing/emergencies). No safety escalation.

## Interfaces / contracts (executor must match)

### Prisma `RefreshToken` model
```prisma
model RefreshToken {
  id        String    @id @default(uuid())
  userId    String
  familyId  String    @default(uuid())
  tokenHash String    @unique
  expiresAt DateTime
  rotatedAt DateTime?
  revokedAt DateTime?
  createdAt DateTime  @default(now())
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@index([familyId])
  @@index([expiresAt])
}
```
Add to `User`: `refreshTokens RefreshToken[]`.

### Redis key formats
- OTP record: `pawcareright:otp:<hmacHex(email)>` → JSON string `{ "codeHash": <hmacHex(code)>, "attempts": <int> }`, TTL `OTP_TTL_SECONDS` (600). `<hmacHex>` = `createHmac("sha256", OTP_HMAC_SECRET).update(x).digest("hex")`. Email normalized (`trim().toLowerCase()`) before hashing so request/verify keys match.
- Rate-limit counter: `pawcareright:rl:otp:<ip>` → integer via `INCR`, `EXPIRE 60` set only when the counter is first created (`count === 1`).

### Token formats / claims
- **OTP code**: 6-digit numeric, `randomInt(0, 1_000_000)` → `String(n).padStart(6, "0")`. Never stored plaintext (HMAC only).
- **Refresh token**: opaque `randomBytes(32).toString("base64url")` (256-bit). Returned to client raw; stored only as `sha256(token)` hex in `RefreshToken.tokenHash`. NOT a JWT.
- **Access token**: JWT (HS256) signed by `@nestjs/jwt`; claims **minimal**: `sub = user.id` (+ standard `iat`/`exp`), `expiresIn = ACCESS_TOKEN_TTL` (15m). No email/household in the JWT (verify response returns those separately; T015 loads the user from `sub`).

### Endpoint request/response shapes (all under `/v1/auth`)
- `POST /auth/otp/request` — body `{ email: string }` (`@IsEmail`). Guard first. Always `200 { ok: true }` (even for unknown/never-seen emails). Body-shape failures still `400 VALIDATION_FAILED` via the global pipe (shape validation ≠ enumeration). Rate-limit overflow → `429 RATE_LIMITED` with `Retry-After: 60`.
- `POST /auth/otp/verify` — body `{ email: string, code: string }` (`@IsEmail`; `@Matches(/^\d{6}$/)`). Success `200 { accessToken, refreshToken, user: { id, email }, householdId }`. Failure (wrong / expired / too-many-attempts / unknown) → `401 UNAUTHORIZED`, **reason not distinguished**.
- `POST /auth/refresh` — body `{ refreshToken: string }` (`@IsString @IsNotEmpty`). Success `200 { accessToken, refreshToken, user: { id, email }, householdId }` (new access + new refresh in same family). Invalid/expired/reused → `401 UNAUTHORIZED`.
- `POST /auth/logout` — body `{ refreshToken: string }`. Always `200 { ok: true }` (idempotent — unknown token still 200, no probing).

### Rotation / reuse / logout semantics (`RefreshTokenService`)
- `issue(userId, familyId?)`: create row `{ tokenHash: sha256(token), userId, familyId: familyId ?? new uuid, expiresAt: now + 30d }`; return `{ token, familyId }`.
- Fresh login (`verifyOtp`) calls `issue(userId)` with NO familyId → brand-new family.
- `rotate(presentedToken)` (in a `$transaction`):
  1. `row = findUnique({ tokenHash: sha256(presentedToken) })`; if `null` → **401**.
  2. if `row.revokedAt != null || row.rotatedAt != null` → **reuse/theft**: `updateMany` set `revokedAt = now` on all `familyId` rows where `revokedAt is null`; **401**.
  3. if `row.expiresAt <= now` → **401** (do not family-revoke on natural expiry).
  4. else valid: set `row.rotatedAt = now`; `issue(row.userId, row.familyId)` (same family); return `{ token, userId: row.userId, familyId }`.
- `revokeFamily(presentedToken)`: `row = findUnique(...)`; if `null` → return (idempotent no-op); else `updateMany` set `revokedAt = now` on family rows where `revokedAt is null`.

### Auto-provision (`AuthService.verifyOtp`, inside `prisma.$transaction`)
- `user = tx.user.findUnique({ email })`.
- If absent: `tx.user.create({ email, locale: DEFAULT_LOCALE, region: DEFAULT_REGION })` → `tx.household.create({ name: DEFAULT_HOUSEHOLD_NAME, ownerId: user.id })` → `tx.membership.create({ userId, householdId, role: "OWNER" })`; `householdId = household.id`.
- If present: `household = tx.household.findFirst({ where: { ownerId: user.id }, orderBy: { createdAt: "asc" } })`; `householdId = household.id`. If somehow no owned household (invariant break; social-auth users arrive T013/14) → throw `InternalServerErrorException` (defensive; do not silently create).

## Risks & the design decisions the planner made (for checker scrutiny)
1. **Storage split**: OTP codes → Redis (TTL-native, HMAC-hashed, single-use, attempt-capped); refresh tokens → Postgres `RefreshToken` (new migration). Decision that OTP HMAC secret and refresh-token SHA-256 are the right hash choices — see #6.
2. **Redis key prefix**: CLAUDE §1a's naming *table* shows `pawcareright-*` (hyphen). I chose colon-namespaced keys rooted at `pawcareright` (`pawcareright:otp:…`, `pawcareright:rl:otp:…`) — the standard Redis idiom AND exactly the key examples the task card uses. They still share the `pawcareright` root prefix. Flagged so the checker can confirm this reading of §1a.
3. **Rotation/reuse semantics**: presenting an already-rotated OR revoked token revokes the ENTIRE family and 401s (theft defense); natural expiry 401s without family-revoke; unknown token 401 on refresh but 200 on logout (no probing). This is a deliberate security-model choice.
4. **Rate-limit isolation in tests**: supertest cannot easily vary `req.ip`, so all requests share one IP counter. Chosen mechanism: `beforeEach` flushes `pawcareright:rl:*` so each test starts clean, and the 429 test fires 6 real sequential requests. No test-only header/backdoor in production code. Guard runs BEFORE the ValidationPipe (Nest guard/pipe order) so even malformed requests count toward the limit — acceptable/desirable for abuse control.
5. **`Retry-After` on 429**: included now (`RATE_LIMIT_WINDOW_SECONDS`) even though T017 owns the throttler — cheap, set on the response in the guard before throwing; the `AllExceptionsFilter` preserves already-set headers. Decision to pre-satisfy T017's later expectation.
6. **Hashing choices / no password-hash lib**: no passwords exist. Low-entropy 6-digit OTP is protected by HMAC-SHA256(server secret) + single-use + 5-attempt cap + 10-min TTL (≤5 guesses over 1e6 space). High-entropy 256-bit refresh token → plain SHA-256 (no salt) is cryptographically sufficient and enables the unique-index lookup. Therefore `node:crypto` only; NO bcrypt/argon2. `@nestjs/jwt@^11` is the one new dep (official Nest JWT tooling for the mandated stack — §2 rule 7 justification).
7. **Response types are local interfaces** (exported from `auth.service.ts`), not `packages/types` Zod schemas — matches the `HealthStatus` precedent (T006) and the card's exhaustive file list (zero `packages/types` churn); no client consumes auth yet. A later task can promote them if mobile/web need inference.
8. **Provisioning defaults**: `locale:"en"`, `region:"US"`, household name `"My Household"` as constants (matching seed conventions). `Accept-Language` parsing deliberately skipped to keep the surface minimal; profile locale can be set by a later task.
9. **OTP attempt counter non-atomicity**: the JSON record read-modify-write on `attempts` is not strictly atomic under concurrent verifies of the same email. Accepted: the 5-attempt cap within a 10-min TTL already makes brute-force infeasible; contention on a single email is negligible. Flagged rather than adding Lua/second-key complexity.
10. **`req.ip` trust**: uses Express default (`trust proxy` OFF). X-Forwarded-For handling is a deploy concern for T017/T106; not enabled here.
11. **Dev transport logging scoped to non-production** (`nodeEnv !== "production"`) so emails+codes are never logged together on a prod path — honors the safety-adjacent logging note.

## Amendment A1 (orchestrator, post-executor) — reuse-revocation rollback bug fixed
Executor attempt 1+2 (turn-limited continuation) delivered the full module, but the auth e2e caught a REAL security defect in `refresh-token.service.ts#rotate`: the reuse-detection family revocation ran INSIDE `prisma.$transaction` and was immediately followed by `throw UnauthorizedException` — the exception aborted the interactive transaction and ROLLED BACK the revocation, so a stolen-token reuse left the family alive (e2e: refresh with RT2 after RT1 reuse returned 200, expected 401).
**Fix (orchestrator, in-scope file):** lookup + reuse/expiry checks moved OUTSIDE the transaction; family revocation on reuse persists BEFORE throwing; the happy-path rotate is atomic via a conditional `updateMany({ id, rotatedAt: null, revokedAt: null })` whose `count===0` → 401 (concurrent-rotation race guard, strictly stronger than the plan's `update`). Unit spec updated to the conditional-write contract + a new race-loss test.
**Re-verified:** api 8 suites / 44 tests green (incl. `rotation: reusing a rotated refresh token revokes the family`); CI-strict `env -u DATABASE_URL -u REDIS_URL turbo run test` green; migrate status clean; seed idempotent; root sequential gates green.
