# Plan — T076: Restore & manage subscription

## Objective (from card)
Add a Restore-purchases action in Settings (outside the paywall), platform-correct
manage-subscription deep links, and a billing-issue banner driven by entitlement status.
Deliver mocked-SDK tests; the banner renders on a BILLING_ISSUE fixture (and not otherwise).

## Safety statement (PRODUCT_SPEC §5 / CLAUDE.md §7)
- This task touches ONLY billing/subscription surfaces. It does NOT go near any §5 safety
  surface: no `<VetDisclaimer/>`, no emergency interstitial, no hotline, no dosing, no
  "diagnos*"/drug/dose copy is added or altered anywhere. All new copy is factual billing
  prose (payment method / Premium / restore). No SAFETY-ESCALATION: implementable without
  weakening any safety surface.
- The banner and manage/restore rows live only on the Settings screen — structurally
  incapable of appearing on the check/intake/result/emergency path.

## Key decisions (each a design choice — scrutinize)
1. **Billing-issue signal = one new `billingIssue: boolean` on the server entitlement
   response, derived from the caller's OWN subscription row status.** The `Subscription`
   row already stores `status` (`billing_issue` written by T073's `computeSubscriptionUpdate`),
   but `GET /billing/entitlement` (`BillingEntitlement`) does NOT expose it today. I add a
   boolean (not the raw status string) because the client only needs "is there a payment
   problem to surface"; leaking the internal audit vocabulary would invite string-branching in
   the UI. `billingIssue` is TRUE only on the **own** branch when `own.status ===
   RC_WEBHOOK_STATUS.BILLING_ISSUE` (still entitling during grace — entitlement stays PREMIUM).
   It is FALSE on the **family** and **none** branches: a family beneficiary cannot fix another
   member's payment method, and a lapsed user sees the paywall, not this banner. The server
   mirror is the single reliable source (RC `customerInfo` billing state is less dependable —
   foundation note). This is the ONE primary decision.
2. **Manage-subscription URL: RC `customerInfo.managementURL` when present, else a
   platform-correct fallback.** RC returns the store-correct URL when available; when absent
   (older SDK / sandbox / native module absent in Expo Go), fall back via `platformOS`:
   iOS → `https://apps.apple.com/account/subscriptions`; Android →
   `https://play.google.com/store/account/subscriptions?package=com.pawcareright.app`
   (§1a bundle id — an identifier, not the display name, so hardcoding the store URL is
   allowed). URL resolution is a **pure** function (`resolveManageSubscriptionUrl`) so the
   per-platform / managementURL-present-vs-absent matrix is unit-testable without `Linking`.
3. **Banner dismissal = per-session, in-memory (non-persisted zustand), never persisted.**
   It is informational; dismiss hides it for the session and it returns next launch if the
   issue persists (mirrors the T075 `upsell-store` pattern). Non-persisted also means no
   secure-store / migration concerns.
4. **Manage-row + banner both read the server `useEntitlement()` query (single source).**
   Manage row is visible when `entitlement.entitled === true` (true during a billing-issue
   grace, so manage + banner co-appear — both help fix payment). Restore row is ALWAYS
   visible. Undefined/loading entitlement → manage row and banner hidden (fail-safe).
5. **Restore-from-Settings reuses the T074 `restorePurchases()` service verbatim** with its own
   Settings-row UI + notice state — no duplication of restore logic, no change to the paywall.

## Risks & edge cases (for the checker)
- **R1 (decision 1) — family beneficiaries never see the banner.** By design: `billingIssue`
  is own-branch only. A family member whose owner has a payment problem sees no banner (they
  can't fix it); the owner does. If product later wants a family-wide notice, extend then, not
  speculatively.
- **R2 — schema change is breaking for every `BillingEntitlement` literal.** Enumerated and
  updated: `pickEntitlement` (3 returns), `billing.service.spec.ts`, `billing.e2e-spec.ts`,
  `quota/entitlement.spec.ts` helper + 2 mock literals, types spec payloads. `entitlementFromBilling`
  reads only `.entitled`, so quota behavior is unaffected.
- **R3 — no store app / managementURL null on sandbox.** `openManageSubscription` always
  resolves to a valid https store URL via the platform fallback even when `fetchCustomerInfo`
  returns `null` (Expo Go / native absent) — `Linking.openURL` opens the web equivalent; never
  throws unhandled (wrap the `openURL` call to swallow rejection).
- **R4 — Android fallback SKU.** We omit `sku` (unknown without customerInfo) and keep only
  `package=com.pawcareright.app`; Play still resolves the account subscriptions page. Bundle id
  is hardcoded per §1a (identifier, not display name) — documented drift point if the id changes
  at C3.
- **R5 — stale billing status after the user fixes payment.** `useEntitlement` reads the server
  mirror, refreshed by RC's next webhook (T073). Until the RENEWAL webhook lands, the banner may
  linger; per-session dismiss (decision 3) lets the user clear it immediately. Acceptable —
  informational, non-blocking.
- **R6 — settings entitlement loading/error state.** Undefined/error `useEntitlement` → banner
  and manage row hidden (fail-safe: never show a billing warning on unknown data). No dedicated
  loading/error UI added (over-engineering for an informational row).
- **R7 — settings-premium-entry.test.tsx renders the banner** (which calls `useEntitlement`);
  without mocking `billing-api` it would throw for lack of a QueryClient. Mock added.
