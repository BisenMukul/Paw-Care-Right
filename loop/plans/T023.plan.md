# Plan â€” T023: Photo upload pipeline (L)

## Objective (from card)
Add `POST /v1/pets/:id/photo-upload-url` (presigned PUT, content-type/size constrained) and `POST /v1/pets/:id/photo-upload-confirm` (enqueues a `pawcareright-images` BullMQ job). The worker uses `sharp` to write a 1600px main + 320px thumb with EXIF stripped, then writes the main key back to `Pet.photoKey`. Introduce a MinIO/S3 storage-service abstraction. Prove it end-to-end with a real-MinIO worker integration test.

## Design summary / three new subsystems
1. **Storage abstraction** (`src/storage/`) â€” thin S3 wrapper over `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`, `forcePathStyle: true` for MinIO.
2. **BullMQ** (`src/queue/` root config + `src/workers/` processor) â€” queue `pawcareright-images`, producer in the confirm endpoint, in-process `@Processor` consumer (same Nest app â€” see R1).
3. **sharp** image processing inside the processor (resize + EXIF strip via default metadata stripping + `.rotate()`).

No Safety Policy surface is touched (no AI output, no disclaimers, no dosing, no emergency copy). Not a SAFETY-ESCALATION.

- Processor `process(job: Job<ImagesJobData>)`:
  ```

  const src = await storage.getObject(key);
  const main = await sharp(src).rotate()
    .resize(1600, 1600, { fit: "inside", withoutEnlargement: true }).jpeg().toBuffer();

  // EXIF orientation then drops the tag â†’ EXIF fully gone.

  await storage.putObject(thumbKey, thumb, "image/jpeg");

  ```

## Risks & the design decisions the planner made (checker: scrutinize these)
- **R1 (same-process processor):** processor registered via `@nestjs/bullmq` `@Processor` inside the SAME Nest app (one process), not a separate worker entrypoint. Matches "simplest now"; a dedicated `apps/api/src/workers` process is a later deploy concern. Consequence: the e2e's in-app worker consumes the job â€” which is exactly what makes the real round-trip test possible.
- **R2 (BullMQ owns its Redis connection):** a dedicated `IORedis(redisUrl, { maxRetriesPerRequest: null })` for BullMQ; the existing `RedisService` client is untouched and unshared. Deviating (reusing it) breaks BullMQ blocking commands.
- **R3 (sharp `^0.33` pin + runtime EXIF fixture):** the EXIF assertion depends on `sharp.withExif`, added in 0.33.0. If unavailable, the fallback is a committed static fixture (not used now). sharp is a native module â€” CI must fetch its linux prebuilt binary during `pnpm i` (ubuntu-latest/node22 â€” supported).
- **R4 (presign size/type enforcement â€” exact mechanism):** the AC "rejected at presign" is satisfied **at the endpoint** by the DTO: `@IsIn(ALLOWED_CONTENT_TYPES)` + `@Max(5MB)` â†’ 400 `VALIDATION_FAILED` BEFORE any URL is signed. The signed URL binds `ContentType` only; content-length is deliberately NOT signed (a plain presigned PUT cannot strictly enforce max bytes without forcing the client to send an exact pre-declared length). The real byte/format check is the worker: sharp rejects non-images and only the resized renditions are ever persisted. This is the one substantive interpretation of the AC â€” flagged for review.
- **R5 (keep original + failed-set as DLQ):** original rendition is retained (enables reprocessing; storage-cost/lifecycle is a deploy task). "Dead-letter queue" is realized as `attempts: 3` + exponential backoff + `removeOnFail: false` (failed jobs retained in BullMQ's failed set) â€” the minimal satisfaction of CLAUDE.md Â§6's intent; a distinct DLQ topic is out of scope.
- **R6 (CI cross-task edit):** editing `.github/workflows/ci.yml` (a T009 file) is required so the acceptance-critical suite runs honestly in CI rather than self-skipping. This is the sole out-of-current-task file edited and must carry a `loop/journal.md` justification. Alternative (skip-if-MinIO-unreachable guard) was rejected because it would leave the AC unverified in CI.
- **R7 (response shapes kept local, not in `packages/types`):** presign/confirm responses are local interfaces + class-validator DTOs, mirroring the existing pets pattern (`PetResponse` is a local interface). This slightly departs from the "all shapes are Zod in `packages/types`" guideline but avoids expanding scope into the shared package; flagged so the checker can accept or reject.
- **R8 (`removeOnComplete` retained briefly):** completed jobs are kept (`age: 3600, count: 1000`) rather than removed immediately, so the e2e's `queue.getJob(jobId).waitUntilFinished(...)` cannot race a just-completed-and-purged job. Bounded retention, not unbounded growth.
- **R9 (`pnpm-lock.yaml` + coverage):** `pnpm i` regenerates the root lockfile (listed). New `*.service.ts` files enter the `collectCoverageFrom` glob and must clear the 80/70 thresholds â€” the unit specs for `storage.service`/`photos.service` are sized for that; `images.processor.ts` is intentionally named `.processor.ts` (outside the coverage glob) but is still directly unit-tested.
