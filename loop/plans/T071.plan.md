## Key decisions (checker: scrutinize)
1. **Product-id scheme (¬ß1a naming).** Three store products, brand-prefixed snake_case, identical on iOS and Android (no `+`, no spaces ‚Äî illegal in identifiers): `pawcareright_monthly`, `pawcareright_annual`, `pawcareright_family_annual`. RevenueCat entitlement id = `plus`; offering id = `default`; package identifiers = the RC reserved `$rc_monthly` and `$rc_annual` for monthly/annual, plus a **custom** package `family` for the family-annual product. These exact strings are the single source of truth in `src/billing/products.ts` and are reproduced verbatim in `docs/store-setup.md`; a unit test asserts constants === documented values so the two can never drift.
2. **Lazy-native guard mirrors the safe-storage / MMKV reconciliation pattern.** `react-native-purchases` is a native module that **cannot load in Expo Go or the jest container**. The service never imports it at module top level. It uses an **injectable loader** (default: a `try { require("react-native-purchases").default } catch { return null }`), exactly like `createSafeStorage({ createMmkv })` + the `eslint-disable @typescript-eslint/no-require-imports // JUSTIFIED:` lazy-require pattern in `active-pet-store.ts`. Loader returns `null` ‚áí the whole service **no-ops gracefully** (no throw, no crash), so boot succeeds in Expo Go/dev.
3. **Configure anonymous, then `logIn` on auth-resolve ‚Äî never before auth resolves.** `initPurchases()` calls `Purchases.configure({ apiKey, appUserID: null })` (anonymous) once. Purchaser identity is applied **only** after the auth store leaves `restoring`: `status==="signedIn"` ‚áí `logIn(user.id)`; `status==="signedOut"` ‚áí `logOut()`. A module-level `lastAppUserId` **dedups** so a `logIn` fires at most once per userId and `logOut` no-ops when already anonymous (zustand `subscribe` fires on every state change). `identifyPurchaser`/`resetPurchaser` are hard no-ops if configure has not run or the native module is absent.
4. **Idempotent configure.** A module-level `configured` flag makes a second `initPurchases()` a no-op (double-mount / Fast Refresh safe). RevenueCat warns on double-configure; the guard prevents it.
5. **Lifecycle wiring via a `usePurchasesInit()` hook mounted in `_layout.tsx`**, sibling to `useNetworkListener()` (called before the `restoring` early-return so hook order is stable). The hook calls `initPurchases()`, applies the current auth state once, subscribes to `useAuthStore`, and returns the unsubscribe. Reads via `useAuthStore.getState()`/`.subscribe()` (non-reactive) so it does not add re-renders.
6. **Public RC SDK keys via `EXPO_PUBLIC_*` with stub defaults**, mirroring `apiBaseUrl`/`googleClientId`: added to `app.config.js` `extra` (`revenueCatIosKey`/`revenueCatAndroidKey` ‚Üê `process.env.EXPO_PUBLIC_RC_IOS_KEY`/`_ANDROID_KEY` ?? a stub) and surfaced through `getConfig()`. RC SDK keys are **public** app-specific keys (safe under `EXPO_PUBLIC_`); the secret webhook auth is server-side and out of scope. `initPurchases()` selects the key by `Platform.OS`.
7. **New dependency `react-native-purchases` (¬ß2 rule 7 + ¬ß3 RevenueCat is locked).** Installed via `npx expo install react-native-purchases` (Expo SDK 57 pairing). No Expo config plugin ships with it (autolinked in a dev client), so `app.config.js` `plugins` is **not** changed. It is another native module ‚áí **dev-client rebuild required**; flag in the doc + journal with the one-line dependency justification.
8. **Global jest mock for `react-native-purchases`.** Once installed, any test transitively importing `_layout` would resolve the package; a stub in `jest.setup.ts` (default export with `configure/logIn/logOut` jest.fns) keeps the suite headless. The "native absent" behaviour is still tested by **injecting a null loader** into the service (decision 2), not by relying on module absence.

## Risks & edge cases (checker: scrutinize)
- **Expo Go / native absence (decision 2):** `require("react-native-purchases")` throws in Expo Go and the jest container. The loader try/catch + `null` no-op path is the safety net; verified by the native-absent test injecting a null loader. Never import the package at module top level.
- **Stub-key boot:** RC's `configure` accepts any non-empty string offline (no network at configure time), so booting with `stub_ios_key`/`stub_android_key` succeeds; only `getOfferings` (T073) would fail against a stub, which is out of scope.
- **logIn before configure / double-configure (decisions 3‚Äì4):** `identify/reset` guard on `native !== null`; `initPurchases` guards on `configured`. Both covered by tests. `logIn/logOut` rejections are swallowed (best-effort) so an identity hiccup never crashes the app or blocks the auth flow.
- **Subscribe fires on every store change:** `lastAppUserId` dedup prevents repeated `logIn` on unrelated state updates (e.g. `markPushAsked`); tested via the dedup case.
- **Auth-resolve ordering:** identity is applied from `useAuthStore.getState()` on mount and via `subscribe`; while `status === "restoring"` neither branch runs, so identity never fires before auth resolves (card requirement). The `_layout` splash early-return does not skip the hook because it is called before the return.
- **Native-module count grows (decision 7):** another dev-client rebuild is required; flagged in `docs/store-setup.md` ¬ß8 and the journal. Not runnable in Expo Go.
- **¬ß1a drift:** the product-id constants test pins the exact strings to the doc; if the executor changes either, the test fails ‚Äî intentional guard.
- **`config.ts`/`app.config.js` default parity:** the stub defaults must be byte-identical in both files so jest (unpopulated `expoConfig`) and a real build behave the same, exactly as `apiBaseUrl` does today.
