# Plan — T058: Notification preferences + quiet hours

## Objective (from card)
Give each user per-notification-type on/off switches and a local-timezone "quiet window"; the push sender must (a) suppress types the user turned off and (b) DEFER pushes that fall inside the quiet window to the window's end. Ships: a Prisma `UserNotificationPrefs` model + migration, a pure DST-correct `computeDeferUntil` helper (midnight-straddling), sender integration, a user-scoped settings API (`GET`/`PUT /me/notification-prefs`), and a mobile prefs screen. Unit tests for the defer math (straddling midnight) and sender tests proving prefs are respected are the acceptance criteria.

## Safety statement
This task touches **notification delivery scheduling only** — it renders no AI output and does not go near any `PRODUCT_SPEC §5` safety surface: no disclaimers, no emergency escalation, no medication dosing copy. The only content the push carries is the reminder titles the user themselves created (unchanged from T057); this task merely gates/reschedules those care reminders. Quiet hours and type-off therefore cannot weaken §5 — they can only delay a **care reminder** (non-emergency by definition; ARCHITECTURE places emergency handling in the in-app interstitial, T049, not in push).
- **Forward safety note (nothing to build now, flag for the checker and future tasks):** the entire notification pipeline today emits ONE push kind — `data.kind === "reminder"` (see `push-sender.service.ts`). **If a future task ever introduces an emergency/red-flag push, that push MUST bypass BOTH quiet-hours deferral AND type-off suppression** (an emergency must never be silenced or delayed by a user preference — CLAUDE §7 rule 4, "fails upward"). This plan builds the gate so it is trivially bypassable later: the defer/suppress logic keys off `kind === "reminder"` semantics and lives in one choke point (`sendForEvent`), so a later emergency path can skip it. No emergency push exists in the repo at T058, so §5 is not weakened. The executor must NOT make the gate unconditional in a way that would also catch a future emergency kind.
- No user-facing copy in this task states or implies medical advice; mobile strings are care-scheduling labels only.

## Key decisions (numbered; re-listed under "Risks & decisions")
1. **Data model = a dedicated 1:1 `UserNotificationPrefs` table** (PK `userId`), NOT columns on `User`. Fields: `disabledTypes String[] @default([])` (the *off* set — default empty ⇒ all types on), `quietStart String?`, `quietEnd String?`, `timezone String?` (the quiet window is an all-or-nothing trio: either all three set or all null ⇒ no quiet hours), plus `createdAt`/`updatedAt`. A separate table keeps `User` lean, is created lazily via `upsert`, and a missing row means "all defaults" (so existing users need no backfill). Requires one migration.
2. **Quiet-hours DEFER is decided per-recipient-user BEFORE the collapse claim, and re-enqueues a delayed copy of the push job (never holds a claim across the window).** In `sendForEvent`'s per-user loop, if the user is inside their quiet window we do NOT take the Redis collapse claim; we enqueue a delayed `PUSH_QUEUE` job for the same `reminderEventId` at the window-end instant and `continue`. Collapse therefore re-evaluates cleanly at window-end: all of a user's same-minute events defer identically (same wall clock, same tz) and, when their deferred jobs fire, the first to run claims and counts every still-eligible enabled event for that user's original due-minute; the rest lose the claim and skip. This is the "never-claim-while-deferred, let the deferred run re-claim" branch — chosen over "winner claims then defers" (which would either hold a 1-hour claim blocking siblings, or need claim-release plumbing). Deferred job id = `` `${eventId}:defer:${userId}:${windowEndMinuteEpoch}` `` — unique per (event, user, window-end) so (a) it never collides with T056's original `jobId = eventId`, and (b) re-deferrals of the same user to the same window-end are idempotent (BullMQ dedups the id).
3. **`computeDeferUntil(prefs, now)` is a pure, DST-correct helper** in `apps/api/src/workers/quiet-hours.ts`. Returns `Date | null`: `null` when there is no quiet window OR `now` is outside it (deliver immediately); otherwise the UTC instant of the next window-end. Midnight-straddling windows (`start > end`, e.g. `22:00–07:00`) and DST are handled via the same two-pass wall-clock→UTC fixed point used in `reminders/next-fire-at.ts` (that file's tz helpers are private, so this module carries its own small copy — Risk 3). Window is **start-inclusive, end-exclusive** (`now == end` ⇒ deliver; `now == start` ⇒ defer).
4. **API = a new user-scoped `notifications` module** exposing `GET /me/notification-prefs` and `PUT /me/notification-prefs` (auth-only, no household scope — prefs are per-user). Quiet hours are modeled to the client as a nested `quietHours: { start, end, timezone } | null` object (clean all-or-nothing validation + a natural mobile shape); the service maps that object to/from the three flat DB columns. Shared Zod shapes live in `packages/types/src/notification-prefs.ts`; the input DTO uses class-validator (repo convention, mirrors `create-reminder.dto.ts`).
5. **Mobile = a dedicated sub-screen** `app/settings/notifications.tsx` reached from the existing Settings tab; toggles are rendered **schema-driven from `REMINDER_TYPES`** (so new types appear automatically), plus a quiet-hours enable switch and in-house HH:mm start/end pickers (fixed 30-min option list — **no new dependency**), timezone defaulted from `Intl.DateTimeFormat().resolvedOptions().timeZone`. Wired via TanStack Query hooks mirroring `households-api.ts`.

## Defer semantics (`computeDeferUntil({ quietStart, quietEnd, timezone }, now)`)
- If `quietStart`/`quietEnd`/`timezone` is null (no quiet window) → `null`.
- Compute `now`'s wall clock `hh:mm` in `timezone`; let `s = parse(quietStart)`, `e = parse(quietEnd)`, `t = parse(now-wall)` (minutes-since-midnight).
- **In-window test:** non-straddling (`s < e`): `s <= t < e`. Straddling (`s > e`, e.g. 22:00–07:00): `t >= s || t < e`. (`s === e` ⇒ treat as never-in-window → `null`.)
- If not in window → `null` (deliver now).
- If in window → the next window-**end** wall-clock instant: for straddling, if `t >= s` (evening side) the end is `e` on **tomorrow's** calendar day, else (`t < e`, early-morning side) `e` **today**; for non-straddling the end is `e` **today**. Convert that `{y,m,d, e.hh, e.mm}` wall clock in `timezone` to a UTC `Date` via the two-pass DST fixed point. Return it.
- Boundaries: `t === s` → defer; `t === e` → not in window → deliver (end exclusive).
- Sender uses `delay = deferUntil.getTime() - now.getTime()` (both from the single captured `now`); jobId per decision 2.

## Risks & decisions (numbered — for the checker to scrutinize)
1. **New table + migration (decision 1).** A schema change + migration is required (unlike T057). Missing row = defaults, so no data backfill. Checker: confirm the migration only adds `UserNotificationPrefs` (+ FK) and nothing else, and that `onDelete: Cascade` matches the `User`-owned-child pattern (`Device`, `Membership`).
2. **Defer-before-claim, re-enqueue delayed job (decision 2) — the main design bet.** Race honesty: because quiet-hours is a deterministic function of wall clock, all of a user's same-minute events defer together and never take a claim while deferred; collapse re-forms at window-end. The jobId `` `${eventId}:defer:${userId}:${windowEndMinuteEpoch}` `` avoids collision with T056's `jobId=eventId` and makes re-deferral idempotent. Residual: an event that (pathologically) becomes eligible for the same original minute *after* the window-end run could miss the collapse — accepted (non-emergency, still visible in-app; same class as T057 Risk 2). Checker: confirm no path double-pushes a user; confirm the deferred job re-runs `sendForEvent(eventId)` and that at window-end `computeDeferUntil` returns null so it proceeds; confirm the claim is NOT taken on the deferred branch.
3. **Duplicated tz math in `quiet-hours.ts` (decision 3).** `next-fire-at.ts`'s zoned-wall-clock↔UTC helpers are private; rather than widen that module's public surface (and risk its tests), this plan copies the ~30 lines. Risk: two copies can drift. Mitigation: `quiet-hours.spec.ts` includes a DST case that pins correctness independently. Checker: verify the copy is correct (two-pass fixed point) and that `next-fire-at.ts` is untouched.
4. **Type-off suppression operates on the collapse GROUP, not the triggering event.** A disabled-type event may still legitimately win the claim and send the enabled siblings; an all-disabled group suppresses with no prune/no send/no defer. Checker: confirm suppression has zero side effects (no `deleteMany`, no `pushQueue.add`) and that a mixed group's count reflects only enabled events.
5. **Quiet window is an all-or-nothing trio, exposed as a nested `quietHours` object.** Partial input (e.g. start without tz) is rejected by `@ValidateNested`/DTO. Storage is three flat nullable columns mapped in the service. Checker: confirm the DTO rejects partial quiet-hours and that `quietHours: null` on PUT clears all three columns.
6. **Determinism of sender defer tests.** They rely on `jest.setSystemTime` to fix `new Date()`. Checker: confirm timers are reset (`useRealTimers`/`clearAllMocks`) so other tests are unaffected, and that non-defer tests do not depend on real wall-clock quiet status (default = no prefs row = never deferred).
