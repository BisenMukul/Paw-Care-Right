# Plan — T018: Mobile auth flow (apps/mobile)

## Objective (from card)
Build the mobile authentication flow in `apps/mobile`: screens welcome → email entry → OTP entry (6 auto-advance cells) → done; Apple (`expo-apple-authentication`) + Google (`expo-auth-session`) social buttons; tokens stored ONLY in `expo-secure-store` (never AsyncStorage); a zustand auth state machine (`restoring | signedOut | signedIn`) with restore-time silent refresh; signed-out `(auth)` vs signed-in `(tabs)` router gating; and JIT device push-token registration (rationale screen first) after sign-in. HTTP goes through `@pawcareright/api-client`. All root gates stay green; §1a display-name-via-constant and §6 mobile rules hold.

## Headless-verification reality (read first — mirrors T008)
This is a headless Linux container: **no iOS/Android device, no emulator, no Expo Go, no live API needed** (all tests mock `expo-secure-store`, the native social/push modules, and `fetch`). Two AC clauses cannot be device-run here and are handled explicitly:
1. **"manual script in journal: cold-start auto-login verified"** — CANNOT run on-device here. The plan provides (a) a written **manual device script** (verbatim text in the Tests section) to be journaled now and executed at the **M1 device milestone**, and (b) an **automated stand-in**: `auth-store.test.ts` unit tests that drive `restore()` against a mocked SecureStore + mocked refresh endpoint (no-token→signedOut; valid refresh→signedIn; failed refresh→signedOut+cleared). State this to the checker; do NOT expect a device boot in this container.
2. **Social sign-in native flows (Apple/Google)** only fully work in a dev build on-device. Headless, the native modules are jest-mocked; tests assert the buttons render and invoke the mocked flow → `store.socialSignIn`. Real Apple entitlements / Google client ids are a build/store concern (C3), out of scope here.

The linchpin automated proofs are the RNTL component tests (OTP behavior + error states), the store unit tests, and the AsyncStorage-absence assertion. **RNTL v14 `render` is async — every test must `await render(...)`** (T008 Amendment A1).

## Interfaces/contracts (executor must match)
**Auth store (`src/auth/auth-store.ts`)**
```ts
type AuthStatus = "restoring" | "signedOut" | "signedIn";
interface AuthUser { id: string; email: string; }
interface AuthState {
  status: AuthStatus;              // initial: "restoring"
  user: AuthUser | null;
  householdId: string | null;
  accessToken: string | null;      // in-memory only; source for getAuthToken
  pushAsked: boolean;              // JIT push flag
  restore(): Promise<void>;
  requestOtp(email: string): Promise<void>;
  verifyOtp(email: string, code: string): Promise<void>;      // throws ApiError on 401 (screen catches)
  socialSignIn(provider: "apple" | "google", identityToken: string): Promise<void>;
  signOut(): Promise<void>;
  markPushAsked(): void;
}
```
`restore()`: read refresh token from SecureStore → none ⇒ `signedOut`; else try `authApi.refresh(token)` → `saveTokens` + `signedIn`; on throw → `clearTokens` + `signedOut`. `verifyOtp`/`socialSignIn`: on success `saveTokens` + set `signedIn`/user/householdId/accessToken; errors propagate. `signOut`: best-effort `authApi.logout(refresh)` (swallow errors) → `clearTokens` → reset.

**AuthTokens (local, mirrors api `AuthService.AuthTokens`; no shared schema exists)**
```ts
interface AuthTokens { accessToken: string; refreshToken: string; user: { id: string; email: string }; householdId: string; }
```
**Endpoints (baseUrl origin + these paths):** `POST /v1/auth/otp/request {email}`→`{ok:true}`; `POST /v1/auth/otp/verify {email,code}`→`AuthTokens` (401 on bad code); `POST /v1/auth/social {provider,identityToken}`→`AuthTokens`; `POST /v1/auth/refresh {refreshToken}`→`AuthTokens`; `POST /v1/auth/logout {refreshToken}`→`{ok:true}`; `POST /v1/devices {expoPushToken,platform}`→device (200). `platform` ∈ `"ios"|"android"`; `expoPushToken` matches `Expo(nent)?PushToken[...]`.

**SecureStore keys:** `pawcareright.auth.accessToken`, `pawcareright.auth.refreshToken` (SecureStore-legal chars).

**api-client usage:** `createApiClient({ baseUrl, getAuthToken })` → `.post<T>(path, body)`; throws `ApiError` with `.httpStatus`/`.code` (import `{ createApiClient, ApiError, isApiError }` from `@pawcareright/api-client`).

**OtpInput props:** `{ value: string; onChangeText(next): void; onComplete(code): void; length?: number /*=6*/; hasError?: boolean; testID?: string }`.

## Risks & the design decisions the planner made (checker: scrutinize each)
- **R1 — Native module versions via `expo install` (STOP condition).** All Expo native deps MUST be pinned by `expo install` for SDK 57, not hand-picked. If `expo install` cannot resolve a compatible version for SDK 57 (e.g. `expo-apple-authentication`/`expo-auth-session`) → BLOCK, do not free-float.
- **R2 — Router structure: keep `(tabs)` as the signed-in group, do NOT nest under `(app)/`.** Chosen for minimum churn: the T008 tab files and their relative imports stay untouched and `app-title.test.tsx` stays green. `(auth)` is added as a sibling group; `push-rationale` is a root-level screen allowed while `signedIn`; the root `_layout.tsx` effect gates via `useSegments()`+`router.replace()`. Trade-off vs. the card's suggested `(app)/(tabs)` nesting: gating lives in one root effect rather than a wrapper layout. If the executor finds the segment-based gate races expo-router mount ordering, fallback: use `<Stack.Protected guard>` (SDK 57) or an `(app)/_layout.tsx` wrapper — flag which path was taken. Router-gate correctness is verified by typecheck + store unit tests + the M1 manual script (RNTL can't meaningfully drive the root redirect headlessly).
- **R3 — SecureStore-only, both tokens persisted, access also in memory (STOP condition).** Decision: persist BOTH access + refresh in SecureStore AND cache access in the store for `getAuthToken`; `restore()` always re-refreshes from the refresh token (simpler + safer than trusting a possibly-expired cached access token). `expo-secure-store` is the ONLY storage module. **No AsyncStorage dependency is added, ever** — if any transitively-required flow seems to need it → BLOCK, do not add it.
- **R4 — AsyncStorage-absence assertion mechanism.** Primary: `expect(() => require.resolve("@react-native-async-storage/async-storage")).toThrow()` (pnpm strict isolation ⇒ undeclared dep is unresolvable) PLUS a positive assertion that verify writes both tokens via `SecureStore.setItemAsync`. Secondary: the `grep` gate in Commands (0 hits). This triangulates "tokens live in SecureStore, never AsyncStorage" mechanically.
- **R5 — Social buttons degrade headless; logic lives in the component.** Apple/Google flow code is inside `social-auth-buttons.tsx` (not separate helper modules) and the native modules are jest-mocked. Google client id + Apple entitlements are placeholders/deferred to C3; the Google button is disabled when `googleClientId` is empty / `request` is null. Tests assert render + mocked-flow → `store.socialSignIn`. Deep native build config is out of scope.
- **R6 — Push registration is JIT, rationale-first, failure-tolerant (§6, §5-adjacent but NOT a safety surface).** Only device-token REGISTRATION is built (foreground/background/cold-start handling is a later task). Permission is requested only from the rationale screen after explaining why; denial/errors are swallowed and the user proceeds to tabs. `projectId` for `getExpoPushTokenAsync` is read defensively (may be undefined in dev) — undefined ⇒ registration returns `"error"`, no crash.
- **R7 — `app.config.ts`: add Apple plugin only, not notifications.** `expo-apple-authentication` plugin + `ios.usesAppleSignIn` are added (Apple-recommended, harmless to config eval). No `expo-notifications` plugin block (registration works without it; avoids config-eval/asset requirements). Fallback: if `expo start`/config eval fails on the Apple plugin (module not installed as a config plugin) → drop the plugin entry (keep the runtime dep) and flag.
- **R8 — api-client consumed via built dist (STOP condition).** `@pawcareright/api-client` resolves through its `exports` map to `dist/index.js` (ESM) — its dist must be built before mobile typecheck/test (Step 2), mirroring the config-package precedent. If Metro/jest cannot resolve the workspace ESM package cleanly (transform/exports issue) → BLOCK rather than duplicating the client. NO `ApiQueryProvider`/TanStack wiring and NO 401→refresh interceptor this task (both are T019) — adding them is out-of-scope scope-creep (§2 rule 2).
- **R9 — Local `AuthTokens` interface is not a forbidden duplicate.** Verified `packages/types` has no auth-response Zod schema (grep clean); the api defines `AuthTokens` as a plain TS interface in `auth.service.ts`, not a shared schema. A minimal local interface in `auth-api.ts` is therefore legitimate. If a shared schema is later found, the executor should import it instead — flag if so.
- **New-dependency justifications (§2 rule 7, one line each; orchestrator journals):** `zustand` = mandated client-state lib (§3); `@pawcareright/api-client` = the mandated shared HTTP/Query layer (§3), now consumed by mobile; `expo-secure-store` = the mandated token store (§6, SecureStore-only); `expo-notifications` = push-token registration (card); `expo-apple-authentication` + `expo-auth-session` (+ `expo-web-browser`/`expo-crypto` peers) = the card's Apple/Google sign-in. All locked-stack or card-mandated, versions pinned by `expo install`.
- **Safety (PRODUCT_SPEC §5):** T018 is an authentication surface — no AI output, no `<VetDisclaimer/>` surface, no emergency-escalation copy, no medication/dosing content. No Safety Policy surface is touched or weakened. No SAFETY-ESCALATION.
