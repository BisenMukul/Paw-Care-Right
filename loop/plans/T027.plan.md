# Plan — T027: Multi-pet switcher

## Objective (from card)
Add a header pet switcher (static when 1 pet, avatar row + dropdown when >1), persist the
active pet per device (MMKV), and expose ONE `useActivePet()` hook that every pet-scoped
screen reads. Home renders the switcher + the active pet's entry; deep-linkable `[id]` route
stays param-driven.

## Reconciliation note (card assumption is wrong — read first)
The card's required reading says `src/api/pets-api.ts` already exports a `usePets` list hook.
It does NOT — that file has only `usePet(id)`, `petsKeys.all`, `petsKeys.detail`, `useCreatePet`,
`uploadPetPhoto`. The backend endpoint `GET /v1/pets` (`pets.controller.ts` `findAll`) already
exists and returns `Pet[]`. So this plan ADDS a minimal client-side `usePets()` list hook that
consumes the existing endpoint. This is a mobile client hook, not a backend/API change — the
constraint "NO api changes" (no `apps/api`, no DTO/endpoint changes) is preserved. See Risk D1.

## Risks & the design decisions the planner made (scrutinize)
- **D1 (main):** Added a client `usePets()` list hook because the card wrongly assumed it existed.
  It consumes the pre-existing `GET /v1/pets`; no backend/DTO change → within "NO api changes". If
  the checker reads "NO api changes" as "do not edit `pets-api.ts`", the task is otherwise
  unbuildable (the switcher needs the pets list) — this is the minimal reconciliation.
- **D2:** Auto-heal reconciliation lives solely in `useActivePet` via `useEffect`, not in the store
  (keeps the store a dumb persisted value; the hook owns list-vs-active consistency).
- **D3:** Dropdown implemented with RN `Modal`/`Pressable` (no action-sheet dependency), and the
  avatar row is also directly tappable — both paths call `setActivePet`. Chosen as the minimal
  composition literally satisfying "header dropdown + horizontal avatars when >1".
- **D4:** Avatars are placeholder circles with the pet's first initial (matching
  `pet-header-card.tsx`'s placeholder), because no mobile photoKey→URL resolver exists and adding
  one would be out-of-scope API work. Honors "avatars with placeholder"; real photos deferred.
- **D5:** The AC phrase "dependent query keys" is verified by mocking `usePet` (whose key is
  `petsKeys.detail(id)`) and asserting it is invoked with the newly-selected id after a switch,
  plus asserting the rendered active-pet name changes. This proves switch → new detail key without
  standing up a live QueryClient.
- **D6:** `usePets` reuses `petsKeys.all` (no new key); existing `invalidateQueries(petsKeys.all)`
  calls (in `useCreatePet`/`useAcceptInvite`) already refresh it by prefix — no extra wiring.
- Prior 14 suites / 79 tests must stay green: Home's edit adds states but keeps `home-add-pet-cta`
  reachable and `strings.home.body`; verify the existing Home test (if any) and add-pet tests are
  unaffected.
