# Plan — T006: NestJS scaffold (apps/api)

## Objective (from card)
Stand up the first app `apps/api` as a NestJS 11 service with: a Zod-env config module (reusing `@pawcareright/config` `defineEnv`), a global `ValidationPipe`, a global exception filter emitting `{error:{code,message,requestId}}` (using `@pawcareright/types` `ErrorCode`/`errorResponseSchema`), request-id middleware, Swagger at `/docs`, `GET /v1/health` that pings Postgres + Redis, a wired `PrismaService` (empty schema OK), and Jest+Supertest tests. All turbo tasks (`dev`/`typecheck`/`lint`/`test`/`build`) must run from repo root.

## Risks & the design decisions the planner made (checker: scrutinize each)
- **R1 — NestJS 11 + CommonJS island in an ESM monorepo.** `apps/api` is NOT `type: module` and its tsconfig overrides the shared base to `module: CommonJS` + decorator metadata. Rationale: NestJS requires `emitDecoratorMetadata`/`experimentalDecorators` and ships CommonJS; the shared base (`ESNext`/`Bundler`, no decorators) cannot compile Nest. This override is confined to the api tsconfigs; the base file is untouched.
- **R2 — Config: custom module, not `@nestjs/config`.** Reuses `defineEnv(apiEnvSchema)` from `@pawcareright/config/env`. Trade-off: no automatic `.env` file loading — mitigated by giving `DATABASE_URL`/`REDIS_URL` local-docker **defaults** in the Zod schema, so `dev` and tests work against the running stack with zero `.env`. Alternative (`@nestjs/config`) rejected: extra dep + its defaults don't propagate to `process.env` for Prisma anyway.
- **R3 — Prisma reads its URL from injected config, not `process.env`.** `PrismaService` passes `datasourceUrl: config.databaseUrl` to `PrismaClient`, and `RedisService` uses `config.redisUrl`. This makes the health ping work deterministically regardless of shell env. `prisma generate` (postinstall) uses `env("DATABASE_URL")` in the datasource block but generate does not resolve the connection, so it succeeds without the var set. If a Prisma version demands the var at generate time, executor should prefix the command with the default URL and flag it — not remove the datasource block.
- **R4 — New dependencies (all locked-stack or its direct tooling; §2 rule 7 justifications, one line each, executor to journal via orchestrator):** `@nestjs/*`+`reflect-metadata`+`rxjs` = the mandated NestJS backend; `@nestjs/swagger` = mandated Swagger; `class-validator`/`class-transformer` = mandated DTO validation; `@prisma/client`+`prisma` = mandated ORM; `ioredis` = the Redis 7 client BullMQ itself depends on (mandated cache/queue layer); `supertest`+`@types/supertest` = mandated API integration testing; `@nestjs/testing`, `@nestjs/cli`/`@nestjs/schematics`, `@types/express` = required Nest build/test tooling. No non-stack packages added.
- **R5 — Single jest config runs unit + e2e; `moduleNameMapper` for `@pawcareright/config`.** House style keeps jest config in `package.json` (no `jest.config.*`). `@pawcareright/config` is consumed as **source TS** (no build step), which ts-jest won't transform through `node_modules`; the two `moduleNameMapper` entries redirect it to the in-repo `packages/config/src` files so ts-jest compiles them. `@pawcareright/types` needs no mapper — it is built (turbo `test` depends on `^build`) and resolves to its CommonJS `dist`. If a mapped path resolves wrong, executor stops rather than deleting strictness flags.
- **R6 — Test-only failing route.** `test/test-throw.controller.ts` is excluded from `tsconfig.build.json` and never imported by `AppModule`; the e2e spec injects it via `Test.createTestingModule({ imports:[AppModule], controllers:[TestThrowController] })` then applies `configureApp(app)`, so it inherits the `/v1` prefix + global pipe/filter without shipping in production `dist`. This is the cleanest way to assert the error envelope without a permanent dev route.
- **R7 — Swagger path vs global prefix.** `setGlobalPrefix("v1")` affects controllers; `SwaggerModule.setup("docs", …)` mounts at literal `/docs` (+ `/docs-json`) independent of the prefix, satisfying "Swagger at `/docs`" while health stays at `/v1/health`. No prefix-exclusion config needed.
- **Safety:** T006 is pure infrastructure scaffolding — no user-facing AI copy, disclaimers, emergency-escalation, or dosing surfaces are touched. No `PRODUCT_SPEC §5` interaction. No SAFETY-ESCALATION.

## Amendment A1 (orchestrator, post-executor attempt 1) — config becomes a built package
**Why:** Executor attempt 1 produced a green typecheck/lint/test but AC1 (`pnpm --filter api dev` serves) FAILED. Root cause: the api aliased `@pawcareright/config/env` to **source TS** (`../config/src/env/index.ts`). Because `defineEnv` is a runtime value, tsc pulled config source into `nest build`, hoisting `rootDir` to the repo root → emit landed at `dist/apps/api/src/main.js` instead of `dist/main.js`, so `nest start` (runs `node dist/main`) threw `MODULE_NOT_FOUND`. Even with correct layout, `node dist/main.js` would `require("@pawcareright/config/env")` → package `exports` → a `.ts` file Node cannot execute. `@pawcareright/types` never hit this because it is built to dist. T006 is the **first runtime consumer** of `@pawcareright/config`, which was source-only until now. Building it is the coherent, symmetric fix (mirrors the existing `@pawcareright/types` tsup setup). Turbo tasks all `dependsOn: ["^build"]`, so ordering is safe.

**Sanctioned scope expansion (supersedes the "do NOT touch packages/**" line for these two files only):**
- `packages/config/package.json` — add `"build": "tsup"` script; add `tsup@^8.5.1` devDep (reuse the exact version types pins — §2 rule 7: the mandated internal-package bundler, already in the repo); change the `.` and `./env` entries in `exports` to dist conditionals mirroring `packages/types` exactly (`{ "types": "./dist/<name>.d.ts", "import": "./dist/<name>.js", "require": "./dist/<name>.cjs" }`). Leave `./eslint`, `./prettier`, `./tsconfig.base.json` subpaths unchanged. Leave the `jest` block (reads `src`) unchanged.
- `packages/config/tsup.config.ts` — NEW. `entry: ["src/index.ts", "src/env/index.ts"]`, `format: ["esm","cjs"]`, `dts: true`, `sourcemap: true`, `clean: true`, `target: "es2022"` (mirror types' tsup config; two entries so `./env` gets its own dist chunk).

**api changes (replaces R5's source-mapping-only story):**
- Keep `apps/api/tsconfig.json` config aliases pointing at **source** (`../config/src/index.ts`, `../config/src/env/index.ts`) so `typecheck` and ts-jest resolve config types WITHOUT requiring config's dist to exist (noEmit — no nesting occurs on typecheck/test).
- In `apps/api/tsconfig.build.json`, **override** `compilerOptions.paths` to point config at **dist** (`../config/dist/index.d.ts`, `../config/dist/env/index.d.ts`). This stops `nest build`/`nest start` from compiling config source → clean `dist/main.js`; runtime `require` resolves via config's dist `exports`. (types alias stays `../types/dist/index.d.ts` in both.)
- Keep the api jest `moduleNameMapper` → config **source** (unchanged): direct `pnpm --filter @pawcareright/api test` then needs no config build.
- Because `dev`/`build` now need config's dist, the executor MUST `pnpm --filter @pawcareright/config build` (or `pnpm build`) before the live `dev` curl check and before `pnpm --filter @pawcareright/api build`. The regression `pnpm build`/`pnpm test` run via turbo already build config first via `^build`.

**Re-verify after A1:** `find apps/api/dist -name main.js` must be exactly `apps/api/dist/main.js` (NOT nested); the live `pnpm --filter @pawcareright/api dev` → `curl /v1/health` must return 200 `{status:ok,db:ok,redis:ok}`; `pnpm build && pnpm test && pnpm typecheck && pnpm lint` from root all green; `pnpm --filter @pawcareright/config test` still green (config's own tests unaffected).
