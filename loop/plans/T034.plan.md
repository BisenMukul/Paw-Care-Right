# Plan — T034: Vision input pipeline

## Objective (from card)
Add a photo-prep service in `apps/api` that turns an intake's photo keys into provider-ready base64 images: fetch bytes from storage by key → downscale to ≤1024px long edge → strip all metadata (EXIF/GPS) → run a stub unsafe-image pre-check (interface + log-only v1) → base64-package with a ≤3-image count cap and a pinned total-byte budget, truncating oversized sets deterministically (intake answer order, drop from the end) with a structured `Logger.warn`. No HTTP surface, no worker wiring (that is T043). Plan only — no code.

## Safety statement (§5-adjacent surface — planned, NOT blocked)
T034 does not touch and does not weaken any PRODUCT_SPEC §5 safety surface: it renders no user-facing copy, no disclaimer, no emergency escalation, no dosing/medication content, and calls no AI provider. It only prepares image bytes.
The one §5-*adjacent* element is the **unsafe-image pre-check hook** (relates to §5 rule 6 "no content that helps harm animals," and CLAUDE §7 rule 6). The card scopes it explicitly as a **stub interface, log-only v1**. This plan implements exactly that:
- The pipeline invokes the injected `UnsafeImageCheck` on the processed bytes of every image and **logs** its verdict.
- v1 is **fail-open by deliberate design**: a `flagged` verdict is logged but does **not** drop the image or block the pipeline. This is a NEW surface, not the weakening of an existing one — there is no pre-existing blocking behavior to remove. Real enforcement (a rule-based unsafe-output/-input detector) is owned by **T038** (red-team detector) and moderation hardening by **T052**; the DI seam here lets a real implementation replace the stub with zero pipeline changes.
- Because this only *adds* a (currently inert) safety seam and removes nothing, it is **not** a SAFETY-ESCALATION. The checker should confirm no blocking/uncertainty behavior was removed elsewhere.

## Placement decision (Risk R1 — the primary decision)
**All new code lives in a new, self-contained `apps/api/src/vision/` module. Nothing else in the repo is touched.**
- The prep runs server-side in `apps/api`, where `sharp` + `@aws-sdk/client-s3` (via `StorageService`) already exist, and where T043's `check-runner` worker (the sole consumer) also lives. `packages/ai` (Option B) would need `sharp` + S3 added to a shared package — rejected as heavy and wrong (§2 rule 2).
- A **dedicated `vision/` module** (not an extension of `photos/`) is chosen because photos/ owns *upload* (presign/confirm/rendition) while this is *vision-input prep for the check flow*; keeping them separate avoids coupling and does not collide with the future `checks/` module (T041/T042) which does not exist yet.
- **The `@pawcareright/ai` workspace dependency is deliberately NOT added** (see R2, R3). Adding it would require, in one task, all of: `apps/api/package.json` dep + `tsconfig.json` alias + `tsconfig.build.json` alias + jest `moduleNameMapper` + lockfile regen — a 5-touch graph change for, at most, a stub interface and a structurally-compatible output type. That is exactly the over-engineering §2 rule 2 forbids. T043 (which must call the provider) is the natural place to add that alias.

## Pipeline semantics (`VisionPrepService.prepare`)
Pinned values (all in `vision.constants.ts`): **max long edge 1024px**, **JPEG quality 80**, **max 3 images**, **total base64 budget 4 MiB** (measured on the base64 payload actually transmitted), **output mime `image/jpeg`**.

1. `const limits = { ...DEFAULT_VISION_LIMITS, ...(input.limits ?? {}) }`.
2. `const allKeys = collectPhotoKeys(input.intake)`; `requestedCount = allKeys.length`.
3. **Count cap (deterministic, drop from the end):** `const capped = allKeys.slice(0, limits.maxImages)`. If `allKeys.length > limits.maxImages`, `this.logger.warn` a structured message `{ checkId, reason: "max_images", requestedCount, kept: capped.length, droppedKeys: allKeys.slice(limits.maxImages) }`.
4. Iterate `capped` in order, accumulating `images: PreparedVisionImage[]` and `totalBase64Bytes = 0`:
   a. `const src = await this.storage.getObject(key)` — infra errors **propagate** (see R5), never swallowed.
   b. `const processed = await sharp(src).rotate().resize(limits? no →` use `VISION_MAX_EDGE, VISION_MAX_EDGE, { fit: "inside", withoutEnlargement: true }).jpeg({ quality: VISION_JPEG_QUALITY }).toBuffer()`. `.rotate()` bakes EXIF orientation; sharp's default (no `.withMetadata()`) re-encode strips ALL metadata incl. EXIF/GPS (same guarantee ImagesProcessor relies on).
   c. `const verdict = await this.unsafeCheck.check({ bytes: processed, key })`. If `verdict.verdict === "flagged"`, `this.logger.warn({ checkId, key, reason: "unsafe_flagged", detail: verdict.reason })` — **v1 fail-open: do not drop, do not throw** (see Safety statement).
   d. `const base64 = processed.toString("base64")`; `const size = base64.length`.
   e. **Byte-budget (deterministic, drop this + all remaining):** if `totalBase64Bytes + size > limits.byteBudgetBytes` AND `images.length >= 1` OR always → if adding would exceed budget, `this.logger.warn({ checkId, key, reason: "byte_budget", totalBase64Bytes, wouldAdd: size, budget: limits.byteBudgetBytes, droppedFromHere: capped.slice(currentIndex) })` and **break** the loop (stop fetching the rest). Edge: if the very first image alone exceeds budget it is dropped too (deterministic; theoretical for a ≤1024px q80 JPEG). Otherwise push `{ base64, mimeType: VISION_OUTPUT_MIME }` and add `size`.
5. Return `{ images, requestedCount, includedCount: images.length, truncated: images.length < requestedCount, totalBase64Bytes }`.

"Journal-visible warning" interpretation (state explicitly): a Nest `Logger.warn` call carrying structured fields (`checkId`, `reason`, dropped keys, sizes). Nest `Logger` output is the process log stream operators read — there is no per-task write to `loop/journal.md` from runtime code (that file is loop-tooling only, CLAUDE §2 rule 6). Tests assert the `Logger.warn` call + fields.

## Risks & design decisions the planner made (for CHECKER scrutiny)
- **R1 — Placement: new self-contained `apps/api/src/vision/` module, not `packages/ai` and not the `photos/` module.** sharp + S3 already live in apps/api; T043 (consumer) is in apps/api; a dedicated module avoids coupling to upload concerns and avoids colliding with the future `checks/` module. Rejected `packages/ai` (would need sharp+S3 in a shared package) and rejected extending `photos/` (different responsibility).
- **R2 — The `UnsafeImageCheck` interface + `LogOnlyUnsafeImageCheck` stub live in apps/api, NOT in `packages/ai`.** The prompt floated putting the hook interface in `packages/ai` for eval/red-team reuse. Rejected for v1: it is a stub, and lifting it would force adding the `@pawcareright/ai` workspace dep + two tsconfig aliases + a jest moduleNameMapper + a lockfile regen — a large graph change for zero behavior, violating §2 rule 2. The DI-token seam (`UNSAFE_IMAGE_CHECK`) already lets a real impl (T038) be injected later; the interface can be promoted to `packages/ai` at that point if sharing is actually needed. Checker: scrutinize whether this local placement is acceptable vs. shared reuse.
- **R3 — Output type `PreparedVisionImage` is defined locally and is structurally assignable to `packages/ai`'s `VisionImage`, rather than importing `VisionImage`.** `{ base64: string; mimeType: string }` satisfies `{ base64?: string; url?: string; mimeType: string }`, so T043 can pass `prepared.images` straight into `provider.generate({ images })` once T043 imports `VisionImage`. This keeps T034 free of the `@pawcareright/ai` alias cost. Risk: if `VisionImage` later gains a required field, the structural link breaks — mitigated because T043 does the actual wiring and would catch it; a code comment will pin the intent.
- **R4 — `prepare` accepts an optional `limits` override (defaulting to the pinned constants).** Needed so budget/count truncation is unit-testable with small sharp fixtures instead of fabricating multi-MB images. Production callers (T043) omit `limits` → pinned defaults. This is a minimal testability seam, not runtime configurability; scrutinize that defaults are the pinned constants and that no env/config path feeds it.
- **R5 — Storage fetch/processing errors PROPAGATE; only over-budget/over-count truncation is silent-and-deterministic.** An infra failure (getObject throws) must not be masked as "fewer photos" — it bubbles to T043's BullMQ retry/backoff (matching ImagesProcessor, which also lets getObject throw). Deterministic truncation (a design outcome) and infra failure (an error) are treated differently on purpose.
- **R6 — Pinned budget = 4 MiB measured on the base64 payload; JPEG quality = 80.** Base64 is what actually travels in the provider request, so the cap is applied there (not raw bytes). 3 images at ≤1024px/q80 are typically ~200–680 KB base64 each (~2 MB worst case), so 4 MiB leaves headroom while staying well under typical Ollama Cloud / Gemini multimodal request limits; it also bounds pathological high-entropy inputs. Quality 80 matches sharp's `.jpeg()` default (what ImagesProcessor already emits), pinned explicitly for deterministic output sizes. Both numbers are design choices — checker should sanity-check against AI_PROVIDERS limits.
- **R7 — v1 unsafe-image pre-check is fail-open (log-only), by deliberate design.** See Safety statement. It adds an inert seam and removes no existing safety behavior; real enforcement is T038/T052. Not a SAFETY-ESCALATION. Checker: confirm nothing here drops/soften an existing §5 guarantee.
- **R8 — Deterministic order is intake answer order, dropping from the end;** count cap applied before fetch, byte budget applied during accumulation with early-break. Same input ⇒ identical included prefix and identical dropped suffix (asserted). No sorting, no shuffling, no dedup of keys (keys are already validated upstream by `completedIntakeSchema`).
