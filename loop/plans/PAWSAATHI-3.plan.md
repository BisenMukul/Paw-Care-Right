## Design decisions the planner locked (scrutinize under "Risks")

1. **Emergency screen `check/emergency/[checkId].tsx` = ZERO-DIFF.** It paints its own opaque red palette edge-to-edge (`SafeAreaView bg-red-700`, inner `bg-red-900`/`bg-red-800`, all text `text-white`). `red-700/800/900` and `white` are theme-INVARIANT Tailwind tokens (identical hex under light or dark OS scheme) and white-on-red-700 ≈ 7.0:1+, so no dark-legibility problem exists to fix. The card permits a dark variant "only as pure additive `dark:` classNames … NOTHING else changing; if any doubt, keep zero-diff." Adding `dark:` tokens here would be pure no-op noise on a screen whose law is "nothing animated, nothing else changing," creating structural/whitespace diff risk for zero legibility benefit → **keep it byte-identical.** Pinned by a new "no `dark:` anywhere" assertion in `emergency-interstitial.test.tsx`.
2. **`vet-disclaimer.tsx` = ZERO-DIFF.** Its text (`text-brand-900`) sits on its OWN opaque `bg-brand-50` container, never on a dark surface — so the pair is a self-contained `brand-900`-on-`brand-50` = **11.67:1 (AAA)** regardless of OS scheme (§1.1). The card's stated trigger for an additive fix — "if its current light tokens fail AA on dark surfaces" — is therefore NOT met. Zero-diff also literally satisfies the scope line "disclaimer subtree byte-identical" and preserves the byte-frozen §5 snapshot subtree. On dark mode it renders as a bright light-mint box, which if anything *increases* the disclaimer's prominence as the §5 trust anchor (§7.6) — never dilutes it. Pinned by a "disclaimer className contains no `dark:`" assertion.
3. **Opaque semantic SAFETY fills on the result screen stay byte-identical; only navigational/guidance surfaces are themed.** The emergency-notice (`bg-red-600`+white), fallback-notice (`bg-amber-100`+`text-amber-950`), and urgency-BANNER (`URGENCY_DISPLAY.containerClass`/`textClass`, opaque semantic fills + white/amber-950) each carry their OWN opaque background, are AA-verified in light (`urgency-contrast.test.ts`), and are theme-invariant → they stay **byte-identical** (same principle as decision 1: a safety signal must "stay loud on both themes"). The guidance `Card` inner text, section headers, page background, and error state ARE themed — the guidance card text is a genuine legibility REQUIREMENT (`text-brand-900` #123a30 on `dark:bg-surface-card-dark` #16241F ≈ 1.1:1, illegible → MUST become `dark:text-ink-dark`).
4. **Only the history-row urgency CHIPS get new dark AA pairs; banners keep their light pairs.** The chips are light-tint pairs (`bg-red-100`/`text-red-900`, …) on browse surfaces (history/entry) — navigational metadata, not a §5 safety fill — where a dark pair improves fidelity. Each tier gets a deep-fill/light-text same-hue dark pair (`dark:bg-{hue}-900 dark:text-{hue}-100`), fresh-AA-verified in a NEW dark section of `urgency-contrast.test.ts` BEFORE wiring. The BANNER pairs (`containerClass`/`textClass`) are untouched (decision 3). No other new dark pair is introduced anywhere — every non-chip dark surface/text reuses an already-verified §1.1a token (`ink-dark`/`ink-muted-dark` on `card`/`page-dark`; white on `accent-dark`; `accent-bright` label; `red-400` error), so `dual-theme-contrast.test.ts` needs NO new rows.
5. **§1.4a fonts adopted across the flow, matched to each Text's EXISTING weight (never swap visual weight), EXCEPT the four frozen subtrees + two zero-diff files.** Mirrors PAWSAATHI-2 (dark+font together). `font-display` = screen titles (`font-bold` headers); `font-display-semibold` = section/step headers (`text-lg font-semibold`); `font-body-semibold` = card titles/labels; `font-body` = body/secondary. The emergency-notice/fallback-notice/urgency-banner/VetDisclaimer subtrees and the emergency screen get NO font token (frozen). **No new user-facing string is introduced anywhere → record-only tone scan surface unchanged; no a11y label needed (all icon-only controls already labelled; scheme-aware color props carry no text).**

## Risks & the decisions the planner made
- **R1 (Emergency screen ZERO-DIFF — decision 1, checker WILL attack):** the screen is self-contained opaque red + white, theme-invariant, already ≥7:1, so no dark-legibility problem exists; per the card's "if any doubt, keep zero-diff" and "nothing else changing," adding `dark:` would be no-op noise with structural-diff risk. If the checker expected a dark variant: the card explicitly gates it on the red staying loud AND nothing else changing — zero-diff is the strictly-safer satisfying reading, and a no-`dark:` test now pins it.
- **R2 (VetDisclaimer ZERO-DIFF — decision 2, checker WILL attack):** its `brand-900`-on-`brand-50` pair is self-contained (own opaque light container) = 11.67:1 AAA on ANY scheme; the card's additive-fix trigger ("light tokens fail AA on dark surfaces") is not met, and zero-diff honors "disclaimer subtree byte-identical" literally. The only counter-argument is visual consistency (a bright box on a dark result screen) — but for a §5 trust anchor extra prominence is acceptable/desirable (§7.6) and never an AA failure. Pinned by a no-`dark:` className assertion. If forced, an additive-only `dark:bg-surface-card-dark`+`dark:text-ink-dark` variant (verified 10.81–12.25:1) is the fallback — but I chose zero-diff as the conservative safety reading.
- **R3 (banners/notices frozen while chips themed — decisions 3–4):** the distinction is principled — opaque semantic SAFETY fills stay theme-invariant-loud; navigational light-tint chips get a fresh-AA dark pair. Any deviation (e.g. theming a safety fill) would need its own AA math and would soften the safety signal.
- **R4 (result snapshot re-record):** the `.snap` changes because `className` is a literal string that now contains `dark:`/`font-*` tokens even under jest's light render. Mitigated by: freezing the four safety subtrees (so their captured classNames are byte-identical), re-recording result ONLY, and a manual additive-only diff + the disclaimer-zero-diff assertion. Any non-additive delta (removed/reordered node, changed disclaimer) ⇒ STOP and report.
- **R5 (fonts on a safety flow — decision 5):** applying `font-*` tokens broadens the diff, but it is presentation-only, matched to existing weights (no visual-weight swap), and excluded from all frozen subtrees; it matches the PAWSAATHI-2 precedent and the "PawSaathi card language" scope. If the checker reads the batch as dark-only, fonts are still additive/reversible and change no copy — but I include them for cross-batch consistency.
- **R6 (scheme-aware ActivityIndicator/placeholder color):** a small call-site `useColorScheme()` read is added to `[category].tsx` and `waiting`; this is presentation (native color prop), not logic, preserves the spinner testIDs, and is the §1.6-mandated way to color native components for dark.
