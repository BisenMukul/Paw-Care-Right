# Plan — T043: check-runner worker (BullMQ triage processor)

## Objective (from card)
Consume `pawcareright-checks` "triage" jobs: load the check + pet, prepare photos, run the
provider-injected triage pipeline, apply the deterministic post-rules safety floors, persist the
`TriageResult` + cost, and fail *upward* to `FALLBACK` (never a silent guess). Integration-tested
with a fake provider across the four card scenarios + rules-floor-preserved + idempotent redelivery.

## Safety statement (PRODUCT_SPEC §5 — this worker is the §5 enforcement point)
- **Post-rules floor application is THE worker's safety job.** Every persisted result goes through
  `applyPostRules(result, { species, rulesFloor })` (T036). The rules `tierFloor` (deterministic
  EMERGENCY_NOW for any matched red flag) can only ever RAISE urgency; the worker must never persist
  a result that lowers a rules floor, **even when the AI says REASSURE and even on the SAFE_FALLBACK
  path** (a provider-failed red-flag check must still surface EMERGENCY_NOW).
- **Fail upward.** `runTriage` never throws and returns `SAFE_FALLBACK` for malformed/low/errored
  output → the worker writes status `FALLBACK` with the (floored) frozen fallback content. Infra
  failures on the final retry also resolve to `FALLBACK`, never a retry-silently-and-guess.
- **No intake logging (T041 N1 / T042 carry-forward #4).** The worker MUST NOT log `intakeJson`,
  `freeText`, `photoKeys`, or raw provider text. All structured logs are keyed by `checkId` and carry
  only safe scalars (status, tiers, counts, latency, cost, model id). This is asserted by a test.
- This task changes no disclaimer, emergency-interstitial, or dosing surface — no SAFETY-ESCALATION.

## Key decisions (each expanded/justified under Risks)
1. **rulesFloor is RECOMPUTED**, not read from the stored `redFlagRuleId`: `buildRedFlagIntake(pet,
   intake)` → `evaluateRedFlags(...)` → `evaluation.highest?.tierFloor ?? null`. Deterministic, same
   inputs as create, and yields the `Urgency` floor `applyPostRules` needs (the stored `redFlagRuleId`
   is a rule id, not a tier). (R1)
2. **Provider comes from a DI token** `TRIAGE_TEXT_PROVIDER` (factory `getTextProvider()`), never
   constructed in the processor body; tests override the token with a scripted `FakeTextProvider`. (R2)
3. **`modelId` column = the CONFIGURED text model** from a second token `TRIAGE_TEXT_MODEL_ID`
   (`loadAiEnv().AI_TEXT_MODEL`), because `runTriage`'s `TriageRunResult` does NOT surface the
   responding provider model (only `usage` + prompt `version`). packages/ai is not modified. (R3)
4. **`runTriage` absorbs provider errors** (timeout/HTTP/parse) into `SAFE_FALLBACK` on the FIRST
   attempt — a provider timeout does NOT drive BullMQ retries. BullMQ retries only fire for **infra**
   throws (DB/storage/VisionPrep). The card's "provider timeout→retries→FALLBACK" is therefore tested
   honestly as two distinct paths (see AC mapping). (R4)
5. **Final-fail FALLBACK is handled in-process** via a `try/catch` around orchestration keyed off
   `job.attemptsMade` — non-final attempt rethrows (BullMQ retries per committed enqueue opts), final
   attempt writes `FALLBACK` and returns. (R5)
6. **Jitter is NOT added.** Committed T042 enqueue opts (`attempts:3`, `backoff exponential 2000`) =
   initial + 2 retries = "retry 2". Native BullMQ v5 exponential backoff has no jitter; adding it needs
   a custom `settings.backoffStrategy` on the Worker + changing the committed `backoff.type` to
   `"custom"` in `checks.service.ts` (out of this task's authorized scope, fragile through Nest's
   `WorkerHost`). Minimal-honest choice: keep exponential-no-jitter, document. (R6)
7. **VisionPrepService is invoked when the check has photos** (exercises T034's fetch/downscale/
   safety/log pipeline, its sole consumer) but its output is NOT fed to a provider: `runTriage` is
   text-only (T033). Vision-provider triage is explicitly deferred. (R7)
8. **Orchestration lives in the processor class** (mirrors `ImagesProcessor`), tested by direct-invoke
   with mocked I/O and REAL safety logic (rules + post-rules + runTriage run for real). (R8)

## Processor flow spec (`process(job: Job<ChecksJobData>)`, step-by-step)
1. `const { checkId } = job.data;` `logger.log({ event: "triage_start", checkId, attempt: job.attemptsMade });`
2. Load: `const check = await prisma.symptomCheck.findUnique({ where: { id: checkId }, include: { pet: true } });`
   - If `check === null` → `logger.warn({ event:"triage_missing", checkId })` and **return** (job acks;
     nothing to do — row deleted).
3. **Idempotency / crash recovery** on `check.status`:
   - `DONE` or `FALLBACK` (terminal, `isTerminalCheckStatus`) → `logger.log({ event:"triage_noop_terminal",
     checkId, status })` and **return** (redelivered/duplicate job; no writes, no provider call).
   - `QUEUED` → `assertTransition("QUEUED","RUNNING")`; `await prisma.symptomCheck.update({ where:{id:checkId},
     data:{ status:"RUNNING", startedAt: new Date() } })`.
   - `RUNNING` → this is a BullMQ retry of a prior crashed attempt; do NOT re-transition, do NOT reset
     `startedAt`; proceed.
4. **Wrap steps 5–11 in `try { … } catch (err) { … }`** (see Error/retry semantics for the catch body).
5. Defensive re-parse of intake: `const parsedIntake = parseIntake(check.intakeJson);` — it was validated at
   create, but re-validate fail-closed. If `!parsedIntake.ok` → throw `new Error("intake failed revalidation:
   " + parsedIntake.reason)` (routes into the catch → retries/FALLBACK; do NOT log the reason's payload — the
   reason is a schema message, not user text, so it is safe, but keep it out of any intake-shaped log object).
   Use `const intake = parsedIntake.value;` (a `CompletedIntake`).
6. Build `PetProfileInput` from `check.pet` (`species, sex, ageEstimateMonths, birthDate, weightGrams`) and
   `rulesFloor`: `const evaluation = evaluateRedFlags(buildRedFlagIntake(petProfile, intake));`
   `const rulesFloor = evaluation.highest?.tierFloor ?? null;` (`Urgency | null`).
7. **Vision prep (photos)** — only when `check.photoKeys.length > 0`:
   `const prepared = await visionPrep.prepare({ intake, checkId });`
   `logger.log({ event:"vision_prepared", checkId, includedCount: prepared.includedCount,
   requestedCount: prepared.requestedCount, truncated: prepared.truncated });`
   The prepared images are NOT passed to `runTriage` (text-only, R7). Storage/VisionPrep throws propagate to
   the catch (infra path).
8. Build `TriagePetContext` (whitelisted fields ONLY): `{ name: pet.name, species: pet.species, sex: pet.sex,
   ...(ageMonths derived: ageEstimateMonths ?? monthsSince(birthDate)) , ...(weightKg = weightGrams/1000 when
   present), neutered: pet.neutered }`. `breedLabel` is omitted (optional; needs a breed lookup that is out of
   scope — mirrors the mapper omitting `sizeClass`). Then:
   `const provider = this.textProvider;`
   `const run = await runTriage({ pet: petContext, intake }, { provider });` (never throws).
9. **Post-rules (the safety composition):**
   `const outcome = applyPostRules(run.result, { species: pet.species, rulesFloor });`
10. Derive persistence values:
   - `const target: CheckStatus = run.status === "SAFE_FALLBACK" ? "FALLBACK" : "DONE";`
   - `assertTransition("RUNNING", target);`
   - `const costMicroUsd = Math.max(0, Math.round(run.usage?.costMicroUsd ?? 0));`
   - `resultJson = outcome.result` (schema-valid by construction), `urgency = outcome.result.urgency`,
     `confidence = outcome.result.confidence`, `modelId = this.textModelId`, `promptVersion = run.version`.
   - `failureReason = run.status === "SAFE_FALLBACK" ? (run.failureReason ?? "safe_fallback") : null;`
11. **Persist atomically** — `await prisma.$transaction([`
    `  prisma.triageResult.upsert({ where:{ checkId }, create:{ checkId, urgency, confidence, resultJson,
       modelId, promptVersion }, update:{ urgency, confidence, resultJson, modelId, promptVersion } }),`
    `  prisma.symptomCheck.update({ where:{ id: checkId }, data:{ status: target, completedAt: new Date(),
       costMicroUsd, failureReason } }),`
    `]);`
    Then `await costLog.record({ costMicroUsd, latencyMs: run.usage?.latencyMs, model: this.textModelId,
    status: run.status, userId: check.createdById, checkId });`
    `logger.log({ event:"triage_done", checkId, status: target, finalTier: outcome.finalTier,
    source: outcome.source, appliedRulesFloor: outcome.appliedRulesFloor, attempts: run.attempts,
    costMicroUsd });`
12. `return;`

## Error / retry semantics
- **Provider errors are absorbed, not retried.** A provider timeout/HTTP/invalid-response (or the one-repair
  failure, or malformed output) is caught INSIDE `runTriage` and returned as `status: "SAFE_FALLBACK"` on the
  first attempt. The worker persists status `FALLBACK` with the floored fallback content. No exception leaves
  `process()`, so BullMQ does NOT retry. This is the intended §5 behavior.
- **Infra errors ARE retried.** Throws from `prisma.*`, `visionPrep.prepare`, or `$transaction` propagate to
  the `try/catch` (step 4). Catch body:
  1. `const finalAttempt = isFinalAttempt(job);` where
     `isFinalAttempt(job) = (job.attemptsMade + 1) >= (job.opts.attempts ?? 1)`
     (BullMQ v5: during `process()`, `attemptsMade` counts prior failed attempts, 0 on first run — executor
     verifies against the installed `bullmq` version; the predicate is the contract).
  2. If **not** final → `logger.warn({ event:"triage_infra_retry", checkId, attempt: job.attemptsMade })`;
     `throw err;` (BullMQ schedules the next attempt using the committed `attempts:3 / exponential 2000` opts).
  3. If **final** → fail upward: recompute `rulesFloor` best-effort is unnecessary — reuse the frozen safe
     fallback composed with any floor already computed if available; to stay simple and robust when the throw
     happened before `rulesFloor` was computed, compute the FALLBACK result as
     `applyPostRules(SAFE_FALLBACK, { species: check.pet.species, rulesFloor: floorOrNull })` where
     `floorOrNull` is the `rulesFloor` computed in step 6 if reached, else `null`. Ensure status is `RUNNING`
     first (if still `QUEUED` because the step-3 update threw, `assertTransition("QUEUED","RUNNING")` +
     update), then `assertTransition("RUNNING","FALLBACK")` + the same `$transaction` upsert/update with
     `target:"FALLBACK"`, `failureReason: (err as Error).message`, `costMicroUsd: 0`, then `costLog.record`
     and `logger.error({ event:"triage_infra_fallback", checkId, message })`; `return` (do NOT rethrow).
  4. If the final-attempt FALLBACK persist ITSELF throws (DB fully down) → rethrow; the job lands in BullMQ's
     failed set (`removeOnFail:false` retains it as a dead-letter for inspection). Documented, acceptable.
- **`runTriage`'s `SAFE_FALLBACK` path is NOT the catch path** — it is a normal return handled by steps 9–12
  (status `FALLBACK`, floored content). Keep these two FALLBACK sources distinct in code and tests.

## Risks & decisions (numbered — for the checker to scrutinize)
- **R1 rulesFloor recompute vs stored id:** recomputed via mapper+engine (deterministic, same create inputs);
  the stored `redFlagRuleId` is a rule id, not a tier, so it cannot feed `applyPostRules` directly. Risk: intake
  content drift between create and run — none, both parse the same immutable `intakeJson`.
- **R2 provider DI token:** `TRIAGE_TEXT_PROVIDER` factory `getTextProvider()`. Risk: prod default
  `AI_TEXT_PROVIDER=ollama` reaches real network — acceptable; tests always override the token with a fake.
- **R3 modelId = configured model, not responding model:** `runTriage` does not surface the provider model, and
  packages/ai is frozen. Configured `AI_TEXT_MODEL` is the honest best. Risk: on SAFE_FALLBACK the stored model
  is the attempted model, which is correct-enough for cost/audit. Flag for a future packages/ai enhancement.
- **R4 provider-timeout does NOT retry:** `runTriage` absorbs provider errors → `SAFE_FALLBACK` first attempt.
  The card's "provider timeout→retries→FALLBACK" is realized via the INFRA-failure test; the provider-timeout
  test documents the absorb behavior. This is a faithful reading of §5 rule 5, not a scope reduction.
- **R5 final-fail via `job.attemptsMade`:** predicate `(attemptsMade+1) >= (opts.attempts ?? 1)`. Risk: BullMQ
  version semantics of `attemptsMade` during `process()` — executor confirms against installed `bullmq`; the
  contract (final attempt writes FALLBACK, earlier attempts rethrow) is version-independent.
- **R6 no jitter:** kept committed exponential-2000 (attempts:3 = "retry 2"). Native BullMQ v5 exponential has
  no jitter; adding it needs a custom `settings.backoffStrategy` + changing committed `checks.service` enqueue
  `backoff.type` to `"custom"` (out of authorized scope, fragile through Nest `WorkerHost`). Thundering-herd
  jitter is deferrable and safety-neutral.
- **R7 VisionPrep output unused by the provider:** invoked to exercise/log T034's pipeline (its sole consumer)
  and ready the seam; text-only `runTriage` cannot consume images yet. Vision-provider triage deferred (future
  task / eval). Risk: perceived dead work — mitigated by the truncation/safety logging it performs and by being
  the wiring point T034 was built for.
- **R8 orchestration-in-processor + direct-invoke tests:** mirrors `ImagesProcessor`; real safety logic runs,
  only I/O is mocked. Risk: no live BullMQ retry exercised — mitigated by the `attemptsMade`-driven unit
  assertions and the version-agnostic contract. Coverage target ≥80% on the processor is met by the 9 cases.
