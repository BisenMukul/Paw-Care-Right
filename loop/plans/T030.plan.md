# Plan — T030: TriageResult schema + safe fallback

## Objective (from card)
Add the safety-critical `TriageResult` Zod schema (PRODUCT_SPEC §6.3) to `packages/types`, a pure `parseTriage(raw) → {ok:true,result} | {ok:false,reason}` gate that NEVER throws and NEVER mutates, and a frozen `SAFE_FALLBACK` constant (vet-recommend copy, urgency floor `VET_SOON`, confidence `low`). This is the fail-upward floor of the whole product (PRODUCT_SPEC §5 rule 2). Plan only — no code.

## Safety statement (this task IS a §5 surface — planned, not blocked)
T030 does not weaken any Safety Policy surface; it *builds* one. It hardens the fail-upward floor: schema rejects unsafe shapes, `parseTriage` fails closed, `SAFE_FALLBACK` copy is §7-compliant vet-escalation text with no diagnosis/dosing language. Because the fallback copy is user-facing (CLAUDE §7), the CHECKER must read it as a §5 surface. No emergency-interstitial, disclaimer-component, or dosing logic is touched (those are other tasks). NOT a SAFETY-ESCALATION.

## Ordered steps
1. In `triage.ts`, declare the tier + confidence enums and a tier-severity order:
   - `URGENCY_TIERS = ['EMERGENCY_NOW','VET_24H','VET_SOON','MONITOR','REASSURE'] as const`; `urgencySchema = z.enum(URGENCY_TIERS)`.
   - `CONFIDENCE_LEVELS = ['high','medium','low'] as const`; `confidenceSchema = z.enum(CONFIDENCE_LEVELS)`.
   - `URGENCY_SEVERITY: Record<Urgency, number>` where **lower index = more urgent** (EMERGENCY_NOW=0 … REASSURE=4). Define `HOME_CARE_ALLOWED_TIERS = ['VET_SOON','MONITOR','REASSURE'] as const` and `VET_SOON_FLOOR_SEVERITY = URGENCY_SEVERITY['VET_SOON']`.
2. Declare `possibleCauseSchema = z.strictObject({ name: z.string().min(1).max(120), whyItFits: z.string().min(1).max(400) })`.
3. Declare a reusable bounded-string-array helper inline (e.g. `boundedStrings = z.array(z.string().min(1).max(400)).max(10)`) for `redFlagsToWatch`, `homeCare`, `doNot`, `vetQuestions`.
4. Build the base object with `z.strictObject({...})` (extra keys REJECTED — R2):
   - `urgency: urgencySchema`
   - `confidence: confidenceSchema`
   - `summary: z.string().min(1).max(600)`
   - `possibleCauses: z.array(possibleCauseSchema).max(4)` (allowed empty — SAFE_FALLBACK asserts nothing)
   - `redFlagsToWatch`, `homeCare`, `doNot`, `vetQuestions`: the bounded-strings helper (each may be empty)
   - `followUpHours: z.number().int().positive().max(2160).nullable()` (R5: int, positive, ≤90 days; `null` allowed)
5. Attach cross-field + safety refinements via `.superRefine` (all REJECT — never mutate; R3/R4/R6):
   - **(a) low-confidence tier floor:** if `confidence === 'low'` and `URGENCY_SEVERITY[urgency] > VET_SOON_FLOOR_SEVERITY` (i.e. tier is *less* urgent than VET_SOON → MONITOR/REASSURE) ⇒ issue on `['urgency']`. REJECT, do not auto-raise (R3).
   - **(b) home-care on high-urgency:** if `!HOME_CARE_ALLOWED_TIERS.includes(urgency)` (EMERGENCY_NOW/VET_24H) and `homeCare.length > 0` ⇒ issue on `['homeCare']`. REJECT (R4 — home-care on an emergency delays care).
   - **(c) diagnosis-word gate:** scan every user-facing string value (`summary`; each `possibleCauses[].name`/`.whyItFits`; every element of `redFlagsToWatch`/`homeCare`/`doNot`/`vetQuestions`) with `/diagnos/i`; any match ⇒ issue on that field's path. REJECT (R6 — mechanical CLAUDE §7 rule 1 at the schema boundary). Name it `assertNoDiagnosisLanguage`.
   - Export the refined schema as `triageResultSchema`; `export type TriageResult = z.infer<typeof triageResultSchema>`.
6. `parseTriage(raw: unknown): ParseTriageResult` (R8 — never throws):
   - If `typeof raw === 'string'`: `JSON.parse` inside try/catch; on throw → `{ ok:false, reason: 'INVALID_JSON: <message>' }`.
   - Run `triageResultSchema.safeParse(candidate)`.
   - Success → `{ ok:true, result: parsed.data }`.
   - Failure → `{ ok:false, reason: <flattened human/machine-usable message>, issues: parsed.error.issues }` — `reason` built from the zod issues (path + message) so the T033 repair prompt can consume it. Use a stable join (e.g. `issues.map(i => i.path.join('.')+': '+i.message).join('; ')`).
   - Type: `type ParseTriageResult = { ok: true; result: TriageResult } | { ok: false; reason: string; issues?: z.core.$ZodIssue[] }` (use zod v4's exported issue type; no `any`).
7. `SAFE_FALLBACK` (R9): a `TriageResult`-typed object frozen with `Object.freeze` (deep-freeze the arrays too, or use `as const` + `satisfies TriageResult`):
   - `urgency: 'VET_SOON'` (the floor — see R9 justification)
   - `confidence: 'low'`
   - `summary`: the §7-compliant vet-recommend copy (verbatim in "Interfaces" below)
   - `possibleCauses: []` (we assess nothing)
   - `redFlagsToWatch: []`
   - `homeCare: []` (empty ⇒ satisfies rule (b) for VET_SOON, and avoids any care claim)
   - `doNot`: single neutral no-human-medication line (verbatim below)
   - `vetQuestions: []`
   - `followUpHours: 24`
   - It MUST itself pass `triageResultSchema` (low + VET_SOON satisfies floor (a); VET_SOON is home-care-allowed and homeCare is empty; no "diagnos" substring). A test asserts this.
8. Append `export * from "./triage";` to `src/index.ts`.
9. Self-verify (commands below). Confirm existing `packages/types` specs stay green and the dist barrel rebuilds.

## Tests to write (map to acceptance criteria)
AC = "Property-style tests: malformed / missing / extra-tier inputs all fail closed to the fallback path." Tests target `parseTriage` (the fail-closed gate) and, where sharper, `triageResultSchema.safeParse`. Use small hand-rolled generator loops over fixture tables — deterministic, no `fast-check` (R10, no new dep). Build one `validFixtureFor(tier)` factory producing a schema-valid `TriageResult` per tier (respecting rules a/b), used as the base to mutate.

- **AC malformed → fail closed** → `parseTriage rejects malformed inputs`:
  - Non-object primitives (`null`, `undefined`, `42`, `true`, `[]`) → each `{ok:false}`.
  - Wrong type per field: loop over each field, set it to a type-wrong value (number for `summary`, string for `followUpHours`, object for arrays, etc.) → each `{ok:false}`.
  - JSON-string inputs: valid-serialized fixture → `{ok:true}`; malformed/truncated JSON string (`'{"urgency":'`) → `{ok:false, reason startsWith 'INVALID_JSON'}`.
- **AC missing → fail closed** → `parseTriage rejects a missing required key`:
  - Loop over every required key of a valid fixture, `delete` it → each `{ok:false}` (and issue path includes that key).
- **AC extra-tier / extra-field → fail closed** → `strict schema rejects extra tiers and extra keys`:
  - Extra/typo urgency values (`'EMERGENCY'`, `'vet_24h'`, `'URGENT'`, `'ER'`, `''`) → `{ok:false}`.
  - Extra confidence values (`'HIGH'`, `'unknown'`) → `{ok:false}`.
  - Extra key at top level (`{...valid, foo:'x'}`) → `{ok:false}` (strict).
  - Extra key inside a `possibleCauses` element (`{name,whyItFits,extra:1}`) → `{ok:false}` (strictObject).
- **AC caps** → `arrays and possibleCauses honor defensive caps`:
  - `possibleCauses` length 5 → `{ok:false}`; length 4 → ok.
  - Any bounded string array length 11 → `{ok:false}`.
  - Over-long strings (summary > 600, cause name > 120) → `{ok:false}`.
  - `followUpHours`: `0`, `-1`, `1.5`, `3000` → each `{ok:false}`; `24` and `null` → ok.
- **AC fail-upward floor (rule a)** → `low confidence must be at least VET_SOON`:
  - Matrix: for each tier, `{confidence:'low', urgency:tier}` — EMERGENCY_NOW/VET_24H/VET_SOON → ok (given rule-b satisfied), MONITOR/REASSURE → `{ok:false}` with issue path `['urgency']`. Assert it is REJECTED, NOT silently raised (the returned result, when ok, has `urgency` unchanged — no mutation).
  - `medium`/`high` confidence with MONITOR/REASSURE → ok (floor only applies to low).
- **home-care safety (rule b)** → `home-care forbidden on emergency tiers`:
  - Matrix: `{urgency:'EMERGENCY_NOW', homeCare:['rest at home']}` and same for `VET_24H` → `{ok:false}` path `['homeCare']`; allowed tiers with non-empty homeCare → ok; emergency tiers with empty homeCare → ok.
- **diagnosis-word gate (rule c)** → `rejects the word diagnosis/diagnose anywhere`:
  - Inject `'diagnosis'`/`'Diagnose'`/`'DIAGNOSED'` into each user-facing field in turn (summary, cause name, cause whyItFits, and one element of each string array) → each `{ok:false}` with the offending field in the issue path.
- **SAFE_FALLBACK self-validation** → `SAFE_FALLBACK is itself a valid TriageResult and fails safe`:
  - `triageResultSchema.safeParse(SAFE_FALLBACK).success === true`.
  - `parseTriage(SAFE_FALLBACK).ok === true`.
  - `SAFE_FALLBACK.urgency === 'VET_SOON'` and `confidence === 'low'`.
  - No user-facing string in `SAFE_FALLBACK` matches `/diagnos/i`.
  - `Object.isFrozen(SAFE_FALLBACK)` (and arrays frozen) — mutation attempt does not change it.
- **parseTriage never throws** → `parseTriage never throws on hostile input`:
  - Wrap calls on circular-ish / exotic inputs (BigInt, Symbol-keyed object, deeply nested) in `expect(() => parseTriage(x)).not.toThrow()`.
- **valid fixtures per tier** → `accepts a well-formed result for every tier`:
  - `validFixtureFor(tier)` for all 5 tiers → `{ok:true}`.

## Interfaces/contracts (executor must match)
```ts
export const URGENCY_TIERS = ['EMERGENCY_NOW','VET_24H','VET_SOON','MONITOR','REASSURE'] as const;
export const CONFIDENCE_LEVELS = ['high','medium','low'] as const;
export const urgencySchema: z.ZodEnum<...>;      // z.enum(URGENCY_TIERS)
export const confidenceSchema: z.ZodEnum<...>;   // z.enum(CONFIDENCE_LEVELS)
export type Urgency = z.infer<typeof urgencySchema>;
export type Confidence = z.infer<typeof confidenceSchema>;

export const possibleCauseSchema: z.ZodType<{ name: string; whyItFits: string }>; // strictObject
export const triageResultSchema: z.ZodType<TriageResult>;  // strictObject + 3 superRefine rules
export type TriageResult = {
  urgency: Urgency;
  confidence: Confidence;
  summary: string;
  possibleCauses: { name: string; whyItFits: string }[];
  redFlagsToWatch: string[];
  homeCare: string[];
  doNot: string[];
  vetQuestions: string[];
  followUpHours: number | null;
};

export type ParseTriageResult =
  | { ok: true; result: TriageResult }
  | { ok: false; reason: string; issues?: z.core.$ZodIssue[] };
export function parseTriage(raw: unknown): ParseTriageResult;   // never throws; JSON.parses strings first

export const SAFE_FALLBACK: Readonly<TriageResult>;             // Object.freeze'd
```

## Risks & design decisions the planner made (for CHECKER scrutiny)
- **R1 — Placement in `packages/types`, not `packages/ai`.** Schema+parse+fallback are pure and dependency-free; the card puts the schema here; `ai`/`api`/`mobile` all consume from `types`. Trade-off: `packages/ai` gains a dependency on `types` (already the norm). If the checker expected them in `ai`, note this is deliberate per the card ("schema in packages/types") + reuse.
- **R2 — `.strict()` / `strictObject` everywhere.** The AC demands *extra-tier / extra-field* inputs fail closed; an extra key IS a malformed response. Both top-level and `possibleCauses` objects are strict.
- **R3 — Low-confidence floor: REJECT, not auto-raise.** `parseTriage`'s contract is validate-don't-mutate; silently rewriting a tier would hide model error. The caller's repair retry / SAFE_FALLBACK handles it (fail upward, never silently guess — PRODUCT_SPEC §5 rule 2). This is the single most safety-load-bearing decision — scrutinize.
- **R4 — Home-care on EMERGENCY_NOW/VET_24H → REJECT.** Spec says homeCare is "only for MONITOR/REASSURE/VET_SOON"; home-care copy on an emergency could delay care. Enforced structurally.
- **R5 — `followUpHours`: positive int ≤ 2160 (90d) or null.** Spec says `number|null`; defensive bounds reject nonsense (0/negative/fractional/huge). If the checker deems the 90-day cap arbitrary, it is a defensive guard, not a spec requirement — easy to widen.
- **R6 — Mechanical `/diagnos/i` gate in the schema.** CLAUDE §7 rule 1 forbids "diagnosis"/"diagnose" in user-facing AI values; enforcing at the schema boundary is defense-in-depth (the model, not just prompt, is caught). Risk: false-positive on a legitimate word — none in the pet-triage domain contains the substring "diagnos" except diagnosis/diagnose themselves. Strong recommend keep.
- **R7 — Single `triage.ts` file (fallback co-located), no separate `triage-fallback.ts`.** Keeps the safety surface in one reviewable file; SAFE_FALLBACK depends on the schema for self-validation anyway.
- **R8 — `parseTriage` accepts `unknown` AND JSON strings.** Providers return text (AI_PROVIDERS §3); accepting a string and `JSON.parse`-ing it (try/catch → `INVALID_JSON` reason) removes a foot-gun at the caller. Never throws on any input.
- **R9 — SAFE_FALLBACK floor = `VET_SOON`, not EMERGENCY, not MONITOR.** EMERGENCY over-alarms and erodes trust / desensitizes (PRODUCT_SPEC metric: watch over-triggering, §8 line 171); MONITOR/REASSURE under-alarms and is unsafe. VET_SOON is the documented low-confidence floor (§6.3) — the correct "we don't know, see a vet soon" tier. `confidence: low`, empty assessment arrays, neutral no-dosing `doNot`. Copy is verbatim above for the checker.
- **R10 — Property-style via hand-rolled generator loops, not `fast-check`.** CLAUDE §2 rule 7 (no new dep without justification). Table-driven loops over field/tier matrices give deterministic, flake-free "property" coverage of the fail-closed paths; a new fuzzing dep is unjustified for this bounded schema.
