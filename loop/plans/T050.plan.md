# Plan — T050: Check history + detail

## Objective (from card)
Per-pet symptom-check history list (chip + date + category) with tested empty/paginated
states; make T044's recent-checks placeholder live; and open the detail via deep link
`pawcareright://checks/:id`, where detail reuses the existing read-only result screen.

## Safety statement (PRODUCT_SPEC §5 / CLAUDE §7)
This task adds **list/detail navigation UI only**. It introduces NO new AI output, diagnosis
wording, or dosing surface.
- **Detail reuses the result screen unchanged.** The deep-link route is a thin alias that
  `Redirect`s to the already-§5-reviewed `app/check/result/[checkId].tsx` (disclaimer,
  emergency-first, fail-upward behavior all preserved — not re-implemented).
- **Chips fail upward, never downward.** Chip derivation precedence: red-flag → EMERGENCY
  tier chip (surfaces urgency, never buries it — §7 rule 4); terminal-with-result → that
  result's urgency; terminal-without-a-valid-result or FALLBACK → `SAFE_FALLBACK.urgency`
  (`VET_SOON`, §7 rule 5); still-running/queued & non-red-flag → a neutral "In progress"
  status chip (NOT a reassuring tier). No path invents a calmer tier for an unknown check.
- Chip labels **reuse the existing §7-reviewed** `strings.check.result.tierLabel` copy;
  only a neutral "In progress" status label and list-chrome copy are added.
Not a SAFETY-ESCALATION; safe to plan.

## Key decisions (see Risks for the scrutinizable ones)
1. **List placement / entry:** new route `app/check/history/[petId].tsx` (path param, mirrors
   `pets/[id]`, `check/[category]`). Entry point = the T044 recent-checks section on
   `app/check/index.tsx` made live (top 3 + "See all" → the history route). A pet-home entry
   link is **out of scope** (not required by the AC; avoids scope creep).
2. **List hook:** `useInfiniteQuery` (TanStack v5, cursor pagination) reusing the existing
   `checksKeys.list(petId)` key (already invalidated by `useCreateCheck`). No new shared Zod
   schema — the page type is built from the existing `CheckResponse` type (T047 D3 precedent:
   hooks type via `apiClient.get<T>`, no runtime parse of GET responses).
3. **Chip:** deterministic `deriveCheckChip(item)` — precedence in the Safety statement.
   The list endpoint's items already carry `status`, `redFlag?`, and `result?` (with
   `urgency`), so no API change is needed; the chip derives honestly from that payload.
4. **Deep-link route:** `app/checks/[id].tsx` renders `<Redirect>` to
   `/check/result/[checkId]` (read-only reuse is inherent — one detail screen, no
   duplication). Expo-router file mapping resolves `pawcareright://checks/:id` → this file
   (scheme already configured via `DEEPLINK_SCHEME` in `app.config.ts`). The native URL→route
   resolution is on-device behavior; the unit test asserts the alias redirects to the result
   screen carrying the id (documented on-device verification deferral).

## Interfaces / contracts (the executor must match these)

### `src/checks/check-history.ts`
```ts
import {
  INTAKE_CATEGORIES, SAFE_FALLBACK, isTerminalCheckStatus,
  type CheckResponse, type Urgency,
} from "@pawcareright/types";

export type CheckChip =
  | { kind: "tier"; urgency: Urgency }
  | { kind: "status"; status: "in-progress" };

export function deriveCheckChip(item: CheckResponse): CheckChip {
  if (item.redFlag !== undefined) return { kind: "tier", urgency: "EMERGENCY_NOW" };
  if (item.result !== undefined) return { kind: "tier", urgency: item.result.urgency };
  if (isTerminalCheckStatus(item.status)) return { kind: "tier", urgency: SAFE_FALLBACK.urgency };
  return { kind: "status", status: "in-progress" };
}

// Look up the human label from schema data; fall back to the raw id (never throws).
export function getCategoryLabel(category: string): string {
  return INTAKE_CATEGORIES.find((c) => c.id === category)?.label ?? category;
}

// Deterministic YYYY-MM-DD (matches the app's existing date convention; locale-free).
export function formatCheckDate(iso: string): string {
  return new Date(iso).toISOString().slice(0, 10);
}
```
`"EMERGENCY_NOW"` and `SAFE_FALLBACK.urgency` are typed `Urgency` enum values (data), not
user-facing copy.

### `src/api/checks-api.ts` (additions)
```ts
import { useInfiniteQuery } from "@tanstack/react-query"; // add to existing import

export interface CheckListPage {
  items: CheckResponse[];
  nextCursor: string | null;
}

export const CHECKS_LIST_PAGE_SIZE = 20;

export function useChecksList(petId: string) {
  return useInfiniteQuery({
    queryKey: checksKeys.list(petId),
    queryFn: ({ pageParam }: { pageParam: string | undefined }) => {
      const params = new URLSearchParams({ limit: String(CHECKS_LIST_PAGE_SIZE) });
      if (pageParam !== undefined) params.set("cursor", pageParam);
      return apiClient.get<CheckListPage>(`/v1/pets/${petId}/checks?${params.toString()}`);
    },
    initialPageParam: undefined as string | undefined,
    getNextPageParam: (lastPage: CheckListPage) => lastPage.nextCursor ?? undefined,
    enabled: petId.length > 0,
  });
}
```

## Risks & the decisions the planner made (for the checker to scrutinize)
1. **Deep link is a redirect alias, not a second detail screen.** `app/checks/[id]` redirects
   to `/check/result/[checkId]`. This keeps one §5-reviewed detail screen (DRY) at the cost of
   a brief URL change on open. Alternative (duplicating the result screen under `checks/[id]`)
   was rejected to avoid divergence of safety surfaces.
2. **Native scheme→route resolution is not unit-tested** (RNTL/jest can't exercise
   `getInitialURL` native mapping). The AC is satisfied by (a) the file-route existing at the
   exact card path `checks/[id]`, (b) the scheme being configured, and (c) a redirect test;
   on-device verification is explicitly deferred and noted in code.
3. **Chip derivation for terminal-without-result / FALLBACK uses `SAFE_FALLBACK.urgency`
   (VET_SOON), and QUEUED/RUNNING uses a neutral "In progress" status chip.** This is a
   deliberate fail-upward choice (§7 rule 5) rather than showing a blank or reassuring chip.
   Red-flag always wins to EMERGENCY (§7 rule 4).
4. **`formatCheckDate` returns locale-free `YYYY-MM-DD`.** Chosen for deterministic tests and
   consistency with the app's existing date convention; locale/region date formatting is
   T110/i18n territory and out of scope.
5. **`useInfiniteQuery` on the existing `checksKeys.list(petId)` key.** Assumes the invalidation
   in `useCreateCheck.onSuccess` (same key) correctly refetches the infinite list — it does
   (TanStack v5 refetches all pages on invalidate). No key/shape change to that mutation.
6. **Load-more is an explicit button, not `FlatList.onEndReached`.** Deterministic to test;
   satisfies cursor pagination without scroll-position simulation.
