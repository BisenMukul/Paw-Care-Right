## Design decisions the planner locked (scrutinize under "Risks")

1. **POTTY/FOOD "dedicated screens" = restyled presentation of the ONE existing logger, no new routes.** POTTY and FOOD are already `ActivityType`s in `ACTIVITY_TYPE_CONFIG` (POTTY = countWithChips `pee|poop|both`; FOOD = stepperWithUnit `meals|grams`). The mockup's "Log pee / Log poop" + QUANTITY stepper is realized by the existing 7-tile chip grid (tap 1) → quantity sheet (tap 2), restyled — NOT split into per-type routes (that is a router-target change, forbidden). The ≤2-tap contract and the `app/activity/[petId].tsx` undo/flush machinery are byte-frozen; only className/structure of the presentational shell changes.
2. **"TODAY'S INTAKE / Meals today / kcal" aggregate = OMITTED (un-backed).** The activity screen fetches NO logs (only `usePet` + the local `activity-recents-store`); there is nothing "already-fetched" to aggregate into "meals today", and kcal/`/650 kcal` targets are absent from the data model. Fabricating them would (a) require new API/data-fetch on this screen (a logic change) and (b) present an un-backed health/wellbeing claim — exactly the class PAWSAATHI-1 omitted (the "AI Health Score"). So no intake panel, no kcal, no fabricated "+45 kcal" quick-add tiles. The mockup's "Quick add" role is already served by the honest **recents row** (last real combos). **No new user-facing strings are introduced anywhere in this batch → the record-only tone scan surface is unchanged.**
3. **Care hub = the existing agenda (Today / Upcoming / new-reminder / care-plan link) restyled.** The mockup's "Health looks great · Run AI checkup" hero and the care-tools grid belong to a services surface (a later batch) and the hero is a wellbeing claim → OMITTED. No subtitle "…wellbeing" line is added (new string + wellbeing tone). Section headers "Today"/"Upcoming" adopt the Bricolage section-header role.
4. **Decorative kind tints stay decorative; no new AA text pair is introduced.** The timeline restyle moves each entry's `getKindDisplay().colorClass` tint from the whole-row background onto a small **rail dot** that contains only an emoji glyph (self-colored, no text-over-tint) — so those light Tailwind tints need no `dark:` variant and create no new normal-text contrast pair. All *text* stays on §1.1/§1.1a verified pairs (`brand-900`/`brand-700` light, `ink-dark`/`ink-muted-dark` dark). `category.*` colors are NOT wired to any small text. **Net: zero NEW light AA pairs → `dual-theme-contrast.test.ts` gains no new pair rows (only the equivalence assertion of follow-up 2).**
5. **pet-home / check-result / paywall / weight-chart snapshots must all stay BYTE-IDENTICAL.** No component in any of those four snapshots' render trees is touched this batch (pet-home = `app/pets/[id]` → PetHeaderCard/PrimaryButton/QuickActions/AnimatedGradientBackground; none in scope). The SecondaryButton follow-up only changes the *dark* icon/spinner hex — jest renders light (`useColorScheme()→null`), so even where SecondaryButton appears its captured `color` prop is unchanged (`#1f6350`). **Do NOT run `jest -u`.** Run the snapshot suites; they must pass unmodified. Any snapshot diff (incl. pet-home) is an unexpected regression → STOP and report, do not re-record. (The card's "pet-home may shift" is reconciled: verified tree-disjoint ⇒ expected delta set is EMPTY.)
