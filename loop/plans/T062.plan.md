## Key decisions (the minimum design choices — scrutinize these)
1. **Consistency check is a real BullMQ repeatable job, not just tests.** New trio in `apps/api/src/workers/`: `reminder-consistency.contract.ts` (queue/job/scheduler-id/pattern constants), `reminder-consistency.service.ts` (pure-ish scan logic taking an injectable `now: Date`, mirroring `ReminderSchedulerService`), `reminder-consistency.processor.ts` (thin `WorkerHost` + `onApplicationBootstrap` `upsertJobScheduler`). Its own queue `pawcareright-reminder-consistency` and hourly pattern `0 * * * *` — a SEPARATE scheduler id from the minute-tick so the existing `ReminderSchedulerProcessor` is left byte-for-byte unchanged (no second `upsertJobScheduler` bolted onto it).
2. **Detector is log-only, no auto-repair, household-agnostic full scan, bounded 24h window** (carry-forward wording). `checkConsistency(now)` returns a structured `ConsistencyReport` (so unit tests assert the return value, not `Logger` internals); the processor logs a one-line summary + one id-only line per discrepancy. It never writes/updates/deletes any row.
3. **Comparison keyed on `${reminderId}:${dueAt.getTime()}`** — the exact merge key `RemindersService.agenda` already uses — so "predicted" (from `occurrencesBetween`) and "materialized" (`ReminderEvent`) sets are compared on identical semantics. Two discrepancy classes: (a) **orphan event** = a materialized `SENT|DONE|MISSED|SNOOZED` event in-window whose key has no predicted occurrence; (b) **missing materialization** = a predicted occurrence with `dueAt <= now - GRACE` that has no event. A `GRACE_MS` (5 min) skirt on the "missing" side avoids false-positives on occurrences the minute-tick has not yet had time to materialize.
4. **Detector scans ALL reminders (active and inactive), no household batching.** `occurrencesBetween` already respects `COUNT`/`UNTIL`, so exhausted/deactivated series self-limit; a full `reminder.findMany({ select: id/rrule/timezone/startAt })` over a bounded window is honest and S-sized for beta scale. Documented as a known scale limitation (Risk R5), not fixed here.
5. **Master tz matrix lives in the header of `apps/api/src/reminders/timezone-matrix.spec.ts`**; every other new spec's header points to it with a one-line "see timezone-matrix.spec.ts §matrix, row X". Matrix axes: **Component × Scenario × tz-pair × DST-direction**, each cell tagged `[NEW: file]`, `[PRE-EXISTING: T053/T055/T056 spec]` (intentionally not re-tested), or `[N/A]`.
6. **DST fixtures are independently derived.** Before writing any fixture the executor runs a throwaway `node -e` Intl script (NOT committed) to confirm each offset, per the T053 pattern. Canonical 2026 transitions: US spring-forward **Mar 8**, fall-back **Nov 1** (`America/New_York` EST −05:00 / EDT −04:00); EU **Mar 29** / **Oct 25** (`Europe/Paris` CET +01:00 / CEST +02:00); `Asia/Kolkata` no DST (+05:30 always). A fixture that disagrees with the Intl script is a TEST bug (fix the fixture); a fixture that agrees but the machinery disagrees is a MACHINERY bug (STOP — see Risk R1).

## Interfaces/contracts (executor must match)
```ts
// reminder-consistency.service.ts
export interface ConsistencyDiscrepancy {
  reminderId: string;
  dueAtEpochMs: number;
  eventId?: string;            // present for orphan-event class
  kind: "orphan_event" | "missing_materialization";
}
export interface ConsistencyReport {
  from: Date;
  to: Date;                    // === now
  remindersScanned: number;
  predictedInWindow: number;
  materializedInWindow: number;
  discrepancies: ConsistencyDiscrepancy[];
}
export declare class ReminderConsistencyService {
  checkConsistency(now: Date): Promise<ConsistencyReport>;
}
```
- Window: `from = new Date(now - CONSISTENCY_WINDOW_MS)`, `to = now`.
- Predicted set: for each reminder, `occurrencesBetween(parsed.value, startAt, timezone, from, now)` → keys `${id}:${dueAt.getTime()}`. Skip a reminder whose `parseRRule` fails (defensive, log `warn` id-only).
- Materialized set: `reminderEvent.findMany({ where: { dueAt: { gte: from, lte: now } }, select: { id, reminderId, dueAt, status } })` → keys.
- Orphan = materialized key ∉ predicted. Missing = predicted key ∉ materialized AND `dueAt.getTime() <= now - CONSISTENCY_GRACE_MS`.
- Final `logger.log({ event: "reminder_consistency_check_done", remindersScanned, predictedInWindow, materializedInWindow, orphanCount, missingCount })` + per-discrepancy `logger.warn({ event: "reminder_consistency_discrepancy", kind, reminderId, dueAtEpochMs, eventId })`.

## Risks & the decisions the planner made
- **R1 — THE SWEEP FINDS A REAL BUG (protocol).** If a NEW test fails and the executor has confirmed (via the raw Intl script) that the fixture is correct, the failure is a genuine defect in existing Phase 5 machinery. The executor MUST STOP, write `loop/plans/T062.blocked.md` naming the failing `it(...)`, the expected-vs-actual instant, and the suspected root cause, and return — it must NOT edit machinery to make the test pass (that is a separate fix-forward decision for the orchestrator). Only a fixture that disagrees with the Intl script is fixable in-task (it is a test bug).
- **R2 — New job is production code with runtime side effects (a queue + repeatable scheduler).** Mitigated by isolating it on its own queue/scheduler id and keeping `workers.module.ts` changes purely additive; the existing minute-tick processor is untouched, so no existing behavior changes.
- **R3 — Consistency detector false positives after a legitimate reminder edit** (a tz/rrule change within the 24h window makes pre-edit events look like orphans). Accepted: the job is log-only, so a benign discrepancy is a `warn` line, not an action. Documented in the service doc-comment and covered by the orphan-event test (which uses exactly this scenario).
- **R4 — Grace-skirt tuning.** `GRACE_MS = 5min` chosen so a just-due occurrence the minute-tick hasn't yet materialized is not flagged missing. If too tight it false-flags; too loose it hides real drift for 5 min. 5 min >> 1-min tick cadence with margin — the honest minimum.
- **R5 — Full-table scan not household-batched / not paginated.** Acceptable at beta scale (matches the carry-forward's "household-agnostic full scan, bounded window"); flagged as a future optimization, not implemented here.
- **R6 — Mobile bucketing test asserts CURRENT (device-clock) behavior, not a "fix".** This is a characterization test documenting that `care.tsx` buckets by device `new Date()`; it makes no claim that device-vs-server drift is corrected. If the checker expects a behavior change, that is a NEW task, not T062.
- **R7 — File count for an S task.** Six test files + one production trio is at the upper edge of "S", justified because the card is an explicit cross-machinery consolidation sweep; each file is small and single-subject, mirroring existing spec structure.
