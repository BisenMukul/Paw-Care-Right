## Key decisions (each a design choice — scrutinize)
1. **Gate pinning = launch snapshot, NOT `staleTime: Infinity` on `useAppConfig`.** In TanStack Query
   `staleTime` is per-observer: it stops the gate's OWN observer from refetching, but the gate still
   shares the `["app-config"]` cache with the paywall observer (`usePaywallConfig` selects the same
   query, T079 decision 5). If any observer refetches, the cache updates and re-renders ALL observers —
   so `staleTime` alone does NOT pin the gate. Therefore freeze the value read at first mount:
   `const [launchMinSupportedVersion] = useState(() => config?.minSupportedVersion ?? DEFAULT_APP_CONFIG.minSupportedVersion);`
   The lazy `useState` initializer runs once, capturing the launch-time cached-or-default config
   (`useAppConfig` seeds `initialData`, so it is never undefined even on the first render). Subsequent
   refetches update `config` but the gate ignores them — the update decision is fixed at launch.
   This deliberately preserves the paywall variant's SWR behaviour (unchanged) while pinning ONLY the
   gate. No change to `app-config-queries.ts`, no `staleTime` added anywhere.
2. **Freeze direction is symmetric and fail-open by construction.** The snapshot fixes the decision at
   launch in BOTH directions: a later refetch reporting a higher min-version cannot raise the gate over
   an active session, and a later refetch reporting a lower one cannot drop it. The safety-critical
   direction (never raise over live children mid-session) is what the pinning test asserts. On the next
   cold launch the fresh config is honoured normally (new mount → new snapshot). Documented in Risks.
3. **Checklist is a standalone human runbook under `docs/qa/`.** New directory `docs/qa/`. Structure:
   `§1 Prerequisites` (dev-client build carrying the 3 native modules — `react-native-purchases`,
   `react-native-mmkv`, `react-native-nitro-modules`; `EXPO_PUBLIC_RC_IOS_KEY`/`EXPO_PUBLIC_RC_ANDROID_KEY`
   real vs stub; RC dashboard config per `docs/store-setup.md`; a reachable API URL for webhooks —
   ngrok/deployed; sandbox tester accounts — an App Store sandbox Apple ID + Play license-tester
   accounts) → `§2 How to read a step` (each step = action → expected UI → server-side check → PASS/FAIL
   box) → `§3` numbered scenarios ①–⑨ → `§4 Sandbox limitations & honesty notes` → `§5 Sign-off table`.
   Every scenario cites the real route/testID/label (harvested from the actual T071–T079 code, listed
   in "Interfaces" below) and the exact server assertion. No listing/marketing copy (that is M10). No
   `Paw Care Right +` hardcoding rules apply to code, not prose — the doc may use the display name in
   English text; it must use identifiers (`pawcareright_monthly`, `com.pawcareright.app`, entitlement
   `plus`) when naming console values.
4. **Sweep is verify-first, minimal-diff.** The ONLY guaranteed code/test change is the gate pinning
   (`update-gate.tsx` + `update-gate.test.tsx`). Test consolidation is opt-in: the executor may edit a
   Phase-7 test file ONLY on finding CLEAR duplication (two files asserting the identical thing) or a
   straggler `TODO` lacking a task ref; absent that, tests are left untouched (no churn). Any such edit
   is enumerated in `loop/journal.md`. Default expected diff: doc + gate + gate test + loop bookkeeping.

## Risks & the decisions the planner made (so checker can scrutinize)
- **R1 (SAFETY) — launch snapshot freezes BOTH directions.** After the snapshot, a same-session refetch
  can neither raise NOR drop the gate; the fresh value is honoured only on the next cold launch (new
  mount). This is deliberate: it makes the cold-launch-only guarantee enforced rather than emergent and
  guarantees no mid-session interposition over live children. The safety-relevant direction (never raise
  over an active flow) is the one that matters; the reverse is a benign consequence.
- **R2 — `staleTime: Infinity` would NOT suffice.** Documented in decision 1: per-observer `staleTime`
  cannot stop the shared `["app-config"]` cache from updating under the gate when the paywall observer
  (or a future online/focus bridge) refetches. Only a value snapshot pins the gate. Choosing `useState`
  lazy init over a `useRef` guard because it is lint-clean (no first-render mutation) and reads once.
- **R3 — snapshot relies on `initialData` being present at first render.** `useAppConfig` seeds
  `initialData` (cache-or-default), so `config` is defined on the first render and the snapshot is never
  captured from undefined. If a future refactor removed `initialData`, the fallback
  `?? DEFAULT_APP_CONFIG.minSupportedVersion` still yields the permissive `"0.0.0"` (fail-open). Both
  paths keep the gate off by default.
- **R4 — checklist accuracy drift.** Every route/testID/label is copied from the current code (step 1),
  not assumed; if any T071–T079 identifier differs at execution time, the executor MUST use the actual
  code value, not this plan's snapshot. The plan's "Interfaces" list is the reconciliation target.
- **R5 — sandbox honesty.** Webhook-dependent scenarios (②③④⑦) cannot pass on the stub-key dev boot;
  the doc must state the reachable-API-URL prerequisite up front so a tester never wastes a pass
  expecting a webhook that can't arrive. Store decline-simulation gaps are called out rather than
  presented as always-runnable.
- **R6 — no-churn discipline.** The sweep must resist rewriting green tests. Only clear duplication or a
  task-ref-less `TODO` justifies a test edit, each journaled. Expected default diff is exactly: the doc,
  `update-gate.tsx`, `update-gate.test.tsx`, and loop bookkeeping.
- **R7 — mobile test-count baseline.** Adding cases to the existing `update-gate.test.tsx` (rather than a
  new file) keeps the file count stable while raising the assertion count; if the checker counts files,
  this is expected and intentional.
