# Plan — T074: Paywall screen

## Objective (from card)
Build the Paw Care Right + paywall: onboarding placement (after first check result,
before the second check) + a Settings entry; a screen with three plan cards (annual
highlighted, family explainer), a 7-day-trial CTA, restore link, and legal links; A/B
copy variant driven by remote `/config`; purchase/restore via the T071 RC service.
Prices come only from the SDK offering. The paywall is a dismissible prompt — it NEVER
gates a check and is structurally incapable of interposing on the Emergency path (T075
owns quota enforcement; this task is placement only).

## Safety statement (PRODUCT_SPEC §5 / CLAUDE.md §7 — LOAD-BEARING)
This task touches a §5 surface indirectly (it adds a screen on the check-start seam), so
the emergency guarantee is designed structurally, not by copy:

- **The paywall trigger lives in EXACTLY ONE file: `apps/mobile/app/check/index.tsx`
  (the check-ENTRY screen), which is strictly UPSTREAM of the intake → submit →
  red-flag → `onEmergency` → `/check/emergency/[checkId]` path.** None of
  `check/[category].tsx`, `use-check-submission.ts`, `check/waiting/*`,
  `check/result/*`, or `check/emergency/*` import the trigger or the paywall. The
  red-flag branch (`use-check-submission.ts` → `onEmergency`) therefore has no paywall
  code on it and cannot be interposed on.
- **The paywall is always dismissible** ("Maybe later" → `router.back()`); it is a pushed
  modal over the entry screen, never a hard gate. It cannot block STARTING a check either.
- **No health/medical claims, no "diagnos*", no drug names/dosing** in any paywall copy
  (§7 rules 1–2). A test asserts the rendered text contains none of these tokens.
- **No `<VetDisclaimer/>`/emergency/red-flag copy is altered.** Emergency interstitial,
  result screen, and disclaimer components are untouched.
- No SAFETY-ESCALATION: the task can be implemented without weakening any safety surface.

## Key decisions (each is a design choice — scrutinize)
1. **Trigger placement = check-entry screen only, one-time, dismissible modal.** In
   `check/index.tsx`, a `useEffect` redirects to `/paywall?source=onboarding` when
   `premiumStatus === "free"` AND `completedCheckCount >= 1` AND `!paywallShown`. This is
   "after first check result, before second": the entry screen is where a *second* check
   begins, and it is upstream of all AI/red-flag logic. Redirect is `router.push` (modal),
   so entry stays mounted underneath and the user can dismiss. Shown-once via a persisted
   MMKV flag (decision 6). This is the ONLY structural place the AC's "never blocks
   Emergency" can be guaranteed and is the launching agent's directed placement.
2. **Dismissibility / no enforcement.** The paywall never blocks a check. "Maybe later"
   dismisses; there is no quota gate here (T075 owns server enforcement; the existing
   `check-submit-quota` state in `check/[category].tsx` is T075's surface and is NOT
   wired to this paywall in scope).
3. **Entitlement source = client premium store fed by RC `getCustomerInfo()`.** New
   non-persisted zustand `usePremiumStore` with `status: "unknown" | "entitled" | "free"`.
   The trigger fires ONLY when `status === "free"` (definitively not entitled). `"unknown"`
   (pre-load, or Expo Go where the native module is absent) NEVER triggers — fail-safe:
   we never nag or block on unknown state. The store is refreshed by the trigger hook on
   mount and set from `customerInfo` after every purchase/restore. (We do NOT modify
   T071's `use-purchases-init.ts`, to keep that passing file/test stable.)
4. **Remote config endpoint = new minimal public `GET /v1/config`** returning
   `{ paywall: { variant: "A" | "B" } }`. Variant is read from a validated env var
   `PAYWALL_VARIANT` (default `"A"`), letting ops flip the variant without a client
   deploy. No DB, no auth. Client `usePaywallConfig()` defaults to `"A"` on any
   error/offline (offline-safe).
5. **Variant mechanism = server sends the variant ID only; all copy lives client-side**
   in `strings.ts` (`strings.paywall.variants.A` / `.B`). The server never sends arbitrary
   prose — safest for §7 review and i18n-ready (T110). Both variants are snapshotted.
6. **Shown-once flag = persisted MMKV store** `usePaywallShownStore` (`createSafeStorage`
   MMKV pattern, memory fallback in Expo Go/jest), mirroring `active-pet-store.ts`. Set
   the moment the onboarding paywall is shown, so it never nags on subsequent entries.
7. **Price sourcing = exclusively from the RC offering.** `priceString` / `introPrice`
   from `getOfferings().current` — never hardcoded. The 7-day-trial CTA reflects the
   monthly package's `introPrice` when present, and degrades to a plain subscribe label
   when absent. If offerings are empty/unavailable (Expo Go), the screen shows a graceful
   "unavailable" state (no crash, no fake prices).
8. **Billing SDK extension (T071 service).** Extend `apps/mobile/src/billing/purchases.ts`
   (same lazy-guarded `native` pattern) with `fetchOfferings`, `purchasePackage`,
   `restorePurchases`, `fetchCustomerInfo`, `isEntitled`. `purchasePackage` RESOLVES (never
   rejects) to a typed union `PurchaseOutcome = {status:"success", customerInfo} |
   {status:"cancelled"} | {status:"pending"} | {status:"error"}`, mapped from the thrown
   RC error's fields: `e.userCancelled === true` → cancelled; `e.code ===
   "PAYMENT_PENDING_ERROR"` → pending; else error. `isEntitled(customerInfo)` checks
   `customerInfo.entitlements.active[RC_ENTITLEMENT_ID] !== undefined`.
9. **Legal URLs = mobile config only** (`app.config.js` `extra` + `src/config.ts`
   `AppConfig`), defaulting to `https://pawcareright.app/terms` and `/privacy` (§1a web
   placeholder until T102). Opened via `expo-linking`. Avoids a cross-package
   `packages/config` rebuild.

## Risks & edge cases (for the checker)
- **R1 (decisions 1/3) emergency-path structural guarantee.** The whole AC rests on the
  trigger living ONLY in `check/index.tsx`. Checker MUST verify no other check-flow file
  imports `use-paywall-trigger`/`paywall`, and that `paywall-emergency-safety.test.tsx`
  proves a red-flag second check reaches the interstitial with eligible state active.
- **R2 (decision 3) Expo Go / native null.** `fetchCustomerInfo` → null ⇒ status stays
  `"unknown"` ⇒ onboarding never auto-fires in Expo Go (devs reach it via Settings). Prices
  unavailable ⇒ `paywall-unavailable` state (no crash, no fake price). Accepted.
- **R3 (decision 8) RC error-code string drift.** Pending is matched by the literal
  `"PAYMENT_PENDING_ERROR"`; cancel by `e.userCancelled === true` (the robust signal).
  Mitigation: unit test pins both; a comment references `PURCHASES_ERROR_CODE`. If the value
  drifts, worst case a pending shows as a generic error (safe, non-blocking).
- **R4 purchase pending → app restart.** No local pending persistence; on relaunch, RC
  `getCustomerInfo` reflects the real entitlement and the premium store/trigger settle from
  truth. Accepted (RC is source of truth).
- **R5 restore on a new device.** Handled by `restorePurchases`; success+!entitled shows a
  neutral "no purchases found" (`paywall-restore-none`), never an error.
- **R6 variant flicker.** `usePaywallConfig` seeds `initialData {variant:"A"}` so the screen
  never renders an undefined/loading variant; a later server `"B"` swaps copy once. Snapshots
  mock the hook to a fixed value, so no flake.
- **R7 checkCount staleness.** `completedCheckCount` derives from the entry list's first page;
  if stale (0) the onboarding paywall shows late or not at all — safe either direction (never
  blocks). Not defended further.
- **R8 module name collision.** New api module is `remote-config` (NOT `config`) to avoid
  colliding with the existing env `ConfigModule`/`AppConfigService`.
- **R9 modal over emergency.** Because the trigger is `push` (modal) from the ENTRY screen and
  the emergency route is reached from intake/submit, the modal can never sit above the
  emergency interstitial; and even the entry modal is dismissible. Verified by R1's test.
