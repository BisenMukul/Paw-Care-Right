# Plan — T039: Cost controls & quotas

## Objective (from card)
Ship a Redis-backed per-user quota service (checks + food lookups) with entitlement-aware limits mirroring SPEC §7/F8 exactly (interfaces only; billing wiring is P7); add an advisory prompt-caching flag on the static triage system prompt; add per-check cost logging (costMicroUsd) plus a daily-aggregate metric hook. No DB model, no endpoints, no worker (T041/T042/T043), no RevenueCat (P7).

## Safety statement (read before coding)
Quotas are **availability/cost controls, NOT a PRODUCT_SPEC §5 safety surface** — this task adds no user-facing AI copy, no disclaimers, no emergency/dosing logic, so it does not weaken any §5 surface. Not a SAFETY-ESCALATION. Two safety-adjacent guardrails to respect:
- **Prompt-caching change must be behavior-neutral.** The `cacheable` flag added to the provider request is a pass-through hint only. It must NOT alter `runTriage`'s fail-upward control flow (parse → one repair → frozen `SAFE_FALLBACK`), temperature 0, schema validation, or any parsed output. If honoring it would change any of those, do not wire it — stop and flag.
- **T042 design note (carry-forward, do NOT implement here):** when T042 wires quota into `POST /checks`, the deterministic red-flag Emergency interstitial (CLAUDE §7 rule 4 / SPEC §5.3) must render **before** any quota rejection. Quota enforcement must never gate or delay the emergency path. This service returns a plain typed result (no throw), leaving that ordering to T042 — record it, don't build it.

## Key decisions (the design choices the checker must scrutinize — see Risks)
1. **Placement:** new `apps/api/src/quota/` module (Redis lives in the api; T042/T043 consume it). Cost logging lives in the same module (`CostLogService`), not `packages/ai`.
2. **"Persisted" cost, with no SymptomCheck table yet (deps are only T029,T035):** persistence for T039 = (a) a structured Nest `Logger` line per run and (b) a Redis **daily-aggregate** counter. T043 later ALSO writes `costMicroUsd` onto the `SymptomCheck` row — that is out of scope here. The accept criterion "cost persisted on fake runs" is satisfied by asserting the Redis daily aggregate equals the summed `costMicroUsd` of fake runs.
3. **Cost input is a plain structural type (`CostLogEntry`), not `TriageRunResult`.** Avoids adding an `apps/api → @pawcareright/ai` dependency edge now (that edge lands at T043 when triage is wired into the worker). T043 maps `TriageRunResult.usage.costMicroUsd` → `CostLogEntry`.
4. **Consume semantics = single atomic `INCR`, `allowed = newCount <= limit`, no compensating decrement.** Race-safe (one op), fails in the safe direction (a rejected over-limit attempt cannot serve work). Rejected attempts leave the counter above the limit — harmless, the metric stays "exceeded" until the window resets.
5. **Windows day | month | total.** SPEC F8 free checks is "1 **total**" (lifetime, survives reinstall per §7) → `total` window, no TTL. The card's "daily/monthly" describes the counter mechanism; SPEC supplies the exact numbers. Both are honored by a generic window system (see Risk R1).
6. **Prompt-caching flag = optional advisory `cacheable?: boolean` on `TextGenerateOptions`**, set `true` at the two triage `provider.generate(...)` calls (the system segment is static/deterministic). Ollama Cloud (OpenAI-compat) exposes no cache flag today and Gemini never touches text triage (AI_PROVIDERS §3/§4), so provider impls currently ignore it — an honest documented pass-through for T040+ real runs. `BuiltTriagePrompt` is intentionally NOT changed (avoids the `build.spec` snapshot churn).

## Interfaces / contracts (executor must match these shapes)
```ts
// quota.types.ts
export type EntitlementTier = "FREE" | "PREMIUM";
export type QuotaMetric = "checks" | "foodLookups";
export type QuotaWindow = "day" | "month" | "total";

export interface QuotaLimit { window: QuotaWindow; limit: number | null; } // null = unlimited
export interface Entitlement { tier: EntitlementTier; bypassQuota: boolean; }

export interface QuotaConsumeResult {
  allowed: boolean;
  metric: QuotaMetric;
  window: QuotaWindow;
  limit: number | null;      // null = unlimited
  used: number;              // count AFTER this consume; 0 when unlimited/bypass
  remaining: number | null;  // null = unlimited
  unlimited: boolean;
}

export interface CostLogEntry {
  costMicroUsd: number;
  latencyMs?: number;
  model?: string;
  status?: string;   // e.g. "OK" | "REPAIRED" | "SAFE_FALLBACK"
  userId?: string;
  checkId?: string;
}

// entitlement.ts
export const ENTITLEMENT_RESOLVER = "ENTITLEMENT_RESOLVER"; // DI token (string or Symbol)
export interface EntitlementResolver { resolve(userId: string): Promise<Entitlement>; }

// quota.service.ts
class QuotaService {
  constructor(redis: RedisService) {}
  consume(userId: string, metric: QuotaMetric, entitlement: Entitlement, now?: Date): Promise<QuotaConsumeResult>;
}

// cost-log.service.ts
class CostLogService {
  constructor(redis: RedisService) {}
  record(entry: CostLogEntry, now?: Date): Promise<void>;
  getDailyAggregate(now?: Date): Promise<number>;
}
```

## Quota semantics (implement exactly)
- **Keys:** `pawcareright:quota:{metric}:{window}:{userId}:{bucket}` where `bucket = windowBucket(window, now)`:
  - `day` → `YYYY-MM-DD` (UTC), `month` → `YYYY-MM` (UTC), `total` → `all`.
- **TTL (`QUOTA_TTL_SECONDS`):** `day` = 172800 (48h — margin past the UTC midnight rollover + auto-cleanup); `month` = 3456000 (40d — longest month + margin); `total` = `null` (persistent; SPEC §7 "counters survive reinstall"). Call `redis.expire(key, ttl)` **only when `used === 1`** and `ttl !== null`.
- **`consume(userId, metric, entitlement, now = new Date())`:**
  1. `{ window, limit } = resolveLimit(entitlement.tier, metric)`.
  2. **Bypass path:** if `entitlement.bypassQuota === true` OR `limit === null` → return `{ allowed:true, metric, window, limit, used:0, remaining:null, unlimited:true }` and **do not call Redis at all** (no INCR, no key created).
  3. Otherwise `used = await redis.incr(key)`; if `used === 1` set TTL as above.
  4. `allowed = used <= limit`; `remaining = Math.max(0, limit - used)`; return with `unlimited:false`.
- **Reset:** natural via the date-derived bucket in the key — a new day/month is a new key. No cron. `total` never resets.
- **Isolation:** per-user (userId in key) and per-metric+window (metric+window in key) are structurally independent — no shared counter.
- **No throwing:** `consume` returns a typed result; HTTP/402 mapping is T042's job.

## Cost logging & daily-aggregate hook semantics
- **`record(entry, now = new Date())`:**
  1. Emit ONE structured Nest `Logger` line (`new Logger(CostLogService.name).log(...)`) carrying the entry fields (never `console.log`). A run is always logged, even when cost is 0 (SAFE_FALLBACK / no provider usage).
  2. `micro = Math.max(0, Math.round(entry.costMicroUsd))`. If `micro === 0` → return after logging (aggregate unchanged).
  3. `total = await redis.incrBy(costDailyKey(now), micro)`; if `total === micro` (first write of the day) → `redis.expire(costDailyKey(now), COST_AGGREGATE_TTL_SECONDS)`.
- **`getDailyAggregate(now = new Date())`:** `redis.get(costDailyKey(now))`; return `0` when null else `Number(raw)`.
- **Key:** `pawcareright:cost:daily:{YYYY-MM-DD}` (UTC, global — not per-user). `COST_AGGREGATE_TTL_SECONDS` = 3024000 (35d, so a monthly metrics job/admin can read it).
- **The "metric hook":** this global daily counter IS the extension point a later metrics sink (PostHog, T078) reads/subscribes to. Do **not** build a sink abstraction now (over-engineering = defect).

## Tests to write (map to every acceptance criterion)
**AC "Quota service unit tests (boundaries, reset windows, premium bypass flag)"** — `quota.service.spec.ts` (fake Redis `Map`, injected `now`):
- `under-limit consume` — FREE foodLookups: calls 1–4 all `allowed:true`, `remaining` = 4..1.
- `exact boundary` — FREE foodLookups: 5th call `allowed:true, used:5, remaining:0`; 6th `allowed:false, used:6`.
- `checks total limit` — FREE checks: 1st `allowed:true`; 2nd `allowed:false` (limit 1, window total).
- `daily vs monthly windows independent` — PREMIUM checks (month, 30) counts under a `YYYY-MM` key while FREE foodLookups (day) uses a `YYYY-MM-DD` key; assert distinct keys / independent counts.
- `daily window rollover` — FREE foodLookups: exhaust 5 at `now=2026-07-12T…Z` (6th rejected); with `now=2026-07-13T…Z`, next call `allowed:true` (new key).
- `monthly window rollover` — PREMIUM checks: exhaust 30 in `2026-07`; `now` in `2026-08` → `allowed:true`.
- `premium bypass flag` — `{ tier:"PREMIUM", bypassQuota:true }`: `consume` returns `allowed:true, unlimited:true, used:0` and `redis.incr` is **not** called. Also PREMIUM foodLookups (`limit:null`) → `unlimited:true`, no INCR.
- `per-metric independence` — consuming `checks` does not change the `foodLookups` count for the same user.
- `per-user isolation` — user A at limit does not reject user B.
- `TTL set once` — `expire` called with 172800 when `used===1` (day metric), not called on `used===2`; for FREE checks (`total`, ttl null) `expire` is never called.
- `StaticEntitlementResolver` — `resolve("u1")` → `{ tier:"FREE", bypassQuota:false }`.

## Risks & decisions (numbered — for the checker)
1. **SPEC "1 check total" vs card "daily/monthly checks."** Reconciled by a generic `day|month|total` window system: FREE checks uses `total` (SPEC-exact, survives reinstall), PREMIUM checks uses `month` (fair-use 30/mo). Card's "daily/monthly" is read as the counter mechanism, SPEC as the authoritative numbers (SPEC is supreme). If the checker requires FREE checks to be a daily/monthly window, that contradicts SPEC §7 and should be raised, not silently changed.
2. **Consume = atomic INCR without compensating decrement.** Over-limit rejected attempts leave the counter above the limit (still rejected until reset). Chosen for single-op race-safety and safe-direction failure; the alternative (read-modify-write or INCR+DECR) adds races/complexity for a non-security cost control. Accepted trade-off: the counter/used value can exceed the limit by the number of rejected attempts.
3. **`bypassQuota` flag distinct from tier.** It is a hard "skip enforcement" escape hatch (comped/internal accounts) AND the mechanism by which unlimited-limit metrics (PREMIUM foodLookups, `limit:null`) short-circuit. The "premium bypass flag" AC is tested via `bypassQuota:true`. P7 sets these from the RevenueCat mirror.
4. **Cost input decoupled from `TriageRunResult` (`CostLogEntry`).** Keeps `apps/api` free of an `@pawcareright/ai` import until T043. If the checker expects direct `TriageRunResult` consumption, note the dep-edge deferral is deliberate.
5. **`cacheable` is an inert pass-through today.** No provider honors it (Ollama Cloud has no cache flag; Gemini is image-only). It is honest advisory metadata for T040+ real runs, not fake caching. It must not change any triage output or fail-upward behavior.
6. **Daily cost aggregate is a single global Redis counter, not a sink abstraction.** Interpreted as the minimal "metric hook"; a PostHog/T078 sink layers on later. Building an abstraction now would be over-engineering (CLAUDE §2 rule 2).
7. **`now` injected as a defaulted method parameter** (`now = new Date()`) rather than a DI clock, matching the lightweight existing api service style and enabling deterministic window/rollover tests without extra DI wiring.
