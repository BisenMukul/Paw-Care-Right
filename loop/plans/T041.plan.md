## Prisma model spec (transcribe verbatim; adjust only relation-field naming to satisfy `prisma format`)
```prisma
enum CheckStatus {
  QUEUED
  RUNNING
  DONE
  FALLBACK
}

model SymptomCheck {
  id                String        @id @default(uuid())
  petId             String
  createdById       String
  status            CheckStatus   @default(QUEUED)
  category          String
  intakeJson        Json
  photoKeys         String[]      @default([])
  redFlagHit        Boolean       @default(false)
  redFlagRuleId     String?
  redFlagPayloadKey String?
  costMicroUsd      Int           @default(0)
  failureReason     String?
  idempotencyKey    String?
  createdAt         DateTime      @default(now())
  startedAt         DateTime?
  completedAt       DateTime?
  updatedAt         DateTime      @updatedAt

  pet       Pet           @relation(fields: [petId], references: [id], onDelete: Cascade)
  createdBy User          @relation(fields: [createdById], references: [id], onDelete: Restrict)
  result    TriageResult?

  @@unique([createdById, idempotencyKey])
  @@index([petId, createdAt(sort: Desc)])
  @@index([createdById])
}

model TriageResult {
  checkId       String   @id
  urgency       String
  confidence    String
  resultJson    Json
  modelId       String
  promptVersion String
  createdAt     DateTime @default(now())

  check SymptomCheck @relation(fields: [checkId], references: [id], onDelete: Cascade)
}
```

## Status transition semantics (the acceptance-critical contract)
Legal edges — EXACTLY these three (interpret the card literally: "QUEUED→RUNNING→DONE|FALLBACK only"):
- `QUEUED → RUNNING`
- `RUNNING → DONE`
- `RUNNING → FALLBACK`

Everything else is ILLEGAL, including: `QUEUED → DONE`, `QUEUED → FALLBACK` (see R4), any self-edge/no-op (`QUEUED→QUEUED`, `RUNNING→RUNNING`, `DONE→DONE`, `FALLBACK→FALLBACK`), any backward edge (`RUNNING→QUEUED`, `DONE→RUNNING`), and any edge out of a terminal state (`DONE→*`, `FALLBACK→*`). Terminal states: `DONE`, `FALLBACK`.
Transition table the helper encodes:
```
QUEUED   -> [RUNNING]
RUNNING  -> [DONE, FALLBACK]
DONE     -> []
FALLBACK -> []
```

## Index justification table
| Index | Justification |
|---|---|
| `SymptomCheck(petId, createdAt DESC)` | Explicitly required by ARCHITECTURE §3; serves T042 `GET /pets/:id/checks` and T050 per-pet history list (cursor-paginated, newest-first). Also covers the `petId` FK as a leftmost-prefix, so no separate `petId` index is needed. |
| `SymptomCheck(createdById)` | FK index (ARCHITECTURE "every FK"); supports the `@@unique([createdById, idempotencyKey])` lookups and per-user queries. |
| `SymptomCheck` unique `(createdById, idempotencyKey)` | Enforces T042 idempotent replay: a given user replaying the same key resolves to the same check. `idempotencyKey` is nullable → Postgres treats NULLs as distinct, so non-idempotent/legacy inserts are unconstrained (standard pattern). Scoped to the requesting user (`createdById`), the natural idempotency owner — see R5. |
| `TriageResult.checkId` (PRIMARY KEY) | 1:1 with `SymptomCheck` (ARCHITECTURE `SymptomCheck ||--|| TriageResult`); using `checkId` as `@id` both enforces the FK-unique 1:1 and indexes the join. |
| (No standalone `status` index) | The check-runner is BullMQ-driven (jobs carry `checkId`); the DB is never scanned by status. Adding a `status` index now would be an unused index (over-engineering, CLAUDE §2 rule 2). Deferred until a query needs it. |

## Risks & the design decisions the planner made (for CHECKER scrutiny)
- **R1 — Two models, TriageResult keyed by `checkId` as PK.** ARCHITECTURE shows `SymptomCheck ||--|| TriageResult` (1:1) and lists `TriageResult(checkId, ...)` with no surrogate id, so `TriageResult` uses `checkId String @id` (unlike the T011 Membership surrogate-id pattern) to make the 1:1 exact and index the join for free. Reversible if the checker prefers `id @default(uuid()) + checkId @unique`.
- **R2 — `CheckStatus` lives in `packages/types` AND as a Prisma enum; kept aligned by a parity test.** They cannot share source (Prisma enums are schema-side). The helper depends on the types version; the parity test in `check-status.spec.ts` fails loudly if the two drift. Decision per CLAUDE §6 (cross-package enums live in `packages/types`).
- **R3 — `onDelete`: `pet → Cascade`, `createdBy → Restrict`, `TriageResult.check → Cascade`.** A check belongs to its pet, so it dies with the pet (Cascade, mirrors Pet→Household). `createdBy` is the audit "who ran it"; `Restrict` preserves check history and blocks deleting a user who still has checks (mirrors Household.owner in T011) — account-deletion reassignment is a later task. `TriageResult` dies with its check. All reversible.
- **R4 — `QUEUED → FALLBACK` is treated as ILLEGAL.** The acceptance string is read literally ("QUEUED→RUNNING→DONE|FALLBACK only"): the worker must move a check to `RUNNING` before it can `FALLBACK`. If a future path needs a pre-run fallback, that edge is added deliberately then. Checker should confirm this literal reading is intended.
- **R5 — Idempotency scoped to `(createdById, idempotencyKey)`.** The requesting family member is the idempotency owner; `idempotencyKey` nullable so only client-supplied keys are constrained (Postgres NULL-distinct). Alternative scope (household/pet) rejected as broader than the client contract.
- **R6 — Forward-looking columns (`redFlagRuleId`, `redFlagPayloadKey`, `failureReason`, `idempotencyKey`, `startedAt`, `completedAt`, denormalized `urgency`/`confidence`/`category`).** These exceed ARCHITECTURE's abbreviated field list but are mandated by the card's explicit goal that T042/T043/T050 need zero further migration. They are nullable/defaulted and inert until those tasks populate them — not speculative features, just the columns their queries require. Checker should confirm each maps to a named downstream consumer (table above).
- **R7 — New `apps/api/src/checks/` dir holds only the pure helper + spec, no Nest module.** T042 owns module wiring; adding a module now would be unused scaffolding (CLAUDE §2 rule 2). The orphan `.ts` is typechecked/linted/built by tsc and its spec run by jest without being imported anywhere — verified by the command suite.
- **R8 — `id` strategy `@default(uuid())`** to stay consistent with `brandedId = z.string().uuid()` in `packages/types` (same rationale as T011 R7).
