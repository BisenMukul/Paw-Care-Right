# Plan — T069: Photo attachments on logs

## Objective (from card)
Extend multi-photo attach to **VET_VISIT** logs (reusing T066's `HealthLogPhotoPicker` + T023's upload machinery), render **thumbnails** for any timeline entry that carries `photoKeys`, and add a **full-screen viewer** (swipe/paged) opened by tapping a thumbnail. Because photos are currently **never displayed back to the user** (verified: no `getPresignedGetUrl`, no view endpoint anywhere), this task adds one **minimal, household-guarded pet-photo view-URL endpoint** mirroring the existing `PhotosController` upload endpoints, plus the mobile display path. Accept: upload-reuse tests on VET_VISIT + viewer accessible-label test.

## Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)
Photos are **records**, not assessments. Nothing in this task produces AI output, guidance, or interpretive copy. Every new string is a neutral record/a11y label (photo counts, "Close photo viewer"); none contains "diagnos*", a drug name, or a dose pattern. This touches **no** disclaimer, emergency-escalation, or dosing surface. **Not a SAFETY-ESCALATION.**

## Key decisions (checker: scrutinize)
1. **View-URL sourcing — new endpoint `POST /v1/pets/:id/photo-view-urls`.** No view/download machinery exists today (`StorageService` has only `getPresignedPutUrl`; `PhotosController` has only upload/confirm; the check-result screen renders no photos). So we add the minimal thing: a household-guarded endpoint that presigns **GET** URLs. It lives in the existing **`photos` module** (generic pet-photo concern, mirrors the upload endpoints' `@Controller("pets") @HouseholdFromMembership()` posture), not `health-logs`. `POST`+`@HttpCode(200)` mirrors the sibling `photo-upload-url` (a read-ish presign already modeled as POST-200 in this repo). Body `{ keys: string[] }` (≤ `HEALTH_LOG_PHOTO_KEYS_MAX`, ≥1). It returns **both** rendition URLs per key so one round-trip serves thumbnails *and* the viewer.
2. **Renditions: thumb for the grid, main for the viewer, derived server-side.** Stored `photoKeys` are the **original** keys (`uploadIntakePhoto` returns the key from `createUploadUrl` → `pets/<petId>/original/<uuid>.<ext>`). The T023 images worker already produces `pets/<petId>/thumb/<uuid>.jpg` and `pets/<petId>/main/<uuid>.jpg`. The service derives them with the existing `deriveThumbKey`/`deriveMainKey` and signs each — the **client never learns the key scheme**. Response item: `{ key, thumbUrl, mainUrl }`.
3. **Scope guard = `petsService.findOne(householdId, petId)` + namespace check.** Exactly the `confirmUpload` posture: `findOne` gives a 404 (never a leaking 403) for a pet outside the caller's household; every requested key must `startWith(originalKeyPrefix(petId))` or the request is a `400`. A caller can therefore only ever presign their own pet's objects.
4. **Memo-safe thumbnail data flow — a separate memoized child component owns the query.** `TimelineRow` stays `React.memo`'d and keeps calling `getKindDisplay` exactly once per actual render (the T067 render-count AC). Thumbnails are rendered by a **new, separately-memoized** `TimelinePhotoStrip`, embedded in `TimelineRow` **only when `item.photoKeys.length > 0`**. When the strip's URL query resolves, **only the strip re-renders** — `TimelineRow`'s body (and `getKindDisplay`) is not re-invoked, so the append-page render-count proof still holds (and the existing fixtures all use `photoKeys: []`, so the strip never even mounts in that test). `TimelineRow` gains two **stable** props — `petId` (string) and `onOpenPhoto` (a `useCallback([])` from the screen) — so its memo still bails on unchanged rows.
5. **Viewer shape — in-screen RN `Modal` overlay, no new route, no lightbox dep.** `TimelinePhotoViewer` is a presentational component controlled by timeline-screen state `{ petId, photoKeys, index } | null`. It renders a `Modal` containing a horizontally-**paged `FlatList`** of full-screen `expo-image` (main renditions, `contentFit="contain"`), `initialScrollIndex = index`, a close button, and a per-page accessibility label. Simplest correct S-size choice; horizontal paging = the "swipe across the entry's photos" requirement with zero new dependencies.
6. **Single shared query `usePhotoViewUrls(petId, photoKeys)` for both strip and viewer.** Same query key (`["pet-photo-view-urls", petId, photoKeys.join(",")]`) ⇒ the viewer reuses the strip's already-cached URLs **instantly**, and if the presign is stale it refetches on mount. `staleTime = 4 min` (below the 300 s presign expiry) keeps URLs from churning per render while still refreshing before they die. Passing `petId + photoKeys + index` (not URLs) to the viewer keeps expiry handling in one place (decision-6 mitigates the "expiry mid-view" risk).
7. **expo-image everywhere for display (§6).** Reuse the exact `import { Image } from "expo-image"` pattern already in `HealthLogPhotoPicker`. Thumbnails: fixed `h-16 w-16 rounded-md`; a lightweight placeholder while pending/on-error (missing rendition) so a not-yet-processed object degrades to a placeholder, never a crash.
8. **VET_VISIT wiring reuses everything — no new API.** `CreateLogDto` already accepts top-level `photoKeys` for every creatable kind (incl. `VET_VISIT`); the service already persists them. So only the client changes: `AddVetVisitForm` renders the existing `HealthLogPhotoPicker`, and `useAddVetVisit`'s vars become `{ value: VetVisitValue; photoKeys: string[] }`, omitting `photoKeys` from the POST body when empty (byte-identical to `useAddNote`'s pattern).

## Risks & edge cases (checker: scrutinize)
- **Signed-URL expiry mid-view (300 s):** decision 6 sets `staleTime` below expiry and shares the query so the viewer refetches on mount when stale; images already decoded by `expo-image` stay cached. A viewer left open >5 min then paging to an un-decoded page can fail to load → placeholder, no crash. Accepted for S; documented.
- **Missing/not-yet-processed rendition:** a freshly-added photo whose worker job hasn't produced thumb/main yet → the presigned GET 404s at S3 → `expo-image` shows the placeholder. No HEAD pre-check (extra I/O) for S; the older records that dominate the timeline always have renditions. Documented.
- **Deleted object:** same failure mode as above — placeholder, never a throw. The endpoint signs a URL regardless of existence (cheap, avoids per-key HEAD); the client degrades gracefully.
- **6-photo grid layout:** `flex-row flex-wrap gap-2` with fixed `h-16 w-16` tiles caps at `HEALTH_LOG_PHOTO_KEYS_MAX = 6`, wrapping to two rows on narrow screens — no horizontal overflow.
- **Render-count regression (T067):** the strip is a *separate* memoized component, so its query resolving never re-invokes `getKindDisplay` in `TimelineRow`; the new `petId`/`onOpenPhoto` props are stable (string + `useCallback([])`), so `TimelineRow`'s memo still bails on unchanged rows. Existing fixtures use `photoKeys: []`, so that test is untouched.
- **Namespace guard:** the endpoint presigns GETs only for keys under the caller's own pet (`findOne` + `startsWith(originalKeyPrefix(petId))`), mirroring `confirmUpload` — no cross-pet/household object exposure. Verified by the e2e `400`/`404` cases.
- **VET_VISIT body shape change:** `AddVetVisitVars` becomes an object `{ value, photoKeys }`; the existing screen test's assertion must be updated in lockstep (listed in Modify) or it fails loudly — intended signal, not silent drift.
