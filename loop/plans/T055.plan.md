# Plan — T055: Reminders CRUD API

## Objective (from card)
Ship the reminders NestJS module on top of T053's schema + helpers: household-scoped CRUD for `Reminder`, a household-wide **agenda** endpoint that expands each active reminder's rrule in its own timezone across `[from,to]` and merges the result with materialized `ReminderEvent`s, and a **template-instantiate** endpoint that turns a T054 care pack into a batch of reminders for a pet — **idempotently**. Deps: T053 (models + `computeNextFireAt` + `CreateReminderDto` + `@IsRRule`), T015 (household guards/infra). No scheduler/worker/push (T056/T057), no mobile (T059/T060).

## Safety statement (read first)
- This task is **CRUD + a read/expansion endpoint + a data-instantiation endpoint**. It is NOT itself a PRODUCT_SPEC §5 medical-content surface: no AI triage, no user-facing guidance copy, no emergency/hotline logic, no disclaimer components, no dosing. It cannot weaken any §5 safety surface, so it is planned (not blocked). No `SAFETY-ESCALATION`.
- Reminders are **care-critical for timing**; the agenda/DST correctness here is a reliability concern (missed/mistimed care), fully covered by the AC's DST + monthly-31st unit tests. It ships no delivery (that's T056/T057).
- **Template instantiation carries T054's safety copy through UNTOUCHED.** Each created `Reminder` copies the resolved item's `title` verbatim; the T054 items are schema-`.refine`-guaranteed to include the pinned `VET_CONFIRM_SENTENCE` in their `note`. T055 authors **no** new medical/care/dosing copy, invents no titles, and stores the med-name field only if the CRUD caller supplies it (never suggested — CLAUDE §7/§5). The instantiate endpoint neither strips nor rewrites any vet-confirm note; `note` is not persisted on `Reminder` at all in the T053 model, so T055 must not add a column for it (out of scope) — it simply maps `title`/`rrule`/`reminderType`/anchor onto reminders.
- `medNameAsEntered` remains "as entered by the user/vet, never suggested" — reused as-is from the T053 `CreateReminderDto`; T055 adds no default and no medication vocabulary.

## Key decisions (each is a scrutiny point — see Risks)
1. **Agenda is HOUSEHOLD-WIDE `GET /agenda?from=&to=` with an optional `petId` filter** — matches ARCHITECTURE §3 API surface (`GET /agenda?from=&to=`) verbatim, not a per-pet path. (Risk R1)
2. **`occurrencesBetween` reuses `computeNextFireAt` by ITERATION — T053's `next-fire-at.ts` is NOT modified.** The range expander calls the proven single-occurrence helper in a loop, inheriting its DST/monthly-31 correctness exactly. (Risk R2)
3. **`from`/`to` are REQUIRED and the window is bounded (≤ 92 days); over-window or reversed range → 400.** Bounds compute. (Risk R3)
4. **Template instantiation idempotency is app-level, keyed on `(petId, templateKey)` where `templateKey = resolved item.id`** — pre-query existing keys and skip them. No new DB unique index / no migration (schema is T053's domain). (Risk R4)
5. **`nextFireAt` is set to `computeNextFireAt(rule, startAt, startAt, tz)` on create and on any schedule-changing update** (per the T053 documented contract: `after == startAt` returns the anchor). A rule with **no occurrence** (exhausted `COUNT`/`UNTIL`) → **400** on create/update and → **skip that item** during batch instantiation, so the non-null `nextFireAt` column is always satisfiable. Advancing `nextFireAt` past already-fired occurrences is T056's job, not T055's. (Risk R5)
6. **Paused reminders (`active=false`) are excluded from agenda expansion and from their events' inclusion.** Agenda considers only `active=true` reminders in the household. (Risk R6)
7. **`DELETE /reminders/:id` is a hard delete** (Reminder has no `deletedAt`; T053 gave both FKs `onDelete: Cascade`, so events are removed with it). Returns the deleted reminder (200), mirroring `PetsService.remove`'s return-the-resource shape. (Risk R7)
8. **Anchor→DTSTART:** `PLAN_START` → `now + startOffsetDays`; `PET_AGE` → `effectiveBirthDate + startOffsetDays`, where `effectiveBirthDate = pet.birthDate ?? (ageEstimateMonths != null ? now − ageEstimateMonths months : null)`; if a `PET_AGE` item has no derivable birth date → **skip that item**. All instantiated reminders use the tz supplied in the request body. (Risk R8)

## Endpoint specs (all `@HouseholdFromMembership()`; not `@Public()`; 404-no-leak on any cross-household/unknown id)
- `POST /pets/:petId/reminders` → `create`. Body `CreateReminderDto` (T053, reused unchanged). Order: pet-404 (`petsService.findOne(householdId, petId)`) → `parseRRule(dto.rrule)` (defensive; 400 on `!ok`) → `nextFireAt = computeNextFireAt(rule, startAt, startAt, tz)` → **400 if null** (no future occurrence) → persist → 201 `ReminderResponse`.
- `GET /pets/:petId/reminders` → `list`. pet-404 first; cursor page (`take: limit+1`, `orderBy: createdAt desc`, cursor/skip as in `ChecksService.list`). Returns `ReminderListResponse { items, nextCursor }`.
- `GET /reminders/:id` → `findOne`. `findFirst({ where: { id, pet: { householdId, deletedAt: null } } })`; 404 if absent. Returns `ReminderResponse`.
- `PATCH /reminders/:id` → `update`. Household-scoped fetch (404). Apply present (`!== undefined`) fields. If any of `rrule`/`startAt`/`timezone` present → recompute `nextFireAt` from the effective (new-or-existing) rule/startAt/tz; **400 if null**. Returns `ReminderResponse`.
- `DELETE /reminders/:id` → `remove`. Household-scoped fetch (404) → `prisma.reminder.delete` (cascade removes events) → 200 `ReminderResponse` of the deleted row.
- `GET /agenda?from=&to=&petId=` → `agenda`. Query `AgendaQueryDto`. If `petId` present, validate it belongs to the household (`petsService.findOne` → 404). See semantics below. Returns `AgendaResponse`.
- `POST /pets/:petId/reminders/from-template` → `instantiateFromTemplate`. pet-404 → resolve pack → derive/skip → create-if-absent (idempotent). Returns 201 `InstantiateTemplateResponse { created: ReminderResponse[]; skipped: number }`.

## Agenda & expansion semantics
1. **Window guard (service):** parse `from`/`to`; if `to <= from` or `(to − from) > 92 days` → `BadRequestException` (400).
2. **Reminders in scope:** `prisma.reminder.findMany({ where: { active: true, pet: { householdId, deletedAt: null }, ...(petId ? { petId } : {}) } })`.
3. **Virtual occurrences:** for each reminder, `occurrencesBetween(parseRRule(rrule).value, startAt, timezone, from, to)` → one virtual `AgendaEntry` per instant (`virtual: true`, `status: "SCHEDULED"`, no `eventId`). Skip a reminder whose rrule fails to parse defensively (should not happen — validated on write).
4. **Materialized events:** `prisma.reminderEvent.findMany({ where: { dueAt: { gte: from, lte: to }, reminder: { active: true, pet: { householdId, deletedAt: null }, ...(petId ? { petId } : {}) } }, include: { reminder: true } })`.
5. **Merge/dedupe keyed on `(reminderId, dueAt-epoch-ms)`:** an occurrence that has a matching materialized event is emitted with the **event's** status (`virtual: false`, `status: event.status`, `eventId: event.id`); a materialized event with **no** matching virtual occurrence (e.g. rrule edited after the event was created) is still included as its own entry; a virtual occurrence with no event stays virtual. One entry per key.
6. **Response:** `AgendaResponse { from: Date; to: Date; entries: AgendaEntry[] }`, `entries` sorted by `dueAt` ascending. `AgendaEntry { reminderId; petId; type; title; dueAt: Date; status: ReminderEventStatus | "SCHEDULED"; virtual: boolean; eventId?: string }` (service-local).

## Instantiation semantics (anchors, idempotency)
1. Load pet (household-scoped, 404). `now = new Date()`. `ageMonths = petAgeMonths(pet, now)`.
2. Resolve the pack: if `dto.group` present → `resolveCareTemplate(pet.species as Species, resolveLifeStage(pet.species as Species, ageMonths), dto.group)`; else → `resolveCareTemplateForPet({ species: pet.species as Species, ageMonths, countryCode: dto.countryCode ?? null })`. (`pet.species` is Prisma's `Species`; the data package expects `@pawcareright/types` `Species` — both are the `"DOG"|"CAT"` union; cast if the compiler requires it.)
3. **Idempotency pre-query:** `existing = prisma.reminder.findMany({ where: { petId, templateKey: { in: pack.items.map(i => i.id) } }, select: { templateKey: true } })`; build `existingKeys = new Set(existing.map(r => r.templateKey))`.
4. For each `item` whose `id ∉ existingKeys`: `startAt = deriveTemplateStartAt(item, pet, now)` — **skip** if `null` (unanchorable `PET_AGE`); `rule = parseRRule(item.rrule).value`; `nextFireAt = computeNextFireAt(rule, startAt, startAt, dto.timezone)` — **skip** if `null` (exhausted). Build reminder create data: `{ petId, type: item.reminderType, title: item.title, rrule: item.rrule, timezone: dto.timezone, startAt, nextFireAt, templateKey: item.id, active: true }`.
5. Create the surviving items inside a `prisma.$transaction` (loop `create`, small batch), return them as `created`; `skipped = pack.items.length − created.length`. Re-instantiating (all keys already exist) → `created: []`.

## Tests to write (map to acceptance criteria)
- **AC "Agenda expansion unit tests incl. DST boundary (Europe/Paris + America/New_York) and monthly-on-31st"** → `apps/api/src/reminders/occurrences-between.spec.ts`:
  - **Europe/Paris spring-forward** (2026-03-29, clocks 02:00→03:00): a `FREQ=DAILY` 09:00 `Europe/Paris` reminder expanded across `[2026-03-27T00:00Z, 2026-03-31T00:00Z]` → asserts 2026-03-28 09:00 = `08:00Z` (CET +01) and 2026-03-30 09:00 = `07:00Z` (CEST +02): local time-of-day stays 09:00 while the UTC offset shifts one hour across the boundary.
  - **America/New_York spring-forward** (2026-03-08): `FREQ=DAILY` 09:00 `America/New_York` across `[2026-03-06, 2026-03-10]` → 03-06 09:00 = `14:00Z` (EST) and 03-09 09:00 = `13:00Z` (EDT).
  - **Fall-back** (at least one): Europe/Paris 2026-10-25 or America/New_York 2026-11-01 `FREQ=DAILY` 09:00 → offset shifts back, local 09:00 preserved.
  - **Monthly-on-31st:** `FREQ=MONTHLY;BYMONTHDAY=31` from a 2026-01-31 anchor across `[2026-01-01, 2026-05-01]` → occurrences on Jan 31 and Mar 31 only; **Feb and Apr absent** (no roll-forward).
  - **Window/bounds:** `from > to` → `[]`; inclusive boundaries (an occurrence exactly at `from` and exactly at `to` are included); a `COUNT`/`UNTIL`-bounded rule stops within the window.
- **AC "template instantiation idempotent"** → `reminders.service.spec.ts` "instantiateFromTemplate: second call with the same pet/pack creates no new reminders" (mock pre-query returns the ids created on the first pass → `created` empty, `skipped` = all) **and** `apps/api/test/reminders.e2e-spec.ts` "POST from-template twice → `GET /pets/:petId/reminders` count is unchanged after the second call".
  - Also `reminders.service.spec.ts` "anchor derivation: PET_AGE item → `startAt = birthDate + offsetDays`; PLAN_START item → `startAt = now + offsetDays`" and `template-anchors.spec.ts` unit cases (incl. `PET_AGE` with only `ageEstimateMonths` → pseudo-birthdate; unknown age → skip).
- **CRUD happy + auth + validation + cross-household** → `reminders.e2e-spec.ts`:
  - happy: create → get → list → patch (changes rrule, `nextFireAt` updates) → delete (subsequent get 404).
  - auth: any endpoint without a token → 401.
  - validation: `POST` with a malformed `rrule` / bad `timezone` / out-of-set `type` → 400; agenda missing `from`/`to` or window > 92 days or `to<=from` → 400.
  - cross-household: user B `GET/PATCH/DELETE /reminders/:id` for user A's reminder → 404 (no leak); `from-template` / `POST reminders` on a pet outside the household → 404.

## Risks & decisions the planner made (numbered — for the checker)
1. **Agenda is household-wide `GET /agenda?from=&to=` (+ optional `petId`).** Chosen to match ARCHITECTURE §3's literal API surface over a per-pet path. Risk: the card text says "expansion … merged with pending events" without fixing the URL; mitigated by ARCHITECTURE being authoritative and the `petId` filter still enabling per-pet views.
2. **`occurrencesBetween` iterates `computeNextFireAt`; T053's file is untouched.** Rationale: directly inherits the DST/monthly-31 correctness that T053 already unit-proved, and keeps the proven helper unmodified (lower risk than extracting internals). Risk: each call re-scans from the anchor, so expansion is O(occurrences × scan) — bounded by the ≤92-day window (≤ ~92 daily occurrences/reminder) and a `MAX_OCCURRENCES` cap; a future perf need can extract shared internals or adopt a library (deferred, journal then). Not a correctness risk.
3. **Required, ≤92-day agenda window.** Bounds compute and makes the guard test crisp. Risk: a client wanting a longer horizon must page by window — acceptable for v1; the mobile agenda (T060) shows today/upcoming, well under 92 days.
4. **Idempotency is app-level on `(petId, templateKey=item.id)` via pre-query, no DB unique index.** Keeps T055 free of a migration (schema is T053's domain) and uses the existing `petId` index. Risk: two *concurrent* instantiations could both pass the pre-query and double-create (no DB constraint); accepted for v1 (instantiation is a rare, user-initiated wizard action) and noted — a partial unique index can be added later in a schema task. The AC's "double-call idempotent" (sequential) is fully satisfied.
5. **`nextFireAt = computeNextFireAt(rule, startAt, startAt, tz)`; null → 400 (create/update) or skip (batch).** Follows the T053 documented `after==startAt` contract and keeps the non-null column satisfiable. Risk: a reminder with a past `startAt` gets a past `nextFireAt`; resolving/advancing that (and backfill of missed fires) is explicitly T056's responsibility, not T055's — so T055 does not try to "roll forward to now" (which would be non-deterministic to test and would pre-empt T056's backfill policy).
6. **`active=false` reminders are fully excluded from the agenda** (both expansion and their events). Risk: a paused reminder's already-materialized future events won't show; acceptable — "paused" means "don't surface," and T055 creates no events anyway.
7. **Hard `DELETE` (cascade).** Reminder has no soft-delete field in the T053 model and both FKs cascade. Risk: no undo/audit trail for a deleted reminder; accepted (no requirement for it; adding `deletedAt` would be a T053 schema change, out of scope).
8. **`PET_AGE` uses `pet.birthDate`, falling back to a pseudo-birthdate from `ageEstimateMonths`, else the item is skipped; instantiated reminders use the request-body `timezone`.** Per T054, `PET_AGE` items only appear in juvenile packs (which require a known young age), so the skip branch is a defensive backstop. Risk: the request must supply a timezone (no tz on `Pet`/`Household`); made a required DTO field so no silent-UTC default — the T059 wizard passes the device tz.
