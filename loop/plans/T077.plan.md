# Plan — T077: Family plan household wiring

## Objective (from card)
Family entitlement already covers all household members/pets (T072/T075) — VERIFY and pin it
with a named test. CLOSE the two real gaps: (1) leaving a household drops the family entitlement,
with a member-facing grace message before they leave; (2) purchase management is owner-only — a
family beneficiary who doesn't own the plan gets a graceful explainer instead of a broken
store-management deep link.

## Safety statement (PRODUCT_SPEC §5 / CLAUDE.md §7)
- This task touches ONLY household-membership and billing/entitlement surfaces. It goes nowhere
  near a §5 safety surface: no `<VetDisclaimer/>`, no emergency interstitial, no poison/vet
  hotline, no dosing/drug/diagnos* copy is added or altered. Every new string is factual
  household/billing prose (leave household / family plan / manage billing).
- The leave button, grace message, and family-managed note live only on the Family and Settings
  screens — structurally incapable of appearing on the intake/check/result/emergency path.
- NOT a SAFETY-ESCALATION: implementable without weakening any safety surface.

## Key decisions (each a design choice — scrutinize)
1. **Removal mechanism = member-initiated self-leave `POST /v1/households/leave`, NOT an
   owner-only remove-member endpoint.** (This is the ONE primary decision.) The card's Do line
   frames the behaviour as *"leaving household drops access (grace message)"* — inherently
   member-initiated — and the grace message only exists as a member-facing confirmation dialog.
   A single self-leave endpoint satisfies BOTH the Do bullet and the "removal revokes" AC (the
   member's own removal from the household flips their entitlement to none). An owner-only
   remove-member endpoint would add RBAC/target-selection surface that no acceptance criterion
   requires and would produce no member-facing grace dialog — per CLAUDE.md §2.2 that is scope
   creep. The checker should confirm the leave endpoint is sufficient to prove "removal revokes".
2. **Leave preserves the one-membership-per-user invariant by re-provisioning a fresh solo
   household** (mirrors `AuthService.provisionNewAccount` lines 183–188 and the
   `acceptInvite` JOIN-REPLACES pattern). In one `$transaction`: create a new
   `Household{ name: DEFAULT_HOUSEHOLD_NAME, ownerId: userId }` + `Membership{ role: OWNER }`,
   then delete the caller's old MEMBER membership row. After commit the user still has EXACTLY
   ONE membership, so `HouseholdScopeGuard` stays satisfied. `DEFAULT_HOUSEHOLD_NAME` is imported
   from `../auth/auth.constants` (already exported) — no string duplication.
3. **Only a MEMBER may leave; an OWNER leaving is a `ConflictException` (409).** An owner leaving
   would strand the shared household's pets, other members, and the family subscription. Owners
   manage/cancel their plan instead (T076). A solo user is their own household's OWNER, so "leave"
   is correctly a no-op-409 for them too. Membership count `!== 1` also → 409 (invariant guard),
   matching `acceptInvite`.
4. **Entitlement revocation is automatic — no billing code changes.** `BillingService.
   getEntitlement(userId, householdId)` already resolves family grants by the caller's CURRENT
   `householdId` (`entitlement.util.ts` / `billing.service.ts`). After leaving, the caller's
   household is their new solo household, whose `subscription.findMany` returns no family row →
   `pickEntitlement` returns `{ entitled:false, source:"none" }`. The server billing layer is
   UNTOUCHED; the "removal revokes" AC is proven end-to-end via the existing resolver.
5. **Owner-only purchase management (graceful) = branch the Settings manage row on
   `entitlement.source`.** `own` → the existing T076 `settings-manage` row (opens the store
   management flow). `family` (entitled via someone else's plan) → a new non-interactive
   `settings-family-note` explainer ("Your Premium comes from your household's family plan. Only
   the plan owner can manage billing.") — NO store deep link, because a beneficiary cannot manage
   a subscription they don't own (their own RC user has no `managementURL`). `none`/loading →
   neither. This reuses T076's `useEntitlement()` (which already returns `source`); no server
   change. A member CAN still buy their OWN subscription via the unchanged paywall (RC purchases
   are per-user) — that path is not blocked.
6. **Leave response reuses a dedicated Zod schema `leaveHouseholdResponseSchema`** (shape
   `{ householdId, name }`, identical to `acceptInviteResponseSchema`) added to
   `packages/types/src/household.ts`. A distinct honest name (vs. reusing the accept schema)
   keeps the contract self-documenting; additive only.

## Risks & edge cases (for the checker)
- **R1 (decision 1) — self-leave vs owner-remove.** The foundation note floated an owner-only
  remove-member endpoint; I chose member self-leave because the Do line's grace-message behaviour
  is member-initiated and self-leave satisfies "removal revokes" with less surface. If product
  later wants owners to eject members, add it then — not speculatively.
- **R2 — member who ALSO holds an own subscription leaves.** Their own row resolves by identity
  (`rcAppUserId`), independent of household, so they KEEP own-premium after leaving; only the
  FAMILY grant drops. Correct and intended. The removal-revokes e2e uses a family-only member so
  the flip to `none` is observable.
- **R3 — invariant during the transaction.** New household + OWNER membership are created before
  the old membership is deleted, all inside one `$transaction`, so the user is never observably at
  0 or 2 memberships at commit. Matches the `acceptInvite` JOIN-REPLACES guarantee.
- **R4 — owner/solo leave.** OWNER (incl. a solo-household user) → 409, never strands pets/subs.
  Documented as decision 3.
- **R5 — T073 stamped-household drift.** A member's own sub (if any) keeps its
  purchase-time `householdId` after they leave (T073 stamps at write time). Acceptable v1 —
  own-sub resolution ignores `householdId`, so entitlement is still correct; documented, not fixed.
- **R6 — family beneficiary still sees the "Upgrade" row in Settings.** Pre-existing T076
  behaviour (the `settings-premium` row is always shown); the card asks only for graceful *manage*
  handling, which the `settings-family-note` provides. Left unchanged to avoid scope creep.
- **R7 — `settings-premium-entry.test.tsx` unaffected.** It mocks `useEntitlement` → `{ data:
  undefined }` (T076), which hits the neither-branch (no manage row, no family note). No edit
  expected; executor should confirm it stays green after the source branch is added.
- **R8 — stale `households/me` after leave.** `useLeaveHousehold` invalidates `householdKeys.me`
  + `billingKeys.entitlement`, so the Family and Settings screens refetch the new solo household
  and downgraded entitlement. Grace message is pre-departure only (informational).
