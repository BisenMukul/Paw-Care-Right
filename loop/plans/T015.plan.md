# Plan ‚Äî T015: Guards, roles & household scoping

## Objective (from card)
Bind a global JWT auth guard honoring `@Public()`, add a `HouseholdScope` decorator+guard that resolves membership and injects `householdId`+`role`, and a role guard requiring OWNER for destructive ops. Prove cross-household access returns 404 (no existence leak) and the member-vs-owner matrix is enforced.

## Key resolution of the ARCHITECTURE ¬ß4 design tension (READ FIRST)
The API table marks `otp/request|verify|social` public but NOT `refresh|logout`. A live access-JWT requirement on `/auth/refresh` is self-defeating (refresh is called precisely when the access token is expired ‚Äî the T019 "401‚Üírefresh‚Üíretry" contract) and would regress the T012 auth e2e (which calls refresh/logout with NO Authorization header ‚Äî a protected contract). **RESOLUTION: mark `refresh` + `logout` `@Public()` too.** The refresh token itself is the credential (opaque, hashed, family-revocable in `RefreshTokenService`); requiring an access token on top is meaningless. This follows the table's spirit (public = "no access-JWT gate"), and is documented under Risks. NOT a blocked task.

## Interfaces/contracts (executor must match)
- `IS_PUBLIC_KEY = "isPublic"`; `Public = () => SetMetadata(IS_PUBLIC_KEY, true)`.
- `HOUSEHOLD_SCOPE_PARAM_KEY = "householdScopeParam"`; `HouseholdScoped = (paramName = "householdId") => SetMetadata(HOUSEHOLD_SCOPE_PARAM_KEY, paramName)`.
- `REQUIRED_ROLE_KEY = "requiredRole"`; `RequireRole = (role: Role) => SetMetadata(REQUIRED_ROLE_KEY, role)`.
- `JwtAuthGuard.canActivate`: if `reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [ctx.getHandler(), ctx.getClass()])` ‚Üí return `true`. Else parse `Authorization: Bearer <t>` (missing/wrong scheme ‚Üí `UnauthorizedException`); `payload = await jwt.verifyAsync(token)` (throws ‚Üí `UnauthorizedException`); require `payload.sub` non-empty string; set `req.user = { userId: payload.sub }`; return `true`. No secret arg ‚Äî the configured `JwtService` already carries it.
- `HouseholdScopeGuard.canActivate`: `paramName = reflector.getAllAndOverride<string | undefined>(HOUSEHOLD_SCOPE_PARAM_KEY, [handler, class])`; if `undefined` ‚Üí `true` (no-op for all non-scoped routes). `userId = (req as RequestWithUser).user?.userId` ‚Üí missing ‚Üí `UnauthorizedException`. `householdId = req.params[paramName]` ‚Üí missing ‚Üí `NotFoundException`. `membership = await prisma.membership.findUnique({ where: { userId_householdId: { userId, householdId } } })`; `null` ‚Üí `NotFoundException` (NEVER 403). Else set `req.householdScope = { householdId, role: membership.role }`; return `true`.
- `RolesGuard.canActivate`: `required = reflector.getAllAndOverride<Role | undefined>(REQUIRED_ROLE_KEY, [handler, class])`; `undefined` ‚Üí `true`. `scope = (req as RequestWithHouseholdScope).householdScope`; missing ‚Üí `ForbiddenException` (fail closed). `scope.role !== required` ‚Üí `ForbiddenException`; else `true`.
- Membership compound key is Prisma-generated `userId_householdId` (from `@@unique([userId, householdId])`, already indexed). No schema change, no new index.

## Risks & the design decisions the planner made (for the checker to scrutinize)
1. **`refresh` + `logout` marked `@Public()`** ‚Äî deliberate deviation from ARCHITECTURE ¬ß4's literal table (justified above; without it T012's e2e and the T019 client contract break). Checker: confirm the auth e2e stays green unmodified.
2. **All three guards registered as global `APP_GUARD` in AppModule, in fixed order** (Jwt ‚Üí HouseholdScope ‚Üí Roles). Relies on NestJS executing multiple APP_GUARD providers in provider-array order. Safety net: `RolesGuard` fail-closes (403) if `req.householdScope` is absent, so a mis-order can only wrongly 403 an owner (a caught test failure), never grant an unauthorized 200. `HouseholdScopeGuard`/`RolesGuard` are no-ops on routes without their metadata, so every existing route is unaffected.
3. **File placement:** JWT/`@Public`/`@CurrentUser` in `src/auth/` (cohere with token issuance + the existing `OtpRateLimitGuard`); generic household/role authorization primitives in `src/common/` (cross-cutting, consumed by every future domain module). Deliberately NO new `HouseholdsModule` ‚Äî that is T021's scope; creating it now is premature (over-engineering is a defect).
4. **404 for cross-household, 403 for insufficient role in one's own household** ‚Äî 404 prevents existence leaks across tenants; within your own household you already know it exists, so a role failure is a legitimate 403 (matches the card's "matrix" language and the leak rule's intent).
5. **`exports: [JwtModule]` added to AuthModule** so the AppModule-scoped `JwtAuthGuard` injects the same secret-configured `JwtService` used for issuance (verify identity of instance; e2e falls back to signing via `AppConfigService().jwtSecret`).
6. **`test-throw.controller.ts` gets `@Public()`** ‚Äî the regression trap; a minimal edit to a build-excluded test file that preserves the error-envelope test's intent (reach the handler, then throw).
7. **`RolesGuard` exact-match on required role** ‚Äî only `OWNER` is ever required today; `MEMBER !== OWNER` ‚Üí 403. No role hierarchy introduced (YAGNI).
