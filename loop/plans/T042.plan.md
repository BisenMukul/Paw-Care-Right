# Plan — T042: POST /checks + status polling API

## Objective (from card)
Build the symptom-check REST surface: `POST /pets/:petId/checks` (validate intake, run deterministic red-flag rules **synchronously before quota**, persist, enqueue AI triage, idempotency-key support, free-tier quota metering), `GET /checks/:id` (status/result/redFlag polling), and `GET /pets/:petId/checks` (cursor list). Wire `@pawcareright/ai` into `apps/api` (deferred from T034) and implement the intake→`RedFlagIntake` mapper (deferred from T032, contract in `loop/plans/T032.plan.md`).

## Safety statement (PRODUCT_SPEC §5 — this task touches an emergency-escalation surface; NOT blocked)
This endpoint is the server side of the §5 "fail-upward" pipeline. The plan encodes three §5-critical invariants that the executor MUST NOT reorder or soften; the checker verifies each:
1. **Rules-before-quota, always.** In `POST`, red-flag evaluation runs and is acted on *before* any quota consumption. A red-flag ("emergency") check is **never** rejected for quota reasons (decision D1: emergency checks bypass quota entirely). This is the T039 carry-forward: "quota rejection must NEVER precede the emergency red-flag path."
2. **Emergency payload in the 201.** When a red flag fires, the `201` body carries `redFlag: { ruleId, payloadKey }` so mobile (T047) renders the Emergency interstitial *before* any AI content. The interstitial copy/hotlines are resolved client-side from `payloadKey` + region (T036) — this task emits only the stable keys, never medical/dosing prose.
3. **No intake logging.** Structured logs are keyed by `checkId`/`ruleId`/`status` only. `intakeJson`, `freeText`, and `photoKeys` are NEVER logged (T041 N1, T033 "freeText sensitive"). No `console.*` — Nest `Logger` only.
Red-flag detection is deterministic (`packages/ai` rules engine, read-only here). This is a normal planned task, not a SAFETY-ESCALATION.

## Key decisions (numbered — see Risks for scrutiny notes)
- **D1** — Red-flag checks bypass quota entirely (option (a)): no `QuotaService.consume` on the red-flag path. Emergency is free + never blocked. Non-red-flag checks consume `checks` quota.
- **D2** — Both paths enqueue the triage job (queue `pawcareright-checks`, payload `{ checkId }`, `jobId = checkId`). Matches ARCHITECTURE §5 (the `enqueue(checkId)` step is *after* the alt block) and SPEC §6 (worker applies a post-rules floor so AI never downgrades the emergency tier). The 201's `redFlag` drives the immediate mobile branch; the worker (T043) adds complementary content.
- **D3** — Reuse existing error codes. The global filter maps HTTP status → `ErrorCode` (`STATUS_TO_CODE`). Quota-exceeded → throw `PaymentRequiredException` → **402 `PAYMENT_REQUIRED`** (already in the enum). Cross-household / missing check → `NotFoundException` → **404 `NOT_FOUND`**. **No new error codes; `packages/types/src/error-codes.ts` is NOT modified.**
- **D4** — Response shapes are **service-local TypeScript interfaces** in the checks service (mirroring the existing `PetResponse` precedent in `pets.service.ts`). **`packages/types` is NOT modified.** T047's client contract is out of scope for this task.
- **D5** — Idempotency check runs **before** `parseIntake` and **before** quota: if a `SymptomCheck` with `(createdById, idempotencyKey)` already exists, return its response immediately (same `checkId`), no re-parse, no re-consume, no re-enqueue. `Idempotency-Key` is read from the request header; absent/empty → `idempotencyKey = null` (Postgres treats NULLs as distinct — non-idempotent semantics, T041 N1).
- **D6** — Idempotent replays return **201** with the existing check (uniform status; simplest, avoids a 200/201 branch). Race on the unique constraint → catch Prisma `P2002`, refetch by `(createdById, idempotencyKey)`, return that row.
- **D7** — jest resolves `@pawcareright/ai` via package-`exports` (no mapper); dist-cjs mapper is a documented fallback only.
- **D8** — `sizeClass` is **not** derivable from the `Pet` model (no size/weight-class field). The mapper omits it (`undefined`). Acceptable: the ACs and red-flag rules under test are sign-based (urinary/breathing/injury), not size-gated. Documented gap.

## Endpoint specs

### `POST /v1/pets/:petId/checks`
- **Auth/scope:** global `JwtAuthGuard` + class `@HouseholdFromMembership()` → `scope.householdId`; user via `@CurrentUser() → { userId }`.
- **Header:** `Idempotency-Key` (optional string). Empty/absent → `null`.
- **Body DTO:** `{ intake: object, photoKeys?: string[] }`.
- **Service flow (ORDER IS §5-CRITICAL):**
  1. `await petsService.findOne(scope.householdId, petId)` → 404 `NOT_FOUND` if the pet is not in the caller's household (no existence leak). Returns pet profile.
  2. **Idempotency short-circuit (D5):** if `idempotencyKey` non-null AND a `SymptomCheck` exists for `(createdById: userId, idempotencyKey)` → return its `CheckResponse` (201). Stop.
  3. `parseIntake(dto.intake)` → if `!ok` throw `BadRequestException` (400 `VALIDATION_FAILED`). Fail-closed.
  4. `redFlagIntake = buildRedFlagIntake(pet, parsed.value)`; `evaluation = evaluateRedFlags(redFlagIntake)`.
  5. **If `evaluation.highest !== null` (RED FLAG):** persist `SymptomCheck` `{ petId, createdById: userId, status: QUEUED, category: parsed.value.category, intakeJson: parsed.value, photoKeys: dto.photoKeys ?? [], redFlagHit: true, redFlagRuleId: highest.ruleId, redFlagPayloadKey: highest.emergencyPayloadKey, idempotencyKey }`. **No quota consume (D1).** Enqueue (D2). Return 201 with `redFlag`.
  6. **Else (NO red flag):** `entitlement = await entitlementResolver.resolve(userId)`; `result = await quotaService.consume(userId, "checks", entitlement)`; if `!result.allowed` throw `PaymentRequiredException` (402 `PAYMENT_REQUIRED`) — no persist, no enqueue. Otherwise persist `SymptomCheck` with `redFlagHit: false, redFlagRuleId: null, redFlagPayloadKey: null` (rest as above). Enqueue (D2). Return 201 (no `redFlag`).
  7. **P2002 race (D6):** wrap the create in try/catch; on Prisma `P2002` (unique `(createdById, idempotencyKey)`) refetch and return the existing check.
  8. **Enqueue:** `queue.add("triage", { checkId }, { jobId: checkId, attempts: 3, backoff: { type: "exponential", delay: 2000 }, removeOnComplete: { age: 3600, count: 1000 }, removeOnFail: false })` (mirror photos options; `jobId = checkId` makes re-enqueue idempotent).
- **201 body (`CheckResponse`):** `{ id, status, category, createdAt, redFlag?: { ruleId, payloadKey } }`.
- **Errors:** 400 (invalid intake), 401 (no token), 402 (quota exceeded, non-red-flag only), 404 (pet not in household / no household).

### `GET /v1/checks/:id`
- **Auth/scope:** same class scope; `scope.householdId`.
- **Service `findOne(householdId, userId?, id)`:** fetch `SymptomCheck` by `id` with `include: { result: true, pet: true }`; if missing OR `check.pet.householdId !== householdId` OR `check.pet.deletedAt !== null` → 404 `NOT_FOUND` (no cross-household leak). Scope by household via the pet.
- **200 body (`CheckResponse`):** `{ id, status, category, createdAt, redFlag?: { ruleId, payloadKey } (present iff redFlagHit), result?: <TriageResult> }`. `result` is included iff `status` is terminal (`DONE`/`FALLBACK`) AND a `TriageResult` row exists: read `result.resultJson` and validate with `parseTriage` (ARCHITECTURE §3 "validated at read"); if `ok` return `parsed.value`, else omit `result` (defensive — no invalid triage surfaces). In T042 no results exist yet; tests seed rows manually to exercise shapes.
- **Errors:** 401, 404.

### `GET /v1/pets/:petId/checks`
- **Auth/scope:** same. First `await petsService.findOne(householdId, petId)` → 404 if not in household.
- **Query:** `paginationQuerySchema` shape from `@pawcareright/types` (`cursor?`, `limit` 1..100 default 20) — validate via a class-validator DTO `ListChecksQueryDto` OR coerce inline; keep `limit` bounded. (Reuse the zod schema for bounds if convenient; a small class-validator DTO is acceptable and matches existing DTO convention.)
- **Query semantics:** `prisma.symptomCheck.findMany({ where: { petId }, orderBy: { createdAt: "desc" }, take: limit + 1, ...(cursor ? { cursor: { id: cursor }, skip: 1 } : {}) })` — uses the T041 `SymptomCheck(petId, createdAt desc)` index. Compute `nextCursor` = the extra row's `id` when `results.length > limit`, else `null`; return exactly `limit` items.
- **200 body:** `{ items: CheckResponse[], nextCursor: string | null }`.
- **Errors:** 401, 404 (pet not in household).

## Mapper spec — `buildRedFlagIntake(pet, intake): RedFlagIntake`
`PetProfileInput` = the fields the mapper reads from the resolved pet: `{ species: "DOG"|"CAT", sex: "MALE"|"FEMALE"|"UNKNOWN", ageEstimateMonths: number|null, birthDate: Date|null, weightGrams: number|null }`.

Profile derivation:
- `species` ← `pet.species` (always present).
- `sex` ← `pet.sex` (maps 1:1 to `@pawcareright/ai` `Sex`).
- `ageMonths` ← `pet.ageEstimateMonths` if non-null; else if `pet.birthDate` non-null → whole months between `birthDate` and now (`(nowYear-y)*12 + (nowMonth-m)`, clamped ≥ 0); else `undefined`.
- `weightKg` ← `pet.weightGrams != null ? pet.weightGrams / 1000 : undefined`.
- `sizeClass` ← **omitted (undefined)** — not on the Pet model (D8).
- `freeText` ← `intake.freeText` (passthrough; T031 keyword channel). Omit when absent.

Signs derivation — scan `intake.answers` (each `{ questionId, type, ... }`); build `signs: Partial<Record<RedFlagSign, boolean>>` setting `true` for each matched row (only `single` answers, `answer.value` a string). Verbatim from `loop/plans/T032.plan.md` R2 table:

| category (intake.category) | questionId | matching answer.value(s) | → RedFlagSign |
|---|---|---|---|
| urinary | `difficulty` | `straining`, `cannot-urinate` | `straining_to_urinate` |
| urinary | `blood-in-urine` | `yes` | `blood_in_urine` |
| breathing | `character` | `labored`, `open-mouth-cat`, `gasping` | `breathing_difficulty` |
| breathing | `gum-color` | `pale-white`, `blue-purple` | `abnormal_gum_color` |
| injury | `what` | `hit-by-vehicle` | `major_trauma` |
| injury | `bleeding` | `heavy` | `uncontrolled_bleeding` |
| injury | `consciousness` | `unresponsive` | `collapse_unresponsive` |

Rules: match by `questionId` + `type === "single"` + `value` in the set; omit signs that don't match (do not set `false`). Category is informational — match on `questionId`+`value` (question ids are unique within their category; a mismatched category simply won't carry those answers). Pure, deterministic, no throws.

## Queue contract (for T043)
- **Queue name:** `pawcareright-checks` (exported `CHECKS_QUEUE`, `checks.contract.ts`).
- **Job name:** `"triage"`.
- **Payload:** `ChecksJobData = { checkId: string }` — minimal; the worker (T043) refetches the `SymptomCheck` by id (never trust intake from the payload; never log it).
- **Options:** `jobId = checkId` (idempotent enqueue), `attempts: 3`, `backoff: exponential 2000ms`, `removeOnComplete: { age: 3600, count: 1000 }`, `removeOnFail: false` (dead-letter retention).
- **Registration:** `BullModule.registerQueue({ name: CHECKS_QUEUE })` in `ChecksModule` (producer only). **No `@Processor` in T042** — T043 consumes.
