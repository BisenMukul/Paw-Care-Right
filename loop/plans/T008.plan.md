# Plan — T008: Expo scaffold (apps/mobile)

## Objective (from card)
Stand up the mobile app `apps/mobile` as an Expo + expo-router + NativeWind + TypeScript-strict project. `app.config.ts` sources `name` from `APP_DISPLAY_NAME`, bundle ids from `BUNDLE_ID` (`com.pawcareright.app`), scheme from `DEEPLINK_SCHEME` (`pawcareright`), slug from `APP_SLUG` — all imported from `@pawcareright/config` (§1a; never hardcode). Deep-link scheme, splash/icon placeholders, a 4-tab shell (Home, Care, Timeline, Settings), Sentry stub OFF, MMKV installed (dep-only, not wired), Jest + `@testing-library/react-native` configured with ONE passing component test that proves `APP_DISPLAY_NAME` renders in a title component. NativeWind must consume the shared Tailwind v3.4 preset via `presets: [require('@pawcareright/config/tailwind')]`. All root turbo gates (`typecheck`/`lint`/`test`/`build`) stay green and are callable from repo root.

## Headless-verification reality (read first)
This is a headless Linux container: **no iOS/Android device, no emulator, no Expo Go app.** The AC "boots to tab shell in Expo Go" **cannot** be device-verified here. The plan replaces it with a headless-equivalent proof triad:
1. **The component test** (`@testing-library/react-native`) renders the title component and asserts `APP_DISPLAY_NAME` text — this is the real automated linchpin of "tab shell + name renders", fully headless.
2. **Metro boots**: `pnpm --filter @pawcareright/mobile start -- --port 8081` prints the Metro "Waiting on …" / dev-server-ready log, then is killed (best-effort; a persistent process — do not treat exit code as gate).
3. `pnpm --filter @pawcareright/mobile typecheck` exits 0.
The on-device Expo Go boot is a **deferred MANUAL check**, to be journaled at the **M1 device milestone**. The plan states this explicitly so the checker does not expect a device boot in this container.

## Risks & the design decisions the planner made (checker: scrutinize each)
- **R1 — Expo SDK version + pinned RN/React.** Recommend the current SDK (SDK 53-era, which pairs React 19 + RN 0.79 — aligning with the React 19 already used by `apps/web`). The executor MUST obtain RN/React/native-module versions via `expo install` (the SDK's own pinning), NOT hand-pick or free-float them. If the installed SDK pairs React 18 instead, match the SDK's peer (still let `expo install` decide) and flag — do not override.
- **R2 — Headless verification strategy (the central risk).** No device/emulator/Expo Go here, so "boots to tab shell in Expo Go" is proven by: (a) the component test rendering the title component with `APP_DISPLAY_NAME` [primary, deterministic], (b) Metro dev server starting, (c) typecheck green. Device boot is a **deferred manual M1 check**. The checker should NOT require an on-device boot in this container.
- **R3 — NativeWind v4 ⇄ Tailwind v3.4 pairing (cross-task anchor).** NativeWind v4 is compatible with Tailwind **v3.4**, not v4's CSS-first `@theme` model — this is exactly why T007 shipped the shared preset as a classic v3 JS object. Mobile consumes the SAME `@pawcareright/config/tailwind` preset via `presets:[require(...)]` in `tailwind.config.js`, plus `babel-preset-expo` `jsxImportSource:"nativewind"` + `nativewind/babel` + `withNativeWind` in metro + `global.css`. If NativeWind rejects v3.4 or the preset is unloadable from Metro → BLOCK (do not bump Tailwind to v4, which would also strand web).
- **R4 — No `apps/mobile/turbo.json` needed (deviates from T007 Amendment A1).** T007 needed a turbo override because its tsconfig explicitly `include`s Next-generated `.next/types`. Here I **keep expo-router typed-routes OFF** and use a **glob-only** tsconfig `include` (`.expo/types/**/*.ts` is a glob that tolerates absence), so mobile typecheck has NO dependency on any generated file and no build-ordering race. Therefore no per-app turbo.json. **Fallback**: if the executor finds typecheck DOES require generated Expo types (e.g. it enables typed routes), add `apps/mobile/turbo.json` `{ "extends": ["//"], "tasks": { "typecheck": { "dependsOn": ["^build","build"] } } }` mirroring T007 A1, and give mobile `build` a real typegen step (`expo customize` / `expo export`) instead of the no-op. Flagged so the checker can weigh which path the executor took.
- **R5 — `app.config.ts` imports the ESM constant from `@pawcareright/config`.** Expo evaluates `app.config.ts` via Node + a TS loader; the `.` export resolves to the built dist (`import`/`require` conditions both exist — verified `packages/config/dist/index.{js,cjs}` present). Requires config built first (covered by the explicit config build command). Mirrors how `apps/web` imports the same constant. Low risk; if the loader cannot resolve the workspace ESM package, flag and BLOCK rather than duplicating the string.
- **R6 — "Sentry stub off" = omit the plugin entirely.** No `@sentry/react-native` dep, no plugin entry, no DSN, no runtime import — just a one-line comment marking the future insertion point. This is the lightest reading of "stub off" and adds zero runtime surface / zero new dep. Real Sentry wiring is a later task.
- **R7 — ESLint reuses the shared flat config; CJS tooling files ignored.** `apps/mobile/eslint.config.mjs` spreads `@pawcareright/config/eslint` and ignores `babel.config.js`/`metro.config.js`/`tailwind.config.js`/`jest.config.js`/`.expo` (CommonJS `module.exports`/`require` files that would trip the shared TS/import rules). No `eslint-config-expo` added (the shared base lints the authored TSX fine). Fallback if a shared rule fights RN/JSX: add a narrow override in the mobile eslint file only — never relax the shared base.
- **R8 — Placeholder assets are binary PNGs copied from the Expo default template.** `icon.png`/`splash-icon.png`/`adaptive-icon.png`/`favicon.png` cannot be hand-authored in a text editor; the executor copies the stock Expo template assets. They are NOT needed for headless verification (component test / Metro / typecheck don't require them). **Fallback**: if copying is infeasible, OMIT the `icon`/`splash` keys from `app.config.ts` and let Expo use its built-in defaults, leaving a comment noting the placeholder intent — still satisfies "placeholders" without blocking any gate.

No `src/storage.ts` is created this task. `newArchEnabled` is left to the SDK default (SDK 53 = New Arch ON) — do not override.
