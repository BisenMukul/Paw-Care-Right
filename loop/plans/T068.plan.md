# Plan — T068: Vet visit prep summary

## Objective (from card)
Add `GET /pets/:id/vet-summary`: a deterministic, structured **plain-text** record digest of the pet's last 90 days (weight trend, symptom checks with tiers, medications given as-entered, notes) with a mandatory disclaimer footer, capped at ≤2,500 chars; plus a mobile "Prepare vet summary" action that fetches it and opens the native share sheet. Golden-file test proves the generated text for a fixture pet.

## Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)
This is a **RECORD digest for a vet, not advice** — it assembles only facts the owner already logged; it produces **no AI output and introduces no new safety surface**.
- No "diagnosis"/"diagnose" token anywhere (§7 rule 1). The disclaimer footer is phrased **without** the `/diagnos/` substring on purpose (avoids tripping the §7/T038 scanners and needs no warning-context exemption); tier labels and section headings are neutral record nouns.
- Medications render **only** `medNameAsEntered`/`medDoseAsEntered` verbatim as entered (§7 rule 2); when absent, a neutral "Medication given" label. No drug name, dose, or schedule is ever computed, formatted, or suggested — the builder has no medication vocabulary of its own.
- The disclaimer footer is **mandated by the card** and is the ONE block the truncation algorithm may never drop or shorten; it is appended after truncation and always terminates the output. It lives as an SSOT constant in `@pawcareright/types` (mirrors `MEDICATION_STATIC_COPY`) so the generator, the golden test, and any future UI render byte-identical copy.
- Symptom-check tiers are shown as recorded assessments (record framing), not new guidance; this does not touch emergency escalation, disclaimers, or dosing prohibitions.
This task does **not** weaken any Safety Policy surface. Not a SAFETY-ESCALATION.

## Key decisions (the design choices the checker must scrutinize)
1. **Section order (fixed, keep-priority high→low):** `[header, weight trend, symptom checks, medications given, notes]`, then the disclaimer footer. Header = pet name, species, age (see D3), and a static "Health record summary — last 90 days" line (no generated-on timestamp — see D4).
2. **Truncation priority = greedy keep-order fill against a char budget.** `budget = VET_SUMMARY_MAX_CHARS (2500) − footer.length − join separators`. Header + weight trend are always emitted (bounded, tiny). Then checks, meds, notes are appended **in that order**; each list section's items are added newest-first while they fit the remaining budget. When an item would overflow, that section emits a single deterministic `"… (older entries omitted to keep this summary short)"` marker (only if the marker itself fits) and **no further lower-priority section is added**. The footer is appended last and its length is always reserved, so it is never truncated. Consequence: **notes drop before meds, meds before checks**; checks (most clinically relevant) survive longest. This guarantees `output.length ≤ 2500` and `output.endsWith(footer)`.
3. **Age rendering is data-derived, never "now"-derived.** If `ageEstimateMonths` is set → `"~N months"`; else if `birthDate` set → its `YYYY-MM-DD`; else omit the age clause. No age is computed from the current clock (keeps golden deterministic).
4. **No generated-on timestamp in the text.** The digest prints only dates that come from the data rows (entry dates), never `Date.now()`, so the golden output is a pure function of the fixture. The "last 90 days" label is static text; the 90-day *window filter* is applied by the service before calling the pure builder, not inside it.
5. **Dates formatted as UTC `YYYY-MM-DD` (`toISOString().slice(0,10)`).** The API is timezone-less and already serializes `occurredAt` as UTC ISO; UTC slicing is the repo's existing convention (`formatCheckDate`) and the only choice that makes the golden stable. Documented tz-drift trade-off under Risks.
6. **The generator is a PURE function `buildVetSummary(input): string` in the api** (mirrors `downsample-weight-series.ts`: pure, unit- and golden-testable, no prisma/clock). The service `vetSummary()` gathers the four data sets + pet, maps them to the input shape, and calls it. Tier labels and section headings are constants **local to the builder file** (server-only copy, no drift because the golden imports the builder); only the disclaimer footer is promoted to `@pawcareright/types`.
7. **Checks are read directly via prisma inside `HealthLogsService`** (same posture as its existing direct `reminderEvent` read for MED_GIVEN) rather than injecting `ChecksService` (which drags in quota/queue deps). Household scope is already enforced by the leading `petsService.findOne(householdId, petId)`. Tier = the check's terminal `result.urgency` parsed via `parseTriage`; a check with no terminal/parseable result renders `"awaiting assessment"`. Follow-up escalation tiers are **not** folded in (kept minimal; the escalation still lives on the check itself — no safety surface touched). Noted under Risks.
8. **Endpoint returns JSON `{ summary: string }`** (consistent with the repo's JSON envelope + `{error:{...}}` format) rather than a raw `text/plain` body. The mobile share sheet shares `summary`. No request body/query ⇒ **no new class-validator DTO is needed**; the inherited `@HouseholdFromMembership()` guard + `@CurrentHousehold()` param + Swagger `@ApiOkResponse` satisfy §6.
9. **Mobile entry point = the Timeline tab header** (the pet's health-record surface, already active-pet-scoped). A "Prepare vet summary" button in the `SectionList` `ListHeaderComponent` triggers an on-demand fetch-then-share. No new screen/route.
10. **No new indexes / no migration.** All four gathering queries reuse existing access patterns already indexed by T064/T042 (`HealthLog(petId, kind, occurredAt)`, `SymptomCheck(petId, createdAt)`, `ReminderEvent` via `reminder.petId` + `status/completedAt`). Verified against the existing `weightSeries`/`list`/checks queries.

## Risks & edge cases (checker: scrutinize decisions 1–10)
- **Golden brittleness:** any wording/format/order change breaks `build-vet-summary.spec.ts`. Mitigated by (a) a documented regeneration policy in the test header, (b) the fixture using only fixed UTC dates and data-derived values (no clock), so the golden is a pure function of inputs.
- **tz drift in dates (decision 5):** UTC `YYYY-MM-DD` slicing can render an entry logged at 23:30 local as the next/previous calendar day. Accepted for a record digest (matches the API's existing UTC convention); documented, not silently chosen.
- **Unicode length counting:** `String.length` counts UTF-16 code units, so an astral char (emoji) counts as 2 toward the 2500 cap. The builder itself emits no emoji and content is predominantly ASCII, so the cap stays conservative; the overflow test proves `≤2500` under `.length` — the exact metric the enforcement uses.
- **Footer never truncated:** the algorithm reserves `footer.length` up front and appends it after truncation; the overflow + empty + golden footer tests all assert `endsWith(VET_SUMMARY_DISCLAIMER)`. If the executor instead appended the footer *before* budgeting, the overflow test fails loudly.
- **Empty sections:** a pet with no data must still yield a valid, footer-terminated, ≤2500 summary — covered by `EMPTY_FIXTURE`.
- **Checks without a terminal result:** rendered `"awaiting assessment"`, never a fabricated tier (parse via `parseTriage`, fail-closed). Follow-up escalation deliberately omitted (decision 7) — no safety surface, noted for the checker.
- **now-dependence of the 90-day window:** lives only in the service (`Date.now()`), never in the pure builder; the e2e seeds recent dates so rows fall in-window, and unit/golden tests bypass the clock entirely.
