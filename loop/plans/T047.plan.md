Card (docs/PHASES.md:234-236): Submit → immediate red-flag branch (T048/T049) or
polling screen (1.5s backoff, calm copy, cancel-safe); offline guard (block submit
w/ clear message). **Accept:** polling stops on DONE/FALLBACK; red-flag branch
bypasses polling; offline submit blocked with retry affordance.

## Objective

Replace the T045 `handleSubmit` stub with the real submit → branch → load flow:
1. Add typed check-response shape + a client-visible "terminal status" helper to
   `packages/types` (single source of truth, CLAUDE §6 — no hand-written duplicate).
2. Add mobile TanStack hooks `useCreateCheck` (mutation, `Idempotency-Key`) and
   `useCheck` (polling query that stops on terminal status).
3. Add a `useCheckSubmission` orchestration hook: offline guard → mutate → red-flag
   vs polling branch → 402/error handling with retry.
4. Add the calm, cancel-safe polling screen `/check/waiting/[checkId]`.
5. Wire the intake route to it; register the new route; add §7-clean strings.

T047 builds NO AI content, disclaimer, emergency-interstitial, or result UI — it only
*routes* to the contract routes that T048/T049 own. See "Flow & navigation contract".

## Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)

- **§7 rule 4 (emergency-before-AI) is the load-bearing invariant here.** A `201`
  whose body carries `redFlag` MUST route to the emergency interstitial route
  (`/check/emergency/[checkId]`, T049) and MUST NOT mount the polling/AI-result path.
  This is the mobile half of "Emergency interstitial *before* any AI content." The
  red-flag → emergency navigation and the "no polling mount for red-flags" assertion
  are explicit acceptance tests below.
- **Polling copy is neutral/calm and pinned in-plan** (see Strings). No diagnosis /
  "diagnose" wording (§7 rule 1), no reassurance that could downgrade urgency, no
  medication/dosing text (§7 rule 2).
- **402 (quota) copy is §7-clean and safety-positive**: it never discourages seeking
  care. Pinned copy includes an explicit "you can still reach a vet anytime" line.
  Upgrade hint (SPEC F8) is a neutral "See plans" affordance — no medical claim.
- **Uncertainty fails upward (§7 rule 5):** provider/mutation errors show a plain
  retry affordance — never a silent retry-and-guess, never fabricated AI content.
- This task authors no §5 safety *surface* (disclaimer / hotline / emergency copy);
  those remain T048/T049. Nothing here weakens a Safety Policy surface, so the task
  is plannable (not SAFETY-ESCALATION).

## Key decisions (numbered; scrutinize in Risks)

- **D1 — Hooks live in `apps/mobile/src/api/checks-api.ts`, NOT `packages/api-client`.**
  The established repo pattern (`pets-api.ts`, `households-api.ts`) puts TanStack
  hooks in the mobile app because they bind the mobile `apiClient` singleton
  (`apps/mobile/src/api/client.ts`). `packages/api-client` is deliberately
  platform-agnostic infra (client factory, query-client, errors, `online`) with no
  domain hooks and no `apiClient` instance; hosting checks hooks there would break
  that boundary. This diverges from the card-hint wording ("api-client hooks") in
  favor of repo reality + the `pets-api` precedent.
- **D2 — Shared response shape goes in `packages/types` as Zod** (`check.ts`):
  `checkRedFlagSchema` + `checkResponseSchema` (+ inferred `CheckRedFlag`,
  `CheckResponse`). Mirrors how `Pet` is a Zod-inferred type consumed by `pets-api`.
  T042 kept `CheckResponse` service-local (its D4); T047 needs the client shape and
  §6 forbids hand-writing a duplicate interface. `createdAt` is modeled as
  `z.string()` (JSON-serialized ISO; the field is unused by T047 but included for
  T048). `result` is `triageResultSchema.optional()` (reuse existing schema).
- **D3 — No runtime `.parse()` of the GET response in T047.** Hooks type the
  response via `apiClient.get<CheckResponse>` (exactly the `pets-api` posture).
  Defensive result-schema validation on render is a T048 concern.
- **D4 — "Terminal status" knowledge is added to `packages/types`**
  (`TERMINAL_CHECK_STATUSES` + `isTerminalCheckStatus` in `check-status.ts`), reused
  by the mobile hook. Avoids hardcoding `"DONE"/"FALLBACK"` strings in mobile. The
  server-side `apps/api/src/checks/check-status.ts` keeps its own copy (server
  transition logic, out of T047 scope — not refactored here).
- **D5 — Idempotency-Key: one UUID per submission attempt, reused across retries.**
  Generated lazily via `expo-crypto` `randomUUID()` on first `submit`, held in a
  `useRef`, cleared only after a successful branch navigation. A blocked-offline or
  errored attempt keeps the key, so the retry is an idempotent replay (T042 returns
  the same check). A brand-new intake submission gets a fresh key.
- **D6 — Both terminal statuses route to the result screen.** The waiting screen
  treats `DONE` and `FALLBACK` identically → `/check/result/[checkId]` (T048 renders
  the safe fallback for `FALLBACK`). This satisfies "handles FALLBACK same as DONE"
  and §7 rule 5 (fallback is a first-class terminal outcome, not an error).
- **D7 — Cancel-safe = navigate away, don't cancel the server job.** The waiting
  screen's back/cancel affordance does `router.replace` to pet home; the check keeps
  running server-side (T043 worker) and is reachable later via history (T050).
  Leaving unmounts `useCheck`, which stops polling. No abort call.
- **D8 — `photoKeys` are flattened from the intake.** The POST DTO takes a top-level
  `photoKeys` array (server stores it for the worker); the keys live inside
  `photoPrompt` answers. A pure `extractPhotoKeys(intake)` helper flattens them.
- **D9 — `router.replace` (not `push`) for both branches.** After submit the intake
  form should not remain on the back stack (you can't edit an already-submitted
  check). Emergency and polling both replace the `[category]` screen.

## Hook & helper specs

### `checkRefetchInterval` (pure, exported from `checks-api.ts`)
```ts
export const CHECK_POLL_INTERVAL_MS = 1500;
// TanStack v5 refetchInterval-as-function: receives the Query; reads its data.
export function checkRefetchInterval(
  query: { state: { data?: CheckResponse } },
): number | false {
  const status = query.state.data?.status;
  if (status === undefined) return CHECK_POLL_INTERVAL_MS;      // no data yet → keep polling
  return isTerminalCheckStatus(status) ? false : CHECK_POLL_INTERVAL_MS;
}
```
Type the param compatibly with TanStack's `Query` (the executor may widen to the
library type and read `query.state.data`); the pure test constructs a minimal
`{ state: { data } }`.

### `useCheck(checkId)`
```ts
useQuery({
  queryKey: checksKeys.detail(checkId),
  queryFn: () => apiClient.get<CheckResponse>(`/v1/checks/${checkId}`),
  enabled: checkId.length > 0,
  refetchInterval: checkRefetchInterval,
});
```

### `useCreateCheck(petId)` (mirrors `useCreatePet`)
```ts
useMutation({
  mutationFn: (vars: { intake: CompletedIntake; photoKeys: string[]; idempotencyKey: string }) =>
    apiClient.post<CheckResponse>(
      `/v1/pets/${petId}/checks`,
      { intake: vars.intake, photoKeys: vars.photoKeys },
      { headers: { "Idempotency-Key": vars.idempotencyKey } },
    ),
  onSuccess: () => { void queryClient.invalidateQueries({ queryKey: checksKeys.list(petId) }); },
});
```
`checksKeys = { all: ["checks"], detail: (id) => ["checks", id], list: (petId) => ["checks","pet",petId] }`.

### `useCheckSubmission({ petId, onEmergency, onPolling })`
State machine `state: "idle"|"submitting"|"offline"|"quota"|"error"`; internal
`idempotencyKeyRef` (D5), `pendingIntakeRef`. Uses `useCreateCheck(petId)`,
`useIsOffline()`, `expo-crypto`.
- `submit(intake)`: store `pendingIntakeRef`; if `petId` undefined → `state="error"`;
  else if `useIsOffline()` → `state="offline"` (NO mutate); else set key if null
  (`Crypto.randomUUID()`), `state="submitting"`,
  `mutateAsync({ intake, photoKeys: extractPhotoKeys(intake), idempotencyKey })`.
  - success `(check)`: `check.redFlag !== undefined` → clear key, `onEmergency(check.id)`;
    else clear key, `onPolling(check.id)`.
  - error `(e)`: `isApiError(e) && e.code === "PAYMENT_REQUIRED"` → `state="quota"`;
    else `state="error"`. (Import `isApiError` from `@pawcareright/api-client`.)
- `retry()`: re-invoke `submit(pendingIntakeRef.current)` (reuses same key → idempotent).

## Flow & navigation contract

Submit (review "Continue") → `useCheckSubmission.submit(intake)`:
- **offline** → blocked, `strings.check.submit.offlineBlocked` + retry (no network call).
- **online → `201` with `redFlag`** → `router.replace({ pathname: "/check/emergency/[checkId]", params: { checkId } })`
  — bypasses polling (§7 rule 4). **T049 route.**
- **online → `201` without `redFlag`** → `router.replace({ pathname: "/check/waiting/[checkId]", params: { checkId, petId } })`.
- **`402`** → `quota` state (calm copy + "See plans"), retry available.
- **other error** → `error` state + retry.

Waiting screen `/check/waiting/[checkId]` (**T047, created here**):
- `const { data } = useCheck(checkId)`; polls every 1.5s until terminal (D4).
- `useEffect`: when `data && isTerminalCheckStatus(data.status)` →
  `router.replace({ pathname: "/check/result/[checkId]", params: { checkId } })` (both
  DONE and FALLBACK, D6). **T048 route.**
- Cancel/back affordance → `router.replace({ pathname: "/pets/[id]", params: { id: petId } })`
  (D7, cancel-safe). Renders calm title/body + `ActivityIndicator`; SafeAreaView;
  offline note non-blocking.

**Contract routes T048/T049 MUST implement (documented here, not created here):**
`/check/result/[checkId]` (T048, params `{ checkId }`) and
`/check/emergency/[checkId]` (T049, params `{ checkId }`). Precedent: T044 documented
a forward route contract and the checker accepted it; typed-routes is NOT enabled
(no `experiments.typedRoutes`), so string/object hrefs to not-yet-created routes do
not fail typecheck.

## Risks & decisions

1. **D1 divergence from card hint.** Card hint says "api-client hooks"; repo puts
   domain hooks in `apps/mobile/src/api` (they need the mobile `apiClient` singleton).
   Chose repo reality + `pets-api` precedent. Checker: confirm this is the intended
   layering and that no domain hook belongs in the platform-agnostic package.
2. **D2 adds a Zod schema to `packages/types`.** Minimal but crosses into a package
   T042 chose to defer (its D4 kept the shape service-local). Justification: §6
   forbids a duplicate hand-written client interface; `Pet` sets the precedent.
   `createdAt` typed `z.string()` (JSON), unused by T047. Risk: schema drift from the
   server `CheckResponse` interface — mitigated by keeping fields identical and by
   `check.spec.ts` round-trip over a realistic body.
3. **D4 duplicates terminal-status knowledge** across `packages/types` (new) and
   `apps/api/src/checks/check-status.ts` (existing server copy). Accepted: additive,
   no behavior change; server refactor is out of scope. Alternative (mobile-local
   predicate) was rejected to avoid hardcoding status strings in the app.
4. **Navigating to not-yet-existing routes** (`/check/emergency/*`, `/check/result/*`).
   Safe at typecheck time (typed-routes not enabled) and in tests (router mocked).
   At runtime these resolve once T048/T049 land — same staged-build pattern the
   checker accepted for T044. Risk if the loop runs the app before T048/T049: a
   missing-route navigation. Deemed acceptable per card intent; the contract route
   names are pinned above so T048/T049 match.
5. **`refetchInterval` typing.** TanStack v5's function form receives the `Query`
   object; the exact generic type is verbose. The pure helper is typed against a
   minimal structural `{ state: { data?: CheckResponse } }`; if TS rejects assigning
   it to `refetchInterval`, widen the param to the library's `Query` type (no `any`,
   no `@ts-ignore`). The unit test targets the structural signature directly.
6. **D5 idempotency-key lifecycle.** Key persists through offline-block and errors so
   retries replay; cleared on success so a fresh intake gets a fresh key. Risk: if the
   user edits answers and resubmits within the same mounted screen after an error, the
   old key is reused — acceptable/desired (server dedupes by key+user; a genuinely new
   check comes from a new screen mount). Checker: confirm this matches "retry reuses
   the key" intent.
7. **`expo-crypto` dependency** already in `apps/mobile/package.json` (`~57.0.0`) — no
   new dependency added.
8. **Cancel-safe semantics (D7).** Back does not abort the server job; the check
   completes and surfaces in history (T050). If the checker expects an explicit
   client-side cancel/abort, that is not in the card ("cancel-safe" = leaving is
   harmless) and would be over-engineering.
