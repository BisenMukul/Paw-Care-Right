## Risks & the design decisions the planner made (so checker can scrutinize)
1. **Env parameterization = interpolation-with-defaults.** `docker-compose.yml` uses `${VAR:-default}` so it runs with zero setup, and the keys are documented in `.env.example` (T003 established `.env.example` as "every key used across the repo"). Alternative rejected: hardcoding creds inline (less flexible, duplicates values).
2. **Ports = standard defaults** (5432/6379/9000/9001) exposed directly; conflicts handled by documenting the `*_PORT` override vars rather than pre-offsetting. Risk: a port already bound on the dev laptop — mitigated by the override note.
3. **Bucket bootstrap = `minio/mc` sidecar `createbuckets`**, `restart:"no"`, gated on `minio` healthy, using `mc mb --ignore-existing` (idempotent). Reused for the independent AC3 check via `docker compose run --rm`.
4. **minio healthcheck = `curl -f .../minio/health/live`** (curl is in the minio image; `mc` is not). Executor MUST verify curl exists in the pinned tag; if absent, fall back to the image-documented healthcheck (stated as a controlled fallback, not free improvisation).
5. **No redis volume; named volumes for postgres (`pgdata`) and minio (`miniodata`).** Redis is a dev cache/queue — persistence unneeded. Risk: dev loses redis data on `down` — acceptable pre-BullMQ.
6. **Omit top-level `version:` key** — deprecated/ignored by Compose v2+ (host has Compose v5 CLI).
7. **Teardown = `docker compose down` after verification** (laptop/CI hygiene); named volumes intentionally persist.
8. **Pin the minio image tag** rather than `:latest` for reproducibility; executor picks a current documented dated tag.
