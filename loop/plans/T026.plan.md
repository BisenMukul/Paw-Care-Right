# Plan — T026: Household invites (M, api + mobile)

## Objective (from card)
`POST /households/invites` (OWNER-only) mints a short code + deep link (`pawcareright://join/:code`, 7-day expiry); `POST /households/invites/accept` joins the caller into the inviting household as MEMBER; mobile gains settings→Family screen (member list + invite share sheet) and a `join/[code]` deep-link handler route.

---

## ⚠️ CENTRAL DESIGN COLLISION — resolved FIRST (checker: scrutinize here)
Every user is auto-provisioned their OWN household (T012) and `HouseholdScopeGuard`'s from-membership mode **404s whenever memberships ≠ 1** (documented deferral, guard lines 94–112). If an accept simply *added* a membership, the joiner would have 2 memberships → every `/v1/pets` + photo route breaks with 404.

**Resolution = JOIN-REPLACES (option a).** Accepting an invite makes the joiner *leave* their current household so memberships stay **exactly 1**. This preserves the guard invariant everywhere with zero change to existing scoped routes. The rule "**exactly 1 membership per user**" becomes an explicit v1 system invariant — documented in a code comment in `households.service.ts` and here. Multi-household selection remains deferred (guard already cites T027/T028).

Leave semantics inside the accept transaction:
- Joiner is **OWNER** of their current (auto-provisioned) household → the household is deleted (its sole membership cascades away). Blocked if that household still has non-deleted pets (would strand/delete the user's own pet data) → **409 CONFLICT**.
- Joiner is **MEMBER** of their current household (previously joined someone else's) → only their membership row is deleted; the household (someone else's) is untouched. The pets-present block still applies uniformly for v1 simplicity/safety.

This option was chosen over (b) multi-household selector — that touches every scoped route + mobile state and is explicitly NOT this card.

### acceptInvite transaction — exact order (single `prisma.$transaction`)
Pre-checks (may run before txn, re-validated inside):
1. Load invite by `code`. If **not found**, or `expiresAt < now`, or `usedAt != null` → **`NotFoundException` (404)** — uniform, see Risk R1.
2. Load joiner's memberships. If `length !== 1` → **`ConflictException` (409)** (unsupported multi/zero-household state; invariant guard).
3. `sole = memberships[0]`. If `sole.householdId === invite.householdId` → **`ConflictException` (409)** ("already a member of that household" — covers self-accept and re-accept).

Inside `$transaction`:
4. **Claim invite atomically:** `updateMany({ where: { id: invite.id, usedAt: null }, data: { usedAt: now, usedById: joiner } })`. If `count !== 1` → throw `NotFoundException` (lost a race → uniform 404). This is the single-use guarantee (the reuse-blocked AC).
5. **Pets guard:** count non-deleted pets in `sole.householdId` (`{ householdId, deletedAt: null }`). If `> 0` → throw `ConflictException` (409, "remove/rehome pets before joining another household"). Throwing rolls back step 4.
6. **Leave current household:** if `sole.role === "OWNER"` → `household.delete({ where: { id: sole.householdId } })` (cascades the sole membership; no pets, asserted). Else → `membership.delete({ where: { id: sole.id } })`.
7. **Join target:** `membership.create({ userId: joiner, householdId: invite.householdId, role: "MEMBER" })`. The `@@unique([userId, householdId])` is a second backstop against a double-join race.
8. Return `{ householdId: invite.householdId, name: <target household name> }`.

FK/cascade facts relied on: `Household → Membership` = onDelete Cascade (schema line 62); `Pet → Household` = onDelete Cascade (line 113); `HouseholdInvite → Household` = onDelete Cascade (new). Deleting a *household* is allowed — `Household.owner` Restrict blocks deleting the *User*, not the household.

## Risks & the design decisions the planner made (checker: scrutinize)
- **R1 (anti-probing uniform 404):** invalid / expired / used invites ALL return `404 NOT_FOUND` with the same message, so a caller cannot brute-force/probe which 8-char codes exist or are live. Distinguishable business errors (own-household, pets-present, unsupported state) are `409 CONFLICT` — those require an authenticated caller acting on a *valid* code, so they leak nothing about code space.
- **R2 (JOIN-REPLACES invariant):** the central collision resolution. "Exactly 1 membership per user" is now an explicit invariant. Weigh vs. multi-household: rejected as out-of-card scope. Data-loss guard = the pets-present 409 (never silently deletes a household with pets).
- **R3 (code alphabet/length):** 8 chars over a 32-symbol unambiguous alphabet ≈ 1.1×10¹² space; single-use + uniform-404 make guessing impractical. No global rate-limit added beyond the app-wide `ThrottlerGuard` default (accept is authenticated). Codes are uppercase-only; the DTO regex is uppercase — deep links always produce uppercase, so no case-normalization transform is added (kept minimal).
- **R4 (always-new invites):** `createInvite` always mints a fresh row rather than reusing an unexpired one — simpler, no read-modify race; multiple live invites per household is acceptable.
- **R5 (accept status = 200):** accept is an action, not resource creation → `@HttpCode(200)`; create-invite stays default 201. e2e assert accordingly.
- **R6 (GET /households/me added):** the card's "list members" mobile screen needs a data source; ARCHITECTURE §4 already lists `GET /households/me`, so it is added here as that source (not scope creep).
- **R7 (safety):** T026 touches NO Safety Policy surface (no AI output, disclaimer, emergency escalation, or dosing) — CLAUDE.md §7 / PRODUCT_SPEC §5 unaffected. Not a SAFETY-ESCALATION.
