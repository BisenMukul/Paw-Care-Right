# Plan — T051: Follow-up loop

## Objective (from card)
Schedule a follow-up push from a result's `followUpHours` (queue interface now; P5 consumes), add `POST /checks/:id/followup` accepting `better|same|worse`, and on `worse` create a tier+1 escalation (persisted, exposed in the check response, floored at `EMERGENCY_NOW`). API tests for all three responses, worse-path escalation, and duplicate-idempotency.

## Safety statement (read first)
- Escalation ONLY ever RAISES urgency: `escalatedTier = raiseUrgency(finalUrgency)` steps up exactly one tier and no-ops at `EMERGENCY_NOW` (fail-upward, PRODUCT_SPEC §5 rule 3 / CLAUDE §7). It can never lower a tier.
- Idempotency is **first-write-wins** on a one-row-per-check table: once an escalation row exists it is never mutated or removed, so a later `better`/`same` can never un-escalate a prior `worse` (§5 never-downward).
- **No new user-facing medical/guidance copy is authored in this task.** The API returns only the `escalatedTier` enum + the raw `response`; the "tier+1 guidance / vet CTA" is rendered later by mobile reusing existing per-tier result copy. Nothing here emits dosing, drug names, or "diagnos*" language.
- The `check-runner` worker's §5-critical persist/transaction/fallback behavior stays **byte-identical**. Scheduling is a purely additive, post-commit, `try/catch`-wrapped best-effort call; a queue failure never fails the triage job nor changes any persisted safety state. A missed follow-up push is acceptable graceful degradation.

## Key decisions (see Risks for rationale — each is a scrutiny point)
1. New one-to-one table `CheckFollowUp` (`checkId` PK), `response` and `escalatedTier` stored as `String` (mirrors the existing `TriageResult.urgency` String precedent) — **no new Prisma enum**.
2. Idempotency = first-write-wins via the `checkId` primary key; duplicate POST returns the existing state with `200`.
3. `escalatedTier` uses a new exported pure helper `raiseUrgency` added to `packages/types` (single source of the tier constants); `apply-post-rules.ts` is left untouched.
4. New `pawcareright-followups` BullMQ queue; a **delayed** job (`jobId = checkId`) is enqueued by the worker. **No processor** — P5 (reminders/push) consumes it.
5. Follow-up allowed only on a terminal check with a schema-valid result → `409 CONFLICT` otherwise; endpoint returns `200`.
6. Mobile / api-client / timeline rendering are OUT OF SCOPE; the `CheckResponse` Zod schema is extended so future surfaces read `followUp` with zero client-code change.

## Data model & migration spec
Add to `schema.prisma`:
```prisma
model CheckFollowUp {
  checkId       String   @id
  response      String   // "better" | "same" | "worse" (validated by Zod at the boundary; String mirrors TriageResult.urgency)
  escalatedTier String?  // an Urgency value, set ONLY when response == "worse"
  createdAt     DateTime @default(now())

  check SymptomCheck @relation(fields: [checkId], references: [id], onDelete: Cascade)
}
```
And on `SymptomCheck`, add the back-relation field (one-to-one, mirrors `result TriageResult?`):
```prisma
  followUp  CheckFollowUp?
```
`checkId` is the PK → naturally unique/indexed (no extra `@@index` needed), and it gives the first-write-wins idempotency (a second insert raises Prisma `P2002`). Migration is generated by the migrate command below; verify the generated SQL only creates the `CheckFollowUp` table + FK and touches no other table.

## Endpoint spec
`POST /v1/checks/:id/followup`  (class already has `@HouseholdFromMembership()`; global `JwtAuthGuard` applies)
- Body: `{ "response": "better" | "same" | "worse" }` (`FollowUpDto`).
- `@HttpCode(200)` on both create and idempotent replay.
- Swagger: `@ApiOkResponse`, `@ApiBadRequestResponse`, `@ApiConflictResponse`, `@ApiNotFoundResponse`, `@ApiUnauthorizedResponse` (inherited).
- Returns the updated `CheckResponse` (now carrying `followUp`).

`ChecksService.submitFollowUp(householdId: string, id: string, response: FollowUpResponse): Promise<CheckResponse>`:
1. `findFirst` check `where: { id, pet: { householdId, deletedAt: null } }`, `include: { result: true, followUp: true }`. Null → `NotFoundException` (no leak, mirrors `findOne`).
2. If `check.followUp` already exists → return `toResponse(check)` (idempotent, first-write-wins, 200) — before any state/precondition checks.
3. Precondition (uniform for all three responses): require terminal status AND a schema-valid result. Compute `parsed = parseTriage(check.result?.resultJson)`. If `!isTerminalCheckStatus(check.status)` or `!check.result` or `!parsed.ok` → `ConflictException("check is not ready for follow-up")` (409 → `CONFLICT`).
4. `escalatedTier = response === "worse" ? raiseUrgency(parsed.result.urgency) : null`.
5. Create the row; wrap in `try/catch` for `P2002` (concurrent duplicate) → re-fetch the existing check with `followUp` and return it (mirrors `persistCheck`'s P2002 handling).
6. Attach the created row to the in-memory check (`check.followUp = created`) and return `toResponse(check)`.

`toResponse` addition (only when `check.followUp` present):
```ts
response.followUp = {
  response: check.followUp.response as FollowUpResponse,
  ...(check.followUp.escalatedTier ? { escalatedTier: check.followUp.escalatedTier as Urgency } : {}),
};
```
Extend the service `CheckResponse` interface with `followUp?: { response: FollowUpResponse; escalatedTier?: Urgency }` and `CheckRow` with `followUp?: CheckFollowUp | null` (Prisma type). Add `followUp: true` to the `findOne` and `list` `include`.

## Scheduling spec (queue contract for P5)
`followups.contract.ts` is the single source shared by the producer (worker, this task) and the future P5 consumer:
- Queue name: `pawcareright-followups`; job name: `followup-prompt`; payload: `{ checkId: string }`.

In `check-runner.processor.ts`:
- Constructor: add `@InjectQueue(FOLLOWUPS_QUEUE) private readonly followUpQueue: Queue<FollowUpJobData>`.
- Private `scheduleFollowUp(checkId: string, result: TriageResult): Promise<void>`:
  - `const hours = result.followUpHours; if (hours === null) return;`
  - `const delay = hours * 3_600_000;`
  - `try { await this.followUpQueue.add(FOLLOWUP_JOB_NAME, { checkId }, { jobId: checkId, delay, removeOnComplete: true, removeOnFail: false }); this.logger.log({ event: "followup_scheduled", checkId, delay }); } catch { this.logger.warn({ event: "followup_schedule_failed", checkId }); }`
  - Both log entries are checkId-keyed with safe scalars only (satisfies the existing log-privacy test).
- Call `await this.scheduleFollowUp(checkId, outcome.result)` AFTER the happy-path `$transaction` + `costLog.record`, and AFTER the catch-path fallback `$transaction` + `costLog.record`. Do not move or alter any existing line inside/around the transactions.
- `jobId = checkId` makes re-enqueue idempotent; the delayed job is inert until P5 attaches a worker.

`workers.module.ts`: change the registration to `BullModule.registerQueue({ name: IMAGES_QUEUE }, { name: CHECKS_QUEUE }, { name: FOLLOWUPS_QUEUE })`.

## Escalation semantics
- `raiseUrgency(tier)` (new, in `triage.ts`): uses `URGENCY_TIERS` (ordered most→least urgent, so severity === index) + `URGENCY_SEVERITY`; returns `URGENCY_TIERS[sev-1]` for `sev>0`, else `tier` (no-op at `EMERGENCY_NOW`). Pure, total, never throws. Semantically identical to the private `raiseOne` in `apply-post-rules.ts` (which stays as-is).
- `worse` → `escalatedTier = raiseUrgency(finalUrgency)`; `better`/`same` → `escalatedTier = null`.
- Ceiling: a `worse` follow-up on an `EMERGENCY_NOW` result yields `escalatedTier === "EMERGENCY_NOW"`.

## Risks & decisions the planner made (numbered — for the checker)
1. **Table vs columns / String vs enum.** Chose a one-to-one `CheckFollowUp` table (mirrors `TriageResult`) with `response`/`escalatedTier` as `String` — matching the repo's existing choice to store `TriageResult.urgency` as a validated `String` rather than a Prisma enum. Avoids a case-mapping layer and keeps the timeline (P5) join clean. Risk: `String` is not DB-enforced — mitigated by Zod/`@IsIn` at the boundary and a `String?` escalatedTier only ever written from `raiseUrgency`.
2. **Idempotency = first-write-wins.** The `checkId` PK gives a single follow-up per check; a duplicate/late differing response returns the original state (200) and never mutates the escalation. Rationale: safety (§5) forbids un-escalating; a genuinely new concern is a new symptom check, not a follow-up amendment. Risk: a user cannot correct a mis-tap — accepted for v1; a future task could allow upward-only re-escalation.
3. **`raiseUrgency` duplication.** Added a new exported helper in `packages/types` instead of editing the §5-critical `apply-post-rules.ts` to export its private `raiseOne`. Deliberately avoids modifying T036 code. Risk: two identical implementations — flagged for a future dedupe; both are trivially unit-tested.
4. **No API-authored guidance copy.** API returns only `escalatedTier` + `response`; the "tier+1 guidance + vet CTA" is rendered by future mobile reusing per-tier copy. Shrinks the §5 copy surface to zero in this task. Risk: the card literally says "creates escalation card ... guidance" — interpreted as *creating the escalation data*, not authoring new copy; the tier fully determines the guidance shown.
5. **Best-effort scheduling.** `scheduleFollowUp` is post-commit and `try/catch`-wrapped so a Redis/queue outage can never fail an already-persisted triage or trigger a retry that (on redelivery) would skip scheduling via the terminal-no-op guard. Risk: a follow-up push can be silently missed on queue failure — accepted as graceful degradation (non-safety-critical; the result itself is safe and stored).
6. **409 precondition uniform for all three responses.** A follow-up requires a terminal, schema-valid result even for `better`/`same` (which don't strictly need the urgency). Simpler, single precondition; the common case always passes. Risk: an unreadable-result FALLBACK corner returns 409 — acceptable and safe.
7. **200 (not 201).** The endpoint is an idempotent action returning the updated resource, not a nested collection create; `@HttpCode(200)` keeps create and replay uniform.
