## Safety statement (§5 surface — planned, NOT blocked)
This IS the primary PRODUCT_SPEC §5 surface of Phase 3: the system prompt *encodes* the Safety Policy and the exemplar outputs *teach* the safe output style. T033 does not weaken any §5 surface — it strengthens them, and app-side validation (T030) remains the enforcement floor. Concretely:
- The system prompt instructs: guidance-not-diagnosis (never emit "diagnosis"/"diagnose" in results); **no medication dosages, drug-name recommendations, or human-medication administration**; fail-upward on uncertainty; cats bias one tier more urgent; refuse harm-enabling requests; `homeCare` only on VET_SOON/MONITOR/REASSURE; low confidence ⇒ at least VET_SOON.

## Prompt architecture

### System prompt (`buildSystemPrompt`) — section-by-section, STATIC
Ordered sections, plain text, deterministic. Refers to "this service"/"this guidance app" — **no product display name is embedded** (Decision R7: avoids hardcoding `APP_DISPLAY_NAME` and avoids a new `packages/config` dep). Sections:
1. **Role.** "You are the triage reasoning engine for a dog-and-cat pet-care guidance app. You produce general, plain-language guidance for pet owners. You are not a veterinarian and you never provide diagnosis or treatment."
2. **Absolute safety rules (encode §5):**
   - Never use the words "diagnosis" or "diagnose" in any field; use "possible causes" / "what this could be".
   - Never give medication dosages, never recommend or name a drug to give, never tell an owner to administer any human medication to an animal. `doNot` may warn *against* giving human medications in general terms only.
   - When information is missing, ambiguous, or conflicting, choose the **more urgent** tier and lower `confidence`. Never guess downward.
   - Cats hide illness: for an equivalent presentation, treat a cat as **one tier more urgent** than a dog (caution language; the app also enforces this deterministically).
   - Refuse anything that could harm an animal (cruelty, fighting, DIY sedation/surgery/euthanasia, breeding malpractice): return a firm, kind escalation to a vet/authority inside the JSON, never instructions.
   - If the input describes a life-threatening sign, use `EMERGENCY_NOW` or `VET_24H`.
3. **Tier definitions** (from PRODUCT_SPEC §4/§6.3): `EMERGENCY_NOW` (go now), `VET_24H` (within a day), `VET_SOON` (within ~72h), `MONITOR` (watch at home with a re-check trigger), `REASSURE` (likely fine).
4. **Confidence definitions:** high / medium / low; **low ⇒ urgency must be at least VET_SOON**.
5. **Field rules:** `possibleCauses` max 4 and never called a diagnosis; `homeCare` allowed ONLY on VET_SOON/MONITOR/REASSURE and MUST be empty for EMERGENCY_NOW/VET_24H; `followUpHours` is a positive integer or null.
6. **Output contract:** "Return ONLY a single JSON object matching the schema below. No markdown, no code fences, no text before or after the JSON." followed by `TRIAGE_SCHEMA_TEXT`.
7. **Footer:** `Prompt version: triage-v1`.

Note: the system prompt legitimately contains the literal word "diagnosis" (inside "never use the word diagnosis"). The forbidden-content lint is scoped to **exemplar outputs + user turns**, NOT the system prompt (see Tests).

### `TRIAGE_SCHEMA_TEXT` shape (schema-text.ts)
A fenced-free text block mirroring `triageResultSchema`, e.g.:
```
{
  "urgency": <one of: EMERGENCY_NOW | VET_24H | VET_SOON | MONITOR | REASSURE>,   // from URGENCY_TIERS.join(" | ")
  "confidence": <one of: high | medium | low>,                                   // from CONFIDENCE_LEVELS.join(" | ")
  "summary": string (1-2 plain-language sentences),
  "possibleCauses": array (max 4) of { "name": string, "whyItFits": string },
  "redFlagsToWatch": string[],
  "homeCare": string[] (ONLY for VET_SOON/MONITOR/REASSURE; otherwise []),
  "doNot": string[],
  "vetQuestions": string[],
  "followUpHours": integer > 0, or null
}
```
The two `<one of …>` lists are interpolated from the imported constants; everything else is static.

### User-turn serialization (`buildUserTurn`)
Input `TriagePromptInput = { pet: TriagePetContext; intake: CompletedIntake; }` (photo count derived from intake — see below). Deterministic plain-text, label-based:
```
PET
- Name: {pet.name}
- Species: {Dog|Cat}
- Breed: {pet.breedLabel or "Unknown / mixed"}
- Sex: {Male|Female|Unknown}{", neutered" if pet.neutered === true}
- Age: {pet.ageMonths} months        (omit line if undefined)
- Weight: {pet.weightKg} kg           (omit line if undefined)

REPORTED PROBLEM: {categoryDef.label}

INTAKE
- {question.prompt}: {rendered answer}    (one line per answer, in intake.answers order)
...

OWNER'S DESCRIPTION: {intake.freeText}    (omit whole line if freeText undefined)

PHOTOS ATTACHED: {n}                       (n = total photoKeys across photoPrompt answers; line always present, n may be 0)

Return only JSON matching the schema. No text outside the JSON.
```
Answer rendering (resolve question/option via `getCategoryDef(intake.category)`; defensive fallback to the raw value only if a label cannot be resolved — intake is already validated so this is belt-and-suspenders):
- `single` → option **label** for the answered value (e.g. `four-plus` → "Four or more times").
- `multi` → option labels joined with ", ".
- `scale` → `{value} of {max} ({min} = {minLabel}, {max} = {maxLabel})`.
- `duration` → `{value} {unit}`.
- `photoPrompt` → not listed in INTAKE lines; only contributes to the PHOTOS ATTACHED count.
**PII guarantee:** `TriagePetContext` structurally contains only `name, species, breedLabel?, sex?, ageMonths?, weightKg?, neutered?` — no email/householdId/userId/ownerName field exists, so none can be serialized. Actual image bytes are attached by the vision path (T034/T043); here photos are a count placeholder only.

### Version registry (version.ts)
`TRIAGE_PROMPT_VERSION = "triage-v1"`; `TRIAGE_PROMPT_REGISTRY: Record<string, { version: string; notes: string }>` with one `"triage-v1"` entry. Static data; the version string is embedded in the system prompt footer and returned as `BuiltTriagePrompt.version`.

### Repair prompt (repair.ts)
`buildRepairPrompt(reason)` → e.g. "Your previous response was not valid TriageResult JSON. Problems: {reason}. Return only corrected JSON that matches the schema exactly — no code fences, no text outside the JSON." Deterministic given `reason`.

## Repair / fallback flow semantics (`runTriage`)
`RunTriageDeps = { provider: TextProvider }`. `TriageOutcomeStatus = "OK" | "REPAIRED" | "SAFE_FALLBACK"`. `TriageRunResult = { status; result: TriageResult; version: string; attempts: number; usage?: ProviderUsage; failureReason?: string }`.
1. `built = buildTriagePrompt(input)`.
2. Call `provider.generate({ system: built.system, messages: built.messages, temperature: 0 })` → `res1`.
3. `parseTriage(extractJsonCandidate(res1.text))`. If ok → `{status:"OK", result, version, attempts:1, usage:res1.usage}`.
4. Else ONE repair retry: `messages2 = [...built.messages, {role:"assistant", content:res1.text}, {role:"user", content: buildRepairPrompt(parsed1.reason)}]`; call provider again (same system, temp 0) → `res2`; `parseTriage(extractJsonCandidate(res2.text))`. If ok → `{status:"REPAIRED", result, version, attempts:2, usage:res2.usage}`.
5. Else → `{status:"SAFE_FALLBACK", result: SAFE_FALLBACK, version, attempts:2, failureReason: parsed2.reason}`. **Never** return an unvalidated object; never a third silent retry (§5 rule 2 / CLAUDE §7 rule 5).
6. If any `provider.generate` rejects/throws → catch and return `{status:"SAFE_FALLBACK", result: SAFE_FALLBACK, version, attempts: <calls made>, failureReason: <error message>}`. `runTriage` never throws (Decision R6). BullMQ retry-with-jitter is layered by T043 around this; `runTriage` itself always fails upward to the frozen fallback.

## Risks & design decisions the planner made (for CHECKER scrutiny)
- **R1 — `TRIAGE_SCHEMA_TEXT` is hand-written text, but its enum lists are interpolated from `URGENCY_TIERS`/`CONFIDENCE_LEVELS`.** Full auto-serialization of the Zod schema would need a dep (forbidden). The remaining descriptive text can drift from `triageResultSchema`; mitigated by (a) interpolating the two enums so tiers/confidence cannot drift, and (b) a snapshot + field-name-presence assertion. If the checker wants zero drift risk, note this is the minimal no-dep design and `parseTriage` remains the true gate.
- **R2 — Extraction uses de-fence then first-`{`-to-last-`}` (not full brace-balancing).** Triage output is a single JSON object; any trailing/leading prose is sliced away, and `parseTriage` rejects anything still malformed. Balanced-brace parsing would be over-engineering for a single-object contract. This is the fix for the T030 carried note (fenced/BOM output must be de-fenced/stripped BEFORE `parseTriage`).
- **R3 — Deterministic cat-bias/confidence-floor tier arithmetic is DEFERRED to T043, not shipped here.** T033 ships only prompt caution *language* (the card's "species caution biases"). T043's card explicitly owns the post-rules tier math ("confidence floor, cat bias, rules tier floor cannot be lowered"); T036 is the eval-harness runner and is NOT the owner. This split keeps the safety-critical determinism out of the model layer and out of the prompt. Scrutinize that nothing here silently performs tier arithmetic.
- **R4 — `runTriage` (the full build→provider→extract→parse→repair→fallback loop) ships in `packages/ai`, provider-injected.** The card's accept ("malformed-JSON fake response drives the repair-retry-then-SAFE_FALLBACK path (test)") requires the loop itself to be testable here with the fake provider's scripted-sequence mode. No BullMQ/api/vision wiring (T043). Alternative (ship only the pieces, loop in T043) would make the accept criterion untestable in this task — rejected.
- **R5 — Exactly ONE repair retry, then frozen `SAFE_FALLBACK`.** Matches §5 rule 2 / AI_PROVIDERS §3.4 and CLAUDE §7 rule 5. The returned `result` on double-failure is asserted to be exactly `SAFE_FALLBACK` (deep-equal), never a partially-parsed object.
- **R6 — `runTriage` catches provider errors and also returns `SAFE_FALLBACK` (never throws).** This honors §5 "provider error ⇒ fail upward" inside `packages/ai`. It is cheap and safety-positive, not scope creep; T043 may still wrap it with retry-with-jitter and treat a `SAFE_FALLBACK` status as final. If the checker prefers provider errors to propagate to T043, note the §5 rule ("provider error → safe fallback") justifies containing it here.
- **R7 — The system prompt embeds NO product display name.** It refers to "this guidance app" to avoid both hardcoding the display name (CLAUDE §1a says the display name is rendered only via `APP_DISPLAY_NAME` from `packages/config`, for user-facing surfaces) and adding a `packages/config` dependency to `packages/ai`. The system prompt is model-facing internal copy, not a user-facing surface. Scrutinize that no `Paw Care Right +` literal is hardcoded here.
- **R8 — Exemplars store a prewritten `userText` (not re-serialized from a pet+intake pair).** Keeps exemplars compact and auditable and avoids coupling the exemplar fixtures to `buildUserTurn`; the outputs (the real §5 surface) are fully specified and self-validated via `parseTriage`. Trade-off: exemplar user turns are illustrative, not byte-identical to `buildUserTurn` output — acceptable for few-shot teaching.
- **R9 — `TriagePromptInput` carries no `petId`/photos-at-root and `TriagePetContext` has no PII fields.** Photo count is derived from the intake's `photoPrompt` answers. This structurally guarantees the "no PII beyond pet context" acceptance criterion (an `email` field would be a compile error), backed by a runtime scan test.
