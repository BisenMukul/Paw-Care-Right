# Plan — T059: Mobile care plan setup wizard

## Objective (from card)
Post-pet-creation prompt that shows a suggested care plan (from species/age/region), lets the
owner review it as a list (toggle items on/off, edit each item's start date), and on confirm
creates the corresponding reminders; a second entry point lives on the Care tab. Acceptance is
component tests proving the wizard's suggestions match the resolved template pack for fixture
pets and that every rendered item shows its vet-confirm note.

## Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)
This task renders **care-scheduling content only** — it surfaces `resolveCareTemplate*` items
(titles + the schema-enforced `VET_CONFIRM_SENTENCE` note) and creates the user's own reminders.
It touches **no** AI output, no `<VetDisclaimer/>` / diagnosis surface, no emergency interstitial,
and no medication dosing/drug-recommendation copy. The template content is **read-only** here:
the wizard never edits titles/notes/rrules — it only toggles which items to instantiate and edits
per-item start dates. The vet-confirm note is a hard requirement of the AC and is rendered
verbatim from each item's `note` (which structurally contains `VET_CONFIRM_SENTENCE`, enforced in
`packages/data`). No new user-facing string states or implies medical advice; all copy is
scheduling labels. §5 is not weakened.
- **Do NOT** add any copy that recommends a drug/dose, and **do NOT** relocate the note out of any
  item row — every visible suggestion row must render its note.

## Key decisions (numbered; re-listed under "Risks & decisions")
1. **Suggestions come from a new server endpoint `GET /pets/:petId/reminders/template-suggestions`,
   NOT from the mobile importing `resolveCareTemplate*` directly.** The deciding constraint is
   consistency with instantiation: the wizard's reviewed list must line up 1:1 with what
   `POST …/from-template` will create so the idempotent skip-existing keys (templateKey = item.id)
   and the derived default start dates match exactly. The endpoint reuses the *same* server code
   path the POST uses — `resolveCareTemplateForPet`/`resolveCareTemplate` for the pack,
   `petAgeMonths` + `deriveTemplateStartAt` for default start dates, and the same
   `reminder.findMany({ templateKey: { in } })` for an `alreadyExists` flag. Importing the resolver
   on the client would force the mobile to re-derive `ageMonths` (birthDate vs ageEstimateMonths)
   and the anchor→startAt math (which lives in `apps/api`, not `packages/data`), inviting drift
   between the reviewed dates and the created dates. The client passes the **same** `countryCode`
   (device region) to both GET and POST so both resolve the identical pack. (Offline cost: the
   Care-tab entry can't fetch when offline — handled by the standard offline screen state; the
   post-creation entry is always online because the pet was just created over the network.)
2. **The toggle + date-edit capability is an additive extension to the existing `from-template`
   POST: an optional `selections?: { templateKey; startAt? }[]`.** The current endpoint instantiates
   the *entire* resolved pack with server-derived dates (verified in `reminders.service.ts`); it
   supports neither a subset nor per-item date overrides. When `selections` is present, only the
   listed templateKeys are instantiated and a provided `startAt` overrides the derived start; when
   absent, behaviour is byte-for-byte unchanged (all items, derived dates). Skip-existing /
   idempotency semantics are preserved unchanged. This is the minimum additive change; the
   alternative (calling `POST /pets/:petId/reminders` per item) was rejected because that endpoint
   sets no `templateKey`, so re-confirming would duplicate reminders (no idempotency).
3. **Date editing uses an in-house day stepper — no new dependency.** `apps/mobile/package.json`
   ships no date picker; mirroring T058's in-house time picker, each item row exposes −1w/−1d/+1d/+1w
   controls that shift an ISO date. Minimal, deterministic, testable.
4. **The wizard is the post-creation prompt.** `add-pet/done.tsx` currently `router.replace`s to
   pet home after a successful create; it will instead replace to the wizard (forwarding the
   `localPhoto` handoff), and the wizard's Confirm **and** Skip both `router.replace` onward to
   `/pets/[id]` (forwarding `localPhoto`). This keeps the existing UX handoff intact while making
   the wizard skippable ("Not now"). The Care-tab entry routes to the same wizard for the active pet.

## Files to create/modify (exhaustive — executor may touch NOTHING else)

### Create — shared types
- `packages/types/src/care-plan.ts` — Zod single-source-of-truth for the new shapes (uses
  `reminderTypeSchema` + `speciesSchema`; `lifeStage`/`group` are `z.string()` to avoid a
  `types → data` cycle, since the life-stage/protocol-group enums live in `packages/data`):
  - `careTemplateSuggestionItemSchema` = `{ templateKey: string.min(1); title: string.min(1);
    note: string.min(1); reminderType: reminderTypeSchema; defaultStartAt: z.iso.datetime().nullable();
    emphasis: z.boolean(); alreadyExists: z.boolean() }`.
  - `careTemplateSuggestionsSchema` = `{ species: speciesSchema; lifeStage: z.string(); group:
    z.string(); items: careTemplateSuggestionItemSchema.array() }`.
  - `templateSelectionSchema` = `{ templateKey: z.string().min(1); startAt: z.iso.datetime().optional() }`.
  - `instantiateFromTemplateInputSchema` = `{ timezone: z.string().min(1); group: z.string().optional();
    countryCode: z.string().optional(); selections: templateSelectionSchema.array().optional() }`.
  - `instantiateFromTemplateResultSchema` = `{ created: z.object({ id: z.string(), templateKey:
    z.string().optional(), title: z.string() }).array(); skipped: z.number().int() }` (narrow read
    projection of the API's `InstantiateTemplateResponse`, which is a structural superset).
  - Export inferred types `CareTemplateSuggestionItem`, `CareTemplateSuggestions`, `TemplateSelection`,
    `InstantiateFromTemplateInput`, `InstantiateFromTemplateResult`.
- `packages/types/src/care-plan.spec.ts` — parses a valid suggestions payload; rejects an item
  missing `note`; accepts `defaultStartAt: null`; accepts a selection with/without `startAt`;
  rejects a selection with empty `templateKey`; accepts input with `selections` omitted.

## Endpoint & contract specs the executor must match
- **GET `/v1/pets/:petId/reminders/template-suggestions?group=&countryCode=`** → `200
  CareTemplateSuggestions`; `401` unauth; `404` pet not in caller's household. Read-only (no writes).
  `defaultStartAt` is `null` for an unanchorable PET_AGE item; `alreadyExists` reflects existing
  reminders whose `templateKey` equals the item id.
- **POST `/v1/pets/:petId/reminders/from-template`** body extended with optional `selections`:
  present ⇒ only listed templateKeys instantiated, per-item `startAt` overrides the derived start;
  absent ⇒ unchanged. Response shape unchanged (`{ created, skipped }`). Idempotent replay: a repeat
  with the same selections creates nothing (all skip-existing) and increments `skipped`.
- `RemindersService.templateSuggestions(householdId, petId, { group?, countryCode? }):
  Promise<CareTemplateSuggestions>`.
- Reuse unchanged: `resolveCareTemplate`/`resolveCareTemplateForPet`/`resolveLifeStage` +
  `VET_CONFIRM_SENTENCE` from `@pawcareright/data`; `petAgeMonths`/`deriveTemplateStartAt` from
  `./template-anchors`; `getDeviceRegionCode` from mobile `src/checks/region.ts`; `apiClient` from
  mobile `src/api/client`; `useActivePetStore` from `src/pets/active-pet-store`.

## Risks & decisions (numbered — for the checker to scrutinize)
1. **Server-sourced suggestions (decision 1).** Chosen for exact consistency with instantiation
   (same pack ids + same derived default dates ⇒ skip-existing lines up). Risk: an offline Care-tab
   entry can't load suggestions — mitigated by the standard offline screen state; the post-creation
   entry is always online. Checker: confirm `templateSuggestions` and `instantiateFromTemplate`
   resolve the pack through the identical branch logic and that the wizard sends the **same**
   `countryCode` to GET and POST.
2. **Additive `selections` on `from-template` (decision 2).** The main API bet. Checker: confirm the
   `selections`-absent path is behaviourally unchanged (all items, derived dates, same skip-existing
   and `$transaction` gating), that idempotency (skip-existing on `templateKey`) is preserved with
   selections, and that a per-item `startAt` override is validated (`@IsDateString`) and still passes
   through `computeNextFireAt` (a rule with no future occurrence is skipped, not created).
3. **Re-pointing `add-pet/done.tsx` to the wizard (decision 4).** Touches the T024 flow + its test.
   Checker: confirm the create/idempotency logic in `done.tsx` is otherwise untouched, `localPhoto`
   is still forwarded through to pet home, and `add-pet-done.test.tsx` asserts the new target.
4. **Default enable/date policy.** Rows are seeded enabled only when not already existing and the
   default date is anchorable; unanchorable rows (`defaultStartAt === null`) default to today and are
   off until the user opts in — the client always sends an explicit `startAt` per selected row, so no
   selected row is dropped server-side as unanchorable. Checker: confirm an unanchorable item can be
   enabled+dated and is then created (not silently skipped), and that an already-existing item is not
   pre-enabled (avoids a pointless no-op create that skip-existing would drop anyway).
5. **New shared shapes as `packages/types` Zod (not service-local).** `lifeStage`/`group` are typed
   as `string` to avoid a `types → data` import cycle (the enums live in `packages/data`). Checker:
   confirm no new `packages/types → packages/data` import was introduced and that the api service's
   return type is the inferred `CareTemplateSuggestions`.
6. **Care-tab entry uses the active pet.** Minimal (no pet-picker built). Checker: confirm the empty
   state when `activePetId` is null and that the CTA routes to `/care-plan/[petId]` for the active pet.
