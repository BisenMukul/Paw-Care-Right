## Interfaces / shapes

### schema.prisma (target shapes â€” ids are UUID to match `brandedId` = `z.string().uuid()` in packages/types)
```
enum Role { OWNER MEMBER }

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  appleSub  String?  @unique
  googleSub String?  @unique
  locale    String
  region    String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  ownedHouseholds Household[]  @relation("HouseholdOwner")
  memberships     Membership[]
  devices         Device[]
}

model Household {
  id        String   @id @default(uuid())
  name      String
  ownerId   String
  owner     User     @relation("HouseholdOwner", fields: [ownerId], references: [id], onDelete: Restrict)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  memberships Membership[]
  @@index([ownerId])
}

model Membership {
  id          String   @id @default(uuid())
  userId      String
  householdId String
  role        Role     @default(MEMBER)
  user        User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  household   Household @relation(fields: [householdId], references: [id], onDelete: Cascade)
  createdAt   DateTime @default(now())
  @@unique([userId, householdId])
  @@index([userId])
  @@index([householdId])
}

model Device {
  id            String   @id @default(uuid())
  userId        String
  expoPushToken String   @unique
  platform      String
  lastSeenAt    DateTime @default(now())
  createdAt     DateTime @default(now())
  user          User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@index([userId])
}
```
Notes: `appleSub`/`googleSub` nullable + `@unique` â†’ Postgres treats multiple NULLs as distinct, so many null rows are allowed (correct for passwordless-or-social). `platform` kept `String` (T016 owns any platform enum; do not invent one now). Membership uses a surrogate `id` PLUS the composite unique â€” do not make the composite the PK (T016 upserts, later relations reference membership by id cleanly).

## Risks / explicit design decisions (for CHECKER scrutiny)
- **R1 (CENTRAL â€” CI without editing T009 workflow):** The api jest suite self-applies the committed migrations via a `globalSetup` that runs `prisma migrate deploy`. This is the crux: CI's `build` job runs `pnpm test` against a fresh empty `postgres:16` with NO migrate step, and `.github/workflows/ci.yml` is OUT OF SCOPE. `migrate deploy` applies only pending migrations â†’ full apply on CI's empty DB, no-op locally when already migrated. Chose `migrate deploy` (real committed migrations, authoritative) over `db push` (would bypass migration history the card wants). Contingency: if ts-jest cannot transform a `.ts` globalSetup, use a `.js`/`.cjs` globalSetup wrapper that shells out â€” no new tsconfig, no workflow edit. **If it turns out CI genuinely cannot be made green without a workflow step, STOP and flag as a sanctioned cross-task edit â€” do not silently edit ci.yml.**
- **R2 (shadow DB):** `prisma migrate dev` needs a shadow database and auto-creates/drops one; this requires the `pawcareright` role to `CREATE DATABASE`. The default postgres superuser-ish role in the docker image typically can. If the executor hits a shadow-DB permission error, add a `shadowDatabaseUrl` to the datasource + a matching env var â€” but that would touch env.schema/env config (out of the listed files), so flag it first rather than expanding scope silently. Note: `migrate deploy` (used in tests/CI) needs NO shadow DB, so CI is unaffected regardless.
- **R3 (onDelete decisions):** `Membership.user`/`Membership.household` â†’ `Cascade` (deleting a user/household removes its memberships). `Device.user` â†’ `Cascade`. `Household.owner` â†’ `Restrict` (can't delete a user who still owns a household; forces explicit ownership transfer later). These are minimal, reversible defaults; checker should confirm they match intended lifecycle. No cascade was invented beyond join/child rows.
- **R4 (seed runner dep):** `prisma db seed` needs a TS runner for `seed.ts`. The api currently has NO `ts-node`/`tsx`. Decision: add `tsx` as a devDependency (fast, ESM/CJS-agnostic, actively maintained) OR reuse ts-jest's runtime is not viable for a standalone script. Justify the added dep in `loop/journal.md` per CLAUDE Â§2.7. This adds `tsx` to `apps/api/package.json` devDeps and updates `pnpm-lock.yaml`. Alternative avoiding a dep: precompile is over-engineering â€” prefer `tsx`. `prisma.seed` config â†’ `"tsx prisma/seed.ts"`.
- **R5 (no-console in seed):** Shared eslint base sets `no-console: "error"` and `apps/api/eslint.config.mjs` merely re-exports it. PREFERRED: seed writes no logs at all (idempotency is proven by exit code + row counts, not stdout) â†’ eslint.config.mjs stays untouched, file list minimal. If progress logging is deemed necessary, add a scoped flat-config override `{ files: ["prisma/seed.ts"], rules: { "no-console": "off" } }` to `apps/api/eslint.config.mjs` (the base's own NOTE sanctions per-file overrides). Do NOT relax the rule globally.
- **R6 (seed idempotency mechanism):** Household/Membership have no natural business-unique key, so idempotent upsert relies on fixed UUID constants for the dev household + membership (upsert by `id`), plus upsert-by-email for the user and upsert-by-composite for the membership. Constants are obviously-dev fixtures. Checker: confirm re-run produces zero new rows.
- **R7 (id strategy):** `@default(uuid())` chosen (not cuid) to stay consistent with `packages/types` `brandedId` = `z.string().uuid()`. If checker prefers cuid, note it would break future branded-id validation â€” uuid is the safer default.
- **R8 (typecheck/lint scope of seed.ts):** `apps/api/tsconfig.json` includes only `src`+`test`, so `prisma/seed.ts` is NOT typechecked by `pnpm typecheck` and NOT built â€” acceptable for a standalone dev script run by `tsx`. It IS linted by `eslint .`, so it must satisfy strict lint rules (no unused, import order, no-console). Do NOT add `prisma` to tsconfig include (would pull the script into the Nest build; out of scope).
