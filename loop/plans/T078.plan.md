# Plan — T078: Analytics event layer

## Objective (from card)
Add a PostHog analytics layer: a shared, typed event map (SPEC §8) in a packages-level
wrapper, plus mobile + api emitters keyed on user/household ids, with a consent flag respected
(default on) and an off switch pre-wired in mobile settings for T091. Prove the map is
type-tested (unknown event = compile error) and that the three key flows emit
(`first_check_completed`, `paywall_view`, `trial_start` via webhook).

## Safety statement (PRODUCT_SPEC §5 / CLAUDE.md §7)
- This task adds telemetry ONLY. It touches no §5 safety surface: it adds/edits no
  `<VetDisclaimer/>`, no emergency interstitial, no poison/vet hotline copy, and no
  dosing/drug/`diagnos*` text. No AI prompt or safety string is modified.
- **No health-content in payloads.** Event properties carry ids + coarse enums ONLY. Explicitly
  FORBIDDEN as event props: `intakeJson`/`freeText`/`photoKeys`/photos, `summary`,
  `possibleCauses`, `redFlagsToWatch`, `homeCare`, `doNot`, `vetQuestions`, email, breed, pet
  name, or any free text. `first_check_completed` carries the coarse 5-value `urgency` tier enum
  (see decision 7) because SPEC §8 explicitly tracks "% checks ending in EMERGENCY tiers"; that
  is a category, never symptom text. Server emitters log ids-only on failure (CLAUDE §6).
- NOT a SAFETY-ESCALATION: implementable without weakening any safety surface.

## Key decisions (each a design choice — scrutinize)
1. **New workspace package `packages/analytics`** holds the shared typed event map + a
   transport-agnostic emitter, consumed by BOTH `apps/api` and `apps/mobile` (and web later).
   Chosen over folding into `packages/types`/`packages/config` so the fetch transport + emitter
   logic lives in one place with one owner. Scaffolding mirrors `packages/types` exactly
   (pure-TS package built by tsup, ts-jest tests).
2. **Zero new EXTERNAL npm deps — a fetch-based PostHog HTTP-capture transport**, not
   `posthog-node`/`posthog-react-native`. PostHog remains the vendor (LOCKED stack, CLAUDE §3);
   we call its documented `POST {host}/capture/` endpoint via global `fetch` (available in Node
   20+ and React Native/Expo Go — no native module). Only internal `workspace:*` deps are added
   (`@pawcareright/analytics`, `@pawcareright/types`) — no §2r7 external-dep justification burden.
3. **Typed map = a TS interface + generic `capture`** so an unknown event name is a compile
   error with no zod/tsd runtime dep. Verified by a ts-jest `@ts-expect-error` spec (decision
   also satisfies `pnpm typecheck`).
4. **Consent (default ON) gated client-side on the mobile emitter** via an MMKV-persisted
   `consent-store` (`enabled: boolean`, default `true`), with a functional off switch pre-wired in
   Settings. The two SERVER events (`first_check_completed`, `trial_start`) are id-only behavioral
   analytics and are NOT gated by a per-user server consent column in v1 — server-side consent
   propagation/deletion is explicitly T091's scope (the card says "pre-wired for T091"). Adding a
   consent column + migration now is out-of-scope creep. Documented under Risks.
5. **Stub-safe everywhere:** an empty PostHog key ⇒ the transport is a no-op (mirrors the
   RevenueCat `stub_*_key` pattern). No key in tests/CI ⇒ no network, no throw.
6. **Instrumentation points:**
   - `first_check_completed` — SERVER, in `check-runner.processor.ts` AFTER the terminal-status
     persist (both the DONE/FALLBACK success path and the infra-fallback path), emitted ONLY when
     this is the user's first terminal check. "First" is detected by
     `symptomCheck.count({ where: { createdById, status: { in: TERMINAL_CHECK_STATUSES } } }) === 1`.
     `createdById` is the leading column of the existing `@@unique([createdById, idempotencyKey])`
     btree, so this count is index-supported — NO new index/migration.
   - `paywall_view` — MOBILE, a mount-once `useEffect` in `app/paywall.tsx`, using the already-read
     `source` param (`"onboarding" | "settings"`, defaulting undefined → `"onboarding"`).
   - `trial_start` — SERVER, in `rc-webhook.service.ts` after the committed, non-duplicate
     transaction, when the handled event is an `INITIAL_PURCHASE` with `period_type === "TRIAL"`.
7. **Trial detection = RC `period_type === "TRIAL"`** (added to `rcWebhookEventSchema`, additive;
   `.passthrough()` already preserved it, we now type it). Double-emit across RC retries is
   prevented by the existing `ProcessedWebhookEvent` dedupe (a replay short-circuits before the
   emit decision) PLUS emit-after-commit. Emission is fire-and-forget and never blocks the 200 ack.

## Risks & the decisions the planner made (so checker can scrutinize)
- **R1 (decision 4) — server events not consent-gated in v1.** `first_check_completed` and
  `trial_start` emit regardless of the mobile consent toggle, because wiring a per-user server
  consent column is a migration the card scopes to T091 ("off switch pre-wired for T091"). The
  mobile emitter (the surface the user toggles) IS gated now. If the checker reads the card as
  requiring server-side honoring, this is the point to flag — but v1 emits only ids, and SPEC
  §5.6 classes analytics as behavioral, not medical.
- **R2 (decision 7) — `urgency` in `first_check_completed`.** Included because SPEC §8 tracks
  EMERGENCY-tier rate; it is a coarse 5-value enum, never free text. Could be dropped if the
  checker deems any tier signal too sensitive; the AC does not require it.
- **R3 — consent race at startup.** The MMKV consent store falls back to memory (default
  `enabled: true`) before hydration; worst case an early event emits under the default-on policy,
  which is the intended default. No event is lost due to consent flapping.
- **R4 — event loss on crash / offline.** Fire-and-forget capture with no local queue means an
  event can be lost if the process crashes mid-send or the device is offline. Acceptable for v1
  analytics (not a billing/entitlement path); documented, not engineered around.
- **R5 — webhook double-emit.** Prevented by the existing `ProcessedWebhookEvent` dedupe (replay
  short-circuits before the emit decision) + emit-after-commit. A crash between commit and emit
  loses the event (R4), never double-counts.
- **R6 — "first" count under concurrency.** Two of a user's checks completing simultaneously could
  both observe `count === 1` and emit twice. Extremely rare (a user rarely has two checks racing
  to terminal at once, and the free tier caps checks); acceptable v1 imprecision, documented.
- **R7 — Expo Go / native module.** The fetch transport uses global `fetch` only (no native
  module), so it works in Expo Go; MMKV consent store already has the memory fallback pattern
  (`createSafeStorage`) so it never crashes at load.
- **R8 — new package registration.** `pnpm-workspace.yaml` globs `packages/*`, so
  `packages/analytics` is auto-included; `pnpm i` must run so `@pawcareright/analytics` resolves
  in api + mobile. Turbo pipelines are task-based (no per-package registration needed).
- **R9 — bundling PostHog key in mobile.** `EXPO_PUBLIC_POSTHOG_KEY` ships in the client bundle;
  that is correct for a PostHog project (write-only ingest key), mirrors the RC public keys already
  in `extra`. No server secret is exposed.
