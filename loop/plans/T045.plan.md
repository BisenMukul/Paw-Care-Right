# Plan — T045: Mobile: dynamic intake form (L)

## Objective (from card)
Build the schema-driven symptom-intake form at `app/check/[category].tsx` (the T044 route target): a `QuestionRenderer` that renders any typed question def (single/multi/scale/duration/photoPrompt) purely from `packages/types` data, a stepped flow with a progress indicator, an optional free-text step, and a review screen that assembles + validates a `CompletedIntake` and hands it off to T047. Adding a new question to a category must require ZERO mobile code change.

## Safety statement
This task is symptom-**collection** UI only. It renders NO AI output, NO triage result, NO "diagnosis"/"possible causes", NO medication/dosing content, and NO emergency interstitial — those surfaces are T047 (submit/poll), T048 (result + `<VetDisclaimer/>`), T049 (emergency). Therefore `<VetDisclaimer/>` is **not** required on this screen (there is no AI result to disclaim), and no §5 Safety-Policy surface is created, altered, or weakened.

Every question/option/scale-label rendered comes from `INTAKE_CATEGORIES` in `packages/types` — already §7-reviewed schema DATA, not component copy. The only new user-facing prose is neutral form chrome (progress, next/back, review, invalid-category) plus the **free-text prompt copy**, which is the T031 keyword channel and §7-relevant. It is pinned in this plan and must be neutral, non-leading, and contain no medical/diagnosis/dosing language:
- `intake.freeText.title` = `"Anything else you'd like to add?"`
- `intake.freeText.placeholder` = `"Add any other details in your own words."`
- `intake.freeText.optional` = `"This step is optional."`
No mobile string may introduce medical advice, drug names, dosing, or the word "diagnos*". This is NOT a SAFETY-ESCALATION — the card is plannable and touches no protected safety surface.

## Key decisions
1. **Renderer is a data-driven switch.** One `QuestionRenderer` switches on `question.type` and delegates to a presentational sub-component per type (`SingleQuestion`, `MultiQuestion`, `ScaleQuestion`, `DurationQuestion`, `PhotoPromptQuestion`). Every sub-component is driven entirely by the `QuestionDef` prop + current `Answer` + `onChange` — ZERO per-category / per-question-id logic. New question *instances* of the 5 existing types render with no code change (the AC). A brand-new question *type* (a 6th) would legitimately need a new case — out of scope, not the AC.
2. **Flow = one question per screen step** (mirrors the T024 add-pet wizard UX: progress + Back/Next chrome), driven by an internal `stepIndex`. Steps are `[...categoryDef.questions, freeText, review]`; `total = questions.length + 2`. This satisfies the card's explicit "progress indicator" + "free-text step" + "review screen". Because the number of steps is dynamic per category, steps are internal state, NOT per-file routes (unlike add-pet's static 5 files).
3. **Ephemeral local state, no store, no persistence.** Answers live in `useState<Record<string, Answer>>` inside `IntakeForm`; free-text in `useState<string>`. Justification vs T024's MMKV draft: a symptom check is a single short sitting (spec §I flow), not a resumable multi-session onboarding, so persistence is unnecessary complexity. No new store file, no MMKV.
4. **Handoff to T047 via an `onSubmit` callback prop.** `IntakeForm` is a presentational component: `{ categoryDef, onExit, onSubmit }`. The review step assembles the candidate, validates with `parseIntake` (fail-closed), and calls `onSubmit(validatedIntake: CompletedIntake)` only when validation passes. The route screen `[category].tsx` supplies a **stub** `onSubmit` for T045 (`// TODO(T047): submit + navigate to red-flag/polling`); T047 replaces the stub with the real mutation. `petId` stays in the route screen's closure (it is a T042 path param, NOT part of `CompletedIntake`). This callback seam makes the whole flow unit-testable without a router and gives T047 a clean, single insertion point.
5. **`photoPrompt` renders a T046 stub.** T045 shows the prompt + a neutral "photos come next" placeholder and a Skip; it emits NO answer (all real photoPrompt questions are `required:false`, so required-gating is satisfied). No camera, no picker, no upload, no `photoKeys` — honest and minimal; T046 replaces the stub and starts emitting `{ photoKeys }`.
6. **Injectable `categoryDef` prop is the mutation-resistance test seam** (mirrors T044's injectable `categories` prop on `CategoryGrid`). In production the route screen passes `getCategoryDef(param)`; tests inject a synthetic `CategoryDef` to prove the renderer/flow are data-driven.

## Renderer specs per question type (exact interaction + emitted answer shape)

`QuestionRenderer` props:
```ts
interface QuestionRendererProps {
  question: QuestionDef;              // from @pawcareright/types
  answer: Answer | undefined;         // current answer for this question (or none)
  onChange: (answer: Answer | undefined) => void;
}
```
It renders the question `prompt` (testID `intake-question-prompt`) and, if present, `helpText`, then switches on `question.type` to the matching sub-component (explicit case per type; a defensive `return null` fallback for a malformed/unknown type — never throw). Each sub-component receives the narrowed question + `answer` + `onChange` and OWNS building the correctly-typed `Answer`.

1. **single** → radio list. One `Pressable` per `question.options`; testID `intake-option-${question.id}-${option.value}`; selected style when `answer?.type==="single" && answer.value===option.value`; label `option.label`. Press → `onChange({ type:"single", questionId: question.id, value: option.value })`. Never emits `undefined` (radio has no deselect). **Answered** ⇔ an answer exists.
2. **multi** → checkbox list. One `Pressable` per option toggling membership in a `values` array; testID `intake-option-${question.id}-${option.value}`; checked style when value ∈ `answer.values`. Enforce `maxSelections`: when `answer.values.length === question.maxSelections`, unselected options are visually disabled and taps are ignored (selected options still deselect). If `maxSelections` defined, render a hint `strings.intake.maxSelectionsHint(n)` (testID `intake-multi-hint-${question.id}`). Emit `onChange({ type:"multi", questionId, values })`; if the toggle empties `values`, emit `onChange(undefined)` (multiAnswerSchema requires ≥1). **Answered** ⇔ answer exists (≥1 value).
3. **scale** → a row of numeric buttons `min..max` inclusive (count `max-min+1`). Button testID `intake-scale-${question.id}-${n}`; selected style when `answer?.value===n`. Render `minLabel` under the min end and `maxLabel` under the max end (testIDs `intake-scale-minlabel-${id}` / `intake-scale-maxlabel-${id}`). Press → `onChange({ type:"scale", questionId, value: n })`.
4. **duration** → a numeric `TextInput` (testID `intake-duration-value-${question.id}`, `keyboardType="number-pad"`) + a unit picker: one `Pressable` per `question.units` (testID `intake-duration-unit-${question.id}-${unit}`, selected style for the chosen unit). This sub-component keeps minimal local state for the in-progress raw text + selected unit, seeded from `answer`. On any change: parse the text with `Number.parseFloat`; if it is a finite value `> 0` AND a unit is selected → `onChange({ type:"duration", questionId, value, unit })`; otherwise `onChange(undefined)` (durationAnswerSchema requires `value` positive + a valid `unit`). No unit is preselected. **Answered** ⇔ positive value + unit both present.
5. **photoPrompt** → **T046 stub**. Renders the prompt + a neutral placeholder `strings.intake.photoStub` inside a container testID `intake-photo-stub-${question.id}`. Emits NO answer (does not call `onChange`). No camera/picker/upload. (All real photoPrompt questions are `required:false`, so this never blocks Next. A `required:true` photoPrompt is out of scope until T046 — see Risk 5.)

## Flow & state spec (`IntakeForm`)

Props: `{ categoryDef: CategoryDef; onExit: () => void; onSubmit: (intake: CompletedIntake) => void }`.
State: `stepIndex: number` (0-based), `answers: Record<string, Answer>`, `freeText: string`.
Derived: `questions = categoryDef.questions`; `total = questions.length + 2`; step `k`:
- `0 .. questions.length-1` → question step (renders `QuestionRenderer` for `questions[k]`).
- `questions.length` → **free-text step**.
- `questions.length + 1` → **review step**.

Wrap content in `KeyboardAvoidingView` + `SafeAreaView` (like `WizardScaffold`).

- **Progress indicator** (testID `intake-progress`): `strings.intake.stepOf(stepIndex+1, total)`.
- **Back** (testID `intake-back`): if `stepIndex>0` → `setStepIndex(stepIndex-1)`; if `stepIndex===0` → `onExit()`.
- **Next / required-gating** (`PrimaryButton`, testID `intake-next`): on a question step, `disabled` when `questions[k].required` and that question is unanswered (`answers[q.id]` absent). Optional questions never block. Free-text step: Next always enabled. Press → `setStepIndex(stepIndex+1)`.
- **Answer wiring**: `QuestionRenderer` `answer={answers[q.id]}`, `onChange={a => setAnswers(prev => a ? {...prev, [q.id]: a} : omit(prev, q.id))}`.
- **Free-text step**: `TextInput` (testID `intake-freetext-input`, `multiline`, `maxLength={2000}`), title `strings.intake.freeText.title`, placeholder `.placeholder`, note `.optional`; value `freeText`, `onChangeText={setFreeText}`. Optional — Next enabled regardless.
- **Review step**: heading `strings.intake.review.title`. For each `q` in `questions` order with an answer, render a row (testID `intake-review-row-${q.id}`) showing `q.prompt` + `describeAnswer(q, answers[q.id])` (human labels, never raw values) + an Edit `Pressable` (testID `intake-review-edit-${q.id}`) → `setStepIndex(index of q)`. If `freeText.trim()` non-empty, render a free-text row (testID `intake-review-freetext`) + edit (testID `intake-review-edit-freetext` → `setStepIndex(questions.length)`).
  - **Validation (fail-closed)**: compute `candidate = buildIntakeCandidate(categoryDef, answers, freeText)` then `result = parseIntake(candidate)`. If `!result.ok`: render `strings.intake.validationError` (testID `intake-validation-error`) and render the submit button **disabled**. If `result.ok`: no error, submit enabled.
  - **Submit** (`PrimaryButton`, testID `intake-submit`, label `strings.intake.submit`): `disabled={!result.ok}`; `onPress` → `onSubmit(result.value)` (only reachable when ok). Never assembles or submits an invalid intake.

`buildIntakeCandidate(categoryDef, answers, freeText)` → `{ category: categoryDef.id, answers: Answer[], freeText?: string }`: iterate `categoryDef.questions` in order, push `answers[q.id]` where present (preserving schema order), and include `freeText` only when `freeText.trim().length > 0` (trimmed). Returns a plain object (typed `unknown`/candidate) that `parseIntake` validates. Note `parseIntake`/`completedIntakeSchema` validate against the shared global `INTAKE_CATEGORIES` by design (server-authoritative) — see Risk 6.

## Risks & the decisions the planner made (for the checker to scrutinize)
1. **R1 (stepped flow vs scrollable).** Chose one-question-per-step with a progress indicator, matching the T024 wizard UX and the card's explicit "progress indicator". A scrollable single-form would also satisfy "renderer" but not naturally the progress requirement. Steps are internal state (dynamic count), not per-file routes.
2. **R2 (ephemeral local state, no store/persistence).** A check is a single short sitting, so `useState` in `IntakeForm` is used instead of a persisted zustand/MMKV draft (unlike add-pet). If the app is killed mid-check the answers are lost — deemed acceptable for a short flow and honest to the spec; revisit only if a resumable-check requirement appears.
3. **R3 (`onSubmit`/`onExit` callback seam).** `IntakeForm` is presentational and receives its submit + exit behavior as props; the route screen supplies a T045 stub `onSubmit`. This keeps the flow router-free and unit-testable and gives T047 a single insertion point. The stub does nothing in T045 (button is enabled on a valid real intake but its handler is a documented `// TODO(T047)` seam) — acceptable because T045→T047 land sequentially before any release; flagged so the checker confirms the button is correctly gated (disabled on invalid) rather than falsely "working".
4. **R4 (`photoPrompt` stub emits no answer).** Since all real photoPrompt questions are `required:false`, a no-answer stub never blocks the flow and keeps T045 honest (no fake camera UI). T046 replaces the stub and begins emitting `photoKeys`.
5. **R5 (`required:true` photoPrompt unsupported).** No real category marks a photoPrompt required; if one were added before T046 it could not be completed in this flow (Next would block). Out of scope, documented; the mutation-resistance synthetic def keeps its photoPrompt optional (matching every real one).
6. **R6 (`parseIntake` validates against the GLOBAL schema).** `parseIntake`/`completedIntakeSchema` look up `getCategoryDef` from the module-level `INTAKE_CATEGORIES` (server-authoritative, by design). Consequence: (a) in production the injected `categoryDef` is always a slice of that global, so validation passes and the real "new question added to packages/types" scenario submits with zero mobile change; (b) the mutation-resistance test injects a global-divergent synthetic def, so it proves **rendering + collection** (the mobile half of the AC) and asserts review summaries — it deliberately does NOT assert `onSubmit`, because parseIntake (correctly) rejects a def that isn't in the shared schema. The same divergence is reused as the fail-closed test (submit disabled + validation error). The packages/types half of "new question accepted" is covered by that package's own `parseIntake` tests.
7. **R7 (`DurationQuestion` local state).** The duration sub-renderer keeps minimal local state for in-progress raw text + selected unit (seeded from `answer`) because a half-typed value has no valid `Answer` representation; it emits a full `Answer` only when value>0 and a unit are both present, else `onChange(undefined)`. Slightly less "pure controlled" than the other sub-renderers; contained to this one type and unit-tested.
8. **R8 (injectable `categoryDef` prop).** `IntakeForm` takes `categoryDef` as a prop (not read internally from params) purely to enable the injected-schema test and keep the component presentational — mirroring T044's `CategoryGrid.categories` prop. The route screen supplies the real def.
