# Plan — T067: Timeline UI

## Objective (from card)
Build the per-pet Timeline tab: an infinite, virtualized list of health-log entries with kind icons/colors, month section headers, kind filter chips, check entries that deep-link to the result detail, and empty/offline states. Prove (via render counts) that appending a page does not re-render existing rows.

## Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)
This task renders existing timeline records only — it introduces **no AI output and no new safety surface**.
- No "diagnosis"/interpretive copy anywhere; rows show recorded facts (date, kind, verbatim note/reason/weight).
- MED_GIVEN rows show `medNameAsEntered` / `medDoseAsEntered` **as entered** (record framing, §7 rule 2) — never a suggested drug or dose; if both are absent, a neutral "medication given" record label is shown. No dose is ever computed, formatted, or recommended.
- CHECK_REF rows only *link back* to the already-§5-reviewed `/check/result/[checkId]` screen; they render no result content inline. That destination already handles a missing/deleted check by failing upward (error state → "contact a vet"), so a dangling link degrades safely.
- All user-facing copy lives in `strings.ts`; emoji glyphs are presentation-only (mirrors `category-icons.ts`).
This task does **not** weaken any disclaimer, emergency-escalation, or dosing-prohibition surface. Not a SAFETY-ESCALATION.

## Key decisions (the design choices the checker must scrutinize)
1. **`SectionList`, not `FlatList`.** Month headers are true section boundaries; `SectionList` (VirtualizedList-backed) gives native virtualization **and** per-section headers without injecting header sentinel items into a flat data array (which would complicate `keyExtractor` and stable memoization). This is the app's first `SectionList`/`FlatList` (everything else uses `ScrollView`); it is justified by the AC's explicit virtualization requirement.
2. **Render-count test technique = spy on a once-per-render call the real memo'd row makes.** `TimelineRow` is `React.memo`'d and calls `getKindDisplay(item.kind)` **exactly once** in its body. The test `jest.spyOn`s `getKindDisplay`, renders page 1 (item refs A,B), clears the spy, then re-renders with page 1+2 where A,B are the **same object references** plus new C,D **in the same month**. Assert the spy is called exactly **once per newly-appended row** (2), never for A,B — proving existing rows did not re-render. This exercises the *real* memoization (no row mocking). It fails if (a) rows aren't memo'd, (b) `onPress` isn't a stable `useCallback`, or (c) page-1 item refs change — exactly the regressions the AC guards.
3. **Tab already exists → replace the placeholder screen; do not touch `_layout.tsx`.** `app/(tabs)/timeline.tsx` and the `timeline` tab entry already exist; this task fills in the screen only.
4. **Pet source = active-pet store (no in-tab pet switcher, no "All").** The API is strictly pet-scoped (no all-pets timeline endpoint), so `PetFilterChips`'s `null`="All" semantics don't apply. The timeline reads `useActivePetStore().activePetId` (same precedent as the home tab / care-plan link). If it's `null`, show a "no pet" message. The tab's "filter chips" are therefore **kind** filter chips (filter the active pet's timeline by entry kind).
5. **Query key = `healthTimelineKeys.pet(petId)` prefix + a `kind` segment.** Correct caching requires the selected kind to be part of the key (so switching kind re-queries and each kind caches independently). Add `healthTimelineKeys.list(petId, kind)` built **on top of** `pet(petId)` (`[...pet(petId), kind ?? "all"]`). T066's mutations invalidate `healthTimelineKeys.pet(petId)`, which — with React Query's default `exact: false` prefix match — invalidates **every** kind variant, so invalidations still compose. This honors the T066 binding ("use exactly `healthTimelineKeys.pet(petId)`") as the composing prefix while keeping filtering correct.
6. **Dangling CHECK_REF handling = navigate when checkId is well-formed, disable when it isn't.** If `checkRefValueSchema` parses a uuid `checkId`, the row is pressable and navigates to `/check/result/[checkId]` (that screen already handles a deleted check gracefully — navigate-and-404). If the value is malformed/absent, the row renders non-pressable (no crash). Only CHECK_REF rows are interactive in this task; other kinds are non-navigating (their detail screens are out of scope here).
7. **Month bucketing + header label use the DEVICE-LOCAL clock (T062), not UTC.** `groupTimelineByMonth` buckets by `getFullYear()`/`getMonth()` (like `care.tsx`'s `localDayKey`), **not** `toISOString().slice(...)` (which `formatCheckDate` uses and which would misbucket entries near a month boundary in non-UTC zones). Header title is a locale-free `"YYYY-MM"` string (consistent with the app's numeric-date convention; avoids month-name i18n, deferred to T110).

## Risks & edge cases (checker: scrutinize decisions 1–7)
- **Append re-render (AC core):** relies on (a) `React.memo` on `TimelineRow`, (b) a `useCallback`-stable `onPressCheck`, (c) TanStack returning stable page-1 object refs when a page is appended. The render-count test fails loudly if any breaks. If a future change makes item refs unstable, this test is the regression tripwire.
- **tz month boundaries:** buckets/labels are device-local (decision 7); an entry at 23:30 local on the last day of a month must not leak into the next UTC month. Covered by the tz-boundary unit case.
- **Duplicate keys from two sources:** the merged feed contains persisted `HealthLog` rows and the MED_GIVEN projection over `ReminderEvent`; ids come from different tables. `keyExtractor` uses the composite `${kind}:${id}` so a (theoretically) colliding raw id across tables can't produce a duplicate React key.
- **Pull-to-refresh vs cursor reset:** `useInfiniteQuery.refetch()` refetches **all** loaded pages from page 1 using stored `pageParams` (it does not naively re-request with a stale cursor), so a refresh is safe — but it also re-fetches every loaded page. To keep scope minimal and avoid extra virtualization/test surface, pull-to-refresh is **deferred**; recovery is via the error/offline retry button (`refetch`). Documented so the checker sees the trade-off was deliberate.
- **Filter-key composition:** decision 5 appends a `kind` segment; if the executor instead made the key exactly `pet(petId)` with `kind` only in the queryFn, kind changes would serve stale cached data and chips wouldn't re-query — the "kind chip re-queries" test guards this.
- **Dangling CHECK_REF:** a well-formed-but-deleted `checkId` still navigates; the result screen's existing error state is the safe landing (no crash, fails upward). A malformed value disables the row. Both covered.
- **Empty vs no-pet vs offline precedence:** ordering is no-pet → loading → offline(no data) → error(no data) → list(+empty). Mis-ordering (e.g. showing empty while offline with no cache) is a UX bug; the offline test with `data: undefined` guards it.
