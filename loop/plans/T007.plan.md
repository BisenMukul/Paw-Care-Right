## Risks & the design decisions the planner made (checker: scrutinize each)
- **R1 — Next.js 15 + React 19.** Current stable major, App Router (mandated by CLAUDE.md §3 "Web = Next.js App Router"). React 19 is Next 15's paired peer. Actively maintained. If the installed Next minor pairs with React 18 instead, executor should match Next's peer requirement (still `^15`) and flag — not downgrade Next.
- **R2 — Tailwind v3.4.x with a classic JS preset (cross-task architectural decision).** Chosen over Tailwind v4 **specifically so the SAME `@pawcareright/config/tailwind` preset serves BOTH web (T007) and mobile/NativeWind (T008)**. NativeWind is compatible with Tailwind v3.4, not v4's CSS-first `@theme` model. A single classic JS preset (`{ theme: { extend } }`) is `require`/`import`-able by both Next's PostCSS pipeline and Metro+NativeWind. Consumed via `presets: [preset]` in each app's `tailwind.config`. This is the ONE decision with cross-task blast radius — if the executor finds Tailwind v3.4 unavailable or the preset unshareable, STOP and block rather than adopt v4 (which would strand T008).
- **R3 — Preset shipped as a plain `.mjs`, not through tsup.** `packages/config/tailwind-preset.mjs` mirrors the existing `eslint.config.mjs`/`prettier.config.mjs` pattern: Tailwind loads config via its own loader (jiti), so the preset must stay a directly-loadable JS file, NOT a tsup-built dist artifact. Hence the new `"./tailwind"` export points straight at the source `.mjs`. No tsup/exports change to `.`/`./env`.
- **R4 — `SITE_URL` = placeholder `https://pawcareright.app` in `src/site.ts`.** CLAUDE.md §1a lists `pawcareright.app` as the placeholder web domain until T102. Centralized in one constant so sitemap/robots share it and T102 changes one line. Not user-facing prose, so no `APP_DISPLAY_NAME` involvement. Checker: confirm no other file hardcodes the domain.
- **R5 — `lighthouserc.json` lives at `apps/web/` (not repo root).** Co-located with the app it measures; T095 (the budget-enforcement task) consumes it from the web app dir. If T095's later plan expects repo-root, that is a T095 concern; present-and-valid here satisfies AC2. Flagged so the checker can weigh the location.
- **R6 — eslint composes `@next/eslint-plugin-next` on the shared flat config.** Gives App-Router lint rules on top of the shared base. Risk: the Next plugin's flat-config export must be ESLint-9-flat compatible. Fallback if it is not: `apps/web/eslint.config.mjs` re-exports ONLY `@pawcareright/config/eslint` (one line, like `apps/api`), dropping `@next/eslint-plugin-next` from deps. Either way lint must pass; executor picks the working variant and the plan permits dropping the Next plugin dep if unused.
- **R7 — `next-env.d.ts` is generated, not authored.** Next writes it on first build; it must be gitignored (via `apps/web/.gitignore`) and is NOT in the authored file list. The tsconfig `include` references it so typecheck works post-build. Executor must not hand-write or commit it.

## Amendment A1 (orchestrator, post-executor) — apps/web/turbo.json (typecheck depends on own build)
**Why:** `apps/web/tsconfig.json` `include`s `.next/types/**/*.ts` (Next.js auto-manages/expects this). Those files are generated by `next build`. Root `turbo.json` `typecheck` only `dependsOn ["^build"]` (upstream), NOT the app's own build. Running gates SEQUENTIALLY (`pnpm typecheck && pnpm lint && pnpm test && pnpm build`, as the M0 gate does) is green — a lone typecheck sees an empty `.next/types` glob and passes. But a SINGLE concurrent turbo invocation (`turbo run typecheck lint test build`, a form CI/T009 may use) races web:typecheck against web:build and reads half-written `.next/types/*.ts` → `TS6053 File not found`.
**Fix (localized, idiomatic Turborepo-for-Next; new file inside apps/web only):** `apps/web/turbo.json` with `{ "extends": ["//"], "tasks": { "typecheck": { "dependsOn": ["^build", "build"] } } }`. Now web's typecheck always runs after its own build, so `.next/types` exists. Verified: `pnpm turbo run typecheck lint test build --force` (concurrent, cache cleared) → 16/16 tasks successful. Sequential gate remains green. No other file changed; no root turbo.json edit.
