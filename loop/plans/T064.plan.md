# T064 · Timeline API — Execution Plan (PLANNER / Fable-tier)

Deps: T063 (HealthLog model + `packages/types/src/health-log.ts`), T015 (household guard).
Size: M · API only. No mobile, no api-client hooks (those ride T065/T066).

---

## 1. Objective

Deliver the health-timeline read/write API over the T063 `HealthLog` model:

1. **Cursor list** `GET /pets/:petId/logs` — merged kinds, newest first, optional single-`kind` filter, keyset pagination.
2. **Create** `POST /pets/:petId/logs` — record a new log for the user-creatable kinds only (`WEIGHT`, `MEAL`, `NOTE`, `VET_VISIT`), `valueJson` validated by the T063 Zod schemas, `photoKeys` (max 6).
3. **Weight series** `GET /pets/:petId/weight-series` — ascending `{t, grams}[]`, optional `from`/`to`, **downsampled to ≤200 points** when the raw series exceeds 200.

All routes reuse the T015 `@HouseholdFromMembership()` posture exactly as `ChecksController`/`RemindersController` do. Response shapes are **service-local interfaces** (mirroring `CheckResponse`/`ReminderResponse` — the codebase does NOT put these in `packages/types`; only Zod validators live there, and the T063 ones already exist).

## 2. Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)

This task touches **no** Safety Policy surface: it does not alter disclaimers, emergency/red-flag escalation, AI output, or dosing logic. It renders **records as entered** only. Concrete guardrails carried into the plan:

- No user-facing advice/interpretation copy is produced anywhere in the API; the timeline returns stored records verbatim.
- `MED_GIVEN` and `VET_VISIT` stay record-only. The create endpoint **rejects** `MED_GIVEN` and `CHECK_REF` (system/projection kinds) so no client can inject a fabricated "med given" record; the med-name/dose fields remain the free-text "as entered" strings already defined in T063 — nothing here suggests, requires, or computes a drug or dose.
- Fail-closed validation: `valueJson` is parsed through `parseHealthLogValue` (T063); an unknown/invalid payload is a `400`, never a silent coerce.

Not a SAFETY-ESCALATION block — proceed.

## 3. Key decisions (numbered; scrutinize under Risks)

**D1 — Keyset cursor on `(occurredAt DESC, sourceRank ASC, id DESC)`.**
Offset pagination is unstable under inserts (the failing case the Accept criteria test). Cursor is an **opaque base64url(JSON)** of `{ o: occurredAt ISO, s: sourceRank (0|1), i: id }`, decoded + Zod-validated; a malformed cursor is a `400`. `sourceRank`: HealthLog rows = `0`, projected MED_GIVEN events = `1` (fixed, arbitrary) — this makes ties across the two sources deterministic. Page size mirrors `paginationQuerySchema` (limit 1..100, default 20).

**D2 — MED_GIVEN = READ-TIME PROJECTION of `ReminderEvent`, NOT a written-through HealthLog row.** (This is the card's central decision.)
Chosen over write-through despite write-through's simpler single-table cursor, because:
1. It honors the **already-committed T061/T063 contract** — `health-log.ts` states verbatim that MED_GIVEN is "a read-time PROJECTION of `ReminderEvent(type=MEDICATION, status=DONE, completedAt)` … not a persisted HealthLog row." Reversing a shipped cross-task contract is exactly what the CHECKER flags.
2. **Single source of truth / §7 record accuracy.** `ReminderEvent` is the truth of whether a dose was given. `RemindersService.snoozeOccurrence` sets a previously-`DONE` event back to `SNOOZED` and **clears `completedAt`**. Under projection the timeline entry then correctly disappears automatically. A write-through row would persist and falsely show the med as "given" — a §7 record-integrity defect — and fixing it would force a *second* coupled hook into `snoozeOccurrence`.
3. **Minimal blast radius (§2).** T064 touches **no** `reminders.service.ts` write path and needs **no backfill decision** (all existing completed med events project for free).
Accepted cost: a two-source merged keyset (only in the unfiltered case) + one additive `ReminderEvent` index (D6). Both are specified and tested below.

**D3 — CHECK_REF is an ordinary persisted HealthLog kind that no source creates yet.**
Nothing in the current codebase writes `CHECK_REF` rows (auto-creating them on check completion is a T041-worker edit — out of scope, §2). Decision: the create endpoint **rejects** `CHECK_REF`; the list read path **includes** any `CHECK_REF` rows if present (rank-0 HealthLog rows, standard keyset + kind-filter). This keeps the merge to exactly **two** sources (HealthLog table + MED_GIVEN projection), never three. No SymptomCheck union.

**D4 — Public-creatable kinds = `{WEIGHT, MEAL, NOTE, VET_VISIT}`; `{MED_GIVEN, CHECK_REF}` are system/projection kinds and are `400` on create.** The `kind` **filter** on the list accepts all six.

**D5 — Single create route `POST /pets/:petId/logs` with a per-kind-validated body, NOT four sibling routes.**
The card phrase "create endpoints per kind" is satisfied by one route whose body is validated **per kind** (each kind's own T063 `strictObject`), rather than by endpoint proliferation. This matches ARCHITECTURE §4's exact route (`POST /pets/:id/logs`, singular) and the entrenched codebase pattern (`ChecksController`/`RemindersController`: thin DTO + Zod parse in the service). Four routes would be over-engineering (§2) for no Accept-criteria benefit. DTO carries `kind` (`@IsIn(publicKinds)`), `occurredAt` (ISO), `photoKeys` (array, ≤6), `value` (object); the service calls `parseHealthLogValue(kind, value)` and maps a failure to `400`. `photoKeys` are validated via `healthLogPhotoKeysSchema` (max 6) — same trust model as checks (T023 keys are trusted post-upload; no per-key ownership re-check).

**D6 — Weight-series downsampling = uniform contiguous-bucket mean, pure function.**
Simplest-correct for a line chart (card's steer over LTTB). `downsampleWeightSeries(points, max=200)`: if `points.length <= 200` return a copy unchanged; else split the time-ascending array into ≤200 equal-count contiguous buckets and emit one point per bucket = `{ t: mean(t), grams: round(mean(grams)) }`. Deterministic, side-effect-free, unit-testable.

## 4. Indexes (new query patterns — §6 requires an index in the same migration)

- `HealthLog(petId, kind, occurredAt DESC)` — powers the **single-kind filtered** list page. (Existing `HealthLog(petId, occurredAt DESC)` from T063 already powers the unfiltered page — keep it.)
- `ReminderEvent(reminderId, status, completedAt DESC)` — powers the MED_GIVEN projection query (`status=DONE` events for a pet's med reminders, newest by completion). Additive index only; no column/behavior change to the T060-owned model.

### List merge algorithm (executor contract — implement exactly)

Unified item: `{ id, kind, occurredAt, value, photoKeys }` with an internal `sourceRank` (HealthLog=0, MED_GIVEN=1). Total order = `occurredAt DESC, sourceRank ASC, id DESC`.

- **Single-kind filter, kind ≠ MED_GIVEN:** query `HealthLog` where `{ petId, kind }`, keyset predicate below (s=0), `orderBy [{occurredAt:desc},{id:desc}]`, `take limit+1`.
- **Filter = MED_GIVEN:** query the projection only (below), keyset with s=1.
- **No filter:** run BOTH source queries with `take limit+1`, merge-sort by total order, take `limit+1` to compute `hasMore`, slice `limit`.
- **Projection query:** `reminderEvent.findMany({ where: { status: "DONE", completedAt: { not: null }, reminder: { petId, type: "MEDICATION" } }, orderBy:[{completedAt:"desc"},{id:"desc"}], take: limit+1, ...keyset })`. Map each → `{ id: event.id, kind: "MED_GIVEN", occurredAt: event.completedAt, value: { reminderEventId: event.id, medNameAsEntered: reminder.medNameAsEntered ?? undefined, medDoseAsEntered: reminder.medDoseAsEntered ?? undefined }, photoKeys: [] }` (pull the med fields via `include: { reminder: true }`).
- **"Comes-after-cursor" per source** given cursor `(o, s, i)`:
  - HealthLog (rank 0): `occurredAt < o` **OR** (`occurredAt == o` **AND** `s == 0` **AND** `id < i`). (When `s == 1`, no equal-`occurredAt` HealthLog row qualifies.)
  - MED_GIVEN (rank 1): `completedAt < o` **OR** (`completedAt == o` **AND** `s == 0`) **OR** (`completedAt == o` **AND** `s == 1` **AND** `id < i`).
  Build these as conditional Prisma `where.OR` clauses; branch on `cursor.s`.
- `nextCursor` = from the last **returned** item: `encodeCursor({ o: occurredAt, s: sourceRank, i: id })`; `null` when `!hasMore`.

## 7. Test plan (named cases)

**`downsample-weight-series.spec.ts`**
- `passthrough_at_or_below_threshold` — length 0, 1, 199, **exactly 200** → returned unchanged (identity of values, ascending order preserved).
- `downsamples_above_threshold` — **exactly 201** and 1000 → result length ≤ 200, strictly time-ascending, first bucket includes earliest point, last includes latest.
- `bucket_mean_values` — known input → asserts each output `grams` = rounded mean of its bucket, `t` = mean of bucket times.
- `pure_no_mutation` — input array + elements untouched after call.

**`timeline-cursor.spec.ts`**
- `roundtrip` — encode→decode identity for `{o,s,i}` incl. both `s` values.
- `rejects_garbage` — non-base64 / non-JSON / missing field → typed error.

**`health-logs.service.spec.ts`** (mocked prisma/pets)
- `create_rejects_system_kinds` — `MED_GIVEN`/`CHECK_REF` → `400`, no DB write.
- `create_rejects_invalid_value` — bad `value` for kind (e.g. WEIGHT negative) → `400`.
- `create_persists_valid` — VET_VISIT with photoKeys≤6 persists; >6 keys → `400`.
- `list_merges_two_sources_newest_first` — interleaved HealthLog + DONE med events → correct total order.
- `list_filter_med_given_projects_only` — filter=MED_GIVEN returns only projected events, no table rows.
- `list_filter_single_kind_table_only` — filter=WEIGHT ignores med events.
- `list_projection_excludes_non_done_and_null_completed` — SNOOZED / PENDING / null-`completedAt` events never appear (the §7 snooze-after-complete guarantee).
- `weight_series_ascending_and_downsampled` — >200 WEIGHT logs → `sampled:true`, ≤200 ascending points; ≤200 → `sampled:false`.

**`health-logs.e2e-spec.ts`**
- `create/list/weight-series happy` per route.
- `auth 401` (no token) + `404` (pet in another household — no leak) per route.
- `validation 400` (bad body / bad cursor / bad limit) per route.
- `pagination_stability_under_insert` — page 1 (limit N) → insert a new log **older** than page-1 cursor → page 2 via `nextCursor` returns no duplicate and no skipped row; inserting a **newer** row does not shift/duplicate an already-returned page.
- `pagination_equal_occurredAt` — several logs sharing one `occurredAt` paginate cleanly via the id tiebreak with no dup/skip.
- `filter_by_each_kind` — each kind filter returns only that kind.
- `empty_and_terminal_cursor` — pet with no logs → `{items:[], nextCursor:null}`; cursor of last page → `[]`.

## 9. Risks / edge cases (CHECKER: scrutinize)

- **R1 (D2 reversal risk):** Choosing projection over write-through is a live cross-task decision; if a later task needs MED_GIVEN rows physically present (e.g. offline export), projection must be revisited. Justified now by the T063 contract + §7 snooze integrity.
- **R2 (merged-cursor correctness):** Equal `occurredAt` **across** the two sources is the sharp edge — the `sourceRank` tiebreak + the per-source "comes-after" predicate table in §6 must be implemented verbatim; covered by `pagination_equal_occurredAt` and `list_merges_two_sources_newest_first`. Fetching `limit+1` from **each** source before merge is required for correctness (do not fetch `limit` total split across sources).
- **R3 (dangling backlinks):** `MED_GIVEN.reminderEventId` / `CHECK_REF.checkId` have no FK. Projection starts *from* `ReminderEvent`, so MED_GIVEN can't dangle (a deleted med reminder cascades its events away). Persisted `CHECK_REF` rows are rendered with their stored `checkId` **without** a join to SymptomCheck — a missing referent renders the record as-is, never a 500.
- **R4 (downsample boundary):** `> 200` means `length >= 201` downsamples; `length == 200` passes through. Off-by-one guarded by explicit 200/201 cases.
- **R5 (empty / terminal pages):** `nextCursor` is `null` only when `!hasMore`; a cursor past the end yields `[]`, never an error.
- **R6 (ReminderEvent index ownership):** the added `ReminderEvent` index touches a T060-owned model — additive index only (no column/behavior change); flagged so the checker confirms it's not scope creep into reminders logic.
- **R7 (N+1 in projection):** must `include: { reminder: true }` on the med query to read `medNameAsEntered`/`medDoseAsEntered` in one round-trip — no per-event reminder fetch.
