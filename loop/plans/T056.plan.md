# Plan — T056: Scheduler worker

## Objective (from card)
Minute-tick scan of active Reminders that are due (`nextFireAt <= now`): materialize the due `ReminderEvent` (idempotent), enqueue a push job, mark it `SENT`, then advance `Reminder.nextFireAt` to the next occurrence via `computeNextFireAt` in the stored timezone. On boot, backfill occurrences missed while the worker was down (`< 24h` old) as `MISSED` **without** pushing, skip anything older than 24h, and advance past them so a restart never floods pushes.

## Safety statement
Pure scheduling / notification-plumbing infrastructure. It renders **no** user-facing medical copy, produces **no** AI output, and touches **none** of the Safety Policy surfaces (disclaimers, emergency escalation, dosing) in `PRODUCT_SPEC §5` — so §5 is not weakened; this task is safe to plan. Reminder `title`, `type`, and `medNameAsEntered` are user content: the worker must **log ids only (reminderId / eventId / ISO dueAt / counts), never titles or med names** (mirrors the check-runner's "no intake-shaped logs" rule). The med tracker never *suggests* anything; this worker only fires whatever a reminder already stores, so it introduces no dosing/administration guidance.

## Key decisions (each restated & numbered under "Risks & decisions")
1. **Tick mechanics = BullMQ Job Scheduler**, not `@nestjs/schedule` (not installed → adding it violates CLAUDE §2 rule 7) and not `setInterval`. `bullmq@5.80.1` (verified in `pnpm-lock.yaml`) exposes `queue.upsertJobScheduler(id, { pattern }, { name })`; register a once-per-minute scheduler `"* * * * *"` on a dedicated `pawcareright-reminders` queue at boot. Registration is idempotent by scheduler id.
2. **Scan/materialization model = scan Reminders by the `nextFireAt` cursor, materialize events at fire time.** T055 keeps future agenda *virtual* — nothing pre-materializes `ReminderEvent`s — so the tick queries `Reminder WHERE active = true AND nextFireAt <= now` (matches ARCHITECTURE §6's "recompute nextFireAt" intent) and creates the event on the fly. This requires a new index (see Files: `schema.prisma` + migration) per CLAUDE §6.
3. **State flow: create the event directly in `SENT`** (with `sentAt = now`) at fire time — the card says "enqueue push jobs → mark SENT" as one tick step; `SENT` means "enqueued for delivery", and T057 (the sender) owns actual delivery/receipts. Backfilled events are created in `MISSED` (no `sentAt`, no push). Existing event at `(reminderId, dueAt)` is never overwritten.
4. **Backfill bounds by a 24h window, reusing the proven helpers.** Missed occurrences are enumerated only within `[max(nextFireAt, now-24h), now]` via `occurrencesBetween` (self-capped at 500), marked `MISSED`; occurrences older than 24h are never enumerated (skipped); `nextFireAt` is then advanced to the first occurrence strictly after `now` via `computeNextFireAt`. A backfill therefore creates at most a bounded number of `MISSED` rows and zero pushes.

## Tick semantics (`tick(now)`, "fire" mode — step by step)
1. `reminders = prisma.reminder.findMany({ where: { active: true, nextFireAt: { lte: now } }, select: { id, rrule, timezone, startAt, nextFireAt } })`.
2. For each reminder, wrapped in its own `try/catch` (failure isolation — log `reminder_fire_failed` with `reminderId` only and `continue`; never rethrow to fail the whole tick):
   a. `dueAt = reminder.nextFireAt` (guaranteed `<= now`). `parsed = parseRRule(reminder.rrule)`; if `!parsed.ok` → log warn + `continue` (defensive; rrule validated on write).
   b. **Create event (idempotent):** `try { event = prisma.reminderEvent.create({ data: { reminderId, dueAt, status: "SENT", sentAt: now } }); created = true }` — on `isUniqueConstraintViolation(e)` (P2002 on `@@unique([reminderId,dueAt])`) → `created = false` (a duplicate/redelivered tick; do NOT push again). Any other error → rethrow into the per-reminder catch.
   c. **Enqueue push (only when `created`), post-create, best-effort:** `pushQueue.add(PUSH_JOB_NAME, { reminderEventId: event.id }, { jobId: event.id, removeOnComplete: true, removeOnFail: false })`. `jobId = event.id` makes the enqueue idempotent. A queue failure is logged (`push_enqueue_failed`, ids only) and does not undo the event (mirrors `scheduleFollowUp`). Because sub-daily cadence does not exist (rrule min granularity is DAILY), at most one occurrence per reminder is ever due per tick — no catch-up loop, no flood.
   d. **Advance the cursor (guarded, idempotent):** `next = computeNextFireAt(parsed.value, reminder.startAt, new Date(dueAt.getTime()+1), reminder.timezone)`. If `next !== null` → `prisma.reminder.updateMany({ where: { id, nextFireAt: dueAt }, data: { nextFireAt: next } })`. If `next === null` (series exhausted) → `prisma.reminder.updateMany({ where: { id, nextFireAt: dueAt }, data: { active: false } })` and log `reminder_series_exhausted`. The `nextFireAt: dueAt` guard makes a concurrent/duplicate tick's advance a 0-row no-op (idempotent progress; also lets a crashed-before-advance prior tick recover on the next tick).
3. Log `reminder_tick_done` with counts (`due`, `fired`, `skippedDuplicate`, `failed`) — ids/counts only, never titles.

## Backfill semantics (`backfill(now)`, runs once at `onApplicationBootstrap`, before the scheduler is registered)
1. `reminders = prisma.reminder.findMany({ where: { active: true, nextFireAt: { lte: now } }, select: { id, rrule, timezone, startAt, nextFireAt } })` (same index).
2. For each reminder (own `try/catch`):
   a. `parsed = parseRRule(...)`; skip on parse failure.
   b. `from = new Date(Math.max(reminder.nextFireAt.getTime(), now.getTime() - BACKFILL_WINDOW_MS))`, `to = now`.
   c. `missed = occurrencesBetween(parsed.value, reminder.startAt, reminder.timezone, from, to)`. For each `o` in `missed`: `try { prisma.reminderEvent.create({ data: { reminderId, dueAt: o, status: "MISSED" } }) } catch (e) { if !isUniqueConstraintViolation(e) throw }` — **no `sentAt`, no push**. Occurrences older than `now-24h` are never enumerated (never created) — that is the "skip stale flood" behavior.
   d. **Advance to the future:** `next = computeNextFireAt(parsed.value, reminder.startAt, new Date(now.getTime()+1), reminder.timezone)`; `updateMany({ where: { id }, data: next ? { nextFireAt: next } : { active: false } })` (unconditional here — boot is single-threaded; `nextFireAt` is guaranteed to move strictly past `now`, so the very next minute-tick will not re-fire the backfilled window).
3. Log `reminder_backfill_done` with counts (`behind`, `missedCreated`, `deactivated`) — ids/counts only.

Rationale: after boot-backfill every behind reminder has a strictly-future `nextFireAt`, so the minute-tick only ever fires genuinely-fresh occurrences (SENT + push, ≤1 min late). No push is emitted for anything that came due while the worker was down.

## Push queue contract (for T057)
- Queue: `pawcareright-push` (`PUSH_QUEUE`). Job name: `reminder-push`. Payload: `{ reminderEventId: string }` (minimal — T057 loads the event/reminder/devices and performs its own per-user collapse-with-count).
- Enqueue opts set by this producer: `{ jobId: reminderEventId, removeOnComplete: true, removeOnFail: false }` (`removeOnFail:false` = DLQ retention). **Retry/backoff, batching (100/chunk), receipt polling, DeviceNotRegistered pruning, and per-user collapse are T057's concern** — not set here (mirrors `followups.contract.ts` leaving the consumer inert).

## Tests to write (map to acceptance criteria)
All in `reminder-scheduler.service.spec.ts`; mock prisma = `{ reminder: { findMany, updateMany }, reminderEvent: { create } }`, mock pushQueue = `{ add: jest.fn() }`; construct `new ReminderSchedulerService(prisma, pushQueue, ...)`.
- **AC "fires within tick"** → `it("fires a due reminder within the tick: creates a SENT event, enqueues a push, advances nextFireAt")` — given one active reminder `nextFireAt = now`, `tick(now)` asserts `reminderEvent.create` called with `{ status: "SENT", sentAt: now, dueAt: now, reminderId }`, `pushQueue.add` called once with `{ reminderEventId: <id> }` and `opts.jobId === <id>`, and `reminder.updateMany` advances `nextFireAt` to the next occurrence with the `{ nextFireAt: <dueAt> }` guard.
- **AC "DST transition correctness"** → `it("advances nextFireAt across a Europe/Paris spring-forward, holding 09:00 local")` — daily 09:00 `Europe/Paris`, `startAt` anchored, `nextFireAt = 2026-03-28T08:00:00Z`; `tick(2026-03-28T08:00:30Z)` asserts the advanced `nextFireAt = 2026-03-29T07:00:00Z` (09:00 CEST, i.e. one hour earlier in UTC than the pre-DST occurrence) — proving `computeNextFireAt` is called with the stored `startAt`/`timezone`.
- **AC "restart backfill behavior" (a)** → `it("backfill marks a <24h missed occurrence MISSED without enqueuing a push")` — reminder `nextFireAt = now-2h`; `backfill(now)` asserts `reminderEvent.create` called with `status: "MISSED"` and **no** `sentAt`, `pushQueue.add` NOT called, `nextFireAt` advanced to a future instant.
- **AC "restart backfill behavior" (b)** → `it("backfill skips occurrences older than 24h and does not flood pushes")` — daily reminder `nextFireAt = now-48h`; `backfill(now)` asserts only the occurrences within the last 24h are created `MISSED` (bounded count, not ~48h worth), `pushQueue.add` never called, and `nextFireAt` advanced strictly past `now`.
- **AC "idempotency by event id"** → `it("is idempotent on a double-tick in the same minute: one event, one push, one advance")` — run `tick(now)` twice; second `reminderEvent.create` mock rejects with a P2002-shaped error → assert `pushQueue.add` called exactly once (jobId equal both notional calls) and the second advance `updateMany` (guarded on `nextFireAt: dueAt`) is a no-op path (create-rejected → no re-push).
- **Series exhaustion (supports advance branch + coverage)** → `it("deactivates a reminder whose series is exhausted instead of leaving it due")` — rule with `COUNT=1` already fired; `computeNextFireAt` returns null → assert `updateMany` sets `active: false`, not a past `nextFireAt`.
- **Per-reminder failure isolation** → `it("isolates a failing reminder so other due reminders still fire")` — two due reminders, first's `reminderEvent.create` rejects with a non-P2002 infra error; assert the second reminder still creates its event and enqueues its push, and `tick` resolves (error swallowed + logged, not rethrown).

## Risks & decisions (numbered — for the checker to scrutinize)
1. **Tick mechanics = BullMQ `upsertJobScheduler` every minute** (decision 1). Alternative `@nestjs/schedule` rejected as a new dependency (CLAUDE §2 rule 7). The BullMQ scheduler registration is thin/untested-for-timing; all testable logic lives in `tick`/`backfill`, which take an injectable `now: Date` so the fake-clock ACs bypass BullMQ entirely.
2. **New index `@@index([active, nextFireAt])` + migration added to T053's schema** (decision 2, CLAUDE §6). Without it the per-minute scan is a full table scan. Risk: touching another task's migration surface — mitigated by index-only change (no client-type change, no data migration). Checker: confirm the migration SQL matches the schema edit and index name.
3. **Event created directly as `SENT` at enqueue (not a PENDING→SENT handshake)** (decision 3), per the card's literal "enqueue push jobs → mark SENT". Consequence: `SENT` means "handed to the push pipeline", not "delivered". If the best-effort post-create `pushQueue.add` fails, the event stays `SENT` with no delivery and `nextFireAt` still advances (at-most-once push). Judged acceptable — reminders are non-emergency; the future T062 agenda-vs-fired consistency job can reconcile — and it preserves the "never double-push" idempotency guarantee. Checker: verify no push path can double-send on a duplicate tick.
4. **Backfill window fixed at 24h; >24h occurrences skipped (never materialized), exhausted series deactivate the reminder** (decision 4 + advance branch). `active:false` on exhaustion reuses the existing pause flag rather than adding a schema field; agenda/CRUD already filter `active:true`, so an exhausted reminder disappears cleanly, but this conflates "user-paused" and "series-complete". Also: a reminder created *after* boot with a past first occurrence fires once with a push (not backfilled) — accepted as a single, non-flooding prompt.
