# Plan — T052: Phase 4 E2E + load sanity (final Phase-4 task before M4)

## Objective (from card)
Add an API-level full-lifecycle E2E (check submitted over HTTP → real worker →
terminal, with a FAKE provider; text + photo variants) that runs in CI, and commit a
load-sanity script (50 concurrent sustained submissions, queue drains, no rules-layer
event-loop blocking) with thresholds + a journal-results template. Fold in the carried
hygiene items from the T043/T049/T050 reviews.

## Safety statement
This task adds an end-to-end assertion of the PRODUCT_SPEC §5 pipeline — it does not add,
weaken, or reword any safety surface. The lifecycle E2E strengthens §5 by proving, over a
real HTTP + queue + worker round-trip, that: (a) the deterministic rules floor is applied
by the worker (`applyPostRules`), (b) a terminal check surfaces a schema-valid
`TriageResult` and never a raw/guessed one, and (c) the FALLBACK fail-upward path persists
a floored safe result. NO new user-facing copy, NO disclaimer/emergency/dosing text, NO
prompt changes. Not a SAFETY-ESCALATION. `packages/ai` production code is NOT touched
(so `test:ai-evals` is not required by this task).

## Key decisions (state explicitly for the checker)
1. **Load tool = hand-rolled zero-dependency Node/tsx script** (NOT k6, NOT autocannon).
   The card permits "k6 (or autocannon)"; both add cost. k6 is an external binary needing a
   CI setup action (heavy, and we are NOT running load in CI anyway). autocannon is a new npm
   dependency requiring a §2-rule-7 justification. 50 concurrent sustained POSTs is trivially
   scriptable with `node:http`/`fetch` + `perf_hooks`, reusing already-present deps
   (`bullmq`, `ioredis`, `@prisma/client`, `@nestjs/jwt`, `tsx`) → **zero new dependencies**,
   best satisfies §2 rule 7. Run via `tsx`, committed under `apps/api/test/load/`.
2. **"no event-loop blocking >50ms (rules layer)" is measured honestly as API response-latency
   distribution under load, NOT a client-side event-loop probe.** The rules layer runs INLINE
   inside `POST /checks` (`evaluateRedFlags`); if it blocked the API event loop, it would show
   as p99 latency spikes under 50 concurrent load. The script asserts p95/p99 thresholds +
   0% error rate. The sub-ms rules cost itself is already benched in T031 (0.05ms/eval); we do
   NOT duplicate a synthetic event-loop-delay harness (measuring the *client* loop would be
   wrong). This interpretation is recorded here for scrutiny.
3. **CI scope split:** the lifecycle E2E is a `*.e2e-spec.ts` → it is picked up by the existing
   `pnpm --filter api test` job automatically ("E2E in CI" satisfied, no workflow edit). The
   load script is **committed + runnable locally/manually** (`pnpm --filter api load:sanity`),
   **NOT** wired into per-push CI (too heavy/slow; it needs a running API + a drained worker).
   Its results are journaled by the orchestrator, not asserted in CI.
4. **Live-worker isolation via a dedicated Redis DB index (decision under scrutiny — see Risk 1).**
   The lifecycle suite is the ONLY suite that runs a live `pawcareright-checks` worker. To stop
   its worker from consuming jobs enqueued by other suites (and vice-versa) under jest's parallel
   file execution, the suite pins its whole app instance's Redis to a dedicated logical DB
   (`redis://localhost:6379/15`) set in `beforeAll` before the TestingModule is built.
5. **Carried hygiene items folded in:**
   - **T043 (MUST):** apply the no-op `CheckRunnerProcessor` override to EVERY other
     AppModule-booting e2e suite (10 files) via a shared `overrideCheckRunner()` factory helper,
     so no suite except the new lifecycle one has a live checks worker. This is the
     checker-recommended robust fix and is a prerequisite for determinism now that a live worker
     exists.
   - **T049 (folded in — trivial, safety-relevant):** add CA/GB/AU/NZ end-to-end resolution
     assertions through `resolveRegionHotline` in `packages/data/src/regions/regions.spec.ts`.
   - **T050 (folded in — the cheap one only):** add a two-page `data.pages` fixture assertion to
     `check-history-screen.test.tsx` proving `pages.flatMap` appends rows across pages.
   - **T050 explicitly NOT folded in:** the `useChecksList` direct hook test (screen-level +
     typecheck already cover it; adding a hook-render harness risks scope) and the UTC
     `formatCheckDate` change (acknowledged T110/i18n territory — behavior change out of scope).

## Load script spec (`apps/api/test/load/checks-load-sanity.ts`)
A standalone tsx script (NOT a jest spec — filename has no `.spec`, so jest's testMatch ignores
it; it must still pass typecheck + lint). Run against a locally-running API.

**Header / operator notes (as a top comment):** requires `docker compose up -d` (pg+redis+minio),
API running with `AI_TEXT_PROVIDER=fake` (so the worker drains deterministically without network),
then `pnpm --filter api load:sanity`. Env: `API_BASE_URL` (default `http://localhost:3000`),
`DATABASE_URL`, `REDIS_URL`, `JWT_SECRET` (same as the API).

**Bootstrap (reuse existing patterns, zero new deps):**
- `new PrismaClient()` → seed one user + household + membership + pet (mirror
  `factories`'s createUser/createHousehold or raw prisma inserts). Mint a token with
  `new JwtService({ secret: <jwtSecret> }).sign({ sub: userId })` (mirror
  `resolveJwtService` fallback; read secret from `new AppConfigService().jwtSecret` or env).

**Concurrency model:**
- Constants: `CONCURRENCY = 50`, `DURATION_MS = 20_000`. Maintain exactly 50 in-flight
  `POST /v1/pets/:petId/checks` (benign, non-red-flag intake, PREMIUM/quota-bypass NOT available
  over HTTP → so use a fresh intake each loop and expect some 402s OR seed a premium entitlement;
  **simplest:** measure latency of the POST regardless of 201/402 — both exercise the inline rules
  layer; treat only 5xx/network as errors). Each worker: loop until deadline, `performance.now()`
  around each request, record the duration, immediately fire the next.
- Aggregate: sort durations, compute count, p50, p95, p99, max, and error count.

**Thresholds (asserted; process exits non-zero on breach) — starting values, founder-tunable:**
- `p95 < 400ms`, `p99 < 1000ms` (a rules-layer event-loop stall would spike p99),
- `errorRate === 0` (no 5xx / socket errors),
- (optional, cheap) re-assert the rules bench stays fast by importing `evaluateRedFlags` and
  timing N iterations `< 1ms/op` — keep only if trivial; the authoritative bench is T031.

**Queue-drain check:**
- After the load window, connect `new Redis(REDIS_URL, { maxRetriesPerRequest: null })` + a
  `Queue(CHECKS_QUEUE, { connection })`; poll `queue.getJobCounts("waiting","active")` every 500ms
  until `waiting + active === 0` or a 60s timeout (fail on timeout). NOTE: exclude `delayed`
  (follow-up jobs are delayed 24h and legitimately never drain) and `completed`.

**Journal-results template:** print a fenced block ready to paste into `loop/journal.md`, e.g.:
```
### T052 load-sanity — <ISO date>
concurrency=50 duration=20s provider=fake
requests=<n> errors=<n> p50=<ms> p95=<ms> p99=<ms> max=<ms>
thresholds: p95<400 p99<1000 errorRate=0 -> PASS|FAIL
checks-queue drained (waiting+active==0) in <s>s -> PASS|FAIL
```
**Handoff:** the planner/executor do NOT run the load script (no running API in the gate). The
executor commits it and reports that it is runnable; the ORCHESTRATOR runs
`pnpm --filter api load:sanity` at the M4 gate and pastes the printed block into
`loop/journal.md`. State this in the executor summary.

**Cleanup:** the script deletes the seeded user/household/pet/checks (FK order, mirror
`cleanupUsers`) and disconnects prisma + redis in a `finally`.

## Risks & decisions (numbered — for the checker to scrutinize)
1. **Live-worker cross-suite job stealing (THE decision).** Under jest parallel file execution the
   lifecycle suite's live `pawcareright-checks` worker could consume jobs enqueued by
   `checks.e2e-spec.ts` (which enqueues real jobs but no-ops its own worker), flipping its
   "QUEUED"/manually-staged rows and making it flaky — and conversely. Mitigation: pin the
   lifecycle app instance's Redis to a dedicated DB (`.../15`) via `process.env.REDIS_URL` set in
   `beforeAll` before `compile()`, giving that whole process an isolated queue namespace; combined
   with step-2's no-op override on all OTHER AppModule suites, the DB-0 queue has no live consumer.
   Assumes Redis at `localhost:6379` (same assumption `global-setup.ts` makes for Postgres) and
   that `AppConfigService` (and the suite's own `ioredis`/`QueueEvents`) read `redisUrl` at
   construction — verified in `app-config.service.ts` (constructor calls `defineEnv`). If
   `defineEnv` memoizes env at first import (rather than per-construction), the DB pin could fail;
   the executor must confirm by reading `packages/config/src/env` — if memoized, set `REDIS_URL`
   in a per-file jest setup instead (still within the new spec's ownership) or STOP and write
   `loop/plans/T052.blocked.md`. Restore `REDIS_URL` in `afterAll`.
2. **Wait mechanism.** Chose `job.waitUntilFinished(queueEvents, 30_000)` (proven in photos.e2e)
   over a hand-rolled GET poll loop for determinism; the subsequent `GET` still exercises the full
   HTTP read-path, honoring the card's "poll GET until terminal" intent.
3. **Photo-variant red-flag hazard.** The chosen photo-bearing category's default answers must NOT
   trigger a red-flag rule (else the check takes the red-flag branch and the assertion focus
   shifts). Mitigation: assert `body.redFlag === undefined` on the 201 as an explicit precondition;
   the executor tunes benign answer values deterministically. (Even a red-flag check still reaches
   DONE via the worker, so this is about test clarity, not correctness.)
4. **Load thresholds are provisional.** p95<400ms / p99<1000ms are starting values on local
   hardware; the founder tunes them at M4. The script is manual/journaled, so a threshold miss does
   not gate CI. Documented in the script header.
5. **Scope breadth.** This task touches 14 files across api/data/mobile. All non-lifecycle edits
   are mechanical (wrap a builder) or test-only additions; no production behavior changes. The
   turn-limit-resilient order (hygiene → lifecycle → load → nits) means a partial completion still
   lands the highest-value pieces first for a single conventional commit.
