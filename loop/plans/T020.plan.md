## Coverage strategy (the core design decision — see Risks)
- `collectCoverageFrom: ["src/**/*.service.ts"]` — scopes the report to **services only**, matching the card ("≥80% services") and CLAUDE §6 ("≥80% coverage on services"). This instruments exactly: `auth.service`, `otp.service`, `refresh-token.service`, `social/social-auth.service`, `devices.service`, `health.service`, `config/app-config.service`, `prisma/prisma.service`, `redis/redis.service`. Guards/verifiers/middleware are intentionally out of the coverage denominator (not "services").
- `coverageThreshold.global`: `statements: 80, lines: 80, functions: 80, branches: 70`. Lines/statements/functions at 80 enforce the AC + CLAUDE §6 literally. Branches at 70 (justified in Risks) because the only sub-80 branch risk is defensive infra guards (`del([])` early-return, health try/catch, `resolveJwtService` fallback) whose alternate arms are safety nets — forcing 80 there would mean contrived tests = over-engineering.
- Because these live in the jest block but are inert without `--coverage`, `pnpm --filter api test` stays fast and `pnpm --filter api test:cov` is the durable, self-failing gate (jest exits non-zero below threshold). Coverage is collected across the WHOLE run (e2e + unit), so services exercised only via e2e (prisma, redis wrappers, health success path) still count.
- Executor MUST run `test:cov` BEFORE adding any conditional gap-filler, read the per-file table, and add the minimum unit test(s) (file 7) only for services that land <80% — never lower the threshold.

## Risks & the design decisions the planner made (for CHECKER scrutiny)
1. **`collectCoverageFrom` scoped to `src/**/*.service.ts` (services only).** Chosen to match the card's exact wording ("≥80% services") and CLAUDE §6, and to keep the denominator honest — guards/verifiers/middleware aren't "services". Alternative (all of `src/**`) rejected: would import a different, laxer/stricter target than the AC states and pull in `main.ts`/DTOs. Checker: confirm the glob and that all 9 services appear in the table.
2. **`coverageThreshold` committed but only enforced under `--coverage` via a separate `test:cov` script; `test` stays `jest`.** Card-recommended split: gives a durable, CI-ready, self-failing gate for the AC report without slowing every `pnpm test`/gate_exec run. Risk: gate_exec (`pnpm test`) will NOT catch a future coverage regression — mitigated because the threshold is committed and any `test:cov`/CI run enforces it; the checker runs `test:cov` for this task.
3. **Branches threshold 70 (vs 80 for lines/statements/functions).** The only plausibly-sub-80 branches are defensive infra guards (`redis.del([])` early return, `health` try/catch, `resolveJwtService` catch) whose alternate arms are unlikely-error safety nets; mandating 80 branches would force contrived error-injection tests = over-engineering (CLAUDE §2 rule 2). Lines/functions/statements stay at the AC-mandated 80.
4. **Anticipated gap-filler = `app-config.service.spec.ts`.** `AppConfigService` is a thin 9-getter service; getters like `port`/`nodeEnv` are only read by `main.ts` (never loaded in tests), so it would otherwise sit below 80% functions/lines and drag the aggregate down. A direct getter test guarantees it. This is a NEW test file (test-only, permitted) — not a `src` change. If `test:cov` shows it already ≥80 via indirect use, executor may skip it (minimality) — but the plan authorizes it so the AC is reachable deterministically.
5. **Refactor is strictly mechanical across devices/guards/auth-social** — the three suites that manually create users/households/memberships + mint app JWTs (the heaviest duplicators). auth.e2e uses OTP endpoints (no manual provisioning) so it is a weak fit and left untouched to avoid risk; 3 suites already satisfy the AC. Checker: diff must show identical `it(...)` titles/counts and unchanged `expect`s.
6. **`cleanupUsers` also deletes `device` + `refreshToken` rows** (superset of any single suite's cleanup) so one helper serves all three FK-ordered teardowns; safe because it filters by the created `userIds` only.
7. **Coverage provider left at jest default (ts-jest instrumented).** `jest --coverage` with the existing ts-jest transform works out of the box. Risk: if instrumentation misbehaves, executor may add `"coverageProvider": "v8"` to the jest block (still within file 5, no new dep) — flagged so the checker doesn't treat that one-line addition as scope creep.
