# Plan — T049: Emergency interstitial (full-screen red-flag takeover)

## Objective (from card)
Full-screen takeover shown for red-flag hits: **what was detected** + **"go now" guidance**
(per the rule's `emergencyPayloadKey`), a **region-aware poison hotline**, a **find-an-emergency-vet**
action, and an explicit **acknowledge** that is the ONLY way past (cannot be swiped/back'd away
accidentally). Route is the T047/T048 contract route `/check/emergency/[checkId]`. Ship the two
missing datasets it needs — `packages/data/regions` (hotlines) and `packages/data` emergency-payload
content (the 22 payloadKeys) — plus mobile region resolution. **Accept:** component tests incl. region
hotline resolution + fallback when region unknown; renders before any AI content (flow test).

## Safety statement — THIS IS A PRODUCT_SPEC §5 RULE-4 SURFACE (the emergency interstitial itself)
Planned, not blocked: T049 *builds and strengthens* the §5 rule-4 surface; it does not weaken any
Safety Policy surface. Invariants this plan enforces (each has a test):
- **Emergency-before-AI (§5 rule 4 / §7 rule 4).** The interstitial reads ONLY the check's
  `redFlag.payloadKey` and renders static emergency content + hotline. It renders **no** AI `result`
  fields, even when `data.result` is populated (flow test asserts absence of AI content). It is the
  screen the T047 red-flag branch and the T048 emergency CTA both land on.
- **Fail-upward, never blank (§5 rule 2).** If `payloadKey` is missing/unknown, or the check hasn't
  loaded yet, the screen renders a **GENERIC emergency payload** (a valid go-now emergency screen) —
  never a spinner that hides an emergency, never empty. `resolveEmergencyPayload(key?)` returns the
  specific payload or the generic one.
- **Cannot be dismissed accidentally (card requirement).** The route sets `gestureEnabled: false`
  (blocks iOS swipe-back) under the root Stack's `headerShown:false` (no back button exists); the
  screen blocks the Android hardware back button (`BackHandler` handler returns `true`). The ONLY
  forward navigation is the **acknowledge** button.
- **Region hotlines are FACTUAL data with source comments; unknown region fails safe.** The
  `regions` dataset carries only verified, publicly-published animal-poison hotlines (source comment
  per row). Unknown/undefined region resolves to `FALLBACK_REGION_HOTLINE`, which carries **no
  fabricated number** — it directs the user to find a local emergency vet. (See R-hotline-verify:
  the numbers are publicly-listed values that MUST be confirmed by the founder/vet at review.)
- **§7-clean copy, pinned in-plan.** All 22 payload entries + generic + all screen labels are pinned
  below. No dosing, no drug names as recommendations, no "diagnosis"/"diagnose", always urgent, never
  "wait and see." Naming a hazard (chocolate, xylitol, rat poison) is required and safe; it is not a
  treatment/dose. A `/diagnos/i` + dosing-pattern scan test guards the dataset.
- No medication administration advice; "do not make your pet vomit unless a vet/poison line tells
  you to" is the only vomiting reference (safety-positive). Not SAFETY-ESCALATION.

## Scope decisions (numbered; scrutinize in Risks)
- **D1 — NO API `/config` endpoint; region resolution is on-device against a bundled dataset.**
  The card's phrase "from packages/data/regions via /config" describes infrastructure that does not
  exist and is not a dependency (deps: T031, T044 — neither is an API task). Building a NestJS config
  controller + module + server region detection (Accept-Language) + a mobile config-fetch hook + e2e
  is an undeclared, >M scope and a new API surface. The user-facing feature ("region-aware poison
  hotline + unknown-region fallback") is delivered fully by shipping the dataset in `packages/data`
  and resolving region on-device. A future task can move resolution server-side once a `/config`
  endpoint exists. This is the primary scope call (R1).
- **D2 — the two datasets live in `packages/data`** (honoring "packages/data/regions"): a `regions`
  hotline dataset and an `emergency` payload-content dataset. §5 content → validated, tested,
  single-source (mirrors `toxins`). Mobile gains `@pawcareright/data` as a workspace dependency and
  imports the pure resolvers.
- **D3 — device region via `expo-localization`** (`getLocales()[0].regionCode`, ISO-3166 alpha-2).
  Region-awareness is a hard card requirement and no region source exists; `expo-localization` is the
  standard Expo SDK 57 module (matches `expo ~57.0.4`, huge adoption, actively maintained) and is the
  *safer* choice than fragile Hermes `Intl` (authoritative region, independent of language). Wrapped
  in `apps/mobile/src/checks/region.ts` → `getDeviceRegionCode()` so tests mock the wrapper. §2-rule-7
  justification line is the orchestrator's to add in `loop/journal.md` at commit.
- **D4 — acknowledge navigates to the result route.** `router.replace({ pathname:
  "/check/result/[checkId]", params:{ checkId } })`. The T048 result screen already handles a
  non-terminal (still-running) red-flag check by showing its loading view and polling to terminal,
  then reinforces the emergency. This satisfies "interstitial before any AI content": AI content is
  only reachable *after* acknowledge.
- **D5 — the completeness/parity test lives in `packages/ai`** (only place both `RED_FLAG_RULES` and
  `@pawcareright/data` payloads are importable without a cycle).
- **D6 — "call-vet + emergency-vet-search" = one "Find an emergency vet" maps action** (reusing
  T048's `buildVetSearchUrl("EMERGENCY_NOW")`) **plus** the region poison-helpline **call** button.
  There is no stored vet phone number in v1 (no vet-profile feature exists), so a literal one-tap
  "call your vet" has nothing to dial; the maps search surfaces nearby emergency clinics (with their
  phone numbers) and fulfills both "find" and "call a vet". The direct `tel:` call button is the
  region poison helpline (present only when the region is known).

## Region resolution & /config design (explicit)
- **Resolver (pure, in `packages/data`)**: `resolveRegionHotline(regionCode?)` upper-cases and looks
  up `regionHotlineByCode`; miss/undefined → `FALLBACK_REGION_HOTLINE` (no fabricated number). This is
  the unit the AC's "resolution + fallback" tests target — independent of any device/transport.

## Risks & decisions the planner made (numbered — for CHECKER scrutiny)
- **R1 (primary scope call) — no API `/config` endpoint; on-device resolution.** The card's "via
  /config" describes infrastructure that does not exist and is not a dependency (deps: T031, T044).
  Building it would be a new API surface + undeclared scope for an M card. The user-facing feature is
  delivered by the `packages/data` dataset + on-device resolver; a future task can move it server-side.
  If the checker deems the API path mandatory, this becomes blocked-for-missing-dependency, not
  something to improvise. Documented as the deliberate divergence.
- **R2 — T049 creates the emergency-payload content dataset** (T031's "T036 owns it" pointer was
  stale; T036 is the eval harness and excludes this). Content is §5 copy → placed in `packages/data`,
  validated + tested + single-source, exactly like `toxins`.
- **R3 — the parity test lives in `packages/ai`** (ai→data dep already exists; data↦ai would cycle).
  It is the only cycle-free home able to import both `RED_FLAG_RULES` and the payloads. Adding a spec
  to `packages/ai` is in-scope for the completeness AC.
- **R4 — `expo-localization` added (Expo SDK 57 module).** Required: region-awareness is a hard card
  requirement and no device region source exists; it is the standard, authoritative, testable choice
  (safer than fragile Hermes `Intl`). §2-rule-7 justification line owed in `loop/journal.md` at commit
  (orchestrator). No config-plugin change needed for `getLocales()`.
- **R5 — mobile gains `@pawcareright/data`.** Barrel import pulls the whole data package (breeds/
  toxins) into the mobile module graph; production bundlers tree-shake and the datasets are modest, so
  accepted for v1. A subpath export (`@pawcareright/data/emergency`) is a future bundle-size
  optimization; not done here to keep the diff minimal and consistent with the existing single-barrel
  export.
- **R6 (fail-upward) — generic emergency payload when payloadKey is missing/unknown or not yet
  loaded.** The screen never shows a spinner hiding an emergency and never renders blank; an unknown
  key still yields a valid go-now emergency screen + hotline + acknowledge. Non-red-flag deep-links
  (not a real flow) also fail upward to the generic emergency — safe by §5 rule 2.
- **R7 (safety-load-bearing) — hotline numbers are factual and MUST be verified.** Rows carry a
  `source` field + code comment. The values are publicly-published animal-poison hotlines, but the
  planner cannot dial them; the founder/veterinarian MUST confirm each number (and add regions) at the
  next checkpoint. Unknown/unverified regions intentionally resolve to the **no-number** fallback
  rather than risk a wrong number — the fail-safe direction.
- **R8 — "call-vet" realized via the emergency-vet maps search** (D6): no stored vet phone exists in
  v1, so a literal one-tap vet call has nothing to dial; the maps search surfaces nearby emergency
