Pure data-collection UI. No AI output, no triage, no diagnosis/"possible causes" copy, no medication names/dosing/administration copy — all new strings are neutral photo-affordance prose (CLAUDE §7 clean). This screen renders NO `<VetDisclaimer/>` and NO emergency interstitial (unchanged: those belong to the T047+ AI-result surface, per the `[category].tsx` header comment). The uploaded photos feed the server-side T034 vision pipeline later; this task only collects/uploads them. No Safety-Policy surface (PRODUCT_SPEC §5) is touched — NOT a SAFETY-ESCALATION.

## Key decisions (see Risks for rationale/scrutiny)
1. Upload progress uses `XMLHttpRequest` (RN global) `.upload.onprogress` for the presigned PUT — `fetch` (used by T024 `uploadPetPhoto`) exposes no upload-progress in RN. Blob/PUT/confirm flow otherwise mirrors `uploadPetPhoto` exactly.
2. petId seam = an optional `photoUpload` **capability prop** threaded route → `IntakeForm` → `QuestionRenderer` → `PhotoPromptQuestion`. The schema-driven renderer never learns petId; when the prop is absent it degrades to an "unavailable" render emitting no answer.
3. Per-photo lifecycle is a standalone **pure reducer** (`photo-upload-machine.ts`) so the AC "state machine unit-tested" targets a real, I/O-free module; the component/hook drives it.
4. `photoKeys` in the answer are the S3 object `key`s returned by presign/confirm (verified: `photo-upload-url` returns `{uploadUrl,key}`, `photo-upload-confirm` takes `{key}`; the intake `photoPromptAnswer.photoKeys` are exactly these strings).
5. Offline: do NOT block picking. A failed upload (incl. offline) lands in the retryable `failed` state — reuses the retry path, needs no offline branch; the route already shows the offline banner.
6. On back-navigation the component re-seeds `uploaded` slots from `answer.photoKeys` (mirrors `DurationQuestion` seeding) so the answer/count survive remount even though local preview URIs are gone.

## Upload state machine spec (`photo-upload-machine.ts`)

Types:
- `PhotoUploadStatus = "pending" | "uploading" | "uploaded" | "failed"`
- `PhotoSlot = { id: string; uri?: string; status: PhotoUploadStatus; progress: number; key?: string }` (`uri` absent for answer-seeded uploaded slots; `key` present iff `uploaded`)
- `PhotoUploadEvent = {type:"START"} | {type:"PROGRESS";progress:number} | {type:"SUCCEED";key:string} | {type:"FAIL"} | {type:"RETRY"}`

`photoUploadReducer(slot, event): PhotoSlot` — pure, never throws, unknown/illegal event returns `slot` unchanged:

| From status | Event | To status | Effect |
|---|---|---|---|
| pending | START | uploading | progress=0, key cleared |
| failed | RETRY | uploading | progress=0, key cleared |
| uploading | PROGRESS(p) | uploading | progress=clamp(p,0,1) |
| uploading | SUCCEED(key) | uploaded | progress=1, key set |
| uploading | FAIL | failed | progress unchanged |
| pending/uploaded/failed | PROGRESS | (unchanged) | no-op |
| uploading/uploaded/failed | START | (unchanged) | no-op |
| pending/uploading/uploaded | RETRY | (unchanged) | no-op |
| pending/uploaded/failed | SUCCEED | (unchanged) | no-op |
| pending/uploaded/failed | FAIL | (unchanged) | no-op |

Helpers: `createPendingSlot(id, uri): PhotoSlot`; `createUploadedSlotFromKey(id, key): PhotoSlot`; `collectUploadedKeys(slots): string[]` (keys of `uploaded` slots, in array order).

## Component spec (`PhotoPromptQuestion` interactions)
Props: `{ question: PhotoPromptQuestionDef; answer: PhotoPromptAnswer | undefined; onChange: (a: Answer | undefined) => void; photoUpload?: PhotoUploadCapability }`.
State: `useState<PhotoSlot[]>` seeded from `answer?.photoKeys` via `createUploadedSlotFromKey`; local slot-id counter; `error` string.
Emission: after any slot change, `onChange(keys.length>0 ? {type:"photoPrompt", questionId:question.id, photoKeys:keys} : undefined)` where `keys = collectUploadedKeys(slots)` — via `useEffect` keyed on the joined keys (stable; avoids parent re-render loops).
Render:
- Rationale text (always visible, T024 pattern). No `photoUpload` → render `intake-photo-unavailable-<id>` note, CTAs disabled, emit nothing.
- Two CTAs: `intake-photo-camera-<id>` (Take photo), `intake-photo-library-<id>` (Choose from library); both `disabled` and a `intake-photo-limit-<id>` hint shown when `slots.length >= question.maxPhotos`.
- Per slot tile keyed by `slot.id`: `expo-image` thumbnail when `uri` present else uploaded placeholder; status area — uploading → progress %, failed → error + `intake-photo-retry-<id>-<slotId>`, uploaded → check; always a `intake-photo-remove-<id>-<slotId>`.
- `intake-photo-error-<id>` for permission errors.
Add flow (per CTA): request the matching permission (`requestCameraPermissionsAsync` / `requestMediaLibraryPermissionsAsync`); denied → set error, return. Launch (`launchCameraAsync` / `launchImageLibraryAsync`); canceled/empty → return. `compressImage(asset)` → push `createPendingSlot` → `START` → `photoUpload.upload(uri, p=>dispatch PROGRESS)` → `SUCCEED(key)`; `catch` → `FAIL`.
Remove: drop slot from list. Retry: `RETRY` + re-run `photoUpload.upload` for that slot.

## petId / capability seam design
`intake-photos-api.ts` exports `interface PhotoUploadCapability { upload(localUri: string, onProgress: (progress: number) => void): Promise<{ key: string }> }`. `uploadIntakePhoto(petId,…)` closes over petId. `[category].tsx` (which owns the `petId` route param) constructs `photoUpload = useMemo(() => petId ? { upload: (uri,op) => uploadIntakePhoto(petId, uri, op) } : undefined, [petId])` and passes it to `IntakeForm`, which passes it to `QuestionRenderer`, which passes it only into the `photoPrompt` case. Every other question type is unaffected; the renderer + all sub-components stay pure and schema-driven (no petId knowledge).

## Risks & the decisions the planner made (for the checker)
1. **XHR for progress.** `XMLHttpRequest` is an RN global; if it fails typecheck in the mobile tsconfig the executor must STOP and flag (do not add `any`/`@ts-ignore`). Fallback would be dropping live progress (still show pending/uploading/uploaded), but attempt XHR first.
2. **Capability prop vs context.** Chose prop threading through the generic `QuestionRenderer` — slightly awkward for one question type but keeps the renderer pure and unit-testable without a provider, and preserves the T045 mutation-resistance guarantee.
3. **T045 test edits.** `question-renderer.test.tsx` + one line of `intake-form.test.tsx` reference the now-removed stub testID and MUST be updated; the schema-driven/collection guarantees in those tests must remain intact (only the photoPrompt-specific assertion changes).
4. **onChange emission via effect.** Emitting from a `useEffect` keyed on the joined uploaded keys risks a re-render loop if mis-keyed; key on the string join, not the array identity.
5. **Answer-seeded uploaded slots have no preview URI.** Back-nav shows an "uploaded" placeholder tile rather than the original image — acceptable; the answer/count are what must persist.
6. **Offline not explicitly blocked.** Uploads fail into the retryable state instead; if the checker expects an explicit offline block at pick time, this is the deliberate deviation.
