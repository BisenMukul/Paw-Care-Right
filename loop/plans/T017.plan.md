# Plan — T017: Security baseline (helmet, CORS, rate-limit classes)

## Objective (from card)
Add Helmet, strict CORS (web-admin origin only; mobile needs none), a global throttler
with named route-class definitions (auth/checks/food) plus body-size limits — and prove
it with integration tests asserting 429 + Retry-After and security headers present, WITHOUT
regressing any existing e2e suite.

---

## ⚠️ THE REGRESSION TRAP — resolved first (read before implementing)
`configureApp`/`AppModule` are shared by every e2e TestingModule. A strict global throttler
would 429 the dozens of rapid same-IP requests each suite fires (auth ~30 same-IP calls,
devices fires 5 parallel) → suite collapse.

**Resolution — Option (a), confirmed sound:**
1. Register **one** throttler in `ThrottlerModule.forRoot`: `default` = **100 req / 60 s**.
   Existing suites' worst case is auth (~30 same-IP requests within its window) — a >3× margin.
2. **Do NOT register `auth`/`checks`/`food` as global named throttlers.** In `@nestjs/throttler`
   v6 *every* named throttler in `forRoot` is enforced on *every* route unless `@SkipThrottle`d,
   so a global `auth` (5/min) throttler would throttle ALL routes at 5/min and collapse every
   suite. Instead these are exported **per-route override configs** (`THROTTLE_AUTH` etc.)
   for FUTURE `@Throttle(...)` decorators; checks/food routes do not exist yet, so nothing
   consumes them this task — they are defined only.
3. The deterministic 429 comes from a **test-only** controller (`test/throttle-test.controller.ts`)
   decorated `@Throttle({ default: { limit: 2, ttl: 60_000 } })`, injected only into the new
   security suite — mirrors the existing `test-throw.controller.ts` / `household-scoped-test.controller.ts`
   pattern. It cannot affect other suites (never imported by `AppModule`).

**Cross-suite isolation confirmed:** each `*.e2e-spec.ts` builds its own `Test.createTestingModule`
→ its own Nest app instance → its own **in-memory `ThrottlerStorage`**. Counters never bleed
across suites even if Jest runs suites in parallel workers sharing `127.0.0.1`. Within a single
suite the per-window count stays far under 100.

**T012 OTP guard kept, unchanged:** `OtpRateLimitGuard` (Redis fixed-window 5/min/IP on
`POST /auth/otp/request`) is a route-level guard that runs after the global guards. 5 < 100, so
its 429 always fires before the global default — no double-429, no conflict. Its dedicated e2e
(`auth.e2e-spec.ts` "6th otp/request → 429") stays green untouched.

**Future note (out of scope, §2 rule 2):** throttler storage is in-memory per instance. A Redis
throttler store is a deploy-scale concern for a later task, NOT this one — do not add it.

## Tests to write (map to acceptance criteria)
- **AC "security headers present"** → `security.e2e-spec.ts` "sets baseline security headers on responses":
  GET `/v1/health` → asserts `x-content-type-options === "nosniff"`, `x-frame-options` present,
  `x-dns-prefetch-control` present, and `x-powered-by` **absent** (helmet strips it). Pick these 4 (stable
  across helmet defaults with CSP disabled).
- **AC "hit limits → 429 + Retry-After"** → "throttled test route returns 429 + Retry-After + RATE_LIMITED
  envelope after the class limit": GET `/v1/__test__/throttle/ping` twice → 200; 3rd call → status 429,
  `res.headers["retry-after"]` defined, and `errorResponseSchema.parse(res.body).error.code === "RATE_LIMITED"`.
- **Do "strict CORS"** → "echoes ACAO only for the allowed web-admin origin": request with
  `Origin: http://localhost:3001` → `access-control-allow-origin === "http://localhost:3001"`; second
  request with `Origin: https://evil.example` → `access-control-allow-origin` **undefined** (request still 200;
  CORS omission, not rejection).
- **Do "body size limits"** → "rejects an over-limit request body with 413": POST a JSON body > 1 MB to
  `/v1/__test__/echo`-style route (reuse the throttle test controller with a POST, or assert against any
  existing POST route) → status 413. Assert **status only** (see Risk R4 — 413 may bypass the envelope filter).
- **Regression (the trap)** → no new test; the self-verify command run must show `app`, `auth`, `auth-social`,
  `devices`, `guards`, `schema` e2e suites all green, UNMODIFIED.

## Interfaces/contracts (executor must match)
- `ThrottlerModule.forRoot([{ name: "default", ttl: 60_000, limit: 100 }])` — ttl is **milliseconds** in
  @nestjs/throttler v6.
- Test-route decorator: `@Throttle({ default: { limit: 2, ttl: 60_000 } })` — overrides the `default` named
  throttler for that route only.
- 429 must surface as the existing envelope `{ error: { code: "RATE_LIMITED", message, requestId } }`.
  `ThrottlerException` is an `HttpException(429)`; `AllExceptionsFilter` already maps
  `TOO_MANY_REQUESTS → RATE_LIMITED` — no filter change required.
- `configureApp(app)` signature is UNCHANGED (reads CORS origin via `app.get(AppConfigService)` internally).
- CORS: `{ origin: [webAdminOrigin], credentials: false }` (bearer-token auth, no cookies; mobile is native).

## Risks & the design decisions the planner made (scrutinize these)
- **D1 — Global default 100/60 s (only named throttler globally).** Rationale: worst existing suite is
  ~30 same-IP requests/window (auth); 100 gives >3× headroom while genuinely active in tests. Risk: future
  suite growth could approach 100 — if so, raise the default, don't disable. Strict classes are per-route
  overrides, deliberately NOT global (a global `auth` 5/min would collapse every suite — see trap).
- **D2 — ThrottlerGuard registered FIRST (before Jwt).** It is metadata-independent (throttles `@Public()`
  too), cheaper than auth, and protects token verification. Ordering only ever adds a 429 earlier; cannot
  wrongly grant access.
- **D3 — Retry-After via the stock `ThrottlerGuard`.** @nestjs/throttler v6's guard sets `Retry-After` on
  the response before throwing `ThrottlerException`; headers set on the response survive `AllExceptionsFilter`
  (it only calls `response.status().json()`, preserving prior headers). **Fallback if the installed version
  omits it:** create the minimal `src/common/throttler.guard.ts` subclass (listed conditionally) that sets
  the header, and register it in place of the stock guard. The e2e assertion is the source of truth.
- **D4 — Body limit 1 MB via `useBodyParser` + `{ bodyParser: false }` in `main.ts`.** The card says "body
  size limits" without a number; 1 MB is a safe API default (flagged). Nuance: other e2e suites call
  `createNestApplication()` (no `bodyParser:false`), so Nest's default 100 KB parser runs there and the added
  1 MB parser is a no-op for them — harmless (they send tiny bodies). The security suite passes
  `{ bodyParser: false }` so it exercises the exact 1 MB boundary.
- **R4 — 413 may bypass the exception filter.** Body-parser payload-too-large errors are raised in Express
  middleware, before the Nest handler lifecycle, so they may not pass through `AllExceptionsFilter` (would be
  Express's default 413, not the JSON envelope). The body-limit test therefore asserts **status 413 only**,
  not the envelope shape. This is acceptable for the AC ("body size limits" enforced).
- **D5 — `helmet({ contentSecurityPolicy: false })`.** Helmet's default CSP would break the served Swagger
  UI (`/docs`, covered by `app.e2e-spec.ts`). Disabling CSP keeps all other headers (nosniff, frame-options,
  dns-prefetch-control, HSTS, x-powered-by removal). For a token-auth JSON API this is the standard choice.
