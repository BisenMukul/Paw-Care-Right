# Plan — T060: Mobile agenda & reminder management

## Objective (from card)
Turn the Care tab into the real **agenda** (today / upcoming) driven by the existing
`GET /agenda`, add **complete / snooze** per occurrence (optimistic + rollback), a **create/edit
custom reminder** screen whose **schedule builder maps UI choices → a validator-valid rrule**
(daily / weekly / monthly / every-N), and **per-pet filter chips** spanning the household's pets.
Acceptance = (AC1) an optimistic-complete rollback test on API failure, and (AC2) the schedule
builder emits a valid rrule for daily / weekly / monthly / every-N.

The card's complete/snooze cannot be built on T055 as-is: **there is no endpoint to complete or
snooze an occurrence** (verified in `reminders.controller.ts` / `reminders.service.ts` — only CRUD +
`agenda` + template routes exist). This plan therefore specs the **minimum additive API**
(materialize-on-action complete/snooze) plus the smallest scheduler change needed to re-fire a
snoozed occurrence, and keeps everything else on the mobile side.

## Safety statement (CLAUDE §7 / PRODUCT_SPEC §5)
This task renders **care-scheduling content only**: the user's own reminders and their occurrences.
It touches **no** AI output, **no** `<VetDisclaimer/>` / diagnosis surface, **no** emergency
interstitial, and **no** medication dosing/drug-recommendation copy. Specifically:
- The create/edit reminder form lets the user type a **title** and pick a **type** + schedule. It
  **does not** add a `medNameAsEntered` field, a dose field, or any drug-name suggestion — the
  medication subtype/med tracker is **T061** and stays out of scope. Selecting `type: MEDICATION`
  only records the user's own title; the app never suggests a drug or a dose (§7.2).
- No new user-facing string states or implies medical advice, diagnosis, or a dose; all copy is
  scheduling labels ("Today", "Upcoming", "Mark done", "Snooze", weekday/month-day labels, etc.).
- Push body copy is unchanged (the pinned two forms in `push-sender.service.ts`); snooze re-fire
  reuses that exact sender path and adds no new push copy.
- No Safety Policy surface (disclaimers, emergency escalation, dosing prohibitions) is weakened.
  This is **not** a SAFETY-ESCALATION case.

## Key decisions (numbered; re-listed under "Risks & decisions")

1. **Complete/snooze act on an *occurrence* `(reminderId, dueAt)` via materialize-on-action, keyed on
   the EXACT virtual `dueAt`.** The agenda merge dedupes on `` `${reminderId}:${dueAt.getTime()}` ``
   and lets a materialized event win over a virtual occurrence. So completing/snoozing a
   not-yet-materialized (virtual) occurrence must create the `ReminderEvent` with **exactly** the
   virtual `dueAt` (millisecond-identical), or the next agenda load shows a duplicate row. The client
   sends back the exact ISO `dueAt` it received from `/agenda`; the server re-derives occurrences with
   the **same** `occurrencesBetween`/`computeNextFireAt` used by `agenda`, so the epoch round-trips
   bit-for-bit. Implemented with a single `prisma.reminderEvent.upsert` on the existing
   `@@unique([reminderId, dueAt])` compound key (`reminderId_dueAt`) → **idempotent under
   double-tap/race** (one concurrent call creates, the other updates; both converge to the same
   terminal state). Neither action touches `Reminder.nextFireAt` (decision 4).

2. **Occurrence validation before materializing.** To keep the agenda clean (an event with a bogus
   `dueAt` would surface forever as an orphan entry), the service validates that the posted `dueAt`
   is a genuine occurrence of the reminder: parse the reminder's rrule, then
   `occurrencesBetween(parsed, startAt, tz, dueAt, dueAt)` must return exactly `[dueAt]` (epoch-equal)
   — reuse of the same pure functions the agenda uses, no new tz math. Otherwise `400`.

3. **Two explicit endpoints, household-scoped, returning the resulting `AgendaEntry`:**
   - `POST /reminders/:reminderId/complete` — body `{ dueAt }` → upsert status `DONE`,
     `completedAt = now`, `snoozedUntil = null`.
   - `POST /reminders/:reminderId/snooze` — body `{ dueAt, snoozeUntil }` → upsert status `SNOOZED`,
     `snoozedUntil = snoozeUntil`, `completedAt = null`.
   Both resolve the reminder via `findFirst({ id, pet: { householdId, deletedAt: null } })` (404 if
   not in caller's household — same posture as `findOne`). Both return the merged `AgendaEntry`
   (`virtual:false`, `eventId`, the new status) so the client can reconcile. Chose explicit verbs
   over a generic `PATCH …/occurrences` for clear DTOs + Swagger and a clean optimistic contract.

4. **Snooze re-fire mechanism = a new scheduler pass `refireSnoozed(now)`, NOT a `nextFireAt`
   change.** Because the scheduler already materialized (or will materialize) the occurrence and
   advanced `nextFireAt` strictly past it (`tick` advances with `after = dueAt + 1ms`), a snoozed
   event is invisible to the normal `tick` (which only scans `nextFireAt <= now`). So snooze must
   NOT rewind `nextFireAt` (that would regress the whole series and risk double-firing later
   occurrences). Instead, a second pass in the same minute-tick processor scans
   `ReminderEvent where status = SNOOZED and snoozedUntil <= now`, and for each **atomically**
   flips `SNOOZED → SENT` via a **guarded** `updateMany({ where: { id, status:"SNOOZED",
   snoozedUntil: { lte: now } }, data: { status:"SENT", sentAt: now, snoozedUntil: null } })`. Only
   a `count === 1` result enqueues a push job (jobId `` `${eventId}:snooze:${minuteEpoch(now)}` `` —
   distinct from the original fire jobId so BullMQ never dedupes it away). The 0-row guard makes
   concurrent/duplicate ticks a no-op → **no double-fire**. The push then flows through the
   **unchanged** `PushSenderService` (its `status === "SENT"` gate passes; collapse/quiet-hours/
   receipts all reused). This is the smallest mechanism that satisfies "a snoozed event re-fires at
   snoozedUntil" without touching the sender and without regressing `nextFireAt`. (Collapse-minute
   caveat under Risks R3.)

5. **Per-pet filter chips need NO API change** — `GET /agenda` already accepts an optional `petId`
   (validated as a household pet). Chips = `[All, …household pets]` from the existing `usePets()`
   hook; selecting a pet re-queries `/agenda?petId=`. (T059 left the Care tab active-pet-only; T060's
   chips span the whole household per the card.)

6. **Optimistic complete/snooze follow the T058 mutation shape but add `onMutate` cache patching +
   `onError` rollback** (T058's notification save had no optimistic cache write; the AC here requires
   a true optimistic rollback). Keys are structured so `onMutate` can patch every cached agenda
   window: `agendaKeys.all = ["agenda"]` prefix; `onMutate` cancels + snapshots all `["agenda"]`
   queries, maps the entry whose `reminderId`+`dueAt` match to the new status, `onError` restores the
   snapshot, `onSettled` invalidates `["agenda"]`.

7. **Schedule builder = an in-house pure function `buildRRule(config)` in mobile (no new dep).** It
   maps `{ freq, interval, byDay?, byMonthDay? }` to the exact T053 grammar
   (`FREQ=…[;INTERVAL=N][;BYDAY=…][;BYMONTHDAY=N]`), and its output is asserted valid via
   `isValidRRule` (imported from `@pawcareright/types`) for the AC cases. "Every-N" is `INTERVAL=N`
   over a chosen `FREQ`. Kept as a pure module separate from the RN component so the AC test runs
   without rendering. `startAt`/time-of-day is the reminder's DTSTART anchor (separate field), not
   part of the rrule.

8. **Shared agenda + occurrence-action shapes live in `packages/types` (Zod, single source of
   truth).** Mobile infers `AgendaEntry`/`AgendaResponse` from Zod rather than hand-writing an
   interface; the API service's existing service-local `AgendaEntry` interface is a structural match
   and stays (mirrors T059's service-local + shared-read-type split).

## Endpoint & contract specs the executor must match
- `POST /v1/reminders/:reminderId/complete` body `{ dueAt }` → `200 AgendaEntry`
  (`status:"DONE"`, `virtual:false`, `eventId`). `400` non-ISO `dueAt` or not-an-occurrence;
  `401` unauth; `404` reminder not in caller's household. Idempotent replay (upsert).
- `POST /v1/reminders/:reminderId/snooze` body `{ dueAt, snoozeUntil }` → `200 AgendaEntry`
  (`status:"SNOOZED"`). `400` non-ISO or `snoozeUntil` not in the future or not-an-occurrence.
- Materialized `dueAt` is **epoch-identical** to the virtual occurrence (decision 1).
- `refireSnoozed(now)`: flips `SNOOZED`→`SENT` for `snoozedUntil <= now` (guarded, idempotent) and
  enqueues one push; never advances/rewinds `nextFireAt`.
- Reuse unchanged: `occurrencesBetween`/`computeNextFireAt`, `parseRRule`/`isValidRRule`,
  `PushSenderService`, `usePets`, `apiClient`, `useActivePetStore`.

## Risks & edge cases (numbered — for the checker to scrutinize)
1. **Virtual-vs-materialized `dueAt` drift (decision 1/2).** If the materialized `dueAt` differs by
   even 1ms from the virtual occurrence, the agenda double-renders. Mitigation: server re-validates
   via the same `occurrencesBetween` and stores the posted `dueAt` verbatim (which passed the
   epoch-equality check). Checker: confirm the e2e "complete then GET /agenda shows one row" case and
   that `assertOccurrence` compares epoch equality, not just membership of a window.
2. **Double-tap complete / complete-then-snooze race.** `upsert` on `reminderId_dueAt` is atomic;
   the terminal state is last-writer. Completing an already-`MISSED`/`SENT`/`SNOOZED` event is
   allowed (user acknowledgement). Checker: confirm idempotency test + that no path throws on a
   pre-existing event.
3. **Snooze re-fire collapse-minute caveat (decision 4).** `refireSnoozed` flips to `SENT` but the
   event keeps its **original** `dueAt`, so `PushSenderService` derives its collapse minute from that
   past instant. For a snooze **shorter than `COLLAPSE_CLAIM_TTL_SECONDS` (1h)** where a *different*
   event won the original minute's collapse claim, the re-fire could be collapse-suppressed. Accepted
   for MVP (bounded edge: multiple same-minute reminders + sub-hour snooze + non-winner); snoozes ≥1h
   and the common single-reminder case always re-notify. Documented; **no sender change** (keeps this
   an L task and avoids T057 churn). Checker: confirm `refireSnoozed` never rewinds `nextFireAt` and
   the guarded flip prevents double-fire.
4. **Snooze past the next occurrence.** `snoozeUntil` may fall after the next scheduled occurrence's
   `dueAt`; the two events are distinct rows keyed by distinct `dueAt`, so both fire independently
   (possible two close notifications). Acceptable; snooze never suppresses the series. Checker:
