# Plan — PAWSAATHI-1: Dual-theme foundation + Variant-A home (founder redesign, batch 1/4)

## Objective (from card)
Introduce the PawSaathi visual language as a DUAL theme (keep the current light theme, restyled; add a system/Appearance-driven dark theme), wire the Bricolage Grotesque / Plus Jakarta Sans fonts (non-blocking), dual-theme the canon components, and adopt home Variant A "Companion" — presentation + tokens ONLY, zero logic/store/api changes, every `testID` preserved, and NO fabricated AI-wellbeing surface (§5/§7).

---

## Design decisions the planner locked (scrutinize under "Risks")
1. **Dark-mode strategy = additive `dark:` variants over the existing light classes.** Light appearance stays byte-identical (existing `bg-white`/`text-brand-*` classes are kept and only *appended* with `dark:` tokens). NativeWind `darkMode: "media"` → Appearance/system-driven (founder's "system-following dark theme"); no in-app toggle this batch.
2. **Fonts = weight-matched family tokens.** Named static Google-Fonts families PIN weight on native (RN ignores `fontWeight` once `fontFamily` names a static face), so a single `font-body` token cannot render Plus Jakarta at multiple weights. To avoid a weight regression I expose weight-keyed family tokens (`font-display`, `font-display-semibold`, `font-body`, `font-body-semibold`, `font-body-bold`) and apply the token whose weight matches each Text's existing weight class. The task's "(font-display / font-body)" is read as the naming scheme, not a hard cap.
3. **AI Health Score / wellbeing copy = OMITTED.** The mockup's "AI Health Score" ring, "Active & healthy" status chip, and "Health looks great · No urgent concerns" copy are wellbeing claims with no backend → they would violate fail-upward (§5) and record-only tone (§7). Batch-1 home keeps the *existing* factual content (avatar, name, species·breed, chevron, real agenda "Up next today") in PawSaathi visual language. **No new user-facing wellbeing strings are introduced → nothing new for the safety tone scan.**
4. **Home hero = solid accent surface, not a gradient.** `PetHeroCard` is restyled to an elevated accent-green surface (`bg-brand-700 dark:bg-accent-dark`, cream/white text, larger rounded avatar) rather than a second `LinearGradient` — avoids adding another native-gradient surface + its availability guard. White text on both fills is AA-verified.
5. **Home page background gradient = color-array swap by scheme.** `AnimatedGradientBackground` keeps its reduced-motion + native-availability guards and its single reanimated loop; in dark mode it selects PawSaathi dark stops via `useColorScheme()` (react-native) instead of the light brand stops. (expo-linear-gradient is linear-only; we approximate the mockup's radial with a top-down dark linear gradient.)
6. **Snapshot churn is expected and re-recorded.** Because NativeWind (under jest + the CSS stub) keeps `className` as a literal prop, appending `dark:`/font tokens to canon components changes the className strings captured by `pet-home`, `paywall`, and `check-result` snapshots. These are re-recorded with `jest -u`; the diff must be *additive only*, and the `vet-disclaimer` subtree in check-result must remain **byte-identical** (vet-disclaimer.tsx is out of scope and untouched).

## Risks & the decisions the planner made
- **R1 (font weight-pinning):** named static families ignore `fontWeight` on native → mitigated by weight-keyed tokens applied per existing weight (Decision 2). If the checker prefers the literal two-token reading, that would regress semibold body weight; this expansion is the correct trade.
- **R2 (new deps):** three deps require a one-line `loop/journal.md` justification each (CLAUDE §2.7) and Expo-Go/SDK-57 verification — executor must journal them; `@expo-google-fonts` is already in `jest.config.js` transform allowlist.
- **R3 (dark-mode test mechanism):** under NativeWind 4.2.6 + jest-expo with the `.css` stub, `className` is kept as a literal prop and is NOT resolved to styles, and the full string carries base + `dark:` statically — so a scheme flip is not observable in the tree for className-driven styling. The definitive assertion is therefore className-content (base + `dark:` present), matching the established `touch-targets.test`/`chip.test` pattern. The ONLY runtime-flippable surface is the gradient's `colors` prop (color chosen via `useColorScheme`), which `home-gradient-scheme.test.tsx` flips via `jest.spyOn(react-native,"useColorScheme")`.
- **R4 (health-score exclusion):** Decision 3 — omitting the ring/chip/copy is the safety-correct choice (no backend ⇒ any wellbeing claim violates §5). No honest record-derived replacement is added because deriving a "tasks complete" metric would touch data-shaping the card scopes out; the real agenda "Up next today" already carries the honest signal.
- **R5 (gradient treatment):** Decision 5 — linear dark stops approximate the mockup radial; acceptable within expo-linear-gradient's capability and keeps all existing guards.
- **R6 (snapshot churn):** Decision 6 — additive re-record; the review MUST confirm vet-disclaimer byte-identity and additive-only diffs. `weight-chart` snap re-recorded only if a canon component is in its tree.
- **R7 (hardcoded icon hexes):** several components pass `color="#1f6350"`/`"#2f8f74"` to `Ionicons` (not classes) — these won't respond to `dark:`. Executor computes the icon color from `useColorScheme()` (light `#1f6350` → dark `#2EA57C`/cream) so icons stay legible on dark. This is presentation-only and preserves every prop/testID.
- **R8 (preset shared with web):** adding tokens/fonts to the shared preset also reaches `apps/web`; harmless (unused utilities, fonts unreferenced there) and consistent with the §1.1 precedent that the preset is the shared source of truth.
