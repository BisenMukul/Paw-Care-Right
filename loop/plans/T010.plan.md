# T010 — packages/api-client scaffold — PLAN

## Objective
Create `@pawcareright/api-client`, the third shared *built* package (mirrors `@pawcareright/types` / `@pawcareright/config` build+package pattern exactly). It provides:
1. A platform-agnostic typed fetch wrapper `createApiClient({ baseUrl, getAuthToken?, fetch? })` with auth-header injection and error normalization to the shared `ErrorCode` enum from `@pawcareright/types`.
2. A `createQueryClient()` factory + a thin `ApiQueryProvider` component (TanStack Query, usable by web AND mobile).
3. A **no-retry-on-4xx** retry policy exported as a standalone pure `shouldRetry(failureCount, error)`.
4. A **mobile-only MMKV persister** exported from a **separate subpath** (`@pawcareright/api-client/mmkv-persister`) so `react-native-mmkv` never enters the core/web import graph.

Scope is the PACKAGE + its exports + unit tests + a scratch import-clean proof. **App wiring into `apps/web` / `apps/mobile` is DEFERRED** (lands when auth/data screens need it, ~T019). This task does NOT edit `apps/web` or `apps/mobile`.

Safety: no Safety-Policy surface touched (no user-facing copy, no AI output, no disclaimers/emergency/dosing). Error `message` values come from the server body or a minimal internal status→code fallback map — they are error *metadata*, not UI strings. Not a safety escalation.

## Central design challenge — web/mobile isolation of MMKV (resolved)
- **Core entry `.`** depends ONLY on `@tanstack/react-query`, `@pawcareright/types`, `zod`, and `react` (peer). It NEVER imports `react-native-mmkv` or `@tanstack/react-query-persist-client`.
- **`./mmkv-persister` subpath** is a SEPARATE tsup entry → separate dist chunk. Only mobile imports it. `react-native-mmkv` + `@tanstack/react-query-persist-client` are **optional `peerDependencies`** (mobile provides `react-native-mmkv` already; it will add persist-client when it wires the persister). They are ALSO `devDependencies` of api-client so tsup/`tsc` can resolve their types at build time. tsup externalizes all `dependencies`+`peerDependencies`, so neither is bundled into the chunk. The core `index.ts` barrel does NOT re-export the subpath, so web's `.` import can never pull the native module.

## Tests → Acceptance-criterion map
AC1 "Unit tests for error normalization + no-retry-on-4xx":
- `errors.spec.ts`:
  - `normalizeError` for each status with a schema-valid body `{error:{code,message,requestId}}` → asserts `ApiError.code` == body code, `.httpStatus`, `.requestId`, `.message` for 400/401/403/404/409/429/402/500.
  - `normalizeError` with an UNPARSEABLE / non-conforming body (e.g. `{}`, `"boom"`, missing `requestId`) → falls back to `statusToErrorCode(status)` (400→VALIDATION_FAILED, 401→UNAUTHORIZED, 403→FORBIDDEN, 404→NOT_FOUND, 409→CONFLICT, 429→RATE_LIMITED, 402→PAYMENT_REQUIRED, else→INTERNAL), `requestId` null.
  - `normalizeNetworkError(new TypeError("fail"))` → `ApiError` code `INTERNAL`, `httpStatus` 0, `requestId` null.
  - `isApiError` true for `ApiError`, false for plain `Error`.
- `retry.spec.ts`:
  - `shouldRetry(0, ApiError{httpStatus:400..499})` → `false` (all 4xx incl. 401/404/409/429/402).
  - `shouldRetry(n<MAX, ApiError{httpStatus:500})` → `true`; `shouldRetry(MAX, ...)` → `false`.
  - `shouldRetry(n<MAX, networkApiError{httpStatus:0})` → `true` (network is retryable); at MAX → `false`.
  - `shouldRetry(0, new Error("x"))` (non-ApiError) → `true` (treated as retryable transient) until MAX.
- `client.spec.ts` (strengthens AC, via injected fake `fetch` — no live network):
  - happy 2xx: returns parsed JSON typed as `T`.
  - `getAuthToken` returning a token → request carries `Authorization: Bearer <t>`; returning `null` → no `Authorization` header.
  - non-2xx (e.g. 401 body) → `createApiClient(...).get()` rejects with an `ApiError` carrying `UNAUTHORIZED`.

AC2 "imports clean in mobile and web": the two scratch consumers (Bundler + NodeNext) importing the exports above typecheck exit 0 and are deleted; the web/`.`-only scratch never resolves `react-native-mmkv`; the mobile scratch resolves `./mmkv-persister`.

Plus DoD gates: package build (dual + subpath chunk) + typecheck/lint/test green; root `typecheck && lint && test && build` stays green.

### errors.ts
```ts
export class ApiError extends Error {
  readonly code: ErrorCode;
  readonly httpStatus: number;      // 0 for network errors
  readonly requestId: string | null;
  constructor(args: { code: ErrorCode; message: string; httpStatus: number; requestId: string | null });
}
export function isApiError(e: unknown): e is ApiError;
export function normalizeError(args: { status: number; body: unknown }): ApiError; // parse body w/ errorResponseSchema; on fail → statusToErrorCode(status)
export function normalizeNetworkError(cause: unknown): ApiError;                    // → { code:"INTERNAL", httpStatus:0, requestId:null }
// internal, not exported UI-facing: statusToErrorCode(status:number): ErrorCode
```

### client.ts
```ts
export interface ApiClientConfig {
  baseUrl: string;
  getAuthToken?: () => string | null | Promise<string | null>;
  fetch?: typeof fetch; // injectable for tests; defaults to globalThis.fetch
}
export interface ApiClient {
  request<T>(path: string, init?: RequestInit): Promise<T>;
  get<T>(path: string, init?: RequestInit): Promise<T>;
  post<T>(path: string, body?: unknown, init?: RequestInit): Promise<T>;
  put<T>(path: string, body?: unknown, init?: RequestInit): Promise<T>;
  patch<T>(path: string, body?: unknown, init?: RequestInit): Promise<T>;
  delete<T>(path: string, init?: RequestInit): Promise<T>;
}
export function createApiClient(config: ApiClientConfig): ApiClient;
```
Behavior: builds URL from `baseUrl`+path; injects `Authorization: Bearer <token>` when `getAuthToken` resolves non-null; sets `Content-Type: application/json` for bodied methods; on 2xx parses JSON → `T`; on non-2xx reads body then `throw normalizeError({status,body})`; on `fetch` throw `throw normalizeNetworkError(e)`. No `any` — generics + `unknown` body.

## Risks / design decisions (for CHECKER scrutiny)
1. **MMKV isolation via subpath + optional peers (crux).** `react-native-mmkv` + `@tanstack/react-query-persist-client` are declared ONLY as optional `peerDependencies` (+ devDeps for build-time types), never as `dependencies`, and are reachable ONLY through the `./mmkv-persister` subpath which the `index.ts` barrel does NOT re-export. This is what keeps web from ever resolving the native module. Checker should verify the core `dist/index.*` chunk contains no `react-native-mmkv` reference and the web scratch import stays clean.
2. **`@tanstack/react-query` as a hard `dependency` (not peer).** Chosen so the single QueryClient/provider instance ships with api-client (only api-client creates the client + provider); avoids apps needing to add it before wiring. Risk: if an app later also adds `@tanstack/react-query` directly, pnpm must dedupe to one instance (React context requires it) — flagged for the T019 wiring task. Alternative (peer) rejected because apps have no query dep yet.
3. **Throw vs return for errors: THROW a typed `ApiError`.** So TanStack Query surfaces it in `error` and `shouldRetry` can branch on `httpStatus`. Normalization never throws raw/opaque errors — unparseable bodies and network failures both become `ApiError` (INTERNAL). Aligns with CLAUDE §7.5 "fail to a safe normalized state, never retry-and-guess."
4. **`react` in both peer+dev.** Peer for single-instance at runtime; dev so api-client's own build/typecheck/dts resolves `react`/`@types/react`. Same dual-listing rationale for the two optional native peers.
5. **Sanctioned edit to `packages/config/tsconfig.base.json`** (two `paths` entries). Precedent: T005 added the `@pawcareright/types` alias here. Enables scratch-consumer + package resolution WITHOUT editing `apps/web`/`apps/mobile`. If checker deems this over-broad, the fallback is per-scratch `paths`, but the base alias matches the established pattern and keeps future consumers consistent.
6. **App wiring DEFERRED.** T010 delivers the package + exports + tests + import-clean proof only; it does NOT add api-client to `apps/web`/`apps/mobile` package.json or wire providers. The card's "provider factories" are the exported factories, not their installation. Justification: wiring belongs with the first data/auth screen (~T019); adding it now would be speculative scope + would force premature `@tanstack/react-query` into the apps. Checker: confirm apps are untouched.
7. **MMKV persister is NOT unit-tested.** `react-native-mmkv` is a native module that cannot load in node/jest; importing it at module top would crash the test env. Its correctness is covered by (a) tsup dts build and (b) the mobile NodeNext scratch import. No fake-storage adapter is introduced (would add a file + a design decision the card doesn't require). Risk: the persister's runtime behavior is only proven at app integration; acceptable at scaffold stage.
8. **JSX config.** `tsconfig.json` (and spec) set `"jsx": "react-jsx"`; base tsconfig has none. tsup/esbuild handle `.tsx`; tsc reads tsconfig for dts. If dts of the `.tsx` provider fails, the fallback is authoring the provider with `React.createElement` (no JSX) — but `react-jsx` is cleaner and standard.
9. **`@tanstack/react-query` major version `^5`.** Current major; no v6 as of this repo. Justify in journal per CLAUDE §2 rule 7.
10. **No hardcoded user-facing strings / no `any` / no `console.log`.** Error `message` derives from server body or a minimal internal status map (metadata, not UI copy). Generics keep the fetch wrapper `any`-free. Verified by lint (no-console, no-explicit-any) + §1a grep is N/A (no display name here).
